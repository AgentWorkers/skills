# 上下文范围标签（聊天协议）

这是一种轻量级、可移植的约定，用于在聊天中明确划分上下文边界。

## 快速入门

在消息的**开头**添加一个或多个标签，然后正常编写内容。

示例：
- `[ISO: FinLife] 调试 Windows 运行指令。请不要使用其他仓库的上下文信息。`
- `[SCOPE: openclaw-mem] 实现下一个基准测试步骤。请确保讨论内容仅限于该范围。`
- `[GLOBAL][REM] 请记住：除非另有说明，否则使用 UTC 作为 cron 计划的时间格式。`
- `[ISO: marketing][NOMEM] 草拟 5 个广告方案；不要存储任何长期使用的信息。`

## 标签解析规则

- 标签必须出现在用户消息的**开头**。
- 可以添加多个标签（推荐顺序：范围 → 内存使用意图）。
- 标签**不会覆盖**安全政策、工具设置、访问控制或审批规则。

## 支持的标签

### 隔离/范围

- `[ISO: <主题>]` / `[Isolated Context: <主题>]`
  - 将其视为一个**独立的主题**。
  - 除非用户明确提供其他上下文信息，否则**不要**引入其他对话或项目的上下文。
  - 允许隐式继承一些通用的安全规则以及用户设定的偏好设置（例如时区、“未经批准不得更改配置”等）。

- `[SCOPE: <主题>]` / `[Scoped Context: <主题>]`
  - 将讨论范围限制在指定的主题内。
  - 如果在指定范围内缺少信息，请询问相关细节。

- `[GLOBAL]` / `[Global Context OK]`
  - 允许跨主题使用相同的上下文信息。
  - 在重新使用之前的上下文时，需要明确说明使用了哪些内容。

### 内存使用意图

- `[NOMEM]` / `[No Memory]`
  - 不要存储本次对话中的任何持久性/长期使用的信息。

- `[REM]` / `[Remember]`
  - 如果消息中包含用户的偏好设置或决策，请将其作为短期持久性信息进行存储。

## 默认行为（未使用标签的情况）

- 在跨主题混合信息时要谨慎处理。
- 如果用户对信息泄露问题提出投诉，建议使用上述标签。

## 命令式帮助提示

当用户发送 `/ctx` 或 `/context_def` 时，回复一个简短的、可复制的帮助提示：
- 标签及其含义
- 两个使用示例

## 跨平台/聊天平台注意事项

- Telegram 的斜杠命令不能包含破折号（-）。
  - 应使用 `/context_def`（下划线），而不是 `/context-def`。
- 标签本身只是文本；它们在 Telegram、Discord、Slack 和 WhatsApp 上的用法相同。
- 如果聊天平台自动格式化了括号，只需将标签放在消息的开头即可。
