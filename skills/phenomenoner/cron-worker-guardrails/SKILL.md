# Cron 工作器防护措施（POSIX）

这是一份以可靠性为首要目标的检查清单，适用于 **OpenClaw 的 cron 工作器** 以及所有无人值守的自动化任务。

## 范围（重要说明）

- 本文档主要针对 **POSIX 环境**（提供 bash/sh 示例）。
- 所介绍的 **原则** 是通用的，但在 Windows/PowerShell 环境中需要使用相应的解决方案。

## `NO_REPLY` 规约

在许多 OpenClaw 配置中，返回 `NO_REPLY` 被视为“成功执行且无需人工通知”的标志。

- 如果你的运行环境不支持 `NO_REPLY`，则应将其理解为：**成功时不输出任何信息**。

## 快速入门

1) **优先使用脚本**：将逻辑代码放入仓库中的脚本文件中（推荐路径：`tools/<job>.py` 或 `tools/<job>.sh`）。
2) **cron 任务应仅执行一个简短的命令**：避免使用多行的 `bash -lc '...'` 语句。
3) **确保工作目录（cwd）和环境变量（env）的确定性**：使用 `cd` 命令进入仓库目录，或让脚本自动完成这一操作，并明确记录所需的环境变量。
4) **成功时保持静默**：成功时不要输出任何信息（或返回 `NO_REPLY`）；只有在出现错误时才发出提示。

**另请参阅：**
- `references/cron-agent-contract.md`
- `references/pitfalls.md`

## 为何需要这些防护措施

Cron 任务的失败通常并非由逻辑错误引起，而是由以下常见原因导致的：
- 不稳定的 shell 引用处理（例如 `bash -lc '...'` 中的嵌套引号问题）
- 命令替换操作中的意外问题（如 `$(...)`）
- 用于执行单行命令的脚本中隐藏了错误（如 `python -c "..."`）
- 工作目录或环境变量的变化（导致在本地可以正常运行，但在 cron 中出错）
- 管道操作因错误原因失败（如 `pipefail` 与 `head` 的组合使用）

**有效的解决方法**是：**优先使用脚本**、确保命令执行的确定性，并在成功时保持静默。

## 可移植性建议（仍然适用）

即使在 POSIX 环境中，也**不要** 将与特定机器相关的绝对路径硬编码到脚本中**。建议使用：
- 相对路径
- 明确记录在文档中的环境变量
- 使用能够自动进入仓库目录的简单脚本封装层

## 常见问题及解决方法

### 1) 在查找匹配的 `)` 时出现意外文件结束（EOF）错误

**可能原因**：
- 命令替换操作中的 `$(...)` 没有正确闭合
- `bash -lc ' ... '` 中的嵌套引号使用错误

**解决方法**：
- 将整个多行 shell 块替换为一个独立的脚本。
- Cron 任务应仅执行一个简短的命令，例如：`python3 tools/<job>.py`

### 2) 由于 `pipefail` 与 `head` 的组合使用而错误地判断任务失败（SIGPIPE）

**症状**：
- 命令执行后返回非零状态码，但实际上输出内容是正常的

**解决方法**：
- 在使用 `pipe` 将输出传递给 `head` 时避免使用 `pipefail`；
- 或者更好的做法是在脚本中直接处理输出内容（仅读取所需的数据）。

### 3) 本地可以正常运行，但在 cron 中失败

**常见原因**：
- 工作目录设置不正确
- 缺少必要的环境变量
- 环境变量 `PATH` 不同

**解决方法**：
- 使用 `cd` 命令进入仓库目录（或让脚本自动完成这一操作）；
- 确保所有依赖项都明确列出并记录在文档中。

## Git 推送时的常见问题（非快速前进模式）

**症状**：
- 当自动化脚本尝试将更改推送到长期存在的 PR/feature 分支时，会出现 `! [rejected] ... (non-fast-forward)` 的错误提示。

**保守的解决方法**（不建议强制推送）：
- 接收到拒绝提示后，先获取远程分支的代码，然后将本地的新提交合并到远程分支上（使用 cherry-pick），然后再尝试推送。

## 可移植的防护提示（适用于所有环境）

在 cron 任务的命令行开头添加以下两行提示信息，以增强安全性：

**必须遵循的防护措施**：
- 遵循 `references/cron-agent-contract.md` 中的建议（优先使用脚本、确保工作目录和环境的确定性、成功时保持静默）。
- 同时应用 `cron-worker-guardrails` 中的防护措施。如果需要复杂的逻辑处理，可以编写一个 `tools/*.py` 脚本。
