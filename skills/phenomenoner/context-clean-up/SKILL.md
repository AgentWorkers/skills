# 上下文清理（仅限审计使用）

此技能是一份**运行手册**，用于识别导致提示上下文膨胀的原因，并制定一个**安全、可逆的修复计划**。

**重要提示：** 此技能仅限**审计使用。
- 它不会删除数据、清理会话记录、修改配置文件或调整定时任务（cron jobs）。
- 如果您请求进行更改，系统会提供详细的修复方案及回滚计划，并等待您的明确批准。

## 安全性模型（为何不会被标记为远程代码执行（RCE）风险）

- 该技能不使用任何`exec`工具（因此不会执行任意shell命令）。
- 也不使用`read`工具（因此不会读取任意文件）。
- 如果您需要文件级别的审计信息，可以手动运行捆绑的脚本，并将生成的JSON报告粘贴出来（可选）。

## 快速入门

- 输入命令 `/context-clean-up`，系统将执行审计并生成可操作的修复计划（不会对现有环境进行任何更改）。

（可选操作：手动运行审计脚本，生成JSON报告。）

请注意：此脚本不会由自动化代理执行（不会使用`exec`工具），因此不会被标记为具有远程代码执行（RCE）风险。

```text
python3 scripts/context_cleanup_audit.py --out context-cleanup-audit.json
```

如果您的Python可执行文件不是`python3`版本（在Windows系统中较为常见）：

```text
py -3 scripts/context_cleanup_audit.py --out context-cleanup-audit.json
```

## 关于`NO_REPLY`的说明

某些OpenClaw配置会将`NO_REPLY`作为“成功完成”的标志（即不向用户显示任何通知）。
- 如果您的运行环境不支持该机制，请将成功后的操作理解为：不输出任何信息。

## 常见问题来源（通常会导致上下文膨胀的原因）

按影响程度从高到低排序：

1) 工具执行后的输出结果：大量`exec`命令的输出被直接粘贴到聊天界面中；大型`read`操作（如日志、JSON文件、锁文件）的生成；网络请求返回的长篇内容。
2) 自动化脚本产生的冗余信息：定时任务每次运行时都会输出“OK”等信息；心跳检测（heartbeat）日志包含非紧急警告内容。
3) 自动化脚本中的重复代码或冗余数据：`AGENTS.md`、`MEMORY.md`、`SOUL.md`、`USER.md`等文件内容过多；`SKILL.md`中直接嵌入了大型运行手册，而非通过引用方式链接。
4) 重复生成的总结信息：这些总结内容不断累积历史数据，而非仅保留与系统重启相关的关键信息。

## 不应使用此技能的情况

- “立即删除旧会话记录/清理日志/应用修复措施”：此技能仅用于审计，不支持实际操作。
- “自动修改我的OpenClaw配置”：必须先征得用户同意。
- “调查应用程序代码中的具体错误”：应使用针对特定仓库的调试工具进行处理。

## 工作流程（审计 → 制定修复计划）

### 第0步 - 确定审计范围

您需要知道以下信息：
- **工作区目录**：存放您的工作区/项目文件的路径。
- **状态目录**：运行时系统存储会话记录/内存数据的路径（可称为`<OPENCLAW_STATE_DIR>`）。

常见默认路径（可能因操作系统而异）：
- macOS/Linux：`~/.openclaw`
- Windows：`%USERPROFILE%\.openclaw`

您可以通过`--state-dir`参数或`OPENCLAW_STATE_DIR`环境变量来自定义状态目录的路径。

### 第1步 - 运行审计脚本

该脚本会输出简短的摘要信息，并可生成JSON报告。

```text
python3 scripts/context_cleanup_audit.py --workspace . --state-dir <OPENCLAW_STATE_DIR> --out context-cleanup-audit.json
```

**解读指南：**
- 如果脚本输出大量数据（来自`exec`命令、`read`操作或网络请求），则可能是由于自动化脚本产生的冗余信息。
- 如果脚本中包含大量`System:`或`Cron:`相关的记录，可能是自动化脚本导致的问题。
- 如果`AGENTS.md`、`MEMORY.md`、`SOUL.md`、`USER.md`等文件内容过多，可能是由于自动化脚本中的重复代码或冗余数据造成的。

### 第2步 - 制定修复计划（优先处理风险较低的问题）

制定一个简单的修复计划，内容包括：
- 问题最严重的来源（即输出内容最多的自动化脚本或定时任务）。
- 重复出现且影响较大的问题（如定时任务或心跳检测日志）。
- 可快速解决的方案（且修改操作是可逆的）。

**常见的修复方法：**

#### 方法A：让无操作的自动化脚本完全静默运行

目标：维护脚本在正常情况下应仅输出`NO_REPLY`（或无任何输出），除非出现异常情况。

#### 方法B：保留通知功能，同时减少冗余信息

如果您需要接收警报信息，但希望减少自动化脚本产生的输出：
- 将警报信息通过外部渠道（如Telegram、Slack等）发送给相关人员。
- 同时确保自动化脚本的输出内容保持静默。

详情请参阅：`references/out-of-band-delivery.md`。

#### 方法C：减少自动注入的文件大小

- 仅将与系统重启相关的规则保存在`MEMORY.md`文件中。
- 将冗长的注释或数据文件移至`references/*.md`或`memory/*.md`等文件夹中。

### 第3步 - 验证修复效果

应用修复措施后，请确认：
- 定时任务或心跳检测在成功执行后不再产生任何输出。
- 监控上下文数据的增长速度（应逐渐趋于稳定）。
