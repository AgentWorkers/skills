# 错误报告

请将“it’s broken”转化为一份规范、可操作的错误报告，以便开发人员能够立即处理。

## 功能描述

该工具接收简略的错误描述，并生成以下内容：
- **标题**（简洁且易于搜索）
- **重现步骤**（编号清晰、具体明确）
- **预期行为与实际行为**
- **严重程度和优先级**的分类
- **环境详细信息**
- **建议的调查方向**
- **相关错误模式**（此类错误的常见原因）

## 使用方法

### 仅使用描述：
```bash
bug-report "使用特殊字符输入密码时登录页面崩溃"
```

### 带有上下文的信息：
```bash
bug-report "上传大文件时API返回500错误" --component "upload-service" --env "production" --frequency "always"
```

### 选项：
- `--component` — 受影响的组件/服务
- `--env` — 错误发生的环境：`production`（生产环境）、`staging`（测试环境）、`development`（开发环境）、`local`（本地环境）
- `--frequency` — `always`（始终发生）、`intermittent`（间歇性发生）、`once`（仅发生一次）、`unknown`（原因不明）
- `--user-impact` — `blocking`（导致系统无法使用）、`major`（严重影响）、`minor`（影响较小）、`cosmetic`（仅影响外观）
- `--format` — `markdown`（默认格式）、`json`（JSON格式）、`github`（用于创建GitHub问题）
- `--template` — `standard`（默认模板）、`jira`（Jira模板）、`linear`（线性格式）、`github`（GitHub问题模板）

## 生成规则

### 标题：
- 长度不超过80个字符
- 格式：`[组件] 缺陷的简要描述`
- 应包含导致问题的具体操作和现象
- 例如：`使用特殊字符输入密码时登录页面崩溃`

### 重现步骤：
1. 说明前置条件（是否已登录、数据状态等）
2. 为每个步骤编号
3. 表述要操作的具体内容（例如：“点击‘提交’按钮”而非“提交表单”）
4. 在相关情况下提供输入数据
5. 说明触发错误的步骤

### 严重程度分类：

| 严重程度 | 判断标准 | 例子 |
|----------|----------|---------|
| **严重** | 系统崩溃、数据丢失、安全漏洞、无解决方法 | 应用程序启动时崩溃、数据损坏、身份验证被绕过 |
| **高** | 主要功能失效、对用户影响较大、有解决方法但操作复杂 | 无法上传文件（核心功能）、响应时间过长（超过30秒） |
| **中** | 部分功能失效、有解决方法、影响中等 | 排序顺序错误、分页功能异常 |
| **低** | 仅造成轻微不便、外观问题、很少发生 | 错误信息中包含拼写错误、提示信息显示不正确 |

### 优先级（与严重程度分开）

| 优先级 | 修复时间 |  
|----------|------------|  
| P0 | 立即修复（例如：生产环境崩溃、安全漏洞） |
| P1 | 本次开发周期内/48小时内 |
| P2 | 下一次开发周期内/2周内 |
| P3 | 待处理事项列表中/根据实际情况安排 |

## 建议的调查方向：

根据错误类型，提供相应的调查方向：

| 错误类型 | 建议调查的领域 |
|-------------|-------------------|
| 上传失败 | 输入验证、字符编码、缓冲区限制 |
| 500错误 | 服务器日志、异常处理机制、请求数据大小 |
| 间歇性故障 | 竞态条件、缓存问题、超时设置、连接池问题 |
| 数据错误 | 查询逻辑问题、数据转换错误、时区处理问题 |
| 用户界面问题 | CSS样式问题、视口设置问题、浏览器兼容性问题 |
| 性能问题 | 数据库查询效率问题、内存泄漏、索引缺失 |

## 输出格式（Markdown）：

```markdown
# [上传服务] 上传大文件时API返回500错误

**严重程度：** 高 | **优先级：** P1 | **组件：** 上传服务
**环境：** 生产环境 | **发生频率：** 始终发生
**报告时间：** 2026-03-09

## 描述
当上传大于10MB的文件时，API端点会返回HTTP 500错误。较小的文件可以正常上传。

## 重现步骤：
1. 使用有效账户登录应用程序
2. 进入上传页面
3. 选择大于10MB的文件（测试文件大小为15MB的PNG图片和25MB的PDF文件）
4. 点击“上传”按钮
5. 观察是否收到HTTP 500错误响应

## 预期行为：
文件应成功上传并显示在用户文件列表中。

## 实际行为：
服务器返回HTTP 500内部服务器错误，文件未被保存，且未向客户端返回任何有意义的错误信息。

## 环境信息：
- 浏览器：Chrome 122 / Firefox 123
- 操作系统：Windows 11、macOS 14
- API版本：v2.3.1
- 服务器：生产环境（us-east-1）

## 建议的调查方向：
- 查看服务器日志以获取具体的异常信息/堆栈跟踪
- 检查Nginx/反向代理配置中的文件上传大小限制
- 验证应用程序层面的文件大小验证机制
- 检查用于处理多部分数据（multipart data）的库是否存在内存限制
- 检查上传目录的磁盘空间是否充足
- 使用不同类型的文件进行测试，以确定问题是由于文件大小还是文件类型引起的

## 相关错误模式：
上传大文件失败的原因通常包括：
- 反向代理的请求体大小限制（Nginx：`client_max_body_size`）
- 应用程序框架的限制（Express：`bodyParser.json({limit})`
- 处理多部分数据时内存耗尽
- 上传完成前请求超时
```

### GitHub问题模板输出（使用`--template github`选项）：
生成的报告包含适合在GitHub上创建问题的标签和格式，可以直接用于提交问题。
