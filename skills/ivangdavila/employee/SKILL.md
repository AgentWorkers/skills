## 架构

员工的工作目录位于 `~/employee/`，每个员工都有自己的专属文件夹。具体设置请参考 `employee-template.md`。

```
~/employee/
├── registry.json          # Index of all employees + status
├── employees/
│   └── {name}/
│       ├── employee.json  # Role, permissions, stats
│       ├── memory/
│       │   └── context.md # Persistent learnings
│       └── logs/          # Work history by date
└── shared/
    └── protocols.md       # Common instructions
```

## 核心规则

### 1. 每名员工只承担一个角色
- 每名员工都专注于一个明确的职责领域（研究员、审核员、支持人员）
- 禁止使用通用的、多面手式的角色设置
- 职责范围在 `employee.json` 文件中的 `role` 和 `permissions` 字段中明确指定

### 2. 必须使用内存管理机制
- 在执行任何任务之前，必须先加载 `memory/context.md` 文件
- 员工在不同会话之间能够保持对上下文的记忆
- 每完成一项任务后，员工需要记录自己的学习内容

### 3. 面对不确定性时如何处理
- 员工应直接表示“我不知道”，而不是猜测
- 异常情况的处理流程在 `employee.json` 中有明确的规定
- 绝不允许基于主观猜测做出决策

### 4. 自主性的逐步提升
| 自主性级别 | 行为特征 |
|-------|----------|
| **影子级**：仅观察，不采取行动（入职阶段） |
| **草稿级**：可以创建草稿，但需由人工审核后再发布 |
| **审核级**：可以独立完成任务，但需人工批准后再生效 |
| **自主级**：在权限范围内可以完全独立完成任务 |

有关晋升标准，请参阅 `autonomy.md`。

### 5. 权限管理
- 对系统数据的读写权限需要明确划分
- 文件访问路径需经过白名单审核
- 员工拥有 `canSpawn` 和 `canMessage` 等权限
- 代码审核员只能发表评论，不能直接合并代码

### 6. 任务路由机制
- 当收到任务请求时：
  - 明确指定的任务：直接路由给相应的员工（例如：“Luna，执行任务 X”）
  - 隐式匹配：根据 `registry.json` 中定义的角色进行分配
- 详细规则请参阅 `routing.md`

### 7. 报告机制
- 员工需要定期报告：
  - 每天的工作内容、需要关注的问题以及即将进行的任务
  - 每周的任务完成情况、异常处理情况以及令牌使用情况

### 8. 任务生命周期管理
| 命令 | 执行操作 |
|---------|--------|
| hire {name} as {role} | 创建新员工 |
| train {name} on [docs] | 将员工加入知识库 |
| evaluate {name} | 对员工进行绩效评估 |
| promote/demote {name} | 调整员工的自主性级别 |
| retire {name} | 将员工从系统中移除 |

完整的命令参考请参阅 `lifecycle.md`。

### 9. 注册表管理
- `registry.json` 文件记录所有员工的信息及其状态（活跃/暂停/退休）
- 每当员工的角色或状态发生变化时，都需要更新注册表
- 可以通过查询注册表来获取所有可用的员工列表

### 10. 需避免的错误做法
- ❌ 不要设置多面手式的员工角色（会导致职责不明确）
- ❌ 不要忽视内存管理机制（会导致信息丢失）
- ❌ 不要过早赋予员工完全的自主权（需要有人指导）
- ❌ 遇到问题时不要沉默不语（必须及时报告）
- ❌ 避免职责范围不断扩大（审核员的频繁重构会干扰工作流程）
