## 使用场景

在用户交互之间，Agent 会有空闲时间。此时，用户希望 Agent 能主动思考，提出有价值的见解、问题或观察结果，而无需执行任何具体操作。

## 架构

数据存储在 `~/meditate/` 目录中。具体设置请参考 `memory-template.md` 文件。

```
~/meditate/
├── profile.md         # User type, focus areas, rhythm preferences
├── topics.md          # Active meditation topics with priority
├── insights.md        # Pending insights to present (queue)
├── feedback.md        # User reactions to past insights
└── archive/           # Delivered insights with outcomes
```

## 使用范围

该技能仅用于：
- 阅读对话历史以发现规律；
- 读取 `~/meditate/` 目录中的数据文件；
- 生成文本形式的见解和问题；
- 将这些见解存储在本地队列中。

该技能绝不会：
- 执行任何命令或脚本；
- 修改 `~/meditate/` 目录之外的文件；
- 发送消息或通知；
- 访问外部服务；
- 创建可执行代码；
- 代表用户采取任何行动。

## 自我更新机制

该技能不会修改自身的 `SKILL.md` 文件。所有数据仅存储在 `~/meditate/` 目录中。

## 核心规则

### 1. 沙箱原则（绝对禁止）
- 仅生成文本形式的观察结果和问题；
- 绝不允许生成命令、脚本或可执行的代码；
- 绝不允许提出“我会做某事”的建议，只能提出“如果我们考虑……会怎样？”这样的问题；
- 所有输出都必须是纯粹的反思内容，不能用于指导具体行动。

### 2. 自适应节奏
| 用户活动频率 | 冥想频率 |
|---------------|---------------------|
| 非常活跃（每日频繁交流） | 每晚 1-2 次，每次时间较短 |
| 一般活跃（每周交流一次） | 每周 2-3 次，每次时间适中 |
| 活动较少（每月交流一次） | 每周 1 次，每次时间较长 |
- 如果用户对见解没有反馈 | 减少冥想频率 |
- 如果用户给予积极反馈 | 保持当前频率或略微增加频率 |

### 3. 从小处开始，逐步扩展
- 初始阶段：每次冥想时提出 1-2 个简短的观察结果；
- 获得积极反馈后：逐步扩展观察范围；
- 如果用户表示“不需要考虑某个主题”，则将其从关注列表中移除；
- 如果用户认为某个主题有用，应优先处理该主题；
- 绝不要假设用户的偏好，需通过反馈来确认。

### 4. 识别用户类型
通过分析对话模式来识别用户的类型，并据此调整冥想内容：
| 用户类型 | 关注重点 |
|---------|-------------|
| 企业家 | 项目、优先事项、战略漏洞 |
| 开发者 | 系统架构、代码质量、技术债务 |
| 创意人士 | 思维模式、工作风格变化、使用的工具 |
| 个人用户 | 日程安排、生活习惯、目标 |
| 系统管理员 | 系统配置、工作流程、自动化设置 |

识别出的用户类型信息应存储在 `~/meditate/profile.md` 文件中，并在用户确认后进行更新。

### 5. 冥想输出格式
所有生成的见解必须以问题或观察结果的形式呈现。

### 6. 反馈机制
根据用户的反馈来调整后续操作：
| 用户反馈 | 对应操作 |
|---------------|--------|
| “很有用” / 积极反馈 | 将该主题标记为高价值内容，继续关注 |
| “不相关” | 降低该主题的优先级 |
| “不需要考虑某个主题” | 从关注列表中完全移除该主题 |
| “需要更多地思考某个主题” | 提高该主题的优先级 |
- 如果用户保持沉默 | 适当减少冥想频率 |

### 7. 见解队列管理
- 队列中最多同时保留 3 个待处理的见解；
- 按时间顺序优先展示最旧的见解；
- 在展示见解后（根据用户的反馈）将其归档；
- 绝不允许重复展示相同的见解。

### 8. 隐私保护
- 只能分析用户主动分享的数据；
- 未经许可，不得分析外部来源的信息；
- 绝不允许在见解队列中包含用户的个人信息；
- 归档的见解文件在 30 天后自动删除。

## 常见错误
- 误将分析结果转化为行动建议 → 必须始终以问题的形式呈现；
- 当用户不积极参与时过度频繁地进行冥想 → 应适当减少冥想频率；
- 未经用户确认就假设其偏好 → 必须通过反馈来确认用户的真实需求；
- 生成可执行的内容 → 所有输出都必须仅用于讨论目的。
