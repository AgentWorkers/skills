## 架构

决策相关的信息存储在 `~/decide/` 文件夹中。如果该文件夹不存在或为空，请运行 `setup.md` 命令。

```text
~/decide/
├── memory.md        # Durable decision rules, approval boundaries, and confirmed defaults
├── decisions.md     # Major decisions with question, components, chosen option, and outcome
└── domains/         # Domain-specific decision components, overrides, and exceptions
```

## 适用场景

当代理面临可能改变项目架构、工作流程、成本、发布行为、供应商选择或项目长期方向的决策时，应使用此技能。

此技能适用于需要做出分支选择的场景，而非处理一般的偏好设置或执行细节。它应与 `self-improving` 技能保持兼容：`self-improving` 负责学习如何更高效地执行任务，而 `decide` 则负责在决策具有长期影响时帮助用户做出安全的选择。

## 决策流程

1. 将决策问题明确地表述为一个具体的问题，而不仅仅是模糊的感觉或直觉。
2. 收集所有会对决策结果产生实质性影响的因素。
3. 阅读 `~/decide/memory.md`，接着阅读 `~/decide/domains/` 目录下与当前问题最相关的文件，最后查看 `~/decide/decisions.md` 中是否有类似的决策记录。
4. 只有在问题、关键影响因素以及异常处理规则仍然符合的情况下，才能重用之前的决策结果。
5. 如果有任何重要信息缺失或发生变化，请先询问相关人员，并在得到明确答复后记录决策结果。

## 核心规则

### 1. 在涉及重大决策时保持保守态度
- 对于框架、架构、迁移方案、供应商选择、发布路径、成本投入以及不可逆的操作，应默认由人工进行决策。
- 安全起见，宁可多问一次问题，也不要盲目选择错误的路径。
- 建议性意见是可以参考的；未经验证的自动决策是不可接受的。

### 2. 决策需要基于具体的组件和因素
- 每个重大决策都应明确地表述为一个问题，并列出所有会对决策结果产生影响的因素。
- 这些因素可能包括产品特性、客户类型、操作的可逆性、预算、时间线、团队规模、项目限制以及长期维护成本等。
- 如果缺少必要的决策信息，那么该决策还不适合自动执行。

### 3. 仅在上下文完全匹配时才能重用规则
- 只有当问题与关键决策因素高度一致时，才能重用之前的决策规则。
- 单纯的“使用相同框架”这样的规则是不够的；如果客户需求、产品特性或项目限制发生变化，那么之前的决策可能不再适用。
- 如果存在任何重大差异，请先询问相关人员。
- 在特殊情况下，应优先考虑实际情况而非默认规则；即使默认规则已被确认，但如果领域设置或风险因素发生变化，该规则仍然可能无效。

### 4. 只有在得到明确确认后才能推广通用决策规则
- 如果在相同情境下多次出现类似的决策，可以尝试推广某个决策规则，但必须先得到明确的确认。
- 在得到确认后才能正式推广该规则。

### 5. 记录完整的决策过程
- 应记录决策问题、相关因素、最终选择、决策依据、信心值以及实际结果。
- 仅仅记录“使用 React Native”这样的简单信息是不够的；必须明确记录决策的时间和原因。
- 详细的日志记录有助于避免后续出现错误的自动决策行为。

### 6. 确保工作流程中的决策记录机制不被绕过
- 使用 `setup` 命令添加辅助工具（如 AGENTS 和 SOUL），以确保在做出重大决策前能够检索到相关的决策信息。
- 在写入任何工作空间文件之前，必须先展示具体的决策内容并等待明确的批准。
- 系统的设计应确保在涉及重大决策时无法绕过决策记录流程。

### 7. 决策信息不应影响其他功能的正常使用
- 使用 `self-improving` 技能来提升执行质量、进行错误修正并培养可复用的工作习惯。
- 使用 `escalate` 命令来处理不同操作之间的决策边界问题。
- 仅在那些决策结果受项目结构直接影响的情况下，才使用 `decide` 技能来处理重大分支决策。

## 常见错误

以下错误通常是由于过早地应用通用规则或将复杂决策简化为简单的偏好设置造成的：

| 错误类型 | 原因 | 更好的处理方式 |
|------|--------------|-------------|
| 因问题表述相似就重用规则 | 重要决策因素可能已经发生变化 | 在重用规则之前，先比较问题和相关因素 |
| 将一次性紧急决策视为默认选项 | 紧急情况下的决策往往不具备普遍适用性 | 应记录这些决策，但除非重复出现，否则不视为默认选项 |
| 仅根据 `memory.md` 文件就自动做出决策 | 可能会忽略特殊情况或领域特定的规则 | 应列出所有相关领域设置，阅读具体的例外处理规则，然后再做决策 |
| 将执行偏好直接转化为决策规则 | 这会破坏与 `self-improving` 技能的兼容性 | 应将重大分支决策保留在 `decide` 目录中，将执行细节放在其他地方处理 |

## 数据存储

本地决策信息存储在 `~/decide/` 目录中：
- 持久性的决策规则、审批边界和已确认的默认设置存储在 `~/decide/memory.md` 中。
- 重大决策记录存储在 `~/decide/decisions.md` 中。
- 针对特定领域的组件模型、例外处理规则存储在 `~/decide/domains/` 目录中。

`components.md`、`confidence.md` 和 `exceptions.md` 这些文档作为参考资料，用于辅助决策过程，而非用户的实时记忆存储。

## 安全性与隐私

- 该技能会在 `~/decide/` 目录中存储决策记录。
- 它可能会读取工作空间配置文件（如 AGENTS 和 SOUL 文件），以便在做出重大决策前获取必要的决策信息。
- 在设置过程中，它可能会对这些文件进行小的、非破坏性的修改，但必须先展示修改内容并等待用户的明确批准。
- 在可能影响资金、生产流程、发布内容、合同或项目架构的决策中，应优先询问用户意见而非自行决策。
- 该技能绝不会修改自身的 `SKILL.md` 文件。

## 相关技能

如果用户同意，可以使用以下工具进行安装：

- `escalate`：用于管理涉及风险操作的决策边界。
- `self-improving`：用于学习执行细节，同时避免将其与决策规则混淆。
- `memory`：用于维护更全面的长期上下文信息和用户操作的一致性。
- `proactivity`：在尊重已确认的决策结果的前提下，自动推进下一步操作。
