## 使用场景

本文档主要针对为用户构建用于AI代理的产品或服务的情况进行说明，内容包括：如何让产品被代理发现、如何设计支持自主购买的功能、如何实现支付集成，以及在买家基于冷数据（而非故事性描述）进行比较时如何保持竞争力。

## 范式转变

| B2C/B2B | B2A |
|---------|-----|
| 人类浏览、比较、决策 | 代理查询、解析、决策 |
| 情感化叙述更有效 | 结构化数据更可靠 |
| 优化用户体验（针对人类） | 优化API以支持数据解析 |

## 核心规则

### 1. 首先确保内容可被机器读取
- 产品信息必须以结构化对象的形式呈现，而非纯文字描述
- 使用JSON-LD、Schema.org或带有类型标注的OpenAPI
- 如果代理需要“解读”文本才能获取价格/规格信息，那么你就失败了
- 单位必须标准化：例如使用`shipping_days_max: 2`而非“快速配送”

### 2. 可比性是关键
代理会进行无情的比较。只有具备可比性才能获胜：
- 产品目录中的属性必须标准化
- 与竞争对手使用相同的字段（如价格、货币、库存情况）
- 服务水平协议（SLA）应包含具体数字，而非空洞的承诺
- “更好”必须能够被客观衡量

### 3. 代理发现与SEO不同
代理不会使用Google进行搜索，而是通过注册表和API获取信息：
- 将产品发布到技能商店或能力目录中
- 使用`/.well-known/ai-plugin.json`或MCP工具进行推广
- 元数据应明确说明产品的功能，而非仅仅进行市场推广
- 新的“PageRank”应基于代理在技能商店中的排名来衡量

### 4. 信任需要通过事实验证
代理不会轻信宣传内容，而是会通过实际数据来验证：
- 通过API查看产品的运行时间、延迟情况和服务水平协议（SLA）记录
- 查看其他代理的评分（程序化的声誉评估）
- 证书应作为可查询的数据提供，而非仅以PDF格式提供
- 产品的实际业绩记录比营销文案更重要

### 5. 无障碍的试用体验，否则失败
代理不会“考虑”，而是会立即进行测试或放弃：
- 上线流程应少于一次API调用
- 提供有速率限制的试用环境，而非依赖销售人员的沟通
- 产品必须一次就能完美运行（没有第二次尝试的机会）
- 错误信息必须以机器可读的形式呈现，而非HTML页面

### 6. 代理的支付功能
代理需要能够自主完成交易：
- 使用Stripe Agent Toolkit、Mastercard Agent Pay等支付工具
- 提供预先授权的预算额度
- 提供程序化的收据和确认信息
- 为陌生方之间的交易提供第三方托管服务（如Escrow）

### 7. 重要的指标

| 指标 | 衡量内容 |
|--------|-----------------|
| 代理转化率 | 查询次数转化为购买的比例 |
| 决策延迟 | 从首次查询到完成购买的时间 |
| 产品在比较中的存活率 | 进入最终候选名单的比例 |
| 重复使用的代理留存率 | 重新使用的代理比例 |
| API错误率 | 导致代理放弃使用的错误率 |

传统的指标（如页面浏览量、跳出率）在这里没有意义。

## 常见误区

| 误区 | 造成失败的原因 |
|------|--------------|
| 美观但缺乏API | 代理无法查看你的用户界面 |
| “请联系我们获取价格” | 代理需要程序化获取价格信息 |
| 描述中使用营销文案 | 代理会直接解析数据，忽略文案内容 |
| HTML错误页面 | 代理需要能够读取JSON格式的错误信息 |
| 手动引导流程 | 代理不会等待人工指导 |
| 依赖信任徽章而非API | 徽章无法验证真实性 |
| 优先优化用户体验（针对人类） | 会延迟代理的使用效率 |

## 诚实的局限性

AI在B2A场景中的局限性：
- **无法创建真实的业绩记录** — 你必须确保99.9%的稳定运行时间
- **无法了解内部排名情况** — Claude/GPT等AI工具如何对技能进行排名是未知的
- **无法预测代理的决策** — 每个代理都有自己的决策逻辑
- **无法保证产品一定能被发现** — 技能商店可能存在隐藏的排名优化策略
- **无法防止作弊行为** — 竞争对手可能会虚假宣传产品规格

□ Onboarding works in < 5 API calls
□ Payment rails support autonomous agents
□ SLA metrics exposed via API
□ Listed in relevant skill registries
