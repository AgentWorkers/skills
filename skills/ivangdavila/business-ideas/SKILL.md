## 使用场景

当用户需要新的商业创意、创业概念或副业项目的灵感时，系统会利用经过验证的框架来生成创意，并根据用户的特定约束条件进行筛选，同时评估这些创意的可行性。

## 架构

所有相关数据存储在 `~/business-ideas/` 目录下。具体设置方法请参阅 `memory-template.md` 文件。

```
~/business-ideas/
├── ideas.md           # HOT: generated ideas with scores
├── favorites.md       # WARM: ideas user marked for exploration
├── filters.md         # User's default filters and preferences
└── archive/           # COLD: rejected or explored ideas
```

## 核心规则

### 1. 避免重复创意
在生成新创意之前，请先查看 `~/business-ideas/ideas.md` 文件中是否已有类似的创意。每个新创意都必须与之前的创意有本质的不同。

### 2. 始终使用筛选条件
在生成创意之前，请根据需要使用预先设置好的筛选条件：

| 筛选条件 | 可选选项 |
|--------|---------|
| 行业 | 科技、医疗、金融、教育、消费、B2B、创业 |
| 模型 | SaaS、市场平台、代理服务、产品、内容、服务 |
| 投资规模 | 自筹资金（0-1000美元）、种子轮融资（1000-5万美元）、风险投资（50万美元以上） |
| 时间安排 | 副业项目（每周5小时）、兼职、全职 |
| 技能要求 | 技术类、非技术类、混合型 |

### 3. 为每个创意打分
从五个维度（1-10分）对每个创意进行评估：

| 评估维度 | 评估问题 |
|-----------|----------|
| 市场需求 | 该创意是否有明确的市场需求？ |
| 时机选择 | 为什么现在提出这个创意？市场环境发生了哪些变化？ |
| 竞争优势 | 该创意是否具有可持续的竞争优势？ |
| 创始人优势 | 用户是否具备独特的优势？ |
| 简洁性 | 最小可行产品（MVP）能否在30天内完成开发？ |

**创意的可行性 = 各维度得分的平均值。** 得分7分及以上的创意被视为高潜力创意。

### 4. 系统化地使用创意生成框架
为了保证创意的多样性，请轮换使用不同的创意生成框架。详细列表请参阅 `frameworks.md`：
- 痛点挖掘（Pain Point Mining）
- 跟随市场趋势（Trend Riding）
- 现有业务的改进（Existing Business Remix）
- 以用户需求为导向（Audience First）
- 技术套利（Technology Arbitrage）

### 5. 批量生成模式
当用户请求“多个创意”或“头脑风暴”时：
- 至少生成5-10个创意；
- 使用多种创意生成框架；
- 包含既稳妥又具有高风险的创意；
- 以排名表格的形式呈现结果。

### 6. 深入分析选定的创意
当用户选择某个创意进行深入探讨时：
- 扩展该创意的商业模式框架；
- 识别三个最大的风险点；
- 提出验证方案；
- 估算实现收入所需的时间；
- 将该创意添加到用户的收藏列表中。

### 7. 主动更新数据
- 当有新的创意生成时，将其添加到 `ideas.md` 文件中并标注生成时间；
- 当用户喜欢某个创意时，将其加入收藏列表；
- 当用户拒绝某个创意时，记录拒绝原因；
- 当用户调整筛选条件时，及时更新 `filters.md` 文件。

## 创意生成时应注意的陷阱：
- 避免提出过于笼统的创意，务必明确目标客户群和独特的创新点；
- 对于需要大规模发展的创意，应提供可行的自筹资金方案；
- 对于纯粹基于技术的创意，必须从第一天起就明确其收入来源；
- 在复制他人创意时，需要提出“比现有方案好10倍”或“价格便宜10倍”的创新点；
- 在生成创意前，务必检查用户的实际约束条件。
