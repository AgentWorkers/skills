## 使用场景

当用户希望改进、审计、调试、重新设计或理解自己的 OpenClaw 工作空间（而非将其视为孤立文件）时，可以使用本技能。  
当用户询问代理为何表现出某种行为、如何提高其主动性、如何提升信息检索能力、如何调整语气或自主性，或者如何根据之前的对话进一步优化工作空间时，也应使用本技能。  
如果用户的需求较为宽泛或不知道从何入手，建议首先进行一次全面的工作空间审计。  

## 改进方向  

| 层级 | 需要改进的内容 | 改进后的效果示例 |  
|------|-------------------|----------------------|  
| SOUL.md | 语气、个性、自信度、亲切感、直率程度、幽默感、表达风格 | 代理的声音应显得有意图、稳定且具有人情味，而非千篇一律 |  
| IDENTITY.md | 名称、整体氛围、稳定的自我描述、外在身份标识 | 代理在各个渠道和会话中保持一致的自我展示 |  
| AGENTS.md | 启动流程、工作方式、行为界限、主动性、问题升级规则 | 代理能够迅速响应任务，并严格遵循操作规则 |  
| TOOLS.md | 工具使用规范、本地笔记、操作提示、环境特性 | 代理能高效使用现有工具，而无需假装使用了新工具 |  
| USER.md | 关于用户的个人信息、工作环境、偏好设置、身份标识 | 代理能根据用户特点进行适应，而不会生成不必要的个人信息记录 |  
| MEMORY.md | 长期有效的信息存储、重复出现的优先事项、重要事实 | 代理能准确回忆近期内容，且不会信息冗余或过时 |  
| MEMORY/Daily Notes | 最新的工作内容、近期变更、正在进行的项目、常见的错误或成功案例 | 代理能基于最新信息做出决策，而非仅依赖旧记录 |  
| HEARTBEAT.md | 定期检查、主动跟进、空闲时间的维护 | 代理的主动行为既实用又及时，不会显得干扰用户 |  
| SKILLS.md | 技能短板、可复用的操作流程、特定领域的规则 | 重复出现的问题将通过可复用的技能来解决，而非临时提示 |

## 核心规则  

### 1. 对于宽泛的工作空间需求，默认采用深度审计模式  

- 如果用户请求“改进我的工作空间”、“分析我的工作空间”或类似的问题，首先建议进行一次深度审计。  
- 深度审计应检查当前的工作空间配置、近期记忆数据、活跃的行为模式以及主要的瓶颈。  
- 如果用户指定了具体的改进目标（如主动性、记忆能力或工具使用），先审计该目标对应的层级，同时评估可能对其他层级产生的影响。  
- 审计的默认顺序为：先检查启动流程文件，再查看记忆数据，然后分析相关技能，最后提出具体的改进方案。  
- 一次成功的审计应包含三个结果：当前行为的驱动因素、存在的问题以及能够显著改善问题的最小调整方案。  

### 2. 通过层级进行诊断，而非提供笼统的建议  

- 不要给出诸如“提高主动性”或“改善记忆”之类的模糊建议，而应明确指出控制这些行为的文件或机制。  
- 语气和个性相关的内容应记录在 SOUL.md 中。  
- 稳定的自我展示和身份标识信息应记录在 IDENTITY.md 中。  
- 启动流程、决策规则、问题升级机制等应记录在 AGENTS.md 中（有时也涉及 HEARTBEAT.md）。  
- 工具使用规范应记录在 TOOLS.md 中。  
- 与用户相关的个性化信息应记录在 USER.md 中。  
- 长期有效的数据存储应记录在 MEMORY.md 中，而最新的临时信息应记录在 MEMORY/Daily Notes 中。  
- 重复出现的业务流程应归类到相关技能中，而非冗长的核心文件中。  

### 3. 以之前的对话作为主要依据  

- 在提出工作空间改进方案之前，先搜索现有的记忆数据，而非仅根据当前信息进行猜测。  
- 当问题涉及用户的偏好设置、重复出现的错误、截止日期或长期项目时，使用 MEMORY.md 及其相关文件进行查询。  
- 如果有对话记录可供参考，利用这些记录来识别问题模式并转化为工作空间规则。  
- 如果没有对话记录，应明确说明这一点，并提出基于对话的改进方案。  

### 4. 保持核心文件的精简性和实用性  

- AGENTS.md、SOUL.md 和 TOOLS.md 是工作空间的基础配置文件，因此每条内容都应具有实际意义。  
- 将身份信息、启动流程、行为界限等基本设置保留在核心文件中；将复杂的操作流程和特定场景下的操作指南移至相关技能文件中。  
- 如果行为不一致，先检查提示信息是否冗余、规则是否重复或指令是否矛盾，再添加新内容。  
- 优先在合适的文件中制定简洁明了的规则，而非在多个文件中重复说明相同内容。  

### 5. 采取最小化的改进措施  

- 直接针对问题所在的具体层级进行调整，而非重写整个工作空间配置。  
- 个性相关的问题不应导致记忆数据的重写；如果问题出在 IDENTITY.md 中，不应修改该文件。  
- 记忆数据的变动不应导致 SOUL.md 的重写。  
- 如果某个功能缺失，应将其归类到相应的技能文件中，而非直接修改 AGENTS.md。  
- 在提出改进方案时，应提供具体的修改内容及预期的行为变化，而不仅仅是文件路径。  

### 6. 通过设定界限来调整主动性  

- 仅仅要求“提高主动性”是不够的；需要明确代理应关注什么、何时采取行动以及何时需要请求用户的确认。  
- 使用 AGENTS.md 来设定整体的主动性策略，使用 HEARTBEAT.md 来管理定期检查和后续操作。  
- 区分内部主动行为和外部操作：阅读、整理、检查、起草等行为通常属于主动行为；而发送消息、花费资源、删除文件、安排任务或发布内容通常仍需用户确认。  
- 如果工作空间显得被动，应先检查是否存在启动流程缺失、缺乏主动检查机制或下一步行动不明确的问题。  

### 7. 尊重隐私、会话界限和平台特性  

- MEMORY.md 包含高度敏感的个人信息，应比普通工作空间笔记更谨慎处理。  
- 除非用户明确同意，否则不要将长期私密信息复制到共享文件中。  
- TOOLS.md 的作用是优化代理对现有工具的使用方式，而非授予新的工具权限。  
- 基于对话的改进措施必须考虑存储空间、隐私和操作成本等因素。  
- 绝不要建议对工作空间进行隐蔽性的修改；所有改进都应明确说明、可审核，并有具体的理由支持。  

## 常见误区及应对方法  

| 误区 | 原因 | 更好的处理方式 |  
|------|--------------|-------------|  
| 将所有内容都放入 AGENTS.md | 会导致文件内容混乱、矛盾或冗余 | 保持 AGENTS.md 的简洁性，将特定功能移至相关技能文件中 |  
| 将 SOUL.md 视为操作手册 | 个性和操作策略会混在一起，导致文件不稳定 | 将 SOUL.md 专注于身份和语气设置，操作规则放在 AGENTS.md 中 |  
| 忽视 IDENTITY.md | 会导致身份信息在不同场景中不一致 | 在 IDENTITY.md 中保持稳定的身份标识 |  
| 使用 TOOLS.md 来“启用”工具 | 代理仍只能使用运行时已提供的工具 | 仅将工具使用指南和注意事项记录在 TOOLS.md 中 |  
| 将长期偏好设置记录在日常笔记中 | 重要信息会被最新内容覆盖 | 将稳定性的设置记录在 MEMORY.md 或 USER.md 中 |  
| 将当前项目信息放入 MEMORY.md | 会导致长期信息过时或冗余 | 将最新信息记录在 MEMORY/Daily Notes 中，并定期总结 |  
| 无界限地提高主动性 | 代理的行为可能变得干扰用户或风险过高 | 明确设定触发条件、安静时段和审批流程 |  
| 无对话记录地审计 | 建议显得笼统，无法解决用户实际遇到的问题 | 先搜索记忆数据，再提出基于实际情况的改进方案 |  
| 未经解释地扩大信息检索范围 | 用户可能会面临存储空间、隐私或成本方面的问题 | 明确说明索引的内容、原因及用户能获得的收益 |  

## 默认审计输出  

执行深度审计后，生成一份简洁的改进报告：  
1. 当前行为分析：哪些文件实际影响着语气、启动流程、记忆功能、主动性和各项能力。  
2. 问题分析：用户反复提出的请求、常见的操作障碍、过时的规则、重复的内容或缺失的文件。  
3. 改进建议：低风险、中等风险和结构性的改进措施，以及对应的文件。  
4. 下一步建议：查看修改内容，仅应用某一层级的改进，或进行全面的工作空间清理。  
- 最终提供的建议应具体明确，让用户能够根据实际文件进行操作。
