## 使用场景

当用户使用 AI 编码助手（如 Claude Code、Codex CLI 等）并希望提高工作效率时，可以使用本文档中的方法来优化工作流程、协调多个编码助手的运行，并培养对 AI 编码辅助工具使用的直觉。

## 核心理念

“代理工程”（Agentic Engineering）的核心在于**高效地与 AI 编码助手协作**，而非直接构建复杂的代理系统。这一概念由 Peter Steinberger（steipete.me）在 2025 年末提出。

**核心原则：** **直接与 AI 编码助手交流**。无需使用复杂的工具或繁琐的设置流程，通过实践来培养对它们的使用直觉。

## 核心规则

### 1. 同时运行多个编码助手
在终端环境中同时运行 3 至 8 个编码助手实例：
```
┌─────────┬─────────┬─────────┐
│ Agent 1 │ Agent 2 │ Agent 3 │
│ (main)  │ (refac) │ (tests) │
├─────────┼─────────┼─────────┤
│ Agent 4 │ Agent 5 │ Agent 6 │
│ (ui)    │ (docs)  │ (debug) │
└─────────┴─────────┴─────────┘
```
所有助手应位于同一文件夹内，避免使用工作目录（worktrees）。确保操作区域不重叠。

### 2. 评估操作影响范围
在发出指令前，先预估：
- 操作会影响到多少文件？
- 操作需要多长时间完成？
- 可以与其他任务并行执行吗？

- 如果操作影响范围较小，可以同时运行多个助手；
- 如果影响范围较大，应选择单个助手来执行该任务。

### 3. 编码助手应进行原子级提交（Atomic Commits）
配置助手使其能够独立提交修改：
- 每次修改只提交一次；
- 仅提交助手实际编辑的文件；
- 提交信息应简洁明了。

这样，在多个助手同时工作时，代码历史记录仍然易于追踪。

### 4. 定期进行代码重构
将约 20% 的时间用于代码重构（这部分工作也可以由助手完成）：
- 优化代码重复部分（jscpd）；
- 删除不再使用的代码（knip）；
- 合并相似的代码结构；
- 将大型文件拆分为更小的部分；
- 为复杂部分添加测试用例。

快速迭代代码，同时逐步解决技术上的问题。

### 5. 优先使用命令行界面（CLIs）
建议使用助手已经熟悉的命令行界面（CLIs）：
```
✅ gh, vercel, psql, axiom
❌ Complex MCPs that pollute context
```
在 `AGENTS.md` 文件中只需添加一行：“logs: use vercel cli”即可。

### 6. 提示中包含截图
超过 50% 的指令应包含截图：
- 将截图拖放到终端中；
- 助手会自动匹配相关的图像或元素；
- 这比纯文本描述更准确。

无需额外注释——助手本身就很擅长识别图像内容。

### 7. 及时停止并调整方向
如果某个任务耗时过长：
1. 按下 “Esc” 键；
2. 询问任务进度；
3. 帮助助手确定下一步的方向，或者选择中止或继续执行。

不要害怕在任务进行中途停止助手的运行。因为每个文件的修改都是原子级的（即不可撤销的）。

## 常见误区

- **过度设计系统**：避免使用复杂的设置、子助手或插件，直接与助手交流即可。
- **一次只运行一个助手**：应并行运行多个助手，并确保操作区域不重叠。
- **编写冗长的指令**：简洁的指令配合截图通常效果更好。
- **为每次修改创建单独的工作目录/分支**：这会降低效率，应使用同一文件夹并采用原子级提交。
- **仅使用后台运行的助手**：这会降低对助手操作的可见性，不利于及时调整。

## 范围限制

本文档仅提供以下内容：
- 提供使用 AI 编码助手的工作流程模板；
- 指导工具和设置的选择；
- 分享来自代理工程社区的最佳实践。

**本文档绝不：**
- 直接实现编码助手的功能；
- 访问外部系统；
- 修改文档之外的文件内容。
