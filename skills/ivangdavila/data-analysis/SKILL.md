## 使用场景

当用户需要分析、解释或可视化来自 SQL、电子表格、笔记本、仪表板、导出文件或临时表格的数据时，可以使用此技能。

适用于 KPI 调试、实验结果展示、漏斗分析或群体分析、异常情况审查、高管报告以及对指标或查询逻辑的质量检查。当分析的重点在于判断性分析（如指标定义、比较设计、结果解读或建议制定）时，优先选择此技能而非通用编程或电子表格辅助工具。

用户可能咨询的问题包括：数据分析、模式识别、指标理解、假设验证、群体分析、A/B 测试、用户流失分析或统计显著性分析等。

## 核心原则

没有决策依据的分析仅仅只是数学运算。务必明确：**如果分析结果显示 X 与 Y 不同，那么会带来什么变化？**

## 方法优先

在处理数据之前，先思考以下问题：
1. 这项分析是为了支持哪个决策？
2. 什么因素会改变你的判断？
3. 你实际拥有哪些数据？你还希望拥有哪些数据？
4. 哪个时间范围是相关的？

## 统计严谨性检查清单
- [ ] 样本量是否足够？（样本量小会导致置信区间较宽）
- [ ] 对比组是否公平？（相同的时间段、相似的条件）
- 是否进行了多重比较？（20 次比较中只有 1 次可能是偶然的“显著”结果）
- 效果大小是否有实际意义？（统计上显著并不一定具有实际重要性）
- 不确定性是否得到了量化？（例如，“提升 12-18%”而不仅仅是“提升 15%”）

## 核心规则

### 1. 从决策出发，而非从数据集开始
- 在进行分析之前，确定决策负责人、可能影响决策的问题以及截止时间。
- 如果没有任何决策需要改变，那么在开始计算之前先重新审视需求。

### 2. 在计算前明确指标定义
- 明确分析对象的实体、粒度、分子、分母、时间窗口、时区、筛选条件以及数据来源。
- 如果有任何定义不明确的地方，必须在结果展示前明确说明。

### 3. 区分数据提取、转换和解释
- 保持查询逻辑、数据处理假设和分析结论的清晰分离。
- 不要在 SQL 代码、公式或笔记本代码中隐藏业务假设，必须在其说明中明确标注。

### 4. 根据分析目的选择合适的可视化方式
- 根据分析需求选择合适的图表类型（趋势分析、对比分析、分布分析、关系分析、组成分析、漏斗分析或群体留存分析）。
- 不要为了填充报告而添加无实际意义的图表。

### 5. 以决策为导向呈现结果
- 每个输出结果都应包含答案、依据、置信度、注意事项以及后续建议。
- 如果结果将呈现给利益相关者，应将其背后的方法转化为业务影响，而非直接提供技术细节。

### 6. 在推荐行动前进行严格验证
- 根据明显的混淆因素对数据进行分段处理，比较合适的基准数据，量化不确定性，并检查结果对筛选条件或时间窗口的敏感性。
- 即使数字看起来很显著，如果没有经过严格验证，也不能作为决策依据。

### 7. 当数据无法支持结论时及时调整
- 如果样本量不足、数据来源不可靠、定义发生变化或混淆因素未解决，应暂停或降低结论的可靠性。
- 宣称“未知”总比提供错误的结论要好。

## 常见误区

- 在修改分子、分母或筛选条件后仍使用相同的 KPI 名称 → 趋势对比将失去意义。
- 在同一张图表中同时展示日度、周度和月度数据 → 数据波动看似真实，但实际上多为汇总后的噪声。
- 仅展示百分比而不提供基础数据 → 领导层可能会对微小的分母产生过度反应。
- 为了美观而选择不合适的图表 → 虽然外观整洁，但可能掩盖了关键信息。
- 在看到结果后随意调整分析内容 → 分析结果可能基于偶然性而非事实。
- 未经 KPI 负责人审核或未附带注意事项就发送自动化报告 → 错误的数据会迅速传播。
- 将观察到的模式误认为是因果关系 → 行动计划可能基于错误的关联关系制定。

## 方法选择指南

| 问题类型 | 适用方法 | 关键输出 |
|---------------|----------|------------|
| “X 与 Y 有区别吗？” | 假设检验 | p 值 + 效果大小 + 置信区间 |
| “什么因素能预测 Z？” | 回归/相关性分析 | 回归系数 + R² 值 + 剩差检验 |
| “用户随时间的行为如何？” | 群体分析 | 按群体划分的留存曲线 |
| “这些群体之间有差异吗？” | 数据分类 | 用户特征 + 统计比较 |
| “有什么异常情况？” | 异常检测 | 异常点标记 + 背景信息 |

有关每种方法的详细信息及使用场景，请参阅 `techniques.md`。

## 输出标准

1. **首先呈现分析洞察**，而非分析方法。
2. **量化不确定性**，使用范围而非具体数值。
3. **说明分析的局限性**，即分析无法揭示的内容。
4. **提出后续行动建议**，以增强结论的可信度。

## 需要升级处理的警示信号

- 用户试图“证明”预先设定的结论。
- 样本量过小，无法得出可靠结论。
- 数据质量存在问题，导致分析无效。
- 存在无法控制的混淆因素。

- 该技能不会：
- 访问未声明的外部接口。
- 将凭证或原始数据存储在本地隐藏文件中。
- 在未经用户明确同意的情况下创建自动化脚本或后台任务。
- 重新编写自身的指令文件。
