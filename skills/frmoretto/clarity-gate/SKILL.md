# Clarity Gate v2.1

**功能：** 在文档进入RAG知识库之前，该系统会进行预处理验证，以确保其知识内容的准确性。生成的文档符合Clarity Gate格式规范v2.1的要求。

**核心问题：** “如果另一个大型语言模型（LLM）阅读这些文档，它是否会将假设误认为是事实？”

**核心原则：** “检测现有的内容；强制要求必须明确表达的内容。实际操作中：在错误的假设变成‘幻觉’之前发现它们。”

## 验证代码

Clarity Gate根据FORMAT_SPEC v2.1定义了以下结构和语义检查的验证代码：

### HITL声明验证（§1.3.2-1.3.3）
| 代码 | 检查内容 | 严重程度 |
|------|-------|----------|
| **W-HC01** | `confirmed-by`/`confirmed-date`字段缺失 | 警告 |
| **W-HC02** | 来源信息模糊（例如，“行业报告”、“待定”） | 警告 |
| **E-SC06** | `hitl-claims`结构中的模式错误 | 错误 |

### 正文结构（§1.2.1）
| 代码 | 检查内容 | 严重程度 |
|------|-------|----------|
| **E-ST10** | 当存在声明时，缺少`## HITL验证记录` | 错误 |
| **W-ST11** | 表格行与`hitl-claims`的数量不匹配 | 警告 |

### SOT表格验证（§3.1）
| 代码 | 检查内容 | 严重程度 |
|------|-------|----------|
| **E-TB01** | 没有`## Verified Claims`部分 | 错误 |
| **E-TB02** | 表格中没有数据行 | 错误 |
| **E-TB03** | 必需的列缺失 | 错误 |
| **E-TB04** | 列顺序错误 | 错误 |
| **E-TB05** | 必需列中的单元格为空 | 错误 |
| **E-TB06** | 验证日期格式无效 | 错误 |
| **E-TB07** | 验证日期在未来（超出24小时的宽限期） | 错误 |

**注意：** RFC-001（澄清文档）中可能定义了额外的验证代码，但它们不属于规范的一部分。

## 打包脚本

本工具包含根据FORMAT_SPEC进行确定性计算的Python脚本。

### scripts/claim_id.py

用于生成稳定的、基于哈希的声明ID，以便进行HITL跟踪（§1.3.4）。

---

**算法：**
1. 对文本进行规范化处理（去除多余的空格）
2. 使用管道符将文本与位置信息连接起来
3. 对连接后的文本进行SHA-256哈希运算，取前8个十六进制字符
4. 在前面加上“claim-”前缀

**测试用例：**
- `claim_id("Base price is $99/mo", "api-pricing/1")` → `claim-75fb137a`
- `claim_id("The API supports GraphQL", "features/1")` → `claim-eb357742`

### scripts/document_hash.py

根据FORMAT_SPEC §2.2-2.4计算文档的SHA-256哈希值，并进行完全的规范化处理。

## 关键区别

现有的工具（如UnScientify和HedgeHunter（CoNLL-2010）**检测**文本中已存在的不确定性标记（“是否表达了不确定性？”）。

Clarity Gate则**强制**在需要表达不确定性的地方明确标注这些标记（“是否应该表达不确定性但实际没有标注？”）。

| 工具类型 | 目的 | 例子 |
|-----------|----------|---------|
| **检测** | “这段文本是否包含模糊表述？” | UnScientify/HedgeHunter会检测到“可能”、“也许”等词语 |
| **强制** | “这个声明是否应该被标记为不确定性的，但实际上没有？” | Clarity Gate会标记“收入将达到5000万美元” |

---

## 重要限制

> **Clarity Gate验证的是格式，而不是事实。**
>
> 该工具仅检查声明是否被正确地标记为不确定的——它无法验证声明是否真实。
>
> **风险：** 大型语言模型可能会在文档中“虚构”事实，然后通过添加来源标记来“通过”Clarity Gate的验证。

## 使用场景

- 在将文档导入RAG系统之前
- 在与其他AI系统共享文档之前
- 在编写规范、状态文档或方法论描述之后
- 当文档包含预测、估计或假设时
- 在发布未经验证的声明之前
- 在不同的大型语言模型会话之间传递文档之前

## 9个验证点

### 与规范套件的关系

这9个验证点指导**语义审查**——需要人工或AI进行判断的内容质量检查。它们回答诸如“这个声明是否应该被标记为不确定的？”以及“这些数字是否一致？”等问题。

审查完成后，会生成一个符合[CLARITY_GATE_FORMAT_SPEC.md](../../docs/CLARITY/Gate_FORMAT_SPEC.md)格式的CGD文件。[CLARITY/Gate_FORMAT_spec.md](../../docs/CLARITY/Gate_FORMAT_SPEC.md)中的C/S规则验证的是**文件结构**，而非语义内容。

**关联说明：**
1. 语义发现（9个点）确定存在的问题
2. 问题会被记录在CGD的状态字段中（`clarity-status`、`hitl-status`、`hitl-pending-count`）
3. 状态一致性通过结构规则（C7-C10）来确保

*示例：* 如果第5点（数据一致性）发现数字冲突，你会将`clarity-status`标记为`UNCLEAR`，直到问题得到解决。然后规则C7确保你不能在`UNCLEAR`的状态下标记为`REVIEWED`。

---

### 认知检查（核心关注点：1-4）

**1. 声明与事实的标注**
每个声明都必须明确标注为已验证或假设性的。

| 失败 | 通过 |
|-------|--------|
| “我们的架构优于竞争对手” | “我们的架构优于表3中的基准数据” |
| “模型提升了40%的性能” | “模型在数据集X上提升了40%的性能” |

**修复方法：** 添加标注：`PROJECTED:`、`HYPOTHESIS:`、`UNTESTED:`、`(~)`、`?“`

---

**2. 不确定性标记的强制使用**
前瞻性陈述需要使用限定词。

| 失败 | 通过 |
|-------|--------|
| “收入将在第四季度达到5000万美元” | “收入**预计**将在第四季度达到5000万美元” |
| “该功能将减少用户流失” | “该功能**预期**会减少用户流失” |

**修复方法：** 添加“projected”、“estimated”、“expected”、“designed to”、“intended to”等限定词”

---

**3. 假设的明确性**
影响解释的隐含假设必须明确表达。

| 失败 | 通过 |
|-------|--------|
| “系统具有线性扩展能力” | “系统具有线性扩展能力[假设同时在线用户数少于1000人]” |
| “响应时间为50毫秒” | “响应时间为50毫秒[在标准负载条件下]” |

**修复方法：** 添加括号中的条件：`[假设X]`、`[在条件Y下]`、`[当Z时]`

---

**4. 表示权威性的未经验证的数据**
包含具体百分比和勾选的表格看起来像测量数据。

**警告：** 包含具体数字（89%、95%、100%）但没有来源信息的表格

**修复方法：** 在数字后添加“(guess)”、“(est.)”、“?”等字样，并添加明确警告：“这些是**预测值**——并非测量结果”

---

### 数据质量检查（补充：5-7点）

**5. 数据一致性**
检查文档中是否存在冲突的数字、日期或事实。

**警告：** 一个部分写着“500用户”，另一个部分写着“750用户”

**修复方法：** 解决冲突或明确说明差异并给出解释。

---

**6. 隐含的因果关系**
没有证据支持的因果关系声明。

**警告：** “更短的提示可以提高响应质量”（虽然看似合理，但未经验证）

**修复方法：** 将其重新表述为假设：“更短的提示**可能**会提高响应质量（这是一个假设，尚未验证）”

---

**7. 将未来状态描述为现在的事实**
将计划中的/期望的结果描述得好像已经实现了一样。

**警告：** “系统每秒处理10,000个请求”（但实际上尚未实现）

**修复方法：** 使用未来时态/条件句：“系统**设计用于**处理……”或“目标：每秒处理10,000个请求”**

---

## 验证流程（8-9点）

**8. 时间一致性**
文档的日期和时间戳必须在内部一致且合理。

| 失败 | 通过 |
|-------|--------|
| “最后更新：2024年12月”（而当前时间是2026年） | “最后更新：2026年1月” |
| v1.0.0的版本日期为2024-12-23，v1.1.0的版本日期为2024-12-20 | 版本顺序正确 |

**子检查：**
1. 文档日期与当前日期是否一致
2. 内部时间顺序（版本、事件的顺序）
3. 参考信息的时效性（“当前”、“现在”、“今天”等表述）

**修复方法：** 更新日期，添加“截至[日期]”之类的限定词，并标记过时的文档**

---

**9. 可外部验证的声明**
具体可以验证的数字应该被标记出来以便进一步核实。

| 类型 | 例子 | 风险 |
|------|---------|------|
| 价格 | “每次调用成本约为0.005美元” | 需要提供来源信息 |
| 统计数据 | “论文平均包含15-30个方程式” | 可能存在很大误差 |
| 比率/比例 | “40%的研究人员使用X” | 需要引用来源 |
| 竞争对手的声明 | “没有竞争对手提供Y” | 可能已经过时 |

**修复方法：**
1. 添加带有日期的来源信息
2. 添加不确定性标记
3. 将相关内容路由到HITL或外部搜索系统
4. 用通用表述代替具体的数字（例如，使用“低成本”而不是“$0.005”）

## 验证层次结构

---

### 第一层：自动化验证

**A. 内部一致性**
- 图表与文本之间的矛盾
- 摘要与正文的不一致
- 表格与文字描述之间的冲突
- 数字的一致性

**B. 外部验证（扩展接口）**
- 用户提供的连接到结构化来源的接口
- 财务系统、Git提交记录、CRM等

### 第二层：两轮HITL验证——强制要求**

**第一轮：派生数据的确认**
- 来自会话中的声明
- 由人工确认其解释，而非事实本身

**第二轮：真正的HITL验证**
- 需要实际验证的声明
- 无法找到来源信息，需要使用人工数据或进行推断

## CGD输出格式

生成Clarity-Gated Document时，应使用[CLARITY/Gate_FORMAT_SPEC.md](../../docs/CLARITY/Gate_FORMAT_SPEC.md) v2.1中的格式：

---

**CGD所需的元素：**
- YAML前置内容，包含所有必需的字段：
  - `clarity-gate-version` — 工具版本（不带“v”前缀）
  - `processed-date` — 年月日格式
  - `processed-by` — 处理者名称
  - `clarity-status` — CLEAR或UNCLEAR
  - `hitl-status` — PENDING、REVIEWED或REVIEWED_WITH_EXCEPTIONS
  - `hitl-pending-count` — 整数≥0
  - `points-passed` — 例如，`1-9`或`1-4,7,9`
  - `hitl-claims` — 已验证的声明列表（可能为空`[]`）
- 结尾标记（HTML注释+状态行）：
  ```
  ```
  <!-- CLARITY_GATE_END -->
  Clarity Gate: <clarity-status> | <hitl-status>
  ```
- 如果状态为REVIEWED，则包含HITL验证记录

**可选/计算字段：**
- `rag-ingestable` — 由验证工具计算得出，不是手动设置的。仅在`CLEAR | REVIEWED`且没有排除块时显示`true`。
- `document-sha256` — 必需字段。64个字符的小写十六进制哈希值，用于完整性验证。具体规则见规范§2。
- `exclusions-coverage` — 可选字段。表示排除块内正文的比例（0.0–1.0）。

**逃逸机制：** 要在文本中写入如`*(estimated)*`这样的标记时，需要用反引号括起来：`` `*(estimated)*` ``

### 声明完成状态（v2.1）

声明的验证状态是通过字段**是否存在**来确定的，而不是通过一个明确的状态字段：

| 状态 | `confirmed-by` | `confirmed-date` | 含义 |
|-------|----------------|------------------|----------|
| **PENDING** | 未提供 | 未提供 | 正在等待人工验证 |
| **VERIFIED** | 提供了 | 提供了 | 人工已经确认 |
| *(invalid)* | 提供了 | 未提供 | W-HC01：部分字段缺失 |
| *(invalid)* | 提供了 | 提供了 | W-HC01：部分字段缺失 |

**为什么没有明确的状态字段？** 因为字段的存在本身就说明了状态——你无法在未提供确认者和确认时间的情况下错误地设置状态。

### 来源字段语义（v2.1）

`source`字段的含义根据声明的状态而变化：

| 状态 | `source` 应包含的内容 | 例子 |
|-------|-------------------|----------|
| **PENDING** | 需要验证的来源（可操作的来源） | “查看Q3规划文档” |
| **VERIFIED** | 找到的内容（证据） | “Q3规划文档，第12页” |

**模糊来源的检测（W-HC02）：** 如“行业报告”、“研究”、“待定”等来源会触发警告。

### 声明ID格式（v2.1）

**通用格式：** `claim-[a-z0-9._-]{1,64}`（字母数字、点、下划线、连字符）

| 方法 | 格式 | 例子 | 使用场景 |
|----------|---------|---------|----------|
| **基于哈希的**（推荐） | `claim-[a-f0-9]{8,}` | `claim-75fb137a` | 确定性高，抗冲突 |
| **顺序式的** | `claim-[0-9]+` | `claim-1`、`claim-2` | 简单文档 |
| **语义化的** | `claim-[a-z0-9-]+` | `claim-revenue-q3` | 便于人类阅读 |

**冲突概率：** 当有1,000个声明且使用8个字符的哈希ID时，冲突概率约为0.012%。当声明数量超过1,000个时，使用12个或更多字符的哈希值。**

**建议：** 使用`scripts/claim_id.py`生成的基于哈希的ID，以确保一致性和抗冲突性。

## 排除块

当内容无法解决（例如没有专家可用、使用过时的文本等）时，应将其标记为排除内容，而不是保持模糊状态：

---

**规则：**
- ID必须符合格式：`[A-Za-z0-9][A-Za-z0-9._-]{0,63}` |
- 不允许嵌套或重复的ID块
- 每个ID只能使用一次
- 必须设置`hitl-status: REVIEWED_WITH_EXCEPTIONS`
- 必须在前置内容中记录`exceptions-reason`和`exceptions-ids`

**重要提示：** 包含排除块的文档**无法被RAG系统接收。这些文档将被完全拒绝（不会被部分导入）。

详细规则请参见[CLARITY/Gate_FORMAT_SPEC.md §4](../../docs/CLARITY/Gate_FORMAT_SPEC.md)。

## SOT验证

在验证Source of Truth文件时，该工具会检查**格式是否符合规范**（[CLARITY/Gate_FORMAT_SPEC.md](../../docs/CLARITY/Gate_FORMAT_SPEC.md)）以及**内容质量**（上述9个验证点）。

### 格式符合性（结构规则）

SOT文档是带有`tier:`块的CGD。它们需要一个`## Verified Claims`部分，并且该部分必须包含一个有效的表格。

| 代码 | 检查内容 | 严重程度 |
|------|-------|----------|
| E-TB01 | 没有`## Verified Claims`部分 | 错误 |
| E-TB02 | 表格中没有数据行 | 错误 |
| E-TB03 | 必需的列缺失（Claim、Value、Source、Verified） | 错误 |
| E-TB04 | 列顺序错误（Claim不是第一列，Verified不是最后一行） | 错误 |
| E-TB05 | 必需列中的单元格为空 | 错误 |
| E-TB06 | 验证日期格式无效 | 错误 |
| E-TB07 | 验证日期在未来（超出24小时的宽限期） | 错误 |

### 内容质量（9个验证点）

这9个验证点也适用于SOT内容：

| 点 | SOT的应用场景 |
|-------|-----------------|
| 1-4 | 检查`## Verified Claims`中的声明是否确实经过验证 |
| 5 | 检查表格中的数值是否一致 |
| 6 | 检查声明是否没有暗示不支持的因果关系 |
| 7 | 检查表格中是否将未来情况描述为当前事实 |
| 8 | 检查日期是否按时间顺序排列 |
| 9 | 标记需要外部验证的具体数字 |

### SOT的特定要求

- **必须包含tier块：** SOT是一个CGD，其中包含`tier:`块，其中包含`level`、`owner`、`version`、`promoted-date`、`promoted-by`等字段 |
- **结构化的声明表格：** `## Verified Claims`部分应包含Claim、Value、Source、Verified等列 |
- **排除块外的表格：** 已验证的声明表格不能位于排除块内 |
- **时效性标记：** 在内容中使用`[STABLE]`、`[CHECK]`、`[VOLATILE]`、`[SNAPSHOT]`等标记
  - `[STABLE]` — 可以直接引用，无需重新验证 |
  - `[CHECK]` — 引用前需要验证 |
  - `[VOLATILE]` — 数据经常变化；必须验证 |
  - `[SNAPSHOT]` — 表示即时数据；引用时需要注明日期 |

## 严重程度级别

| 级别 | 定义 | 处理方式 |
|-------|------------|--------|
| **CRITICAL** | 大型语言模型可能会将假设视为事实 | 使用前必须修复 |
| **WARNING** | 大型语言模型可能会误解内容 | 需要修复 |
| **TEMPORAL** | 发现日期/时间不一致 | 需要验证并更新 |
| **VERIFIABLE** | 可以验证的具体声明 | 需要路由到HITL或外部搜索系统 |
| **ROUND A** | 来自可验证来源的声明 | 需要快速确认 |
| **ROUND B** | 需要实际验证的声明 | 未经验证无法通过 |
| **PASS** | 标记明确，无歧义，已验证 | 无需额外处理 |

## 快速检查清单

| 规则 | 处理方式 |
|---------|--------|
| 具体的百分比（如89%、73%） | 添加来源信息或标记为估计值 |
| 对比表格 | 添加“PROJECTED”标题 |
| 使用“ achieves”、“delivers”、“provides”等词时，如果未经过验证，应使用“designed to”、“intended to” |
| 勾选符号 | 需要验证这些内容是否确实经过确认 |
| “100%”之类的表述 | 几乎总是需要添加限定词 |
| “Last Updated: [日期]” | 需要与当前日期进行核对 |
| 带有日期的版本号 | 需要验证版本顺序 |
| “$X.XX”或“~$X”（价格） | 需要标记以便外部验证 |
| “averages”、“typically”等表述 | 需要标注来源或引用 |
| 竞争对手的能力声明 | 需要标记以便外部验证 |

## 本工具不执行的功能

- 不对文档类型进行分类（使用Stream Coding来完成这项工作）
- 不会重新组织文档内容
- 不会添加深层链接或引用
- 不会自动评估写作质量
- **不会自主验证事实的准确性**（需要通过HITL进行验证）
