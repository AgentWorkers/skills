# Feishu 主题创建

在专门的 **Feishu 主题组** 中创建一个新的主题，然后可以选择性地在该主题中发布 **第一条回复**。

请保持该功能的简洁性。仅用于主题创建和初始回复的生成，而不用于会话路由的复杂操作。

## 已验证的行为

以下工作流程已通过验证：
1. 使用 `feishu_im_user_message.send` 向主题组 `chat_id` 发送消息，会创建一个 **新的顶层主题**。
2. 使用 `feishu_im_user_message.reply` 并设置 `reply_in_thread=true`，会在该主题内发布回复。
3. 可以通过在内容中直接使用 Feishu 标记来 **@** 某位用户：

```text
<at user_id="ou_xxx">Name</at> 你好
```

这是当前推荐的方法。建议优先使用这种方法，而不是依赖隐式的回复路由机制。

## 前提条件

必须满足以下所有条件：
- 目标是一个真实的 **Feishu 主题组**。
- 用户明确表示要发送消息。
- 用户具有在该组中发帖的权限。
- 用户已授权使用 Feishu 消息功能（`feishu_im_user_message`）。

如果用户授权失败或已过期，请停止操作并请求用户重新授权。
如果目标组不是主题组，请告知用户这将导致普通群组消息的生成，而不是主题。

## 默认目标组

默认使用 `TOOLS.md` 中配置的固定主题组：
- `泡菜的话题组`

可以使用 `feishu_chat` 并设置 `action=search` 来查找目标组。
如果没有找到匹配的组：
- 停止操作。
- 告知用户该主题组尚不存在。
- 请求用户先手动创建该主题组。

## 工作流程

### 1. 确定目标组

- 如果用户指定了一个组名，使用该名称。
- 否则，使用默认值 `泡菜的话题组`。
- 使用 `feishu_chat.search` 来查找对应的 `chat_id`。

### 2. 解析用户请求的操作

支持以下四种操作方式：

#### A. 仅创建主题
示例：
```text
开话题：OpenClaw 多线程 SOP
```

仅发送一条顶层文本消息。

#### B. 创建主题 + 发布第一条回复
示例：
```text
开话题：OpenClaw 多线程 SOP｜先写一个 5 点大纲
```

解析为：
- 主题标题：`OpenClaw 多线程 SOP`
- 第一条回复：`先写一个 5 点大纲`

#### C. 创建主题 + 发布第一条回复 + @提及
示例：
```text
开话题：测试 1｜回一条：@我 测试
```

解析为：
- 顶层主题标题/内容
- 然后是一条回复
- 可选地在回复中添加 `@user_id="...">...</at>` 来提及用户

#### D. 创建主题 + 保留之前的上下文 + 提出新问题
示例：
```text
开个新话题聊这个，把前面几条带过去，然后追问：自由现金流有几个指数？有什么区别？
```

解析为：
- 创建一个全新的主题
- 仅保留最相关的上下文
- 将 **上下文和新问题** 放在同一条顶层主题创建消息中
- 确保新问题容易被用户注意到
- 除非用户明确要求，否则不要发送第二条回复

### 3. 创建主题

使用 `feishu_im_user_message.send`：
- `action=send`
- `receive_id_type=chat_id`
- `receive_id=<chat_id>`
- `msg_type=text`
- `content={"text":"<顶层文本>"}`

返回的 `message_id` 即为主题的唯一标识符。

### 4. （可选）添加第一条回复

如果用户需要初始回复，使用 `feishu_im_user_message.reply`：
- `action=reply`
- `message_id=<主题唯一标识符>`
- `reply_in_thread=true`
- `msg_type=text`
- `content={"text":"<回复内容>"}`

如果需要提及用户，可以直接在文本中添加提及：

```text
<at user_id="ou_xxx">Name</at> 回复内容
```

当用户输入 `@我` 时，使用当前的 `open_id` 来生成提及。

### 5. 在新主题中保留上下文时保持简洁

对于 **保留上下文 + 提出新问题** 的情况，建议使用 **一条顶层消息**。
除非用户明确要求，否则不要添加第二条回复，因为两条用户消息可能会导致助手生成两条回复。

建议使用简单的结构：

```text
<标题>

问题：...

前情提要：...
```

或者，如果简短的总结更合适，可以使用以下格式：

```text
<标题>

问题：...

前情提要：
- ...
- ...
```

规则：
- 先提出问题。
- 保持总结简洁。
- 除非绝对必要，否则不要添加额外的指示性文字。
- 目的是让用户能够轻松理解后续操作，而不是试图“欺骗”模型。

示例：

```text
HALO：自由现金流指数有什么区别

问题：
A股里自由现金流目前有几个主流指数？它们分别有什么区别？

前情提要：
- 刚讨论过 HALO 在 A 股的映射，涉及资源、能源、公用事业、电网、交运、央企红利、现金流。
- 这次只是在这个基础上继续追问自由现金流指数。
```

## 解析指南

将以下内容视为可能的触发指令：
- `开话题：标题`
- `开个话题：标题`
- `开个新话题：标题`
- `/topic 标题`
- `/topic 标题｜正文`
- `新开一个话题，然后在话题里回复一条`
- `开话题: 标题, 然后在话题里回复 1 条测试并且 at 我`

有用的解析规则：
- 以第一个 `：` 或 `:` 作为分隔符，将命令和内容分开。
- 将 `｜` 或 `|` 视为标题和第一条回复之间的分隔符。
- 如 `然后回复一条`、`回一条`、`在话题里回复`、`并且 at 我`、`@我` 等短语表示需要在主题内回复。
- 如 `把前面几条带过去`、`把刚才聊的内容带过去`、`新开一个话题聊这个`、`继续追问`、`新的追问` 等短语表示需要携带之前的上下文。
- 如果只有一条内容，将其视为纯主题文本。

不要过度解析。如果消息含义不明确，可以询问用户一个简短的澄清问题。

## 工具推荐

建议优先使用已验证的用户消息发送路径：
- `feishu_im_user_message.send`
- `feishu_im_user_message.reply`

原因：这些方法已经经过了端到端的验证，可以完成以下操作：
- 主题创建
- 主题内的回复
- 用户提及

不要依赖普通的助手回复路由机制来创建新主题。

## 回复用户

操作成功后，简要告知用户以下信息：
- 目标主题组
- 使用的主题文本/标题
- 是否添加了回复
- 是否包含了提及

回复要简洁明了。

## 使用限制

- 该功能仅在 **Feishu 主题组** 中按预期工作。
- 该功能 **不会** 自动创建 Feishu 组。
- 除非用户明确请求，否则不会自动发送消息。
- 该功能用于 **主题的初始化**：创建主题，可选地添加第一条回复，然后让后续对话在该主题内继续进行。
- 如果用户希望发送普通群组消息而非主题消息，请不要强制使用该功能。

## 最小示例

### 示例 1：仅创建主题
用户：
```text
开话题：测试 4
```

操作：
- 查找 `泡菜的话题组`
- 发送顶层消息 `测试 4`

### 示例 2：创建主题 + 发布回复
用户：
```text
开话题：测试 3｜不 @ 的线程回复测试
```

操作：
- 创建顶层消息 `测试 3`
- 在主题内回复 `不 @ 的线程回复测试`

### 示例 3：创建主题 + 发布回复 + @提及
用户：
```text
开话题：测试 2｜@我 再测一次
```

操作：
- 创建顶层消息 `测试 2`
- 在主题内回复：
```text
<at user_id="ou_xxx">Name</at> 再测一次
```

### 示例 4：保留上下文 + 提出新问题
用户：
```text
开个新话题聊这个，把前面几条带过去，然后追问：自由现金流有几个指数？有什么区别？
```

操作：
- 创建一个专注于新问题的新主题
- 发送一条顶层主题创建消息
- 先写 `问题：...`
- 然后添加简短的 `前情提要：...`
- 除非用户明确要求，否则不要添加第二条回复
