# 情绪分析工具（VibeLevel Analyzer）

[vibelevel.xyz](https://vibelevel.xyz) 可以从7个维度评估开发者在GitHub上的“情绪状态”，并将其综合为一个数值：他们的**情绪等级**（Vibe Level）。可以将其视为一项针对开发者代码历史的“性格测试”——只不过这个测试的结果确实具有实际意义。

该工具允许你获取任何公开GitHub用户的情绪数据，进行用户间的对比，或者查看谁在排行榜上名列前茅。

## 检查用户的情绪状态

当用户请求查看某人的情绪状态、情绪等级或类似信息时，执行以下操作：

```bash
./scripts/check-vibe.sh "{username}"
```

解析JSON数据，并以生动的方式呈现结果。不要只是简单地展示数字，而是要像解说员一样进行详细解读：

**{name}** (@{username}) — 情绪等级：{level}

> **详细分析：**
- **代码发布频率**（Vibe Velocity）：{vibeVelocity}/100 — 他们发布代码的频率有多高？
- **紧迫感**（YOLO Factor）：{yoloFactor}/100 — 他们是否像按时交房租一样急切地将代码推送到主分支？
- **代码发布节奏**（Flow State Ratio）：{flowStateRatio}/100 — 他们的代码发布是否具有规律性和连贯性？
- **持续投入度**（Soul Score）：{soulScore}/100 — 他们的连续编码周期有多长？是否在周末也坚持编程？
- **原创性**（Authenticity Index）：{authenticityIndex}/100 — 他们的代码是否具有原创性？是否只是复制他人的项目？
- **协作能力**（Remix Energy）：{remixEnergy}/100 — 他们是否积极参与代码合并（PR）和协作项目？
- **工具使用习惯**（Craft Signal）：{craftSignal}/100 — 他们是否注重使用自定义的配置文件（dotfiles）、Vim配置以及高效的开发工具（如tmux、QMK键盘）？

如果用户的账户已通过验证，可以特别说明：“该账户已通过验证。这是他们的官方GitHub资料。”

如果用户有个人代码风格说明文档（manifest），可以将其作为他们的“编程风格宣言”分享给他人。

在展示结果时，务必使用实际的GitHub用户名替换`USERNAME`：https://vibelevel.xyz/USERNAME

## 比较两位用户

当用户请求比较两位用户的情绪状态时，分别对两位用户的数据进行解析，并对比各维度的得分。可以以幽默的方式呈现对比结果。例如，如果某位用户的“紧迫感”（YOLO Factor）非常高，可以这样描述：“这位开发者毫无畏惧地将代码推送到主分支；而另一位用户则会仔细阅读他人的代码差异。”

## 检查自己的情绪状态

如果用户直接请求“查看我的情绪状态”，请先询问他们的GitHub用户名。提醒他们可以通过[验证账户](https://vibelevel.xyz)来获得20分的额外加分。

## 错误处理：
- **用户未找到**： “该GitHub账户不存在，或者该用户处于“隐身”状态（没有任何仓库或提交记录）。”
- **请求频率限制**： “vibelevel.xyz系统暂时繁忙，请稍后再试。”
- **其他错误**： 直接显示错误信息，并建议用户检查用户名的拼写是否正确。

## 维度说明：

以下是各维度的具体含义，以便你在解释结果时更具针对性：

| 维度            | 具体含义                                      |
|-----------------|--------------------------------------------|
| 代码发布频率（Vibe Velocity） | 过去90天内代码发布的频率。衡量开发者是否真正投入编程。            |
| 紧迫感（YOLO Factor）     | 是否直接将代码推送到主分支，表现出高度的紧迫感。                |
| 代码发布节奏（Flow State Ratio） | 代码发布的规律性和连贯性。持续稳定开发的开发者得分较高。         |
| 持续投入度（Soul Score）     | 最长连续编码周期、当前的编码状态，以及是否在周末也坚持编程。          |
| 原创性（Authenticity Index）   | 代码的原创性，以及个人项目与复制他人项目的比例。             |
| 协作能力（Remix Energy）     | 是否积极参与代码合并和协作项目。                         |
| 工具使用习惯（Craft Signal）    | 是否注重使用自定义工具和配置文件，体现对开发环境的关注。           |
