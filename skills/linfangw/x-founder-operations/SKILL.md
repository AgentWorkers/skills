# X创始人运营指南

本指南为创始人、独立开发者和科技专业人士提供系统的X（Twitter）账户管理方法。

## 适用人群

- **创始人**：在运营公司的同时，需要建立品牌、吸引潜在客户、筹集资金、招聘人才并树立行业领导地位。
- **独立开发者**：希望扩大产品影响力，分享开发过程，并将粉丝转化为实际用户。
- **科技专业人士**：希望打造个人品牌，展示专业能力，扩展职业网络。

该技能可根据您的具体角色、目标和成长阶段进行定制。

## 核心理念

### 三大支柱

1. **真实为先**：您的个人形象应真实反映您的本质，而非刻意塑造的角色。人们关注的是人，而非品牌。
2. **价值优先于推广**：80%的内容应提供真实价值。信任的建立需要时间，但一旦丧失则难以恢复。
3. **数据驱动的迭代**：每条帖子都会产生数据，每个数据点都会影响后续决策。策略应基于实际结果而非直觉制定。

### 数据完整性（不可妥协）

- **严禁伪造推文内容、指标或链接**。
- **严禁为填补数据空白而猜测或编造信息**。
- **必须通过验证流程确认数据**。
- **缺失的数据应标记为`null`，切勿随意假设其值**。

当API调用失败时：立即停止操作，如实报告问题，并严禁使用伪造的数据。详细验证流程请参见[数据验证脚本](scripts/x_data-validator.py)。

## 快速入门

### 1. 设置个人资料

复制并自定义配置模板：

```
config/profile.example.json → config/profile.json
config/goals.example.json → config/goals.json
config/watchlist.example.json → config/watchlist.json
config/content-calendar.example.json → config/content-calendar.json
```

填写您的用户名、角色、产品信息、个人风格和目标受众。有关优化X个人资料的指导，请参阅[个人资料优化](references/profile-optimization.md)。

### 2. 设定目标

在`config/goals.json`中设置：
- 主要目标（品牌建设、吸引潜在客户、筹集资金、招聘人才、树立行业领导地位）
- 成长阶段（初创期、发展期、成熟期）
- 每月和每季度的目标
- 内容发布比例

有关不同成长阶段的实际目标，请参阅[成长基准](references/growth-benchmarks.md)。

### 3. 建立关注列表

在`config/watchlist.json`中指定：
- 5-10个需要每日互动的核心账户
- 3-5个需要监控的竞争对手账户
- 需要关注的行业和媒体账户
- 需要加入的相关X社区

### 4. 开始日常操作

遵循[运营方法论](references/methodology.md)中的PDCA循环：

```
Morning: Scan → Plan → Warm Up (30-45 min)
Daytime: Create → Post → Engage (throughout the day)
Evening: Check metrics → Reflect → Plan tomorrow (15-20 min)
```

## 日常操作流程

详细内容请参阅[方法论](references/methodology.md)。

### 早晨阶段：计划与热身

1. **浏览**：查看夜间收到的提及、核心账户的活动以及热门话题。
2. **计划**：从日历中选择当天的内容并准备推文草稿。
3. **热身**：花15-20分钟回复核心账户中的10-15条推文（这有助于提升您的推文在算法中的排名）。

### 执行阶段：创建推文 → 发布 → 互动

4. 在最佳时间点发布2-3条推文（具体时间请参见[定时计划](references/cron-schedule.md)，间隔2-3小时。
5. **互动**：每条推文发布后30-60分钟内回复所有评论。
6. **全天与核心账户和社区互动**。

### 傍晚阶段：检查与反思

8. **收集当天的数据指标**。
9. 使用[每日回顾模板](templates/daily-review.md)进行快速总结。
10. **规划明天**：起草内容并确定互动目标。

### 周/月/季度评估

- **每周**（周日晚）：评估表现、审核内容、进行A/B测试、规划下周内容。
- **每月**：评估目标进展、调整策略、分析竞争对手。
- **每季度**：全面审视战略、重新设定目标、评估个人形象。

## 内容策略

详细内容请参阅[内容策略](references/content-strategy.md)。

### 五大内容类型

| 类型 | 目标占比 | 目的 |
|--------|----------|---------|
| 技术深度 | 40% | 提供技巧、见解和教程，建立专业权威 |
| 行业动态 | 25% | 发表独到观点、预测趋势，树立行业领导地位 |
| 开发历程 | 25% | 公开分享开发过程和决策，建立联系 |
| 社区价值 | 10% | 提出问题、开展投票、提供资源，增强社区凝聚力 |
| 个人风格 | 适当融入幽默元素、展现个性，增加真实感 |

### 内容制作流程

```
Ideation → Drafting → Review → Scheduling → Posting → Measurement
```

每个步骤都包含质量检查。详细内容请参阅[内容策略：内容制作流程](references/content-strategy.md)。

### 有效的内容公式

- **明确主张 + 证据 + 行动号召**：提出大胆主张，提供具体证据，并引发讨论。
- **故事 + 教训 + 总结**：简要叙述事件，提供可操作的见解。
- **问题 + 背景 + 邀请**：提出有深度的问题，解释原因，并邀请他人分享观点。
- **列表 + 价值 + 行动号召**：列出要点并附简要说明，鼓励读者收藏或关注。
- **反应 + 分析 + 预测**：引用事实、提供独特分析，并预测未来趋势。

### 内容再利用

- 表现良好的推文可转化为5-7条独立推文。
- 热门单条推文可扩展为完整的讨论串。
- 讨论串可转化为博客文章或新闻通讯；博客文章也可转化为讨论串。
- 每3-4周重新发布内容，但使用新的表述方式。

## 讨论串策略

讨论串是X平台上提升影响力的有效工具。详细内容请参阅[讨论串攻略](references/thread-playbook.md)。

### 讨论串架构

```
Hook (stop the scroll) → Context → Value Delivery (one point per tweet) → Summary → CTA
```

### 关键原则

- 最佳长度：大多数创始人适合5-7条推文。
- 在目标受众的时区（08:00-10:00）发布推文。
- 60-70%的推文应包含视觉元素。
- 提前一天准备，周二至周四发布。
- 每周发布1-2个讨论串。

## 多样化内容形式

详细指导请参阅[多模态内容指南](references/multimodal-guide.md)。

### 内容效果

- 原生视频的传播效果是纯文本的10倍。
- 图片的互动效果是纯文本的2-3倍。
- 视觉元素能延长用户停留时间（对算法排名有重要影响）。

### 视觉元素类型

| 类型 | 适用场景 |
|------|----------|
| 屏幕截图 | 产品演示、代码输出、实时数据、仪表盘 |
| 图表 | 系统架构、工作流程、思维模型 |
| 代码片段 | 技术教程、调试过程 |
| 模因 | 行业幽默、个人趣事、文化评论 |
| 数据可视化 | 研究结果、行业趋势、公开分享的指标 |
| 原生视频 | 产品演示、专家访谈、教程 |

### “真实感”原则

所有视觉元素都应显得真实自然。略带瑕疵的截图、手绘图表和真实的工作环境照片能体现真实性。过度修饰或明显由AI生成的视觉元素会显得不自然。

## 产品整合

详细内容请参阅[产品整合](references/product-integration.md)。

### 三种推广方式

| 推广层次 | 内容 | 发布频率 | 方法 |
|------|------|-----------|-----|
| 第一层：背景提及 | 产品作为背景提及 | 默认模式 | 个人简介、个人资料、置顶推文 |
| 第二层：情境关联 | 以产品为例进行讨论 | 约10%的推文 | “当我们开发[产品X]时...” |
| 第三层：直接推广 | 产品作为主题 | 约10%的推文 | 发布新产品发布或重要里程碑的信息 |

### 黄金法则

即使去掉产品提及，推文本身也应具有价值。产品只是证明您能力的工具，而非推广的重点。

### 行动号召优化

- 切勿在主推文中放置链接（否则会导致算法惩罚）。
- 在X平台上，温和的行动号召比强制推销更有效。
- 根据与粉丝的关系阶段调整行动号召：新粉丝可建议“关注以获取更多信息”；已互动的粉丝可建议“尝试使用产品”。

## 互动策略

详细内容请参阅[互动策略](references/engagement-playbook.md)。

### 核心互动群体

识别20-30个关键账户。每天与第一页级的账户互动，每周与第二层账户互动3-4次，第三层账户则视情况而定。具体配置请参见[关注列表示例](config/watchlist.example.json)。

### 回复质量标准

| 回复类型 | 影响效果 |
|-------|------|--------|
| S | 数据/证据 | 非常高 |
| A | 独到见解 | 高 |
| B | 个人故事 | 高 |
| C | 深刻问题 | 中等 |
| D | 机械性回应（如“确实如此！”） | 低 |

### 其他渠道

- **X Spaces**：每两周举办一次音频讨论，邀请核心账户的嘉宾参与。
- **X社区**：在相关社区发布内容，注重提供价值而非推广。
- **私信**：在公开互动3次后，发送真诚的私信（避免推销）。
- **X Lists**：为竞争对手、行业领袖、投资者和潜在客户创建私人列表。

## 个人形象与风格

详细内容请参阅[个人形象与风格指南](references/persona-voice.md)。

### 个人风格设定

在`config/profile.json`中设置以下各项的偏好：
- 技术深度：从浅显到深入
- 幽默风格：从严肃到幽默
- 正式程度：从随意到正式
- 开放程度：从保守到开放
- 表达方式：从谨慎到大胆

### 一致性原则

- 在所有内容中保持一致的语气（算法会识别您的风格）。
- 根据情境调整语气（技术内容、个人内容或推广内容）。
- 建立个人特色：通过重复的主题、口头禅或笑话来建立识别度。
- 双语用户：技术/国际内容使用英语，个人/文化内容使用母语。

## 算法理解

详细内容请参阅[算法指南](references/algorithm-guide.md)。

### 关键规则

- **前30分钟决定传播效果**：前15-30分钟的互动情况决定了80%的传播效果。
- **回复 > 转发 > 收藏 > 点赞**：算法更重视回复（权重约为1倍），点赞权重最低（约0.5倍）。
- **外部链接会降低排名**：链接会减少30-50%的展示次数。建议将链接放在第一条回复中。
- **保持发布节奏的一致性**：突然改变发布频率会触发算法的垃圾信息检测。
- **专注特定领域**：专注于2-3个主题。随意发布其他主题会降低算法排名。
- **负面内容后果严重**：禁言/屏蔽/举报负面内容会严重影响排名。切勿发布攻击性内容。
- **视频优先**：原生视频的传播效果是纯文本的10倍。

## 自动化操作

详细任务安排请参阅[定时计划](references/cron-schedule.md)。

### 日常自动化任务

| 任务 | 时间 | 目的 |
|------|--------|---------|
| 早晨浏览 | 发布窗口前30分钟 | 查看热门话题、核心账户活动、夜间收到的提及 |
| 发布窗口提醒 | 发布窗口开始时 | 根据日历和浏览结果推荐内容 |
| 中午数据收集 | 两次发布窗口之间 | 更新热门话题和发布窗口的指标 |
| 下午发布提醒 | 发布窗口开始时 | 为晚上发布窗口推荐新的内容 |
| 互动跟进 | 发布窗口后30分钟 | 寻找回复机会、挑选合适的推文 |
| 互动情况检查 | 发布窗口后1小时 | 分类并优先处理提及的内容 |

### 数据验证流程

所有自动化数据收集均遵循以下流程：

```
x-timeline → Get tweet list
    ↓
x-tweet-info → Get detailed metrics per tweet
    ↓
Cross-validate → ID, author, timestamp consistency
    ↓
Sanity check → Reasonable data (likes ≤ impressions, no negatives)
    ↓
Report → Mark anomalies, never guess
```

## 竞争对手分析

详细内容请参阅[竞争对手分析框架](references/competitor-framework.md)。

选择5-8个参考账户，分为三类：
- **标杆账户**（2-3个）：规模是您的10-100倍，但处于相同领域。
- **同级别对手**（2-3个）：规模和成长阶段与您相似。
- **新兴对手**（1-2个）：规模较小但增长迅速。

每月分析这些账户的内容策略、互动模式和发布时间，学习他们的成功经验，并将其应用到自己的运营中。

## 应急预案

### API故障
1. 采用指数级重试机制（最多尝试3次）。
2. 如果仍无法解决问题，向用户报告错误。
3. 严禁使用伪造的数据。

### 内容危机
1. 暂停所有自动化发布。
2. 检查最近的内容是否存在问题。
3. 准备应对策略。
4. 立即通知用户。

### 数据异常
1. 在验证报告中立即标记异常数据。
2. 通过其他来源进行交叉验证。
3. 不要在分析中包含异常数据。
4. 通知用户进行调查。
