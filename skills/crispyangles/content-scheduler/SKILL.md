# 内容调度系统

在发布了27篇文章后，我发现其中一半的文章采用了相同的格式，于是我创建了这个系统。

## 核心问题

要么是你发布的文章太少（没有系统来帮助你），要么是发布的文章质量太差（缺乏多样性）。这两种情况都会导致用户参与度下降。这个系统通过提供轮播机制、日历以及从“想法”到“发布”的完整起草流程，解决了这两个问题。

## 内容轮播机制

### 定义你的内容类型（3-5种类型最为合适）

选择与你的内容输出相匹配的类型。以下是我用于构建品牌的内容类型示例：

| 编号 | 类型 | 目的 | 示例 |
|---|---|---|---|
| 1 | **热点观点** | 吸引注意力，引发讨论 | “提示工程并不是一项技能，只是清晰的思考罢了。” |
| 2 | **深度分析** | 展示专业知识，建立信任 | “大多数AI工具为何会失败的三部分剖析” |
| 3 | **问题** | 促进回复，了解受众需求 | “你离不开哪一款AI工具？” |
| 4 | **视觉内容** | 阻止用户快速滚动页面，便于收藏 | 引用卡片、数据可视化、截图 |
| 5 | **故事/进展记录** | 建立联系，展示进展 | “第5天：收入0美元，粉丝17人。这是我的收获。” |

对于**技术博客**，可以替换为以下类型：
1. 教程/操作指南
2. 架构深度分析
3. 观点/热点观点
4. 工具对比/评测
5. 回顾/事后分析

对于**新闻通讯**：
1. 精选链接 + 评论
2. 原创文章
3. 访谈/问答总结
4. 数据分析或趋势报告
5. 背景故事/个人更新

### 轮播规则

很简单：**不要连续发布相同类型的文章**。每次发布后，跳到列表中的下一个类型。当列表中的类型都发布完毕后再重新开始。

这听起来很机械，但实际上并非如此。用户看不到背后的机制，他们感受到的是内容的多样性，而多样性是防止用户取消关注你的关键。

### 跟踪文件

将此文件保存在工作区中。每次发布前请查看它：

```json
{
  "types": ["hot-take", "thread", "question", "visual", "story"],
  "nextType": "question",
  "todayCount": 1,
  "maxPerDay": 4,
  "history": [
    {
      "date": "2026-03-13",
      "type": "thread",
      "title": "Why most people use AI wrong",
      "channel": "twitter",
      "status": "published",
      "notes": "Got 3 bookmarks — thread format works for this topic"
    },
    {
      "date": "2026-03-13",
      "type": "hot-take",
      "title": "Stop asking ChatGPT nicely",
      "channel": "twitter",
      "status": "published",
      "notes": "First post with new voice. 87 impressions in 2 hours."
    }
  ]
}
```

**关键字段：**
- `nextType` — 避免决策疲劳。你已经知道这次要写什么类型的文章了。
- `todayCount` + `maxPerDay` — 避免内容过多导致平台限制。我曾因为一天内发布了14篇文章而触发平台限制，花了三天才恢复。
- `notes` — 这里记录了你的学习过程。一个月后，你会发现哪些内容类型更有效。

## 内容日历

### 每周规划（周一上午15分钟）

```markdown
# Content Plan — Week of March 10

## Slots (adjust times to your audience)
| Day | Time | Type | Topic | Status |
|-----|------|------|-------|--------|
| Mon | 7:30 AM | hot-take | AI and job interviews | ✅ published |
| Mon | 12:00 PM | question | Best AI tool for X? | ✅ published |
| Tue | 7:30 AM | thread | 3 prompt frameworks | 📝 draft |
| Tue | 5:30 PM | visual | Quote card: clear thinking | 🎯 ready |
| Wed | 7:30 AM | story | Day 6 build log | 💡 idea |
| Wed | 12:00 PM | hot-take | Why prompt courses are BS | 💡 idea |

## Notes
- Heavy on hot-takes this week — riding the AI layoffs news cycle
- Question post on Monday bombed — audience prefers A/B choices over open-ended
- Thread needs to be max 4 parts (completion rate drops after 3)
```

**15分钟规则：** 如果规划时间超过15分钟，说明你想得太多了。选择与新闻周期、你的近期经验或受众需求相匹配的主题。不要等待灵感——灵感来自行动，而非规划。

### 批量起草（真正有效的高效方法）

我发现最有效的方法是：
1. **规划阶段**（15分钟）：为接下来的3-5天选择所有主题。
2. **起草阶段**（60-90分钟）：一次性为所有主题起草初稿。
3. **逐个编辑并发布**：逐个完善内容，然后安排发布时间。

为什么这种方法有效？因为在“该写什么？”和“实际写作”之间的切换过程中浪费了很多时间。将这两个步骤分开可以避免这种浪费。

**实际数据：** 批量起草8篇文章大约需要90分钟；如果分散在4天内完成，则总共需要约3小时，但效果却是一样的。

## 起草流程

每篇文章都只处于以下三种状态之一：

| 状态 | 含义 | 规则 |
|---|---|---|
| `idea` | 只有主题，没有内容 | 可以无限期地停留在这个状态 |
| `draft` | 有内容，需要编辑 | **7天期限** — 必须编辑或删除 |
| `ready` | 编辑完成，准备发布 | 48小时内发布，否则内容会过时 |
| `published` | 已发布 | 添加关于发布效果的备注 |

**7天起草规则：** 如果一个草稿7天内未被修改，你有两个选择：要么粗糙地发布，要么删除它。不要拖延到“以后再处理”。我见过23个草稿堆积如山却一个也没发布，因为编辑任务让人感到不堪重负。不要让这种情况发生在你身上。

### 快速检查命令
```bash
python3 -c "
import json, sys
from collections import Counter
with open('content-tracker.json') as f:
    data = json.load(f)
counts = Counter(item['status'] for item in data.get('history', []))
total = sum(counts.values())
for status in ['idea', 'draft', 'ready', 'published']:
    c = counts.get(status, 0)
    bar = '█' * c + '░' * (total - c) if total > 0 else ''
    print(f'  {status:12s} {c:3d}  {bar}')
print(f'  {\"\":12s} {total:3d}  total')
"
```

## 发布节奏指南

根据6周的每日发布经验，我推荐以下节奏：

| 你的情况 | 发布频率 | 原因 |
|---|---|---|
| 从零开始建立粉丝群 | 每天3-4次 | 当没人知道你的存在时，数量很重要 |
| 建立了粉丝群（<1000人） | 每天1-2次 | 一致性比数量更重要 |
| 大型粉丝群（>1000人） | 每周4-7次 | 质量比频率更重要 |
| 新闻通讯 | 每周1-2次 | 尊重读者的收件箱体验 |
| 博客 | 每周1-2次 | SEO更看重一致性，而非数量 |
| 业余项目 | 每周1次 | 根据自己的时间安排合理规划 |

**最常见的两个错误：**
1. 从每天发布开始，结果在第二周就精疲力竭。选择一个你能持续3个月的发布节奏。如果是一周两次，也没关系——一致性会带来长期效果。
2. 当用户参与度低时还增加发布频率。如果最近10篇文章都没有引起关注，问题可能出在内容格式、目标受众或传播方式上。重复同样的内容不会解决问题。

## 用户参与度模式（基于实际数据）

以下是我在6天内发布27篇文章时观察到的模式。虽然具体情况可能有所不同，但这些趋势在大多数平台上都是一致的：
1. **提供选择的问题比开放式问题更有效**。“你会选择A还是B？”这样的问题得到的回复是“你对X有什么看法？”的三倍。
2. **第一句话决定了用户的反应**。80%的用户会根据前10个词决定是否参与讨论。
3. **链接会影响文章的传播范围**。所有平台都会优先展示没有链接的文章。如果必须添加链接，确保链接本身有足够的价值。每4篇文章中最多使用1个链接。
4. **重复的格式确实会降低参与度**。我在连续三天只发布推文时，参与度下降了40%。加入一个深度分析的帖子后，参与度恢复了。
5. **早上发布的文章获得浏览量，晚上发布的文章获得互动**。人们早上浏览，晚上回复。因此，将热点观点安排在早上，问题类文章安排在晚上。
6. **三篇文章规则**：如果连续三篇文章的表现不佳，改变其中一个变量（格式、主题、时间或表达方式）。不要同时改变所有因素，否则你无法确定哪个因素起了作用。

## 安全措施
- **每日发布上限**：在跟踪文件中设置`maxPerDay`，并在每次发布前检查。超过平台规定的限制可能会导致平台限制，恢复需要几天时间。
- **避免包含私人信息的草稿**：发布前仔细检查草稿中是否包含姓名、电子邮件、内部项目细节或API密钥。
- **存档而非删除**：删除草稿时，将其标记为“已存档”，而不是直接删除。未来你可能会需要这些内容。
