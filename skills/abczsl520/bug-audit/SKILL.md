# 错误审计 — 深入分析，然后验证

**切勿** 仅仅使用通用的检查清单。相反，你应该阅读代码，提取所有需要审计的元素，然后对每一个元素进行彻底的验证。

## 第一阶段：深入分析（10-15分钟）

阅读所有项目文件，并创建7个表格。这些表格本身就是审计的内容——在第二阶段中，你需要验证这里发现的所有内容。

### 表格1：API端点

对于服务器端代码中的每个路由：

```
| # | Method | Path | Auth? | Params validated? | Precondition | Returns | Attack vector |
```

对于每个端点，需要询问以下问题：
- 我可以在不进行身份验证的情况下调用这个端点吗？
- 我可以传递0、负数、NaN、非常大的数字、数组或对象吗？
- 我可以跳过某个前置API直接调用这个端点吗？
- 如果我每秒调用这个端点100次会怎样？
- 响应是否会泄露敏感数据（如openid、内部ID、完整用户信息）？

### 表格2：状态机

对于每个布尔值/枚举类型的状态变量（如isGameOver、battleState、Game.running、phase、mode等）：

```
| # | Variable | Set by | Read by | Init value | Reset when? | Can it leak across lifecycles? |
```

对于每个变量，需要询问以下问题：
- 如果游戏/会话结束，这个状态变量会被重置吗？
- 如果我立即开始新的一轮，上一轮的旧状态会影响新游戏吗？
- 在设置这些状态变量时是否存在竞态条件？

### 表格3：定时器

对于每个setTimeout/setInterval调用：

```
| # | Type | Delay | Created in | Cleared in | What if lifecycle ends before it fires? |
```

对于每个定时器，需要询问以下问题：
- 是否有用于清理定时器的处理逻辑？
- 如果游戏结束、用户断开连接或页面切换，这个定时器还会继续执行吗？
- 如果定时器在清理之后仍然执行，它是否会引用已经被销毁的对象？

### 表格4：数值类型

对于每个用户可以影响的数值（如cost、score、damage、lootValue、kills等）：

```
| # | Name | Source (client/server/config) | Validated? | Min | Max | What if 0? | What if negative? |
```

对于每个数值，需要询问以下问题：
- 服务器端的限制是否合理？（例如，击杀上限为200，但最大敌人数量只有50个）
- 客户端发送的数值是否可以直接被服务器信任而无需验证？
- 是否存在浮点数精度问题？（例如，累计计算后可能会出现290402.0000000001这样的结果）

### 表格5：数据流（非常关键！）

对于每一对相关的API（如buy→use、start→complete、pay→deliver、login→action）：

```
| # | Step 1 API | Step 2 API | Token/link between them? | Can skip Step 1? | Can replay Step 1? |
```

最大的错误通常隐藏在这里。对于每一组数据流，需要询问以下问题：
- 我可以在不调用步骤1的情况下直接调用步骤2吗？（例如，在没有完成购买的情况下直接请求raid结果）
- 我可以只调用一次步骤1，然后多次调用步骤2吗？（例如，只购买一次，但多次提交结果）
- 是否存在一次性的令牌来链接这些步骤？如果没有，这就是一个严重的漏洞。
- 我可以先调用步骤1并设置cost为0，然后再调用步骤2并获得高奖励吗？

### 表格6：资源管理

对于游戏中的每种资源（如coins、gems、items、XP、energy等）：

```
| # | Resource | All INFLOWS (APIs/events that add) | All OUTFLOWS (APIs/events that subtract) | Daily limits? | Can any inflow be infinite? |
```

对于每种资源，需要询问以下问题：
- 是否存在没有对应成本的资源流入？（例如，任务完成后可以免费获得coins）
- 资源的流出是否可能为负数？（例如，出售物品后coins数量可能会减少，但如果coins数量超过上限会怎样？）
- 保险箱中的物品是否被排除在所有资源流出之外？（例如，在交易、出售、合并或赠送操作中）
- 是否存在循环操作？（例如，购买物品A后以高于成本的价格出售它，从而形成循环）

### 表格7：并发热点（ToCTOU）

对于每一个读取共享状态后写入共享状态的操作（如balance check→deduct、stock check→reserve、coupon check→redeem）：

```
| # | Operation | Read step | Write step | Atomic? | What if 2 requests hit simultaneously? |
```

这可以帮助发现单次请求测试无法发现的竞态条件。对于每一个这样的操作，需要询问以下问题：
- 读取和写入操作是否是原子的？（例如，SQL中的`UPDATE x=x-1 WHERE x>=1`是原子的；但`SELECT`后紧接着`UPDATE`就不是原子的）
- 是否有可能有两个并发请求同时通过检查并执行写入操作？（这可能导致重复支付）
- 是否有互斥锁或事务机制？如果使用SQLite，是否启用了WAL模式来支持并发读取？
- 对于多步骤的操作流程：请求A是否可以在请求B开始相同操作的过程中执行？

## 第二阶段：验证（主要审计）

逐行检查每个表格中的每一条记录。对于每一条记录，需要判断其属于以下哪种情况：
- 🔴 严重问题：存在可利用的安全漏洞、数据丢失或程序崩溃
- 🟡 中等问题：逻辑错误、数据不一致或性能问题
- 🟢 轻微问题：代码质量问题、边缘情况或用户体验问题
- ✅ 通过验证：所有内容都正确

输出格式：
```
Bug N: [🔴/🟡/🟢] Brief description
- Row: Table X, #Y
- Cause: ...
- Fix: ...
- File: ...
```

**即使你觉得“没有新的问题可发现”也不要停止”。只有当所有表格中的每一条记录都被验证为✅或标记为 🔴/🟡/🟢时，你才能停止。这是一个彻底的审计过程，而不是基于启发式的检查。**

## 第三阶段：红队/蓝队

在完成所有表格的验证后，切换到对抗模式。请阅读`references/redblue.md`以获取完整的攻击和防御策略。

### 结构

该策略分为4个部分：
1. **通用攻击链（5个）** — 适用于所有项目：身份验证绕过、注入攻击、滥用频率、数据泄露、并发/竞态条件
2. **特定类型的攻击链** — 根据项目类型选择相应的部分：
   - 🎮 游戏类项目：跳过支付、经济循环、状态篡改、反作弊绕过
   - 📊 数据工具类项目：数据访问控制、数据完整性、定时任务滥用
   - 🔌 API服务类项目：密钥/令牌滥用、上游依赖关系、响应篡改
   - 🤖 机器人程序类项目：消息注入、机器人状态滥用
   - 🔧 微信相关项目：OAuth与身份验证、WebView兼容性、H5混合页面
   - 📈 平台类项目：跨服务信任、多租户隔离
3. **蓝队防御** — 对于每一个发现的问题，需要验证4个层面：预防 → 检测 → 控制 → 恢复
4. **执行指南** — 为审计人员提供的逐步操作指南

### 执行步骤
1. 根据第一阶段的分析结果，确定项目的类型（一个项目可能属于多个类型）
2. 运行所有5个通用攻击链
3. 运行与项目类型相匹配的特定类型攻击链
4. 对于每一个被标记为🔴的问题，需要验证所有4个蓝队防御层面
5. 对于每一个被标记为🟡的问题，至少需要验证第一个防御层面（预防）

## 第四阶段：补充检查

在完成红队和蓝队的审计后，运行一些通用的安全检查作为最后的保障。请阅读`references/modules.md`并选择与项目相关的部分：
- 🔒 安全相关检查（S1-S3）：CORS、XSS、SQL注入、暴力攻击（如果项目有用户）
- 🔐 加密相关检查（C1）：硬编码的密钥、弱哈希算法、明文存储、不安全的随机数生成（所有项目）
- 📊 数据相关检查（D1-D3）：时区设置、原子操作、浮点数精度（如果项目使用数据库）
- ⚡ 性能相关检查（P1-P2）：内存泄漏、性能瓶颈（如果项目规模较大或需要实时处理数据）
- 🎮 游戏类项目（G1-G4）：状态保护机制、渲染效果、配置设置（如果项目是游戏）
- 🔧 微信相关项目（W1-W3）：ES6兼容性、CDN配置、调试功能（如果项目在微信WebView中运行）
- 🔌 API服务类项目（A1-A3）：接口标准、速率限制（如果项目提供API服务）
- 🤖 机器人程序类项目（B1）：超时处理、重复数据检测、敏感信息的处理（如果项目包含机器人程序）
- 🚀 部署相关检查（R1-R2）：PM2服务器配置、nginx配置、SSL加密、SDK覆盖（所有项目）
- 🧪 错误处理相关检查（E1-E2）：网络错误、服务器错误、优雅的错误处理机制（所有项目）
- 📱 用户体验相关检查（U1-U2）：错误状态显示、边缘情况的用户体验（所有有用户界面的项目）
- 📦 供应链相关检查（SC1）：npm依赖管理、依赖项漏洞、锁文件的完整性（所有使用Node.js的项目）
- 📝 日志记录相关检查（L1）：安全事件日志记录、审计痕迹的完整性（所有有用户的项目）

## 第五阶段：回归测试 + 验证

- 确保修复措施没有引入新的错误
- 在项目被拆分为多个模块后，验证不同文件和函数之间的数据传递是否正常
- 进行实时测试：检查首页是否正常显示、关键API是否能返回JSON格式的数据、登录功能是否正常、核心功能是否能够正常运行

## 第六阶段：归档

更新项目文档，记录以下内容：审计日期、创建的表格、发现的错误及修复情况、下次审计需要注意的关键问题。

## 关键原则
1. **先构建表格，再进行检查。** 构建表格是核心工作；一旦表格完成，后续的验证工作就变得相对简单了。
2. **要彻底检查，而不是基于直觉。** 不要因为“感觉完成了”就停止审计。只有当所有记录都被验证完毕时才停止。
3. **从攻击者的角度思考问题。** 对于每个API，要思考“我该如何利用它？”；对于每个数值，要思考“如果我发送错误的数据会怎样？”
4. **数据流中隐藏着最严重的错误**。相关API之间的关联（或关联缺失）是可利用漏洞的主要来源。
5. **通用检查清单只是辅助工具，而非主要手段。** 它们可以发现已知的问题模式；而表格可以帮助发现项目特有的逻辑错误。
