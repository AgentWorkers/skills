# 游戏质量标准

所有游戏项目都必须遵循这些强制性质量标准。这些标准基于70多个实际出现的漏洞以及行业最佳实践制定。

## 核心原则

> 错误通常源于**组件之间的交互**，而非单个功能的故障。
> 每个功能本身可能是正常的，但它们在组合使用时可能会出现问题。

## 12条通用规则（适用于所有游戏）

### 1. 单一清理入口点 🔄
所有退出路径（死亡/关卡完成/退出/暂停/场景切换）都应调用同一个带有选项的清理方法。

```js
cleanupGameState(opts) {
  // Fixed order: sub-objects → buffs+timers → UI → projectiles → (optional) enemies/controls/events
}
// Every exit: resetBall(), levelComplete(), gameOver(), onShutdown() → calls this
```

**新功能添加时，请在此处添加一行代码。切勿将清理逻辑分散到多个退出路径中。**

### 2. 尊重当前生效的增益效果 ⚡
任何修改属性（速度/攻击力/大小/防御力）的代码都必须先检查是否有生效的临时效果。

```js
// ❌ speed = Math.max(speed, BASE_SPEED);  // ignores slow buff
// ✅ speed = Math.max(speed, this._currentBaseSpeed);  // buff-aware baseline
```

### 3. 在销毁前缓存数据 📦
在调用 `destroy()`、`dispose()` 或 `remove()` 之前，先提取所有需要的数据。

```js
const { x, y } = obj; const color = obj.getData('color');
obj.destroy();
spawnParticles(x, y, color);
```

### 4. 定时器需遵循生命周期管理 ⏰
跟踪所有的 `setTimeout`、`setInterval`、`delayedCall` 和 `rAF` 定时器，并在清理时取消它们。

```js
this.activeTimers.push(this.time.delayedCall(10000, cb));
// In cleanup: this.activeTimers.forEach(t => t.remove(false));
```

### 5. 逻辑处理需与帧率无关 🖥️
所有依赖时间的逻辑运算都应考虑帧率的变化。切勿假设游戏始终以60帧每秒的速率运行。

```js
// ✅ player.x += speed * (delta / 1000);
```
- 使用 Phaser 的 `update(time, delta)` 方法时，`delta` 以毫秒为单位，需除以1000。
- Three.js 的 `clock.getDelta()` 方法返回秒数。
- 物理引擎应使用固定的时间步长（例如每16.67毫秒更新一次）。

### 6. 场景切换时需进行全面清理 🚪
在场景或关卡切换时，需清理以下内容：事件监听器、定时器、动画更新、音频节点、对象池、WebGL资源（几何图形/材质/纹理）、全局状态以及未完成的请求（如XHR请求）。
验证方法：使用 Chrome 开发工具 → 内存 → 堆内存快照，在切换前后进行对比。

### 7. 音频资源的生命周期管理 🔊
- 在 iOS 平台上，`AudioContext` 必须在用户交互事件中重新启动。
- 当元素隐藏时，暂停所有音频；元素显示时，恢复音频播放。
- 在使用微信 WebView 时，需在自动播放前调用 `WeixinJSBridge.invoke('getNetworkType')`。
- 对短音效进行缓存管理；背景音乐需单独处理。

### 8. 输入操作的安全性 👆
- 购买/消耗资源时，使用互斥锁并禁用相关元素的可视化显示。
- 攻击/射击操作需设置冷却时间。
- 状态切换（暂停/恢复）需通过状态机进行控制。
- 有关多点触控指针ID的跟踪方法，请参考 Phaser 的官方文档。

### 9. 状态保存与持久化 💾
- 保存的数据中应包含版本号，以便游戏更新时进行迁移。
- 只保存有意义的状态数据（不包括粒子效果或临时动画）。
- 在关卡结束时自动保存数据；用户可手动保存；当元素隐藏时也需保存状态。
- 使用 `localStorage` 的存储空间限制为5MB；对于较大的数据文件，使用 `IndexedDB` 进行存储。
- 在微信平台上，使用 `wx.setStorage` 而不是 `localStorage`。

### 10. 网络故障的容错能力 🌐
所有网络请求（排行榜/分享/广告/同步）都应设置5秒的超时机制；如果请求失败，应使用本地缓存作为备用方案，并且不能阻塞游戏流程。

### 11. 资产加载策略 📦
资产加载分为三个层级：关键资源（启动时加载，耗时<2秒）、关卡资源（加载界面时加载）以及延迟加载的资源（在后台按需加载）。
只有遇到关键错误时才显示错误提示；对于非关键错误，应优雅地降级游戏体验。
压缩格式：使用 GLB 和 Draco 格式压缩图像，MP3 和 OGG 格式压缩音频，以及精灵图集。

### 12. 反作弊机制的基础要求 🛡️
客户端是不可信的。服务器需要验证以下内容：
- 单次使用的特殊令牌（绑定用户ID和时间戳）。
- 游戏时长检查（确保玩家无法在3秒内完成30个关卡）。
- 分数范围的合法性检查。
- 有关反作弊机制的实现细节，请参考 `references/anti-cheat.md`。

## 部署前的检查清单

在每次部署前，请执行以下检查：

### 🔴 通用规则（适用于所有游戏）
- 新创建的对象是否已在 `cleanupGameState()` 方法中被清理？
- 新添加的定时器是否已在清理过程中被取消？
- 属性修改是否考虑了当前生效的增益效果？
- 数据是否在销毁前被正确缓存？
- 移动/动画操作是否使用了基于时间差的计算？
- 场景切换时是否存在内存泄漏？（使用开发工具进行验证）
- 背景音频和锁定状态下的音频是否已正确暂停？
- 购买/消耗资源操作是否防止了重复点击？
- 保存的数据是否包含版本号并支持迁移？
- 网络请求是否设置了超时机制并提供了备用方案？
- 资产加载失败时是否能够优雅地降级游戏体验？
- 关键操作（如消耗资源、结算等）是否经过了服务器的验证？

### 🟡 移动平台额外检查
- 多点触控功能是否正常工作（每个手指的操作是否被独立处理）？
- 在 iOS 平台上，`AudioContext` 是否在用户交互后能够正确恢复？
- 是否兼容微信 WebView（避免使用复杂的 CSS 样式，如背景滤镜）？
- 虚拟摇杆/按钮是否不会覆盖游戏界面？
- 纵向屏幕旋转功能是否能够正常处理？

### 🔵 引擎特定的检查
- 有关具体的引擎检查事项，请参考 `references/phaser.md` 或 `references/threejs.md`。
