# FarmOS 任务管理器

用于管理工作订单和任务，查看任务分配情况、检查任务状态以及管理任务流程。

## 适用场景

**该功能支持的操作：** 任务创建、任务分配、状态更新、工作订单处理、行动项安排、跟进事宜以及物资补充/采购请求。

**常用触发语句：** “提醒我……”、“我们需要……”、“应该由某人……”、“创建一个任务”、“哪些任务分配给了我？”、“有逾期的任务吗？”、“将任务X标记为已完成”、“我们的……快用完了”、“我们需要更多……”

**不支持的功能：** 设备维护跟踪（使用farmos-equipment）、排班/休假/设备可用性管理（使用farmos-workforce）、现场观察和侦察报告（使用farmos-observations）。

**最低输入要求：** 任何需要执行的工作描述即可。例如：“我们需要处理12号田地的相关工作”。

## API 基础信息

http://100.102.77.110:8007

## 集成端点（无需认证）

### 仪表盘概览
GET /api/integration/dashboard

返回：按状态、优先级分类的任务数据及近期活动信息。**仅用于统计汇总，不可用于列出所有任务。**

### 任务概览
GET /api/integration/tasks-summary

返回：任务汇总信息：
```json
{
  "total": 15,
  "pending": 3,
  "assigned": 5,
  "in_progress": 4,
  "completed": 3,
  "critical": 1,
  "high_priority": 2,
  "overdue": 1
}
```

### 过滤任务列表
GET /api/integration/tasks?limit=10&status=in_progress&priority=high

查询参数：limit、status（pending|assigned|in_progress|completed|cancelled）、priority（low|normal|high|critical）

返回：包含任务ID、标题、状态、优先级、截止日期和负责人的简化任务信息。

### 单个任务详情
GET /api/integration/tasks/{id}

返回：任务的完整详细信息。

## 需要认证的端点

这些端点需要使用JWT令牌进行身份验证。请参阅下面的认证说明。

### 认证流程

该功能需要使用JWT令牌访问受保护的FarmOS端点。

**获取令牌的方法：** 使用具有相应权限的角色运行认证辅助函数：
```bash
TOKEN=$(~/clawd/scripts/farmos-auth.sh manager)
```

**使用令牌的方法：** 将令牌作为Bearer参数附加在请求头中：
```bash
curl -H "Authorization: Bearer $TOKEN" http://100.102.77.110:8007/api/endpoint
```

**令牌有效期：** 令牌有效期为15分钟。如果收到401错误响应，请重新请求令牌。

**角色权限对照：** 查看`~/.clawdbot/farmos-users.json`文件中的用户角色，以确定使用何种权限级别。如果用户角色没有访问所需数据的权限，应直接告知用户无法操作，而无需尝试使用更高权限的令牌。

### 我的任务（员工视角）
GET /api/tasks/mine
授权方式：Bearer {token}

返回：分配给当前用户的任务列表。

### 创建任务（管理员权限）
POST /api/tasks
授权方式：Bearer {token}
内容类型：application/json

请求体：
```json
{
  "title": "Spray north fields - Section 12",
  "description": "Apply pre-emerge herbicide per agronomy recommendation",
  "priority": "high",
  "due_date": "2026-02-20",
  "equipment_id": 5,
  "estimated_duration_minutes": 180
}
```

### 更新任务状态
POST /api/tasks/{id}/start  — 将任务状态标记为“进行中”
POST /api/tasks/{id}/complete  — 将任务状态标记为“已完成”
POST /api/tasks/{id}/cancel  — 取消任务
授权方式：Bearer {token}

### 分配任务
POST /api/tasks/{id}/assign
授权方式：Bearer {token}
内容类型：application/json

员工ID来自Workforce模块的集成端点。

## 任务模板

### 任务模板列表
GET /api/templates
授权方式：Bearer {token}

### 根据模板创建任务
POST /api/templates/{id}/create-task
授权方式：Bearer {token}

根据模板自动生成新任务。

## 任务状态流程

- 任务初始状态为“待处理”
- 分配给员工后状态变为“已分配”
- 工人开始执行后状态变为“进行中”
- 工人完成任务后状态变为“已完成”

## 任务创建流程

当用户描述需要完成的工作时，机器人应主动提出创建任务的建议。**切勿自动创建任务，务必先征求用户确认**。

### 从对话中自动提取任务信息

当用户提及可执行的工作内容时，尽可能提取相关信息：
| 信号 | 检测方式 | 例子 |
|--------|--------------|---------|
| **接收者** | “告诉杰克……” → 确定接收者名称 | “我需要……” → 确定任务对象 |
| **优先级** | “尽快” / “在雨前” / “现在” → 高优先级 | “有空时” / “某个时候” → 低优先级 | “我们需要在下雨前喷洒” → 高优先级 |
| **工作地点** | 田地编号或名称 | “14号田地” |
| **所需设备** | 设备名称或编号 | “8370R型喷雾器” |
| **截止日期** | 从描述中提取的日期 | “在周四之前” |

### 自然语言处理示例

| 用户输入 | 自动检测内容 |
|--------------|-------------|
| “我们需要在周四之前喷洒14号田地” | 田地：14号，任务：喷洒，截止日期：周四 |
| “提醒我联系种子供应商” | 接收者：种子供应商，任务：联系供应商 |
| “应该有人去检查8号田地的灌溉装置” | 田地：8号，任务：检查灌溉装置 |
| “在放牛之前需要修理北侧围栏” | 地点：北侧围栏，任务：修理围栏 |
| “杰克需要在开始工作前给喷雾器加润滑油” | 接收者：杰克，设备：喷雾器，任务：加润滑油 |
| “我们可能需要这个月修理联合收割机” | 设备：联合收割机，任务：维修，截止日期：月底 | 

### 创建任务前的确认步骤

**务必确认用户需求**：
在创建任务之前，务必询问用户是否同意。示例确认语句：“正在创建‘喷洒14号田地’的任务——截止日期是周四，目前尚未分配接收者。这样可以吗？”
如果用户同意，请确认详细信息后再执行任务；如果用户修改了优先级，请重新确认。

### 不适合创建任务的场景

- 如果对话内容仅仅是普通讨论（例如：“那块田地看起来情况很糟糕”），则不需要创建任务。
- 如果用户只是在提问（例如：“14号田地上次是什么时候喷洒的？”），则不需要创建任务。
- 如果工作已经在执行中（例如：“我正在喷洒14号田地”），则无需创建新任务。
- 如果信息只是状态更新（例如：“14号田地的喷洒工作已经完成”），则无需创建新任务。

### 使用建议

- 当用户表示有工作需要处理时，可以使用“需要我帮忙创建任务吗？”这样的提示。
- 如果信息不明确，可以根据现有信息创建任务（虽然任务描述可能不详细，但总比不创建任务要好）。
- 对于重复性任务（例如“我们总是需要在Y之前完成X工作”），可以建议用户使用任务模板：“需要我创建一个模板，以便下次自动生成任务吗？”
- 创建任务后，可以询问用户是否需要分配具体人员或设置提醒。

## 物资补充管理

任何关于物资库存的描述都应被视为可执行的任务。机器人会记录这些信息，避免信息丢失。

### 检测规则

| 用户输入 | 应采取的行动 | 优先级 |
|--------------|--------|----------|
| “我们的液压过滤器快用完了” | 提出补充任务（标签：采购） | 低优先级 |
| “最后一盒种子处理剂已经用完了” | 创建补充任务（标签：采购） | 高优先级 |
| “完成工作之前还需要更多绳子” | 提出补充任务（标签：采购） | 低优先级 |
| “喷雾器的液压过滤器只剩下2个了” | 提出补充任务，并提供具体数量 | 低优先级 |
| “我订购了5加仑的Roundup除草剂” | 仅确认订单情况（无需创建任务） |
| “我们把润滑油放在哪里？” | 回答问题（无需创建任务） |
| “我们的DEF（润滑剂）用完了” | 创建补充任务（标签：采购） | 高优先级 |

### 回应提示

- **物资不足时**：“了解到这种情况了，我会创建一个补充任务，或者直接通知布莱恩？”
- **物资已用完/只剩最后一个时**：“已记录，情况紧急，现在就创建补充任务。”（仍需确认）
- **已订购物资时**：“收到消息了，谢谢提醒。”（无需创建任务）
- **关于物资的疑问**：直接回答问题，无需创建任务

### 补充任务的格式

创建采购任务时，请使用以下格式：
```json
{
  "title": "Restock: hydraulic filters (planters)",
  "description": "Crew reports low supply — down to 2 remaining",
  "priority": "normal",
  "tags": ["procurement"]
}
```

**所有物资补充任务都应标记为`procurement`**，以便统一管理和批量订购。

对于“最后一盒”或“完全用完”的情况，优先级应设置为“高”。

## 数据完整性

- 始终报告返回的任务总数。
- 如果查询结果数量异常少，应说明原因：“只返回了X条结果，可能数据不完整。”
- 使用`/api/integration/tasks`端点并设置合适的过滤条件来查看任务列表，而非`/api/integration/dashboard`（该端点可能截断数据）。
- 对于分页显示的结果，需提供总数并提示用户是否需要获取更多数据。

## 模块间的关联

在处理任务时，需与其他模块进行关联：
- **任务与天气**：在列出当天或本周的任务时，查看farmos-weather模块的天气信息。例如：“周四有喷洒任务，但周四下午有雨，可能需要调整任务安排。”
- **任务与设备**：在处理田间作业任务时（如喷洒、种植、收割），需考虑天气情况。
- **任务与设备**：在创建任务时，需检查farmos-equipment模块的设备可用性。如果任务需要特定设备，需确认设备是否可用或是否需要维护。
- **任务与观察记录**：当任务与田间观察结果相关时，可查看farmos-observations模块的记录。

### 使用注意事项

- 读取任务信息时使用无需认证的集成端点。
- 创建或更新任务时需使用管理员权限（授权方式：Bearer {token}。
- 优先级分为：低、正常、高、紧急。
- 始终检查是否有逾期的任务。
- 设备ID可链接到farmos-equipment模块查询设备信息。
- 员工ID来自farmos-workforce模块。
- 创建任务时，根据实际情况建议合适的优先级。
