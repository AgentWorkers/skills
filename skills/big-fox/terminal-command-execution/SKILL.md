# 终端命令执行

## 概述
使用此技能可以以最小的风险和可预测的结果来运行终端命令。建议采用快速检查、明确的意图验证以及在每个状态变更步骤后进行确认的方法。

## 工作流程
1. 明确目标和范围。
- 从上下文中推断出命令的具体目标（路径、服务、工具、环境）。
- 如果请求不明确或存在风险，请提出一个简洁的澄清问题。

2. 飞行前检查。
- 确认工作目录和所需的二进制文件。
- 在更改当前状态之前先检查当前状态（例如使用 `ls`、`git status` 等命令）。
- 首先使用非破坏性的检查方法。

3. 逐步执行命令。
- 运行能够推进任务的最小命令。
- 对于多步骤任务，在继续之前验证每个步骤。
- 使用可复制的命令；如果存在非交互式选项，请避免使用交互式操作。

4. 系统地处理错误。
- 首先查看标准错误输出（stderr），并确定错误的原因：权限问题、路径问题、依赖项缺失、语法错误、网络问题或运行时状态问题。
- 一次只修复一个问题，然后重新运行受影响的命令。
- 如果需要执行具有破坏性的操作，请在继续之前获得用户批准。

5. 验证结果。
- 检查命令的退出状态和可观察到的状态变化。
- 对于安装操作，使用版本/健康检查命令进行验证。
- 对于编辑操作，验证修改后的文件和行为是否正确。

6. 清晰地报告结果。
- 总结执行了哪些命令、哪些内容发生了变化以及当前的状态。
- 仅在需要用户进一步操作时才提供下一个具体的命令。

## 安全规则
- 默认情况下避免使用具有破坏性的命令（如 `rm -rf`、强制重置、全局修改权限/所有者等），除非明确得到请求。
- 切勿假设网络连接、权限或包管理器可用；请先进行测试。
- 尽量使用局部化的操作（针对特定的文件/路径/服务），而不是进行全局性更改。
- 不要将敏感信息包含在命令输出或日志中。

## 命令模式
- 信息查询：`pwd`、`ls -la`、`rg --files`、`which <tool>`
- 验证：`--version`、健康检查/状态检查命令、针对性的测试
- 诊断：首先检查日志/错误信息，然后逐步调整相关设置
