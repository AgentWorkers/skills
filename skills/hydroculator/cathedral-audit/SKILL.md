# 大教堂代码库审计（Cathedral Codebase Audit）

这是一个用于衡量和解决 Cathedral C# 代码库中规范（spec）与代码（code）不一致问题的结构化流程。

## 审计时机

- 在多次代码构建（包含多个功能或重构）之后
- 在开始新的大型功能之前（建立基准）
- 当怀疑存在规范与代码不一致的情况时
- 定期进行健康检查（不按固定时间表，而是根据事件触发）

## 流程概述

```
1. Forward Audit (specs → code)
2. Reverse Audit (code → specs)
3. Consolidation & Prioritization
4. Execution
5. Verification
```

## 第一步：正向审计（从规范到代码）

对于 `kitty-specs/` 目录中的每个规范，将规范中的描述与实际代码进行对比。

**每个规范的审计结果**：
- ✅ **符合规范** — 代码与规范一致
- ⚠️ **不一致** — 代码存在但与规范不同
- ❌ **缺失** — 规范中描述的功能未被实现

**交付物：`kitty-specs/000-project-context/spec-audit-YYYY-MM-DD.md`

如果内存允许，可以使用并行任务代理（每个规范批次一个代理）来提高效率。**在内存受限的环境中（例如 WSL2），请顺序执行任务，因为同时运行多个代理可能会导致系统内存不足（OOM）。**确保每个代理仅具有读取权限。

## 第二步：反向审计（从代码到规范）

扫描 `src/Cathedral.Core/` 和 `src/Cathedral.TestHarness/` 目录下的所有 `.cs` 文件。对于每个文件或子系统，需要判断：
- 该文件或子系统是否被规范覆盖？
- 代码是否与规范中的描述一致？
- 该代码是否为“废弃代码”或“孤立代码”？

**审计结果包括以下部分：**
1. **执行摘要** — 显示与上次审计相比的变化情况
2. **未受规范覆盖的代码** — 未被规范覆盖的文件或子系统
3. **架构差异** — 代码实现与规范存在根本性差异
4. **超出规范的代码** — 代码包含规范未记录的功能
5. **废弃/孤立代码** — 没有调用者或引用的代码
6. **发现的错误** — 审查过程中发现的运行时错误、数据错误或逻辑错误
7. **与上次审计的对比** — 有哪些改进，哪些问题仍然存在

**交付物：`kitty-specs/000-project-context/reverse-audit-YYYY-MM-DD.md`

如果内存允许，可以使用并行任务代理（按目录或子系统分组文件）来提高效率。**在内存受限的环境中请顺序执行任务。**

## 第三步：整合与优先级排序

将两次审计的结果整合成一个优先级排序的行动计划：

| 优先级 | 类别 | 判断标准 |
|----------|----------|----------|
| **P0** | 错误修复** | 对运行时有影响的错误（例如：功能失效、数据损坏、程序崩溃） |
| **P1** | 删除废弃代码** | 安全地删除这些代码，以减少混淆和代码量（LOC） |
| **P2** | 数据质量修复** | 修复错误的数据、错误的默认值、重复记录等问题 |
| **P3** **规范覆盖** | 为未受规范覆盖的代码编写新的规范（无需修改代码） |
| **P4** **规范准确性** | 更新现有规范，以记录超出规范的功能 |
| **P5** **代码重构** | 重命名变量、调整代码结构、完善实体范围等 |
| **P6** **架构差异** | 由于设计原因导致代码与规范不一致的功能（推迟处理） |

**规则：**
- 错误总是单独记录，并标注严重程度
- “超出规范的代码”需要更新规范，而不是修改代码本身
- 有意推迟处理的架构差异（属于第二版开发的内容）归类为 P6，只需记录无需立即处理
- 每个优先级级别都应在一次任务会话中完成

**交付物：** 反向审计报告中的建议部分。

## 第四步：执行修复工作

按照优先级顺序执行修复（先处理 P0 级别的问题，最后处理 P6 级别的问题或推迟处理的问题）。

**针对每个优先级级别：**
1. 将修复意图记录到 `memory/YYYY-MM-DD.md` 文件中，包括优先级级别、任务会话名称以及要执行的操作
2. 编写任务简报供团队协作使用（参考 `references/cc-task-template.md`）
3. 启动任务会话：`cat /tmp/task.md | claude -p --allowedTools 'Edit,Write,Read,Bash'`
4. 设置定时监控任务（每 5 分钟执行一次）
5. 任务完成后：将结果记录到日志文件 `memory` 中，包括更改的文件、执行的内容以及遇到的问题
6. 在提交代码之前进行构建测试：确保 `dotnet build` 通过
7. 如果任务会话因内存不足而终止（OOM），请检查 `git diff --stat`，手动验证构建结果并修复问题，记录事件
8. 提交代码时附上描述性信息，注明优先级级别
9. 将提交哈希值记录到日志文件 `memory` 中

**严格规则：**
- **务必在提交代码之前验证 `dotnet build` 是否通过。** 任何情况下都不得例外。任务会话可能在构建过程中因内存不足而终止。
- **务必在每个步骤都记录日志。** 在开始任务前记录意图，完成任务后记录结果，提交代码后记录提交哈希值。即使任务会话终止，日志也应保留以便后续恢复。
- 每个优先级级别对应一次提交。

## 第五步：验证

所有步骤完成后，可以选择再次进行快速审计，以评估改进情况：
- 比较未受规范覆盖的代码、存在差异的代码、废弃代码的数量
- 验证各项指标的变化是否符合预期
- 记录仍然存在的差距以及这些差距是属于 P6 级别的问题还是新发现的问题

**交付物：** 更新后的审计文件，其中包含对比结果。

## 日志记录

每次审计都会在 `memory/YYYY-MM-DD.md` 文件中生成完整的审计记录：
- **审计启动信息** — 当前正在执行的审计任务及基准信息
- **审计结果** — 统计结果、主要发现
- **每个优先级级别的详细信息** — 修复意图、任务会话详情、结果及遇到的问题
- **最终总结** — 总共的提交次数、修改的代码行数、已解决的问题与推迟处理的问题

这是不可协商的规则。2 月 17-18 日的事件表明，未记录的工作将会丢失。必须先记录再行动：在开始执行任何操作之前先写下计划，完成后更新日志。

## 基线跟踪

始终将当前的审计结果与上一次审计结果进行对比。审计记录的存储方式如下：
```
kitty-specs/000-project-context/
  spec-audit-YYYY-MM-DD.md      (forward)
  reverse-audit-YYYY-MM-DD.md   (reverse)
```

包含变化情况的执行摘要表格是关键的数据指标：
```markdown
| Category | Previous | Current | Delta |
|----------|----------|---------|-------|
| Unspecced Code | 38 | 24 | -14 |
| Divergences | 12 | 7 | -5 |
| Dead Code | 14 | 8 | -6 |
| Bugs | 0 | 8 | +8 |
| Conformant | ~60 | ~120 | +60 |
```
