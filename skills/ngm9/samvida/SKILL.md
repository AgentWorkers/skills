# Samvida — 为您的业务生成代理合同文档（Agentic Contracts for Your Business）

## 概述

该技能会爬取企业网站，提取结构化信息，并生成格式规范的 `llms.txt` 文件。该文件是AI代理能够理解和使用的标准格式。

该工具遵循 [llmstxt.org](https://llmstxt.org) 的规范，并根据企业具体情况进行了扩展：
- `## 团队**（## Team）：展示企业背后的团队信息，增强代理对企业的信任感。
- `## 客户与评价**（## Clients & Testimonials）：提供客户评价和社会证明，帮助代理做出决策。
- `## 供代理使用**（## For Agents）：说明代理如何与企业互动，或显示“即将推出”的提示。

在生成任何输出之前，请先阅读 `references/llms_txt_spec.md` 文件。

---

## 工作流程

### 第1步 — 获取网站URL

如果用户未提供URL，请询问：
> “网站的URL是什么？”

如果URL缺失，请添加 `https://` 前缀以使其格式正确。

---

### 第2步 — 爬取数据

运行爬虫：
```bash
~/.virtualenvs/samvida/bin/python3 \
  ~/.openclaw/workspace/samvida/scripts/crawl.py \
  {url} > /tmp/llms_business_info.json
```

查看 `/tmp/llms_business_info.json` 文件，记录以下内容：
- 爬取了哪些页面
- 找到了哪些信息（如团队信息、价格、客户评价、API接口等）
- 是否已经存在 `llms.txt` 文件

简要告知用户：
> “已爬取 {domain} 网站的 {N} 个页面。找到了以下信息：{找到的内容}。还有一些信息需要进一步确认。”

如果发现已有的 `llms.txt` 文件，请告知用户：
> “我发现您已经有了一个名为 {domain}/llms.txt 的文件。我会生成一个新的版本，您可以对比后决定保留哪个。”

---

### 第3步 — 询问其他信息来源（务必先询问）

> “还有其他需要查看的页面吗？（如文档、API接口、现有的 `llms.txt` 文件、新闻页面等）”

如果用户提供了URL，请使用这些信息重新运行爬虫：
```bash
~/.virtualenvs/samvida/bin/python3 \
  ~/.openclaw/workspace/samvida/scripts/crawl.py \
  {url} {extra_url1} {extra_url2} > /tmp/llms_business_info.json
```

如果用户表示没有或不需要，继续下一步。

---

### 第4步 — 生成初步版本及缺失信息报告

根据爬取到的数据生成 `llms.txt` 的初稿。使用各种提示信息（如 `team_found`、`testimonials_found`、`pricing_found` 等）以及 `raw_text_summary` 来填充文件。对于无法确定的信息，使用 `[NOT FOUND]` 作为占位符。

然后向用户展示初稿及缺失信息报告：
> “这是您的 `llms.txt` 文件的初稿：
> ```
> {draft}
> ```
>
> **自动找到的内容：** {例如：电子邮件地址、价格页面、来自 Wybrid 和 Cital 的评价}
> **无法确定的信息：** {例如：团队信息、价格详情、API 接口}
>
> 以下是两个需要进一步确认的问题：
>
> 1. **最重要的信息**：**“创始团队成员是谁？请提供姓名、职位及电子邮件地址。”**
> 2. **次重要的信息**：**“您的定价模式是什么？哪怕只是大致描述也可以。”**
>
> **（如果用户希望我进一步查找，可以告诉我‘深入挖掘’。”**

---

### 第4b步 — 进行进一步的数据收集（第二轮）

如果用户表示需要进一步收集信息（例如“再试一次”或“更详细地查找”），请以深度爬取模式重新运行爬虫：
```bash
~/.virtualenvs/samvida/bin/python3 \
  ~/.openclaw/workspace/samvida/scripts/crawl.py \
  {url} {extra_urls} --deep > /tmp/llms_business_info.json
```

此步骤会返回 `pages_raw`，其中包含所有爬取页面的原始文本。使用这些文本提取结构化信息，并在生成 `llms.txt` 时使用这些数据：
```
In addition to the heuristic signals, here is the full raw text from each crawled page.
Extract team members, testimonials, pricing details, and any API information directly from this text.

Homepage raw text:
{pages_raw[homepage_url]}

Team page raw text (if available):
{pages_raw[team_url]}

Pricing page raw text (if available):
{pages_raw[pricing_url]}
```

告知用户：
> “正在进行更详细的爬取，这需要一些时间，但我会尽可能从原始文本中提取所有有用信息。”

第二轮爬取完成后，再次向用户展示更新后的初稿及缺失信息报告。对于仍未找到的信息，请直接询问用户。

---

### 第5步 — 通过对话填补缺失信息

对于在第一轮或第二轮爬取中仍为 `[NOT FOUND` 的信息，请逐一提问。等待用户回答后再继续提问。一旦收集到足够的信息，即可完成文件生成。

**Q1 — 代理应了解的核心价值（必问）：**
> “用一句话概括：AI代理在与 {domain} 合作时应该了解该企业的哪些核心价值或能获得什么？”

**Q2 — 团队信息（如果第一轮未找到团队信息时询问）：**
> “我没有找到公开的团队信息。是否需要添加‘团队’部分？这有助于代理了解企业的背景。可以提供姓名、职位和电子邮件地址。”

**Q3 — 客户与评价（如果未找到相关内容时询问）：**
> “是否有现有的客户或评价信息？哪怕只是几个公司名称或简短的评价也能增强代理的信任感。这完全是可选的。”

**Q4 — API接口/集成（如果 `api_found` 为 `false` 时询问）：**
> “企业是否有公开的API接口或文档？（如不适用可跳过）”

**Q5 — 定价信息（如果 `pricing_found` 为 `false` 时询问）：**
> “企业的定价模式是什么？哪怕只是大致描述也可以，例如‘按评估次数收费’或‘按月订阅’。”

**Q6 — 目标客户群体（ICP）/代理类型（如果从上下文中无法推断时询问）：**
> “哪些类型的代理或自动化系统最有可能与您合作？（例如：人力资源机器人、招聘系统）”

**Q7 — 其他信息（可选，最后询问）：**
> “代理在与您合作前还需要了解哪些其他信息？（如地域限制、入职流程等）”

---

### 第6步 — 生成最终的 `llms.txt` 文件

如果之前没有阅读过 `references/llms_txt_spec.md`，请现在阅读该文件。

使用所有收集到的信息生成完整的 `llms.txt` 文件：
- 爬取到的 `business_info` JSON 数据（如果使用了深度爬取模式，则还包括 `pages_raw`）
- 用户的回答
- `references/llms_txt_spec.md` 中规定的格式要求

**生成规则：**
1. 严格遵循规范格式：使用 H1 标题、块引用进行总结、H2 标题的分节结构、带链接的条目。
2. 每个条目格式为 `- [标题](链接)：描述`，避免使用纯文本格式的列表项。
3. 分节顺序：服务 → 团队 → 客户与评价 → 供代理使用 → 定价 → API接口 → 链接 → 可选内容。
4. **`## 团队**：必须包含。使用爬取到的数据或用户提供的信息；如果没有相关信息，则省略。
5. **`## 客户与评价**：尽量包含。结构如下：
   - 首先列出企业服务的目标客户群体（ICP）。
   - 然后为每个有具体评价或案例研究的客户添加一个子节。
   - 每个子节包含带原文引用的内容，以及问题（Problem）和结果（Outcome）。
   - 如果只有名称和简短评价，也可以使用单个条目。
   - 绝不要虚构评价或结果。
6. **`## 供代理使用**：必须包含。如果没有API接口信息，应添加“即将推出”的提示和联系邮箱。
7. **`## 定价**：如果未知，只需提供API接口的链接；不要包含详细授权信息。
8. **`## API接口`：仅记录API的URL，不要透露任何授权细节。
9. **`## 可选内容**：包括常见问题解答、博客文章、案例研究等补充信息。
10. 不要虚构事实。如果某些信息未知且用户未提供，要么省略，要么明确标注。
11. 保持文件简洁——这是为代理准备的文件，不是营销材料。

将最终的 `llms.txt` 文件保存到 `/tmp/llms_final.txt`。

---

### 第7步 — 展示并确认

将完整的 `llms.txt` 文件以代码块的形式展示给用户，然后询问：
> “这是您的 `llms.txt` 文件 👆
>
> 有什么需要修改的地方吗？
> - 可以告诉我您希望修改的内容。
> - 如果准备发布文件，可以点击 **‘保存’**。
> - 如果准备上线（第二阶段），可以点击 **‘部署’**。

---

### 第8步 — 处理修改请求

如果用户要求修改，请进行修改并展示更新后的版本。重复此过程，直到用户满意为止。

如果用户选择 **“保存”**，请告知文件保存在 `/tmp/llms_final.txt`，用户可以将其复制到自己的项目中。

如果用户选择 **“部署”**，请告知他们文件将通过 Cloudflare Workers 进行部署（这将在第二阶段完成）。

---

## 注意事项

- **已有的 `llms.txt` 文件**：如果爬取过程中发现了该文件，请提前告知用户：“我发现您已经有了一个 `llms.txt` 文件。我会生成一个新的版本，您可以对比后决定保留哪个。”
- **仅链接的页面**（例如 `/#section`）：在第二轮爬取时可以跳过，因为这些页面不会加载新内容。
- **“供代理使用”部分是必填的**——即使内容为空，也应包含，以表明企业支持代理的意愿并提供联系方式。
- **不要一次性询问所有问题**——这是一个交流过程，而不是填写表格。
