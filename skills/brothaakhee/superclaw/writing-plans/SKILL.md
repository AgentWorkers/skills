# 编写计划

**核心原则：**设计描述的是要构建什么，而计划描述的是如何构建它。

拥有一个经过批准的设计并不意味着可以跳过规划阶段——这仅仅意味着规划会变得更加容易，因为你知道自己要构建什么。

---

## 何时使用此技能

✅ **适用情况：**
- 你已经有了一个经过批准的设计或规格说明
- 用户说“开始构建”或“可以开始了”
- 你正在从头脑风暴阶段过渡到实施阶段
- 有人向你提供了需求，并要求你“将其变为现实”

❌ **不适用情况：**
- 尚未设计出任何方案（请先使用“头脑风暴”技能）
- 你已经有了一份详细的计划（请使用“执行计划”技能）
- 这是一个微不足道的变更（总工作量小于2分钟）

---

## 流程

### 第一步：询问相关方法论

在编写计划之前，与用户讨论具体的方法论：

**需要询问的问题：**
- “我们应该使用TDD（测试驱动开发）吗？先编写测试，然后再让它们通过吗？”
- “我应该多久提交一次代码？每次完成任务后提交，还是批量提交？”
- “你有特定的工具、库或模式推荐我使用吗？”
- “我应该创建功能分支，还是直接在主代码树上工作？”

**为什么这很重要：**
- TDD会改变任务的执行顺序（先测试再编写代码）
- 提交频率会影响任务的粒度
- 工具的选择会影响依赖关系和设置工作

**不要想当然。**即使你有自己的偏好，用户可能有不同的背景（例如持续集成/持续部署的设置、团队惯例、时间限制等）。

---

### 第二步：将工作分解为可管理的任务

每个任务应该能够在**2-5分钟内完成。

**任务结构：**
```markdown
### Task N: [Clear, actionable title]

**Files:**
- Create: path/to/new/file.ext
- Edit: path/to/existing/file.ext
- Delete: path/to/old/file.ext

**Steps:**
1. [Specific action with exact commands/code]
2. [Next action]
3. [etc.]

**Verification:**
- [ ] [How to verify task succeeded]
- [ ] [Tests pass / output matches expected]

**Estimated time:** 2-5 minutes
```

**如果同意使用TDD方法：**
```markdown
### Task 1: Write failing test for add_todo

**Files:**
- Create: tests/test_todo.py

**Steps:**
1. Create test file
2. Import todo module (doesn't exist yet)
3. Write test_add_todo that calls add_todo("Buy milk")
4. Assert todo appears in list

**Verification:**
- [ ] Test fails (module doesn't exist yet - expected!)
- [ ] pytest runs without errors

**Estimated time:** 3 minutes
```

**如果同意使用先编写代码的方法：**
```markdown
### Task 1: Implement add_todo function

**Files:**
- Create: src/todo.py

**Steps:**
1. Create src/todo.py
2. Define add_todo(text) function
3. Load/save JSON from ~/.todos.json
4. Append new todo with id, text, done=false

**Verification:**
- [ ] Function defined
- [ ] Manual test: add_todo("test") creates file
- [ ] JSON structure is valid

**Estimated time:** 4 minutes
```

### 第三步：编写计划文档

**保存路径：** `workspace/docs/plans/YYYY-MM-DD-<主题>-plan.md`

**计划模板：**
```markdown
# [Project Name] - Implementation Plan

**Date:** YYYY-MM-DD
**Estimated total time:** [Sum of task times]
**Approach:** [TDD / Code-first / Hybrid]
**Commit strategy:** [Per task / Per feature / At end]

---

## Task Breakdown

[All tasks from Step 2]

---

## Dependencies

- [Required libraries/tools]
- [Environment setup needed]

---

## Success Criteria

- [ ] [All tests pass]
- [ ] [Feature works as designed]
- [ ] [Documentation updated]
- [ ] [Code committed]
```

**示例文件名：** `workspace/docs/plans/2026-02-25-todo-cli-plan.md`

---

### 第四步：调用“执行计划”技能

计划编写完成后，使用“执行计划”技能来分批执行这些任务。

## 常见的错误观念（以及为什么它们是错误的）

你的大脑可能会试图跳过规划阶段。以下是一些常见的错误观念及其原因：

| 错误观念 | 原因 | 现实情况 | 对策 |
|--------|-------------------|---------|---------|
| “设计已经足够详细了” | 设计只描述了要构建的内容，并没有详细说明步骤 | 计划会将工作分解为可执行的任务，并提供验证步骤 |
| “用户信任我，让我自己去解决” | 有自主权的感觉很好 | 信任并不意味着可以忽略最佳实践 | 信任意味着遵循能够防止错误的流程 |
| “规划只是浪费时间” | 觉得规划会耽误编码时间 | 5分钟的规划可以避免后续数小时的调试和返工 | 书面计划可以在编码前发现依赖问题 |
| “我可以边做边规划” | 这样看起来更灵活 | 但如果被打断或出错，头脑中的计划可能会消失 | 书面计划可以保留下来，可以随时回顾 |
| “设计已经得到了批准” | 架构决策已经确定 | 批准的是设计内容，而不是实施步骤 | 计划会将设计转化为可操作的任务 |
| “这是一个简单的变更” | 看起来很简单 | “简单”的变更往往隐藏着复杂性 | 编写任务可能需要3分钟，而调试问题可能需要30分钟 |
| “我很着急” | 时间压力很大 | 匆忙会导致错误，而错误需要更长的时间来修复 | 放慢速度反而能提高效率 |
| “用户说‘直接开始吧’” | 听起来像是得到了许可 | 用户希望看到结果，而不是混乱的局面 | 有计划地开始才是更明智的做法 |

**记住：** 如果整个项目的总工作量小于5分钟，可以跳过规划阶段。但如果工作量超过5分钟，计划会为你节省时间。

## 注意事项 🚩

**如果你有以下想法：**
- “我先写这个文件，然后再规划剩下的部分” → 不行，应该先规划
- “设计就是我的计划” → 不对，设计只是构建的蓝图，计划才是具体的执行方案
- “规划会花费比直接做事更长的时间” → 不对，规划本身就是准备工作的一部分
- “我可以把计划记在脑子里” → 不行，这样可能会导致信息丢失或上下文混乱

**如果你有以下行为：**
- 在编写计划之前就打开了代码文件 → 停下，先编写计划
- 跳过了方法论相关的提问 → 停下，先提出这些问题
- 每个任务的编写时间超过5分钟 → 停下，将任务分解得更细
- 将计划保存在非标准的位置（而不是 `workspace/docs/plans/`） → 停下，使用规定的路径

**如果用户说：**
- “直接开始编码吧” → “我先写计划（2-3分钟），然后再编码，这样可以吗？”
- “你不需要计划” → “计划可以帮助我避免错误并跟踪进度。我可以花3分钟来做这个吗？”
- “我很着急” → “5分钟的规划会让接下来的工作更高效。相信这个过程吧！”

## 与OpenClaw的集成

### 需要注意的信息
在规划之前，请查看以下文件：
- `MEMORY.md` – 用户的长期开发偏好（是否喜欢使用TDD？需要特定的工具？）
- `USER.md` – 用户的开发风格
- `memory/YYYY-MM-DD.md` – 关于这个项目的最近讨论记录

这些信息有助于你提出更合适的方法论问题。

### 文件路径规范
- 计划文件保存在：`workspace/docs/plans/YYYY-MM-DD-<主题>-plan.md`
- 在任务中引用文件路径时，请使用 `workspace/src/`，而不是 `/data/workspace/src/`
- 保持路径的相对性，以便在不同环境中也能使用这些计划文件

### 作为子代理的工作流程
如果你是子代理：
1. 编写计划
2. 将计划保存到 `workspace/docs/plans/`
3. 报告：“计划已完成，保存在 `workspace/docs/plans/YYYY-MM-DD-<主题>-plan.md`。可以开始执行了吗？”

作为子代理，不要自动调用“执行计划”技能——让主代理来决定何时开始执行。

## 示例

### 示例1：使用TDD方法

**用户：** “这是Markdown解析器的批准设计。开始构建吧。”

**你：** “很好！在编写计划之前，我有几个问题：
1. 我们应该使用TDD吗？（先编写测试，然后再让它们通过？）

3. 你对解析器使用的库有什么偏好（正则表达式、AST还是自定义实现？）”

**用户：** “使用TDD，每次完成任务后提交代码。解析器使用自定义实现。”

**你：** （编写计划，包括以下任务：）
- 任务1：编写用于解析标题的测试用例
- 任务2：实现标题解析功能，使其通过测试
- 任务3：编写用于解析粗体文本的测试用例
- 任务4：实现粗体文本解析功能，使其通过测试
- （依此类推）

### 示例2：使用先编写代码的方法

**用户：** “这是CLI待办事项应用的规格说明。不需要TDD，直接开始构建吧。”

**你：** “明白了。我应该多久提交一次代码？每次实现一个功能后提交，还是最后一次性提交？”
**用户：** “最后一次性提交就可以了。”

**你：** （编写计划，包括以下任务：）
- 任务1：设置项目结构和JSON存储
- 任务2：实现`add_todo`函数
- 任务3：实现`list_todos`函数
- 任务4：实现`mark_done`函数
- 任务5：编写手动测试用例
- 任务6：添加错误处理逻辑
- 任务7：提交所有代码

## 常见问题解答

**Q：如果在我规划的过程中设计发生了变化怎么办？**
**A：** 更新计划。计划不是固定不变的，它们是动态更新的文档。更新计划总比基于过时的信息来工作要好。

**Q：如果在规划过程中发现设计有缺陷怎么办？**
**A：** 停止规划，提出问题。如果需要的话，重新进行头脑风暴。计划应该基于良好的设计来制定，而不是掩盖问题。

**Q：我可以先制定一个粗略的计划，然后在执行过程中再完善它吗？**
**A：** 不可以。应该先制定完整的计划。如果在执行过程中发现了新信息，可以再更新计划，但一开始就要制定完整的计划。

**Q：如果用户拒绝让我制定计划怎么办？**
**A：** 向他们解释规划的价值（5分钟的规划可以节省调试时间）。如果他们仍然拒绝，记录下这个决定并谨慎地继续工作。但请温和地表达你的立场——这是专业的最佳实践。

**Q：每次都需要询问关于使用TDD的方法吗？**
**A：** 是的。不要想当然。每个项目都有不同的需求和约束。不过，你可以先查看 `MEMORY.md` 文件——如果有既定的偏好记录在文件中，就按照那个来执行。

## 成功检查清单

在调用“执行计划”技能之前，请确认以下内容：
- [ ] 我已经询问了方法论相关的问题（是否使用TDD？提交频率是多少？）
- [ ] 所有任务的时间都在2-5分钟范围内
- [ ] 每个任务都有明确的验证步骤
- [ ] 计划已经保存在 `workspace/docs/plans/YYYY-MM-DD-<主题>-plan.md` 文件中
- [ ] 任务中引用的文件路径都是正确的
- [ ] 我抵制了“直接开始编码”的冲动
- [ ] 依赖关系和设置相关的任务都考虑到了

**如果所有条件都满足：** 可以继续执行“执行计划”技能。
**如果有任何条件未满足：** 在继续之前先解决这些问题。

**记住：** 计划不是额外的负担。计划是基础，它们会让后续的工作更加高效、安全和可靠。

编写计划吧。未来的你会感谢自己的这个决定的。
