# 执行计划技能

## 核心原则

**分批执行并设置检查点可以防止故障的连锁反应。**

计划以小批量（3-5个任务）的形式执行，并在每个批次之间设置强制性的检查点。这样可以提前发现错误，允许用户及时纠正错误，并使进度可以恢复。

---

## ⛔ 重要规则

1. **在没有设置检查点的情况下，不要执行超过5个任务。**
2. **不要跳过向内存日志更新进度的步骤。**
3. **即使用户表示“不需要中断”，也不要一次性执行所有任务。**
4. **不要为了“效率”而放弃分批执行的策略。**

检查点是一种安全机制，而不是为了麻烦用户。它们可以在错误扩大之前及时发现它们。

---

## 执行流程

### 第一步：加载计划

**查找计划：**
- 在 `workspace/docs/plans/` 目录中查找相关的计划文件。
- 计划文件的命名格式为：`YYYY-MM-DD-<主题>-plan.md`
- 阅读整个计划文件，了解其范围和任务数量。

**验证计划结构：**
- 每个任务都应包含清晰、可操作的步骤。
- 任务应引用确切的文件路径。
- 任务应按照逻辑顺序排列（先执行依赖项）。

### 第二步：将任务分批

**将任务分成3-5个一组：**
- 第一批：任务1-3（通常包括设置和基础工作）
- 第二批：任务4-6（核心功能）
- 第三批：任务7-9（集成和优化）
- 第四批：任务10及以后（测试和最终步骤）

**遵守逻辑顺序：**
- 不要将相互依赖的任务分散到不同的批次中。
- 在将任务分批之前，先完成某个功能或模块。
- 如果某个任务规模较大，它可以单独作为一个批次。

**宣布分批执行计划：**
```
I'll execute this plan in 4 batches:
- Batch 1 (tasks 1-3): File structure and basic loading
- Batch 2 (tasks 4-6): Core CRUD operations
- Batch 3 (tasks 7-9): CLI interface and routing
- Batch 4 (task 10): Testing and validation

Starting with batch 1...
```

### 第三步：执行批次（使用 sessions_spawn）

**对于批次中的每个任务：**

1. **为每个任务创建一个新的子代理来执行：**
   ```
   sessions_spawn --label "task-N-<short-desc>" --prompt "Execute task N from plan..."
   ```

2. **提供清晰的上下文信息：**
   - 任务编号和描述
   - 需要创建或修改的文件路径
   - 预期的结果
   - 需要运行的命令

3. **隔离任务执行过程：**
   - 新创建的子代理不会受到之前任务的影响
   - 每个任务都从干净的状态开始执行
   - 错误仅限于当前任务范围内。

4. **捕获执行结果：**
   - 任务是否成功完成？
   - 是否出现错误或警告？
   - 创建或修改了哪些文件？

### 第四步：审查批次结果

**批次执行完成后，进行审查：**

- ✅ **成功检查：**
  - 所有文件是否按预期创建？
  - 命令是否无误执行？
  - 测试是否通过（如果适用）？

- ⚠️ **警告信号：**
  - 有任何错误信息吗？
  - 文件是否有意外更改？
  - 是否有依赖项缺失？

- 🛑 **停止条件：**
  - 发生严重错误
  - 下一批的执行依赖于当前问题的解决
  - 需要用户做出决定

**记录审查结果：**
```
Batch 1 complete (tasks 1-3):
✅ Created /workspace/todo/lib/storage.js
✅ Created /workspace/todo/lib/todo.js
⚠️  Warning: Missing dependency 'fs-extra' - need to install
```

### 第五步：设置检查点

**在批次之间必须暂停：**

1. **报告批次执行结果**（如上所述）。
2. **询问用户是否继续执行：**
   ```
   Continue to batch 2 (tasks 4-6)?
   ```

3. **等待用户回复：**
   - ✅ “继续” → 继续执行
   - 🛑 “暂停” → 停止并等待用户指示
   - 🔧 用户要求修改 → 根据用户的修改建议进行调整，然后再询问。

**如果用户表示“直接全部执行，不需要中断”：**
- **不要跳过检查点**。
- 回复用户：“检查点是为了防止错误，而不是为了麻烦你。每次执行完一个批次后我会通知你，以便你可以查看进度或让我继续执行。”

**自动继续选项：**
如果用户表示“如果没有错误就自动继续执行”：
- 仍然需要显示批次执行结果。
- 仅在检测到错误或警告时才暂停。
- 用户可以在任何检查点中断执行过程。

### 第六步：更新内存记录

**每次执行完一个批次后，将结果记录到每日内存日志中：**

文件路径：`workspace/memory/YYYY-MM-DD.md`

```markdown
## Executing <Topic> Plan (HH:MM)

**Batch 1 (tasks 1-3):** ✅ Complete
- Created file structure
- Implemented storage layer
- Basic todo object

**Batch 2 (tasks 4-6):** 🔄 In progress
- Add todo functionality complete
- List todos in progress
```

**这样做的意义：**
- 如果执行过程中中断，进度可以恢复。
- 每日的日志记录了已完成的工作内容。
- 未来的会话可以参考之前的执行情况。

### 第七步：重复执行

**继续执行下一个批次：**
- 执行步骤3中的操作。
- 审查步骤4中的输出结果。
- 设置检查点（步骤5）。
- 更新内存记录（步骤6）。
- 重复上述步骤，直到所有任务都完成。

## OpenClaw 特定集成

### 使用 sessions_spawn 的方法

**为每个任务创建一个新的子代理：**
```bash
sessions_spawn --label "task-3-storage-layer" --prompt "
Execute task 3 from /workspace/docs/plans/2026-02-25-todo-plan.md:

Create /workspace/todo/lib/storage.js with:
- loadTodos() function
- saveTodos() function
- Use fs.readFileSync/writeFileSync
- Store in ~/.todos.json

Return the complete file content.
"
```

**好处：**
- 隔离性：每个任务都有独立的执行环境。
- 清晰性：不会受到之前任务的影响。
- 调试：更容易识别出哪个任务出现了问题。

### 工作区文件路径

- 计划文件的位置：`/workspace/docs/plans/`
- 内存日志的位置：`/workspace/memory/YYYY-MM-DD.md`
- 项目文件的位置：`/workspace/<项目名称>/`

在创建子代理时，请始终使用完整的路径，以避免路径混淆。

### 内存管理

**查看最近的执行上下文：**
在开始执行之前，请查看：
- `MEMORY.md`（主会话的日志）
- `memory/YYYY-MM-DD.md`（今天的日志）
- `memory/YYYY-MM-DD.md`（如果相关，还包括昨天的日志）

**逐步更新进度：**
不要等到任务全部完成才更新日志——每次执行完一个批次后就更新，以便随时跟踪进度。

## 常见的错误观念及反驳理由

以下是一些常见的错误观念及其反驳理由：

| 错误观念 | 真实情况 | 反驳理由 |
|--------|---------|---------|
| “用户不希望被打扰” | 检查点不是为了打扰用户，而是为了防止错误扩大。 |
| “一次性完成所有任务更高效” | 没有安全保障的效率等于需要重新工作。 | 分批执行可以提前发现错误，从而减少返工。 |
| “计划已经很清晰了，不需要检查点” | 明确的计划并不意味着执行过程中不会出错。 | 代码中可能存在错误、依赖关系可能出问题、文件路径可能不正确。 |
| “最后再通知用户结果” | 等到最后，错误可能已经累积。 | 提前通知可以防止故障的连锁反应。如果任务3失败，任务4-10可能会白费力气。 |
| “用户表示不需要更新进度” | 用户只是表示不想频繁收到通知，并不意味着不需要检查点。 | 应将用户的请求理解为“不要频繁打扰”，而不是“跳过所有检查点”。 |
| “只有10个任务，数量不多” | 即使只有10个任务，也有10次出错的可能性。 | 即使只有5个任务，错误也可能累积。分批执行可以降低出错的风险。 |
| “如果出错我会回滚操作” | 回滚操作会浪费时间。 | 预防错误比事后处理错误更重要。应该在任务3阶段就发现并解决问题。 |

## 注意事项：当你试图跳过这些规则时

如果你有以下想法，说明你可能正在违反这些执行规则：

- “我会直接执行所有任务，最后再报告结果。”
- “用户看起来很急躁，我就跳过检查点。”
- “对于简单的计划来说，分批执行太麻烦了。”
- “如果出了问题，我会再通知用户。”
- “检查点会拖慢我的进度。”
- “计划写得很清楚，应该不会出错。”

**如果你有这些想法，请立即停止，并重新阅读上述的重要规则。**

---

## 这项技能的重要性

**如果不使用分批执行和检查点：**
- 任务3可能会无声地失败。
- 任务4-10可能会基于错误的假设继续执行。
- 最终，你可能会面临8个出错的任务，而不仅仅是1个。
- 用户将不得不处理一团糟的局面，而不仅仅是修复一个错误。

**使用分批执行和检查点：**
- 任务3失败后，第一批会立即报告错误。
- 用户可以修复任务3或调整计划。
- 第二批可以从已知正常的状态开始执行。
- 错误可以及时被发现并迅速解决。

**目标：**通过控制执行过程并提供及时的反馈，确保任务能够按计划顺利进行，而不是采取极端的全有或全无的策略。

---

## 总结

1. 从 `workspace/docs/plans/YYYY-MM-DD-<主题>-plan.md` 文件中加载计划。
2. 将任务分成3-5个一组进行执行。
3. 使用 `sessions_spawn` 为每个任务创建一个新的子代理来执行任务。
4. 审查执行结果，检查是否存在错误或警告。
5. 设置检查点，询问用户是否继续执行下一个批次。
6. 将执行结果更新到 `workspace/memory/YYYY-MM-DD.md` 文件中。
7. 重复上述步骤，直到所有任务都完成。

**记住：**检查点是为了安全，而不是为了麻烦用户。分批执行可以提前发现错误。执行过程应该是可控的，而不是盲目推进的。
