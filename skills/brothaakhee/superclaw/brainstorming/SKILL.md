# 集思广益（Brainstorming）

该技能强调在实施之前先进行设计，以避免昂贵的返工并明确需求。

## 核心原则

**严格规定：** 在设计获得批准之前，严禁编写任何代码或开始实施。

这一点没有商量的余地。时间压力、用户权限、项目简单性或紧迫性都不能成为跳过设计步骤的正当理由。

## 流程

### 1. 查看背景信息

在提出问题之前，先回顾现有的背景信息，以避免重复提问：
- **MEMORY.md**：用户偏好、过去的决策、技术背景（仅限主要会议内容）
- **USER.md**：用户资料、偏好、沟通风格
- **memory/YYYY-MM-DD.md**：今天和昨天的活动记录
- **相关文档**：查看 `workspace/docs/` 目录中是否存在相关的设计文档或决策记录

### 2. 提出澄清性问题

一次提出 2-5 个澄清性问题，以了解以下内容：
- 核心功能与附加功能
- 存储/持久化需求
- 用户界面偏好
- 与现有系统的集成点
- 非功能性需求（性能、安全性等）

**规则：**
- 即使用户表示“我知道自己想要什么”，也要继续提问
- 保持问题具体且聚焦
- 在提出下一个问题之前，等待用户的回答
- 根据之前的回答进一步深入讨论

### 3. 提出解决方案

提出 2-3 种不同的解决方案，包括：
- **高层次架构**：主要组件和数据流
- **权衡因素**：每种方案的优缺点
- **推荐方案**：你推荐的方案及其原因

每个方案的描述应简洁明了（3-5 句话）。

### 4. 展示设计方案

用户选定方案后，以易于理解的方式展示详细设计：
**A. 架构概述**
- 组件图（用 ASCII 艺术形式或文字描述）
- 数据流
- 关键依赖关系

**B. 实现细节**
- 文件结构及文件路径
- 主要功能/模块及其职责
- 数据模型/模式

**C. 未解决的问题**
- 任何尚未明确的问题
- 用户需要做出的决策

**D. 下一步行动**
- 设计获得批准后的后续步骤
- 预计的实现复杂性

### 5. 获得批准

明确询问：“这个设计方案可行吗？需要修改吗？”

等待用户的批准。未经明确同意，切勿开始实施。

### 6. 保存设计方案

设计获得批准后，将其保存到：
```
workspace/docs/plans/YYYY-MM-DD-<topic>-design.md
```

保存内容应包括：
- 日期和背景信息
- 第 4 步中的完整设计方案
- 用户的批准确认
- 需要参考的下一项技能（写作计划）

### 7. 调用下一项技能

保存完成后，调用 `writing-plans` 技能：
“设计方案已批准并保存。现在将调用写作计划技能，将设计方案分解为可执行的任务。”

## 对跳过设计的常见抵触理由的剖析

以下是一些常见的跳过设计步骤的借口，以及它们为何是错误的：
| 抵触理由 | 错误原因 | 反驳理由 |
|--------|----------------|---------|
| “用户时间紧迫” | 快速设计（2-5 分钟）可以避免后续的返工（可能需要数小时或数天） | 时间压力实际上会让设计变得更加重要，而不是可以忽略 |
| “用户知道自己的需求” | 用户只知道目标，但不知道实现细节 | 通过提问可以发现隐藏的需求和假设 |
| “项目很简单，不需要设计” | “简单”的项目往往隐藏着最多的假设 | 即使是简单的项目也需要详细的设计 |
| “不需要长时间的讨论” | 设计并不等同于长时间的讨论 | 设计过程可能只需要 2 分钟，而批准只需要一个简单的回答 |
| “用户已经提供了详细的规格说明” | 规格说明只描述了“做什么”，而没有说明“如何做” | 设计需要明确架构、权衡因素和实施方法 |
| “这只是个原型/MVP” | 原型最终也会变成正式产品代码 | 优秀的架构同样需要前期设计 |
| “我们可以之后再重构” | “之后”可能永远不会到来 | 现在就进行设计，否则可能会陷入无限期的调试 |

**重要提示：** 如果你已经开始编写代码，那就说明你跳过了某个步骤。

## 需警惕的警示信号

以下情况表明你可能违反了“严格规定”：
🚩 在获得批准之前就开始考虑文件结构
🚩 在展示解决方案之前就开始考虑使用哪些库
🚩 在用户回应之前就开始编写伪代码
🚩 你心里想着“我就直接开始吧……”
🚩 你试图为某些特殊情况找借口

**遇到这些警示信号时**：立即停止，返回到第 2 步（提问）或第 5 步（获取批准）。

## 与 OpenClaw 的集成

### 背景信息

**主要会议（与用户的直接交流）：**
- 阅读 `MEMORY.md` 以了解长期背景信息
- 查看 `memory/YYYY-MM-DD.md` 中的近期决策记录
- 参考 `workspace/docs/plans/` 目录中的以往设计方案

**共享讨论环境（如 Discord、团队群组）：**
- 可以不阅读 `MEMORY.md`（其中包含私人数据）
- 仅使用 `USER.md` 和当前的讨论内容

### 工作区组织

所有设计方案都保存在：
```
workspace/docs/plans/YYYY-MM-DD-<descriptive-name>-design.md
```

示例文件名：
- `2026-02-25-todo-cli-design.md`
- `2026-02-25-user-auth-design.md`
- `2026-02-25-api-refactor-design.md`

### 技能链式使用

设计获得批准并保存后：
1. 明确说明设计方案已获得批准
2. 确认保存的位置
3. 调用 `writing-plans` 技能
4. **切勿** 自己开始实施

## 特殊情况处理

### “如果用户坚持跳过设计怎么办？”

回答如下：
> “我理解您希望快速推进。进行 2 分钟的设计可以避免后续的返工。让我先问两个问题，然后提出一个方案，再征求您的同意。总共需要 3 分钟，可以吗？”

### “如果只是微小的修改呢？”

即使是微小的修改，也需要进行简单的设计：
> “需要快速设计一下：将文件 Y 中的 X 部分修改为 Z。这样可以吗？”

在实施之前，请先获得用户的确认。

### “如果设计方案是由外部提供的呢？”

如果用户提供了现有的设计方案：
1. 先仔细审阅该方案
2. 提出 1-2 个关于实现细节的澄清性问题
3. 将修改内容保存到 `workspace/docs/plans/` 目录中以供参考
4. 征得用户的批准后再继续
5. 调用 `writing-plans` 技能

### “如果这是对现有设计的改进呢？”

参考之前的设计方案：
> “我们在 2026-02-20-todo-cli-design.md 的基础上进行改进。建议的修改是：[描述修改内容]。这会影响：[相关组件]。可以吗？”

在实施之前，请先获得用户的批准。

## 错误做法的示例

### ❌ 错误的做法：直接开始实施
```
User: "Build a todo CLI"
Agent: *writes complete Python application*
```

### ✅ 正确的做法：先进行设计
```
User: "Build a todo CLI"
Agent: "Great! Quick questions:
1. How should todos be stored? (JSON file / SQLite / plain text)
2. Need due dates or just simple list?"

[After answers...]

Agent: "Here are 2 approaches:

A. Simple JSON file + Python CLI
   - Pros: Easy to inspect/backup, no dependencies
   - Cons: Not ideal for large lists, manual file locking
   
B. SQLite + Python CLI
   - Pros: Better for large lists, built-in queries
   - Cons: Binary format, slightly more complex

I recommend A for simplicity unless you expect >1000 todos. Thoughts?"

[After user picks A...]

Agent: "Design:

Architecture:
- Single Python script: todo.py
- JSON storage: ~/.todos.json
- Commands: add, list, done, delete

Implementation:
```
```python
todo.py (主要命令行接口)
  - add_todo(text) → 将文本添加到 JSON 文件中
  - list_todos() → 读取 JSON 文件并显示列表
  - mark_done(id) → 更新 JSON 文件中的状态
  - delete_todo(id) → 从 JSON 文件中删除对应的条目
  
~/.todos.json 文件结构：
[
  {"id": 1, "text": "...", "done": false},
  ...
]
```

### 成功标准

当你成功遵循了这一技能时，说明你做到了以下几点：
✅ 在获得批准之前没有编写任何代码
✅ 提出了至少 2 个澄清性问题
✅ 提出了多种解决方案
✅ 设计方案已保存到 `workspace/docs/plans/` 目录中
✅ 用户给出了明确的批准（如“可以”、“看起来不错”等）
✅ 调用了 `writing-plans` 技能来规划下一步行动

### 失败情况

如果你违反了这一技能，可能意味着：
❌ 在设计获得批准之前就编写了代码
❌ 直接跳过了设计步骤
❌ 为跳过设计找借口
❌ 因用户的要求而忽略了必要的步骤
❌ 在获得方案批准之前就开始编写伪代码
❌ 设计方案已保存，但在调用 `writing-plans` 技能之前就开始了实施

**一旦发现自己违反了这些规则，请立即停止，并回到正确的流程中。**
