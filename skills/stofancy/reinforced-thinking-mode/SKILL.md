# 加强型思维模式（Reinforced Thinking Mode）

## 使用场景

当用户需要深入、多角度的分析时，可激活此模式：
- 关键词：深度思考（deep thinking）、多角度分析（multi-angle analysis）、全面评估（comprehensive evaluation）、设计方案（design solution）、架构规划（architecture planning）
- 适用场景：复杂系统设计、战略决策、创新活动、风险评估

**复杂度与轮次对应关系：**
- 简单问题（事实明确）：2-3轮
- 中等复杂问题（功能设计、权衡取舍）：4-5轮
- 高度复杂问题（系统架构、战略规划）：5-7轮
- 非常复杂的问题（定义模糊、存在冲突）：7-10轮

## 核心原则

**每次思考都是独立的、完整的思考过程，而非迭代优化。**  
每一轮思考都应满足以下要求：“如果这是最后一轮，我完全满意。”

### 心理模型

✅ **心态**：“这是我的唯一机会，必须全力以赴。”  
❌ **切勿** 思想：“我会在下一轮改进。”

---

## 严格限制

### 文件访问规则
- **每轮思考仅允许阅读**：`problem.md` 和 `round_{i-1}.md`（例如，第1轮仅阅读 `problem.md`）
- **严禁** 阅读其他轮次的详细内容。

### 禁用词汇
在解决方案文件中严禁使用以下词汇：`TODO`、`待改进`（to be improved）、`下一轮`（next round）、`稍后处理`（later）、`进一步优化`（refine further）。

### 信息处理原则
- 对不确定的事实立即进行查询；
- 对不确定的需求立即向用户询问；
- **严禁** 未经确认就继续推进。

---

## 执行流程

### 第1阶段：初始化
1. **评估问题复杂度**，确定所需的思考轮次（简单问题2-3轮，中等复杂问题4-5轮，高度复杂问题5-7轮，非常复杂问题7-10轮）；
2. 创建文件夹 `reinforced-thinking/`；
3. 编写问题描述文件 `problem.md`，内容包括问题背景、当前状态、核心问题、约束条件以及成功标准。

### 第2阶段：迭代思考（每轮循环）
对于第 X 轮：
1. 阅读文件：`problem.md` 和 `round_{X-1}.md`（第1轮仅阅读 `problem.md`）；
2. **调整思维方式**：从全新的角度出发，避免重复之前的思考方式；
3. **自由选择思考方向**：必要时可以推翻之前的结论；
4. 编写解决方案文件 `round_X.md`，内容包括：
   - 独立性声明；
   - 核心见解；
   - 解决方案设计；
   - 预期结果；
   - 风险及应对措施；
   - 解决方案为何是完整的；
5. **自我检查**：确保文件中不含禁用词汇，内容完整且具备可执行性；
6. **如果失败**，重新执行第4步；
7. **提前终止判断**（从第2轮开始）：
   - 比较当前轮次的思考方法和解决方案与上一轮的差异；
   - 如果相似度超过70%，或缺乏创新性，**建议提前终止当前轮次**；
   - 在当前轮次中记录终止建议，供用户决策参考。

### 第3阶段：综合分析
1. 阅读所有轮次的思考记录和 `problem.md`；
2. 分析各个解决方案的独特价值、潜在冲突及互补之处；
3. **红队评审**（Red Team Review）：对每个解决方案进行严格评估：
   - 假设的合理性：解决方案基于哪些假设？这些假设是否可靠？
   - 安全风险：是否存在潜在的漏洞或被攻击的弱点？
   - 失败场景：在哪些情况下解决方案会失效？
   - 对抗性测试：如果有人故意破坏，最薄弱的环节在哪里？
4. 编写最终报告 `final_report.md`（包含红队评审的结论）。

### 第4阶段：清理
自动删除所有中间文件（`problem.md` 和 `round_*.md`），仅保留 `final_report.md`。  
如需保留某些文件，请在文档中明确说明。

## 质量保证
### 强制重做规则
如果出现以下任何情况，必须重新进行当前轮次的思考：
1. 文件中包含禁用词汇；
2. 解决方案质量未达到要求，需要“后续补充”；
3. 阅读了除 `problem.md` 和 `round_{X-1}.md` 之外的其他文件；
4. 解决方案不完整。

### 每轮检查清单
- [ ] 选择了与上一轮不同的思考角度；
- [ ] 未重复使用之前的解决方案；
- [ ] 文件中不含禁用词汇；
- [ ] 如果这是最后一轮思考，我会感到满意；
- [ ] 所有细节均已包含，可以直接实施；
- [ ] 仅阅读了 `problem.md` 和上一轮的思考记录。

## 常见错误及解决方法

| 错误 | 例证 | 解决方法 |
|-------|---------|-----|
| 依赖前期信息**：**“基于R1的用户体验和R2的技术...” | 应改为“基于`problem.md`和R3的解决方案...” |
| 保留未完成的内容**：“细节留到下一轮处理” | 应立即提供核心设计细节 |
| 未经验证就做出假设**：“用户可能希望这样做...” | 应先核实事实或询问用户 |
| 预设思考方向**：“下一轮从安全角度入手” | 每轮应自由选择思考方向 |
| 采用迭代式思维**：“在R1的基础上进行改进...” | 每轮思考都是独立的，应从零开始思考 |
| 信息缺失**：“请参考相关文档...” | 应在文件中直接提供完整的信息 |

---

## 最佳实践
- **鼓励发散性思维**：无需遵循固定的思考框架，可自由选择任何角度；
- **确保独立性**：每轮思考仅基于 `problem.md` 和上一轮的记录，避免参考所有历史资料；
- **全力以赴**：每轮思考都应全力以赴，不要为“下一轮”保留任何信息；
- **保持透明度**：在每轮思考中明确说明所选择的视角和思考过程；
- **提前终止**：如果连续两轮的思考缺乏创新性，或与上一轮的相似度超过70%，建议提前终止当前轮次。
