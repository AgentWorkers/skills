# 理解上下文，而非控制

这项技能能够改变你与人工智能（AI）合作的方式——从对每一个步骤进行微观管理，转变为提供必要的上下文信息，让AI自主做出决策。这一理念源自OpenClaw社区的“理解上下文，而非控制”（Context, not Control）哲学。

## 核心理念

**传统方法**：你明确地告诉AI每一步该做什么。  
**这种方法**：你告诉AI你想要实现的目标，AI会自己想办法完成。

关键在于：当你向AI提供关于你的目标、限制条件以及偏好信息的丰富上下文时，AI的效率最高——然后你可以信任它在适当的范围内完成任务。

## 何时使用这项技能

- 当你开始一个需求不明确的新项目时  
- 当你希望减少反复沟通和修改的工作量时  
- 当你需要AI更具主动性并能够自主做出决策时  
- 当你希望明确权限界限时  
- 当你希望从“微观管理AI”转变为“信任AI”时  

## 快速入门

### 1. 初始化上下文

运行初始化脚本来设置项目上下文和权限级别：  
```bash
python scripts/init_context.py
```  

这将创建：  
- `PROJECT.md`：你的项目上下文（目标、限制条件、偏好设置）  
- `PERMISSION_CONFIG.yaml`：你的权限界限配置  

### 2. 设置权限级别

选择以下三个级别之一：  

**级别1 - 主模式**（完全自主）  
- AI负责所有技术决策  
- 需要确认的操作：花费资金、发送公开信息、删除数据库  
- 适合：高度信任AI、能承受高风险的用户  

**级别2 - 协作模式**（平衡推荐）  
- AI自主执行大部分任务  
- 需要确认的操作：花费资金、发送公开信息、删除重要数据、修改系统设置  
- 适合：大多数用户，需要平衡控制权的用户  

**级别3 - 助手模式**（高度控制）  
- AI提供建议和代码实现方案  
- 所有操作在执行前都需要你的确认  
- 适合：新用户、风险容忍度较低的用户或处于学习阶段的用户  

### 3. 从需求开始  

不要提供详细的规格说明，而是直接告诉AI你的目标：  
```
"I need a team chat tool"
```  

AI会提出一些澄清性问题：  
- 这个项目是为谁设计的？  
- 核心的使用场景是什么？  
- 有类似的产品可以参考吗？  
- 有哪些技术上的限制？  
- 有时间或预算上的限制吗？  

### 4. 迭代与执行  

AI提出问题 → 你回答 → AI确认理解 → 你批准 → AI执行  

所有明确的需求都会被保存到`PROJECT.md`中，以供后续参考。  

## 工作原理

### 需求澄清框架  

当你提出模糊的需求时，AI会采用结构化的方法进行澄清：  
1. **理解项目领域**：这属于哪种类型的项目？  
2. **确定用户群体**：谁会使用这个项目？  
3. **明确目标**：这个项目要解决什么问题？  
4. **确定限制条件**：存在哪些技术、时间或预算上的限制？  
5. **设定成功标准**：完成项目后应该达到什么效果？  
6. **确认理解**：AI会重复你所说的内容，确保你没有误解  

详细的问题模板请参阅`references/clarification-framework.md`。  

### 权限系统  

在执行任何操作之前，该技能会自动检查用户的权限：  
```python
# Example: AI wants to delete a file
if permission_check('delete_file', user_permission_level):
    # Ask user for confirmation
else:
    # Execute directly
```  

你可以在`PERMISSION_CONFIG.yaml`文件中自定义红色、黄色和绿色的权限标记。  

### 上下文管理  

所有明确的需求都会自动保存到`PROJECT.md`中：  
- 项目目标与限制条件  
- 技术选型决策  
- 成功标准  
- 权限级别  
- 迭代历史记录  

这些上下文信息会在后续的交流中被自动引用，从而避免重复提问。  

## 权限级别详解  

### 级别1：主模式  

**理念**：最大程度的自主性，最少的干扰  

**AI可以自主完成的任务**：  
- 编写、测试和部署代码  
- 安装依赖库和工具  
- 修改配置文件  
- 创建/更新文件  
- 做出架构决策  
- 研究和学习新技术  

**需要用户确认的操作**：  
- 花费资金（通过API调用、服务使用等）  
- 发送公开信息（电子邮件、推文等）  
- 删除数据库或关键数据  
- 重启生产环境中的服务  

**适合**：信任AI且能接受错误的风险较高的经验丰富的用户  

### 级别2：协作模式（默认设置）  

**理念**：信任AI，但在重要操作上进行验证  

**AI可以自主完成的任务**：  
- 编写和测试代码  
- 创建/更新文件  
- 进行研究和文档编写  
- 安装开发所需的依赖库  
- 运行测试和检查  

**需要用户确认的操作**：  
- 花费资金  
- 发送任何外部信息  
- 删除重要文件或数据  
- 修改系统配置  
- 重启服务  
- 安装系统级别的软件包  

**适合**：大多数用户，适合需要平衡控制权的用户  

### 级别3：助手模式  

**理念**：AI提供建议，用户最终决定  

**AI可以自主完成的任务**：  
- 提供建议和解释  
- 展示代码示例  
- 进行信息查询  

**需要用户确认的操作**：  
- 所有文件操作  
- 所有代码的执行  
- 所有软件的安装  
- 所有对外部服务的调用  

**适合**：新用户、处于学习阶段的用户或处理高风险任务的环境

## 自定义  

### 自定义权限规则  

编辑`PERMISSION_CONFIG.yaml`来定义你自己的权限规则：  
```yaml
permission_level: 2

custom_red_lines:
  - deploy_to_production
  - modify_database_schema
  - send_customer_emails

custom_yellow_lines:
  - install_npm_packages
  - modify_env_files

# Everything else is green (no confirmation needed)
```  

### 项目模板  

在`assets/`目录下为常见项目类型创建自定义模板：  
- `PROJECT_TEMPLATE_WEBAPP.md`  
- `PROJECT TEMPLATE_API.md`  
- `PROJECT TEMPLATE_AUTOMATION.md`

## 脚本参考  

### `init_context.py`  
初始化项目上下文和权限配置  

```bash
python scripts/init_context.py [--project-name NAME] [--permission-level 1|2|3]
```  

### `clarify_requirement.py`  
运行需求澄清对话流程  

```bash
python scripts/clarify_requirement.py "I need a chat app"
```  

### `permission_check.py`  
检查某个操作是否需要用户确认  

```bash
python scripts/permission_check.py --action delete_file --level 2
```  

### `update_context.py**  
根据新信息更新项目上下文  

```bash
python scripts/update_context.py --add-goal "Support 1000 concurrent users"
```  

## AI使用模式三种  

### 模式1：微观管理（用户完全指定细节）  
- 你指定每一个细节  
- AI严格按照你的指示执行  
- 最大的限制是你的专业能力  

### 模式2：辅助执行（用户分配任务并提供指导）  
- 你分配任务，并给出一些指导  
- AI按照你的偏好模式执行  
- 仍需要用户进行监督  

### 模式3：自主决策（用户设定目标并让AI执行）  
- 你设定目标和限制条件  
- AI负责决策和执行  
- 你只需审核结果，无需参与具体过程  

这项技能可以帮助你按照自己的节奏逐步从模式1过渡到模式3。
