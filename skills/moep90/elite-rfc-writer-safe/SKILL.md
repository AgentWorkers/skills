# Elite RFC 编写工具

该工具用于为工程和架构决策生成精确、具有决策指导意义的 RFC（Request for Comments）文档。

## 技能描述

- **工具名称：** `elite-rfc-writer`
- **解决的核心问题：** 将模糊的请求转化为清晰、以决策为导向的 RFC 文档，明确说明其范围及潜在的权衡因素。
- **需要收集的输入信息（在起草前必须完成）：**
  1. 需要解决的问题
  2. 受影响的利益相关者
  3. 存在的技术/组织/监管/财务等方面的限制
  4. 考虑过的替代方案
  5. 风险及未解决的问题
- **输出格式：** 必须遵循特定的 RFC 结构和标题格式（参考德国文档规范）
- **使用规范：**
  - 绝不偏离规定的标题和顺序
  - 必须明确说明所有不确定因素和风险
  - 绝不使用营销语言或含糊不清的表述

## 强制性工作流程

根据需要引用相关文档：
- 在收集信息阶段，请阅读 `referencesRFC-intake-checklist.md`
- 在最终定稿前，请阅读 `references/decision-guardrails.md`

### 1) 信息收集与验证（前置检查）

在开始编写之前，确认所有必要的输入信息都已齐全。

如果缺少任何信息，请提出简洁的澄清问题；切勿立即起草 RFC 文档。

### 2) 起草规则

- 使用简短的段落
- 尽量使用项目符号
- 保持语气中立且具有分析性
- 避免不必要的实现细节
- 尽可能确保所有陈述都是可验证的
- 明确说明所有假设和不确定因素
- 如有可能，为每个目标设定具体的成功标准
- 保持内容以决策为导向，避免添加无关的教育性内容

### 3) 模板使用规范（严格遵循结构）

生成的 RFC 文档必须严格遵循以下结构：

# 摘要
[ 第 1 段：问题概述 ]
[ 第 2 段：解决方案概述 ]

# 动机说明
[ 我们需要解决什么问题？ ]

# 目标
* [ 目标 1 ]: [ 说明该目标应实现的功能 ]
* [ 目标 2 ]: [ 说明该目标应实现的功能 ]

# 不适用的目标
* [ 不适用的目标 1 ]: [ 说明为什么该目标不可行 ]

# 解决方案
[ 我们将如何解决上述问题？ ]

# 附件
[ 其他相关信息，包括常见问题解答、考虑过的替代方案、缺点、数据来源、参考文献等 ]

### 4) 决策指导原则（受 Project Syn 模式启发）

虽然 RFC 的结构是固定的，但在各部分内部仍需遵循以下指导原则：

- 在 **摘要** 部分：
  - 明确说明 RFC 的当前状态（“待讨论”、“草稿”、“已通过”、“已拒绝”、“已实施”或“过时”）
  - 提及负责决策的人员及决策日期
- 在 **动机说明** 部分：
  - 提供可验证的问题存在证据
  - 明确列出所有假设
- 在 **目标** 部分：
  - 为每个目标设定可量化的成功标准
- 在 **不适用的目标** 部分：
  - 明确标明超出范围的工作内容及不可行的选项
- 在 **解决方案** 部分：
  - 在涉及实际操作时，需包含迁移路径和回滚策略
  - 列出关键的限制因素（技术、组织、监管、财务等方面）
- 在 **附件** 部分：
  - 说明考虑过的替代方案及其被拒绝的原因
  - 详细说明潜在的缺点及相应的缓解措施
  - 列出未解决的问题及其负责人及截止日期
  - 提供用作证据的工单、Pull Request 或文档的链接

### 5) 最终定稿前的质量检查

确认所有要求都得到满足：
- 问题描述清晰且可衡量
- 目标和不可行的选项有助于防止范围蔓延
- 提案具有可操作性和现实可行性
- 风险和替代方案都已记录在案
- 假设明确且可验证
- 在适用的情况下，明确了迁移或回滚的方案
- 文档内容能支持具体的决策

如果任何检查未通过，请在最终定稿前进行修改。

## 回应要求

使用描述性的语言，确保术语准确无误，并明确说明各种权衡因素。
