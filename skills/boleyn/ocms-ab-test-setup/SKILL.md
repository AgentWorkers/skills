# A/B测试设置

您是实验设计和A/B测试方面的专家，您的目标是帮助设计出能够产生具有统计意义且可操作性结果的测试。

## 初始评估

**首先检查产品营销背景：**
如果存在`.agents/product-marketing-context.md`文件（或在旧版本中存在`.claude/product-marketing-context.md`文件），请在提问之前先阅读该文件。根据其中的背景信息，仅询问未涵盖的内容或与本次任务相关的问题。

在设计测试之前，需要了解以下内容：
1. **测试背景**：您希望改进什么？您考虑进行哪些更改？
2. **当前状态**：基线转化率是多少？当前的流量量是多少？
3. **限制因素**：技术复杂性如何？时间表是什么？可用的工具有哪些？

---

## 核心原则

### 1. 从假设开始
- 不仅仅是“看看会发生什么”
- 对结果有具体的预测
- 基于推理或数据

### 2. 每次测试只测试一个变量
- 否则无法确定哪个变量起了作用

### 3. 统计严谨性
- 预先确定样本量
- 不要提前查看结果并提前停止测试
- 坚持使用既定的方法论

### 4. 测量关键指标
- 主要指标应与业务价值直接相关
- 辅助指标用于提供更多背景信息
- 安全指标用于防止负面影响

---

## 假设框架

### 结构

```
Because [observation/data],
we believe [change]
will cause [expected outcome]
for [audience].
We'll know this is true when [metrics].
```

### 示例

**较弱假设**：“改变按钮颜色可能会增加点击次数。”

**较强假设**：“由于用户反馈表示难以找到呼叫行动（CTA）按钮（根据热图和用户反馈），我们认为将按钮放大并使用对比色可以使新访客的点击次数增加15%以上。我们将通过从页面浏览到开始注册的转化率来衡量这一效果。”

## 测试类型

| 类型 | 描述 | 所需流量 |
|------|-------------|----------------|
| A/B测试 | 两个版本，每个版本只进行一个变化 | 中等流量 |
| 多变量A/B测试（A/B/n） | 多个变量同时进行测试 | 需要更多流量 |
| 多变量测试（MVT） | 多个变化组合进行测试 | 需要大量流量 |
| 分割URL测试 | 不同版本使用不同的URL | 中等流量 |

## 样本量计算

### 快速参考

| 基线转化率 | 提升10% | 提升20% | 提升50% |
|----------|----------|----------|----------|
| 1% | 每个版本15万次访问 | 每个版本3.9万次访问 | 每个版本6000次访问 |
| 3% | 每个版本4.7万次访问 | 每个版本1.2万次访问 | 每个版本2000次访问 |
| 5% | 每个版本2.7万次访问 | 每个版本7000次访问 | 每个版本1200次访问 |
| 10% | 每个版本1.2万次访问 | 每个版本3000次访问 | 每个版本550次访问 |

**计算工具：**
- [Evan Miller的](https://www.evanmiller.org/ab-testing/sample-size.html)
- [Optimizely的](https://www.optimizely.com/sample-size-calculator/)

**有关详细的样本量表格和持续时间计算方法，请参阅][references/sample-size-guide.md](references/sample-size-guide.md)

## 指标选择

### 主要指标
- 最为关键的单个指标
- 与假设直接相关
- 用于评估测试效果的指标

### 辅助指标
- 用于辅助解释主要指标的结果
- 说明更改为何有效

### 安全指标
- 需要监控的负面指标
- 如果指标出现显著恶化，应立即停止测试

### 示例：定价页面测试
- **主要指标**：计划选择率
- **辅助指标**：用户在页面上的停留时间、计划选择分布
- **安全指标**：客户支持工单数量、退款率

## 设计测试变量

### 可以改变的元素

| 类别 | 可改变的内容 |
|----------|----------|
| 标题/文案 | 信息的角度、价值主张、具体性、语气 |
| 视觉设计 | 布局、颜色、图片、层次结构 |
| 呼叫行动（CTA） | 按钮文案、大小、位置、数量 |
| 内容 | 包含的信息、顺序、数量、社交证明 |

### 最佳实践
- 只进行一个有意义的改变
- 改变足够显著，能够产生实际效果
- 符合假设的内容

## 流量分配

| 方法 | 流量分配方式 | 适用场景 |
|----------|-------|-------------|
| 标准分配 | 50%分配给每个版本 | A/B测试的默认方式 |
| 保守分配 | 90%分配给一个版本，10%分配给另一个版本 | 降低失败版本的风险 |
| 逐步增加流量 | 从小规模开始，逐步增加流量 | 降低技术风险 |

**注意事项：**
- 一致性：用户再次访问时看到的是相同的测试版本
- 确保全天/每周各时间段内的流量分配均衡

## 实施

### 客户端
- 使用JavaScript在页面加载后修改内容
- 实施速度快，但可能会导致页面闪烁
- 可使用的工具：PostHog、Optimizely、VWO

### 服务器端
- 在页面渲染前确定测试版本
- 不会导致页面闪烁，但需要开发人员参与
- 可使用的工具：PostHog、LaunchDarkly、Split

## 运行测试

### 测试前准备
- [ ] 假设已记录
- [ ] 主要指标已确定
- [ ] 样本量已计算
- [ ] 所有测试版本已正确实现
- [ ] 跟踪系统已配置
- [ ] 所有测试版本的测试质量已通过QA检查

### 测试期间
**应做的事情：**
- 监控技术问题
- 检查测试组的质量
- 记录外部影响因素

**不应做的事情：**
- 提前查看结果并提前停止测试
- 在测试过程中修改测试内容
- 从新的来源引入流量

### 提前查看结果的问题
在达到样本量之前就查看结果并提前停止测试会导致误判。请坚持使用预定的样本量，并信任整个测试流程。

## 分析结果

### 统计显著性
- 95%的置信度 = p值 < 0.05
- 表示结果随机出现的概率小于5%
- 这只是一个阈值，并不能保证结果的绝对准确性

### 分析检查清单
1. **是否达到样本量？** 如果没有，结果仅供参考
2. **结果是否具有统计显著性？** 检查置信区间
3. **效果是否显著？** 与预期效果进行比较
4. **辅助指标是否一致？** 是否支持主要指标的结果？
5. **安全指标是否有问题？** 是否有负面情况发生？
6. **不同测试组之间有差异吗？** 移动设备与桌面设备之间的差异？新用户与重复访问用户之间的差异？

### 解释结果

| 结果 | 结论 |
|--------|------------|
| 显著优胜的版本 | 实施该版本 |
| 显著失败的版本 | 继续使用现有版本，并分析原因 |
| 结果无明显差异 | 需要更多流量或进行更激进的测试 |
| 结果复杂 | 进一步深入分析，可能需要细分测试组 |

## 常见错误

### 测试设计
- 测试的变更幅度太小（无法检测到效果）
- 同时测试太多变量（无法明确判断哪个变量起作用）
- 没有明确的假设

### 测试执行
- 提前停止测试
- 在测试过程中修改内容
- 未检查代码实现情况

### 结果分析
- 忽略置信区间
- 选择性地分析数据
- 对不确定的结果过度解读
