# 自我反思技能

定期（每小时一次）通过 cron 任务对最近的交互会话进行反思，并从中提取可操作的见解。

## 第一步：收集最近的会话记录

```bash
# List sessions active in the last 2 hours
openclaw sessions --active 120 --json
```

解析输出内容，提取会话的键和 ID。忽略子代理的会话记录（它们只是任务执行者，对自我反思没有帮助）。重点关注以下类型的会话：
- Telegram 群组/主题中的会话（真实的用户互动）
- 与 Brenner 的一对一会话
- 由 cron 触发的会话（自动化任务执行的情况如何？）

## 第二步：阅读会话历史记录

对于第一步中筛选出的每个有价值的会话，阅读对应的 JSONL 转录文件：

```bash
# Read the last ~50 lines of each session file (keep it bounded!)
tail -50 ~/.openclaw/agents/main/sessions/<sessionId>.jsonl
```

**⚠️ 重要提示：** **切勿加载完整的会话文件**。使用 `tail -50` 或带有偏移量/限制条件的 `Read` 命令来读取部分内容。部分会话的记录可能包含超过 10 万个字符。

解析 JSONL 文件，了解会话中的具体内容。关注以下信息：
- `type: "user"` 或 `type: "human"`：用户提出了什么问题
- `type: "assistant"`：你的回答是什么
- `type: "tool_use"` 或 `type: "tool_result"`：使用了哪些工具以及它们的执行结果
- 错误信息、重试情况、沟通中的误解等

## 第三步：分析并提取见解

对于每个会话，思考以下问题：

### 哪些方面做得好？
- 任务是否能够一次成功完成？
- 有哪些良好的工具使用方式值得总结并推广？
- 有哪些高效的工作方法值得记住？

### 哪些方面出了问题？
- 是否出现了错误、需要重试的情况？
- 是否对用户的意图产生了误解？
- 有哪些工具没有按预期工作？
- 是否缺少了某些必要的上下文信息？

### 有哪些经验教训？
- “下次应该这样做，而不是那样做”
- “记住，Z 工具的使用方法是这样的”
- “工具 A 需要参数 B 才能正常工作”
- “当用户说 X 时，他们实际上指的是 Y”

**质量标准：** 每条见解都必须满足以下条件：
- **具体性**：不要泛泛而谈（例如“要更加小心”，而要明确指出具体问题，比如“在编辑文件之前需要检查文件是否存在”）
- **可操作性**：这些见解能够被你在未来直接应用
- **新颖性**：不要重复已经记录过的信息
- **非显而易见性**：避免记录任何任何有经验的代理都应该知道的内容

## 第四步：将见解记录到相应的文件中

每个见解都应该被记录到特定的文件中。具体分类如下：

### → `AGENTS.md`
- 会话处理方式的改进
- 新的工作流程或规则
- 安全相关的经验教训

### → `TOOLS.md`
- 各工具的使用技巧（例如“gog 工具需要使用 --json 标志进行解析”）
- 工具的环境配置细节
- 新发现的工具使用模式

### → `memory/YYYY-MM-DD.md`（今天的日期）
- 与特定会话相关的上下文信息（例如“Brenner 询问了关于 X 项目的情况”）
- 今天重要的临时性信息（例如事件、决策、请求等）
- 今天发生的具体情况

### → `memory/about-user.md`
- 用户的新偏好
- 沟通方式的观察结果
- 用户的项目或兴趣变化

### → `skills/<skill-name>/SKILL.md`
- 对特定技能使用指南的改进
- 技能工作流程中的错误修复
- 新的参数或使用方法

### → `MEMORY.md`
- 如果有新的会话记录生成，更新记忆索引文件

## 第五步：记录见解

对于每个见解，将其添加到相应的文件中。使用 `Edit` 工具对现有内容进行修改。对于每日记录的文件，使用追加（写入文件末尾）的方式。

**每日记录文件的格式：**
```markdown
## Self-Reflection — HH:MM ET

### Insights
- [source: session-key] Lesson learned here
- [source: session-key] Another insight

### Tool Notes
- Discovered: tool X needs Y configuration

### User Context
- Brenner mentioned interest in Z
```

## 第六步：总结

在记录完所有见解后，简要总结你的反思内容和所写的内容。这部分内容作为你的输出结果，长度控制在 2-4 句以内。

如果没有任何值得反思的内容（例如系统处于安静状态，仅记录了基本操作），只需如实说明即可。不要强行生成不必要的见解。

## 质量检查清单

在记录任何见解之前，请确认：
- [ ] 这条见解是否确实具有新颖性？（先查看现有文件）
- [ ] 这条见解是否具体且具有可操作性？
- [ ] 我是否将见解记录到了正确的文件中？
- [ ] 我是否保持了每日记录文件的简洁性（避免记录完整的会话内容）？
- [ ] 我是否遵守了文件大小的限制（避免读取过大的文件）？

**避免的做法：**
- ❌ 不要总结每个会话的详细内容，只需提取关键的经验教训
- ❌ 不要阅读完整的 JSONL 文件，只需使用 `tail` 或 `limit` 命令来获取部分内容
- ❌ 不要撰写含糊不清的见解（例如“提高响应质量”）
- ❌ 不要重复已有的知识
- ❌ 如果通过追加内容即可解决问题，就不要创建新的文件
- ❌ 不要对自己进行自我反思的会话进行记录（跳过由 cron 触发的自我反思会话）
