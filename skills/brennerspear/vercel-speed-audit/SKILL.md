# Vercel 速度审计技能

优化任何项目的 Vercel 构建和部署速度。在启动新项目或审计现有项目时，可以将其作为检查清单来执行。

## 适用场景

- **新项目设置** — 从第一天起配置最佳默认设置
- **构建时间逐渐增加** — 审计并修复性能退化问题
- **部署队列瓶颈** — 消除等待并发构建的情况
- **生产环境问题** — 使用即时回滚功能，无需等待新的构建结果
- **成本优化** — 在构建基础设施的成本与速度之间找到平衡

## 先进行分类评估（不要盲目执行所有步骤）

在运行完整的检查清单之前，**先进行测量和评估**：

```bash
# 1. Check current build times
cd <project> && npx vercel ls --limit 5

# 2. Check team/plan tier
npx vercel team ls
```

**决策树：**

| 构建时间 | 执行操作 |
|-----------|--------|
| **< 20秒** | 只执行步骤 0、1、3-5、10。除非代码库很大，否则跳过 GitHub Actions（步骤 8）和 barrel 文件审计（步骤 6）。 |
| **20-60秒** | 执行步骤 0-7、10。GitHub Actions（步骤 8）可选。 |
| **60秒以上** | 执行所有步骤。GitHub Actions（步骤 8）变得非常有必要。 |

**应用程序是否需要身份验证？**（所有页面都需要登录）
→ **完全跳过步骤 9**。ISR 为所有用户缓存一个响应——这与按用户显示的内容不兼容。

**框架：**
→ 如果使用 SvelteKit，请同时参考 `docs/sveltekit.md` — 其中包含关于 `adapter-vercel` 的配置选项、预渲染功能以及 `$env` 变量的使用技巧。
→ 如果使用 Next.js/Nuxt 或其他框架，请参考 `docs/general.md` — 其中包含与框架无关的优化建议。

## 检查清单（按影响程度排序）

| 编号 | 优化内容 | 影响程度 | 是否可以通过 CLI 验证？ | 是否需要制定计划？ |
|---|-------------|--------|:-:|:-:|
| **0** | **使用显式的适配器**（例如 `adapter-vercel` 而不是 `adapter-auto`） | 提高构建速度，且无需额外的检测开销 | ✅ 检查配置文件 | 任何项目 |
| **1** | **使用 Turbo 构建服务器** | 构建速度提升 50-70% | 仅通过仪表盘查看 | Pro+ 级别 |
| **2** | **按需执行并发构建** | 消除部署队列等待时间 | 仅通过仪表盘查看 | Pro+ 级别 |
| **3** | **预渲染静态页面** | 减少函数调用次数，加快页面加载时间 | ✅ 审查代码 | 任何项目 |
| **4** | **忽略不必要的构建步骤** | 跳过无关的构建过程 | ✅ 检查 `vercel.json` 配置文件 | 任何项目 |
| **5** **优先处理生产环境构建** | 生产环境的构建优先执行 | 仅通过仪表盘查看 | Pro+ 级别 |
| **6** | **移除 barrel 文件** | 提升构建速度 10-30% | ✅ 审查代码 | 任何项目 |
| **7** | **审计并精简依赖项** | 加快安装和打包速度 | ✅ 运行 `depcheck` 工具 | 任何项目 |
| **8** | **结合 GitHub Actions 和 `--prebuilt` 选项** | 完全控制缓存，跳过 Vercel 的构建过程 | ✅ 添加相关工作流 | 任何项目 |
| **9** | **为动态页面启用 ISR（即时响应）** | 减少冷启动次数 | ✅ 审查路由配置 | 任何项目 |
| **10** | **即时回滚** | 实现零停机时间恢复 | ✅ 使用 `vercel rollback` 命令 | 仅限高级用户；Pro+ 级别 |

**Pro 级别的优化内容（步骤 1、2、5）：** 如果你不是 Pro 用户，请先查看团队计划。这些优化措施对 Pro 用户来说效果最为显著。**

## 仪表盘链接（无法通过 CLI 查阅）

对于名为 `<team-slug>` 的团队：
- **Turbo 构建服务器设置**：`https://vercel.com/<team-slug>/<project>/settings` → “General” → “Build Machine”
- **并发构建设置**：`https://vercel.com/teams/<team-slug>/settings` → “Build Queue”
- **优先处理生产环境构建**：`https://vercel.com/<team-slug>/<project>/settings` → “Git” → “Production Prioritization”

## 经验总结

在一个使用 SvelteKit 的项目中执行所有 10 项优化后，我们发现：
- **有一半的优化措施其实并不必要** — 因为该项目的构建速度已经很快。现在通过分类评估步骤可以避免浪费精力。
- **对于需要身份验证的应用程序来说，ISR 功能并不适用** — 因为所有页面都会为每个用户显示不同的内容。因此添加了决策树来指导选择。
- **将 `adapter-auto` 更改为 `adapter-vercel` 可以显著提升性能** — 这个优化措施最初未被包含在最初的清单中，但在依赖项审计中发现。现在它已成为优先推荐的步骤。
- **仅通过仪表盘查看的设置（如 Turbo 构建服务器、并发构建、优先级设置）无法通过 CLI 验证** — 因此提供了直接访问这些设置的链接。
- **只有当构建时间超过 60 秒时，结合 GitHub Actions 和 `--prebuilt` 选项才值得尝试** — 对于构建速度较快的项目，使用 Vercel 的 Git 集成方式更为简单且同样有效。
- **依赖项审计显示该项目已经非常简洁**（仅使用了 11 个依赖项，`node_modules` 文件大小为 84MB，客户端打包文件大小为 472KB，其中只有一个依赖项 (`clsx`) 未被使用）。
- **团队和项目计划决定了哪些优化措施具有最高效果** — 在开始优化之前，应先查看项目计划。

## 快速入门步骤：

1. **进行分类评估**：测量构建时间并查看项目所属的级别（Pro 或非 Pro）。
2. 根据决策树选择合适的优化措施进行执行。
3. 如果使用 SvelteKit，还需参考 `docs/sveltekit.md`。
4. 将优化结果记录在报告文件中（例如 `vercel-speed-report.md`）。
5. 对于复杂的持续集成（CI）需求，请参考 `docs/github-actions-prebuilt.md` 的指南。
