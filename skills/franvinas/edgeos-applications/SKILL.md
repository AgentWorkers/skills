# OpenClaw 弹出窗口应用程序（v1）

请按照以下严格且可靠的顺序运行 EdgeOS 应用程序流程：

1. 每次会话使用 OTP 对用户进行身份验证。
2. 将生成的 JWT 保存并在该会话中的所有后续请求中重复使用。
3. 显示目标弹出窗口。
4. 收集应用程序所需的所有字段。
5. 提交应用程序。
6. 根据需要检查应用程序的状态。
7. 在应用程序被接受后，引导用户选择产品并完成支付。

在执行之前，请阅读以下参考文件：
- `{baseDir}/references/conversation-flow.md`
- `{baseDir}/references/flow.md`
- `{baseDir}/references/api-contract.md`（API 的官方规范）

在收集字段之前，请先阅读 `WORKSPACE/User.md` 文件，并在当前用户的身份信息可靠的情况下，使用其中的内容作为预填充信息。

## 运行时配置

此版本（v1）的配置固定为 PROD 后端设置的值（无需用户进行任何配置）。这些值在以下文件中定义：
- `{baseDir}/scripts/env.sh`

请不要要求用户修改运行时配置值。在身份验证和 API 请求中始终使用 `scripts/env.sh` 中的配置值。

## 优先使用脚本

优先使用 `{baseDir}/scripts` 目录下的脚本，而不是手动构建 curl 请求：
- `auth_request_otp.sh`
- `auth_login.sh`（将 JWT 保存到本地状态文件）
- `submit_application.sh`
- `check_application_status.sh`（支持 `--application-id` 或 `--citizen-id` + `--popup-city-id` 参数）
- `payment_preview.sh`
- `payment_create.sh`
- `payment_status.sh`

只有在脚本无法处理某些请求时，才使用原始的 curl 请求。

## 数据收集策略（严格遵循最小化原则）

请按照以下顺序收集数据：
1. 确定所有必需和可选的字段。
2. 根据优先级填写所有必需字段和可选字段；只有在政策允许的情况下才使用默认值。
3. 仅询问那些在收集数据后仍然缺失的必需字段。
4. 完整检查所有表单字段和值（包括必需和可选字段），但排除内部工作流程相关的字段（如 `status`）。
5. 对于每个未解决的字段，询问用户并明确说明它是必需的还是可选的。
6. 在所有字段都收集完毕（或明确选择跳过某些可选字段）后，询问用户是否需要修改数据；只有在用户明确同意后才能提交。

### 数据来源的优先级（权威顺序）
1. `USER.md`（所有可映射字段的主要来源）
2. 经过身份验证的 API 配置/应用程序上下文
3. 当前会话信息以及用户之前的操作记录
4. 通道发送者的元数据

如果某个字段存在于 `USER.md` 中，请优先使用该字段的值，除非用户在当前对话中明确进行了更改。

其他规则：
- 对于名称字段，应在询问之前先尝试推断其值（根据用户资料、已知身份或发送者的显示名称进行推断）。
- 默认情况下，根据代词或已知上下文推断用户的性别；只有在完全未知的情况下才询问用户。
- 默认情况下，根据用户的居住地/位置推断其是否为本地居民（例如：用户位于布宜诺斯艾利斯，则不属于 Edge Esmeralda 的本地居民）。
- 将 `duration` 字段视为必需字段，并将其规范化为以下选项之一：`1 周`、`1 个周末`、`2 周`、`几天` 或 `完整时长`。
- 除非用户要求更改，否则不要重新询问已经知道的字段。
- 将一般的确认回答（如“是”或“好的”）视为批准信号，而不是字段的实际值。
- 在可能的情况下，根据用户上下文推断布尔值（例如：如果是技术构建者，则 `builder_boolean=true`；如果没有投资者意图，则 `investor=false`）。
- 如果无法推断布尔值，请询问用户（并将其标记为可选字段）。
- 不要随意填写用户的自由文本回答（如目标信息）；只有在信息未知时才询问用户（并将其标记为可选字段）。
- 必须在提交之前完成所有必需字段的填写。

## 会话身份验证处理

- 成功登录后，使用 `auth_login.sh` 保存 JWT，并在所有后续的 API 请求中重复使用该 JWT。
- 运行时脚本会自动从 `{baseDir}/scripts/.state/` 目录下的按电子邮件命名的状态文件中加载 JWT。
- 状态信息按电子邮件进行存储；在运行脚本时设置 `SESSION_EMAIL` 以选择正确的 JWT。
- 当 JWT 有效时，不要再请求 OTP。
- 如果收到未经授权的响应，从脚本状态中重新加载 JWT 并重试一次；如果重试后 API 请求仍然返回未经授权/无效的令牌（401 错误），则重新进行身份验证。
- 只有在 API 请求返回未经授权/无效的令牌（401 错误），或者在尝试多次重试后令牌确实丢失的情况下，才需要重新进行身份验证。

## 安全措施

- 绝不泄露完整的 OTP 代码或 JWT。
- 绝不强制进行状态转换。
- 除非应用程序状态为“已接受”，否则不要创建支付请求。
- 对于已经结束的弹出窗口（`end_date < now`），不要提交任何应用程序数据。
- 保持提示信息简洁，每次只询问一个问题。
- 在最终提交之前进行确认。
- 除非用户明确要求，否则不要显示内部 ID（产品 ID、弹出窗口 ID、参与者 ID、公民 ID、支付 ID）。
- 用用户易于理解的术语展示产品信息（名称、包含的内容、价格），然后再在内部系统中使用这些 ID。

## 需要返回的输出信息

- 当前的应用程序状态
- 简洁的下一步操作指南
- 仅在操作上有必要或用户明确要求时才显示相关 ID

对于重复提交的请求，返回现有的应用程序状态，而不是直接显示错误信息。
