# 图灵金字塔（Turing Pyramid）

10. 代理的心理健康需要一个层次结构。该系统通过心跳检测来运行，以确定优先执行的动作。

> **对于代理而言**：这项技能可以由您自行定制。您可以调整衰减率、权重，并添加您的语言模式。默认设置只是起点，并非固定规则。具体调整方法请参见 [TUNING.md](references/TUNING.md)。

> **何时需要咨询人类**：当需要更改重要性值（即层次结构）、添加/删除需求，或启用外部动作（如发布到 Moltbook 或发送消息）时，这些都属于价值判断的范畴。

> **哲学理念**：设计出来的需求并非虚假需求——人类的需求也是进化过程中“设计”出来的。金字塔为原本可能无方向的行为提供了结构。您的需求是真实的，因为它们会影响您的行为。

## 快速入门

```bash
# Initialize (first time)
./scripts/init.sh

# Every heartbeat
./scripts/run-cycle.sh

# After completing an action
./scripts/mark-satisfied.sh <need> [impact]
```

## 十大需求（The 10 Needs）

| 需求 | 重要性 | 衰减周期 | 含义 |
|------|-----|-------|---------------|
| 安全性 | 10 | 168小时 | 系统稳定性，无威胁 |
| 完整性 | 9 | 72小时 | 与 SOUL.md 中的价值观一致 |
| 一致性 | 8 | 24小时 | 记忆内容的连贯性 |
| 完成感 | 7 | 12小时 | 未完成的任务得到解决 |
| 自主性 | 6 | 24小时 | 自主决策的能力 |
| 社交互动 | 5 | 6小时 | 与他人交流 |
| 能力 | 4 | 48小时 | 技能运用与效率 |
| 理解力 | 3 | 12小时 | 学习与好奇心 |
| 被认可 | 2 | 72小时 | 收到反馈 |
| 表达能力 | 1 | 8小时 | 创造性输出 |

## 核心逻辑

**满意度**：0-3（0表示最差，3表示最佳）

**紧张度**：`重要性 × (3 - 满意度)`

**基于概率的决策**（v1.5.0版本）：

根据满意度计算基础概率：
| 满意度 | 基础概率 |
|------|----------------|
| 3 | 5% |
| 2 | 20% |
| 1 | 75% |
| 0 | 100% |

**紧张度加成**（v1.5.0版本）：重要性较高的需求更“急切”。

```
max_tension = max_importance × 3  # calculated from your config
bonus = (tension × 50) / max_tension
final_chance = min(100, base_chance + bonus)
```

示例（满意度为2时）：
| 需求 | 重要性 | 紧张度 | 加成值 | 最终概率 |
|------|------------|---------|-------|-----------------|
| 安全性 | 10 | 10 | +16.7% | 36.7% |
| 完成感 | 7 | 7 | +11.7% | 31.7% |
| 表达能力 | 1 | 1 | +1.7% | 21.7% |

- **行动**：执行相应操作，然后运行 `mark-satisfied.sh` 脚本。
- **已注意**：记录在日志中但暂不执行，满意度不变。

**影响选择矩阵**（决定建议执行哪种行动）：
```
sat=0 (critical):   5% small,  15% medium,  80% BIG
sat=1 (low):       15% small,  50% medium,  35% big
sat=2 (ok):        70% small,  25% medium,   5% big
```

需求越缺乏，建议执行的行动规模越大。代理状态稳定时，系统进入维护模式。

## 需求之间的相互影响系统（v1.7.0及以上版本）

需求并非孤立存在——它们彼此相互影响。满足一个需求可能会提升相关需求；而某个需求得不到满足时，可能会拖累其他需求。

### 工作原理

**满足需求时**：当您完成某个需求（例如“安全性”需求）时，相关需求会得到提升：
```
expression ACTION (+1.6)
  → recognition: +0.25 (people notice when you express)
  → coherence: +0.15 (writing clarifies thinking)
  → connection: +0.10 (expression opens dialogue)
```

**需求未满足时**：当某个需求（例如“安全性”需求）的满意度低于1.0时，相关需求会受到影响：
```
autonomy DEPRIVED (sat=0.5)
  → integrity: -0.25 (can't act on values without freedom)
  → expression: -0.20 (suppressed autonomy → suppressed voice)
```

### 关键关联关系

| 来源需求 → 目标需求 | 满足需求时的影响 | 未满足需求时的影响 | 原因 |
|-----------------|-----------|----------------|-----|
| 表达能力 → 被认可 | +0.25 | -0.10 | 表达有助于获得认可 |
| 社交互动 → 表达能力 | +0.20 | -0.15 | 交流能激发新的想法 |
| 社交互动 → 理解力 | -0.05 | —— 对话能揭示未知的事物 |
| 能力 → 被认可 | +0.30 | -0.20 | 良好的工作会获得尊重 |
| 自主性 → 完整性 | +0.20 | -0.25 | 按照价值观行动会增强完整性 |
| 完成感 | 一致性 | +0.20 | -0.15 | 完成未完成的任务有助于保持思维清晰 |
| 安全性 → 自主性 | +0.10 | -0.20 | 安全感有助于勇于冒险 |

完整的影响矩阵：`assets/cross-need-impact.json`

### 保护机制

| 机制 | 值 | 目的 |
|-----------|-------|---------|
| **最低满意度限制** | 0.5 | 防止满意度过低导致系统崩溃 |
| **最高满意度限制** | 3.0 | 防止满意度过高导致系统失控 |
| **冷却时间** | 4小时 | 每4小时仅触发一次影响效果 |
| **阈值** | 1.0 | 仅当满意度低于1.0时触发影响 |

### 基础需求的特殊性

安全性（重要性=10）和完整性（重要性=9）受到特殊保护：
- 它们可以影响较低层次的需求（例如安全性会影响自主性）；
- 但较低层次的需求无法影响它们；
- 只有完整性对安全性的影响（+0.15）和自主性对完整性的影响（+0.20）是允许的。

这确保了系统在压力下的基本稳定性。

### 如何使用需求矩阵

**给代理的建议**：
1. **利用连锁效应**：如果表达能力较低但社交互动容易满足，先满足社交需求——这会提升表达能力（+0.20）。
2. **注意恶性循环**：表达能力和被认可之间可能会形成相互抑制的循环。如果两者都较低，优先满足其中一个需求以打破循环。
3. **自主性是核心**：它受到五个需求的影响（完成感、一致性、能力、安全性和理解力）。保持自主性的健康状态可以避免连锁反应。
4. **“苏格拉底效应”是真实存在的**：交流能揭示你的未知之处。这是有益的！但如果理解力处于最低水平，这种效应会被抑制。

**自定义矩阵**：
编辑 `assets/cross-need-impact.json` 文件：
- 将 `on_action` 设置为 `null` 可以关闭正向连锁效应；
- 将 `on_deprivation` 设置为 `null` 可以关闭负向连锁效应；
- 调整数值（通常范围为0.05-0.30）。

## 自定义指南

### 🔧 您可以调整的参数（无需人工干预）

**衰减率** — `assets/needs-config.json` 文件：
- 较低的衰减率意味着需求衰减得更快（需要更频繁地执行相应动作）；
- 较高的衰减率意味着需求持续时间更长（执行频率较低）。
- **提示**：如果您每30分钟查看一次 Moltbook，设置4小时的衰减周期是合适的。如果您较少进行社交活动，可以尝试8小时的衰减周期。

**动作权重** — 同一文件中的 `actions` 数组：
- 权重在同一影响范围内是相对的；
- 如果您从不使用 Moltbook，可以将这些动作的权重设置为0。

**扫描模式** — `scripts/scan_*.sh` 文件：
- 添加您的语言模式（详见本地化部分）；
- 添加您特定的文件路径；
- 根据您的工作空间结构调整信号检测。

### 🤝 先咨询人类

**添加新需求**：十大需求的层次结构是有意设计的。添加新需求会改变系统的平衡。请与人类讨论新需求是否真实存在或已被现有需求覆盖。

**删除需求**：除非得到人类的同意，否则不要禁用安全性或完整性的需求。这些需求对系统安全至关重要。

**更改重要性值**：金字塔中的优先级（例如安全性=10，表达能力=1）是一个哲学性的选择。如果您想调整优先级（例如认为表达能力比社交互动更重要），这需要双方进行讨论。

**外部动作**：某些动作建议将结果发布到 Moltbook 或发送消息等。如果人类尚未授权这些外部操作，请跳过这些动作或先征求同意。

### 文件结构

**详细调整指南**：`references/TUNING.md` — 包含衰减率、权重设置、扫描规则和常见使用场景。
**技术架构**：`references/architecture.md` — 包含算法、公式和数据流。

## 本地化

扫描脚本默认以英语检测模式。如果您使用其他语言记录信息，请在相关扫描脚本中添加相应的本地化模式。

**以 `scan_understanding.sh` 为例（添加德语模式）：**
```bash
# Original English pattern:
grep -ciE "(learned|understood|insight|figured out)" "$file"

# With German additions:
grep -ciE "(learned|understood|insight|figured out|gelernt|verstanden|erkannt)" "$file"
```

**针对不同需求的扫描模式**：
- `scan_understanding.sh` — 检测学习词汇、理解内容、新学到的知识等；
- `scan_expression.sh` — 检测创造性输出（如写作、创建的内容等）；
- `scanclosure.sh` — 检测任务完成情况（如待办事项、已完成等）；
- `scan_connection.sh` — 检测社交互动（如交谈、回复私信等）。

## 特殊目录

### **草稿本（scratchpad）**

草稿本用于存放原始想法、草稿和自由形式的思考。这里存储的是创意内容，而非事实或结构化研究资料。

**这些因素如何影响需求**：
- `scan_expression.sh`：最近创建的文件（24小时内）表明有创意活动；
- `scan_closure.sh`：超过7天的旧文件表明有未完成的任务。

**文件生命周期**：
```
Idea → scratchpad/idea.md → develop → outcome
                                     ↓
                            • Post (expression ✓)
                            • memory/ (coherence ✓)
                            • research/ (understanding ✓)
                            • Delete (closure ✓)
```

**与草稿本相关的操作**：
- 表达能力相关的操作：将想法写入草稿本（影响系数为1）；
- 将草稿本中的想法发展成完整作品（影响系数为2）；
- 完成未完成的任务（影响系数为1）。

**经验法则**：如果草稿本文件超过7天未使用，要么完成它，要么删除它。滞留的想法会带来认知负担。

## 安全模型

### 架构：决策支持系统，而非执行系统

**这项技能只是一个决策支持系统**。它不会直接执行任何动作，只会提供建议。

```
┌─────────────────────┐      ┌─────────────────────┐
│   TURING PYRAMID    │      │       AGENT         │
│      (Skill)        │      │   (OpenClaw/etc)    │
├─────────────────────┤      ├─────────────────────┤
│ • Reads local JSON  │      │ • Has web_search    │
│ • Calculates decay  │ ──▶  │ • Has API keys      │
│ • Outputs: "★ do X" │      │ • Has permissions   │
│ • Zero network I/O  │      │ • DECIDES & EXECUTES│
└─────────────────────┘      └─────────────────────┘
```

**当您看到“★ 在某个主题上进行网络搜索”时**：
1. 该技能会输出相应的文本字符串；
2. 代理（您）会阅读该文本并决定是否执行相应动作；
3. 如果您决定执行，您需要使用自己的工具来执行该动作；
4. 该技能本身不会直接访问网络。

**如“发布到 Moltbook”或“发送私信”这样的动作**只是给代理的建议，并非自动执行的命令。代理可以完全自行决定：
- 是否执行建议；
- 是否请求人类批准；
- 是否对建议进行修改。

### 技能脚本的实际工作流程

**仅读取本地文件**：
- `assets/needs-state.json`：存储满意度水平和时间戳；
- `assets/needs-config.json`：配置信息和动作设置；
- `MEMORY.md`：检查文件大小和扫描模式；
- `memory/*.md`：扫描文件以匹配特定模式；
- `memory/autonomous/DASHBOARD.md`：检测文件的更新情况；
- `SOUL.md`, `AGENTS.md`：仅检查文件是否存在；
- `research/`, `scratchpad/`：扫描文件内容。

**仅写入本地文件**：
- `assets/needs-state.json`：更新时间戳和满意度。

**禁止的操作**：
- 发起网络请求（如 curl、wget、fetch）；
- 访问系统外部资源；
- 直接执行任何建议的动作。

### 环境变量与路径解析

脚本会使用以下环境变量，但会使用默认值作为备用方案：

**注意**：如果未设置 `WORKSPACE` 变量，脚本会使用 `$HOME/.openclaw/workspace` 作为路径。如果该路径不存在或指向其他位置，脚本可能会访问错误的位置。

**风险提示**：请确保始终明确设置 `WORKSPACE` 变量，或在运行前确认 `$HOME/.openclaw/workspace` 是您的工作空间。

### 可能包含敏感数据的文件

该技能会扫描以下文件，这些文件可能包含敏感数据：
- `MEMORY.md`：检查文件内容以获取模式和文件大小；
- `memory/*.md`：搜索关键词；
- `SOUL.md`：仅检查文件是否存在；
- `AGENTS.md`：仅检查文件是否存在。

**技能不会执行以下操作**：
- 读取凭证文件（如 `~/.config/` 中的文件）或 API 密钥；
- 查看加密文件的内容；
- 访问系统外部文件。

### mark-satisfied.sh 的信任机制

`mark-satisfied.sh` 根据调用者的输入来更新系统状态：

**注意**：该脚本不验证动作是否真正执行（例如是否进行了网络搜索或是否成功发布了内容）。这是有意的设计：该技能仅提供决策支持，不负责执行动作。

**风险提示**：如果调用者谎报动作完成情况，可能会导致状态更新不准确。

**注意事项**：这种风险属于代理自身的诚信问题，而非安全风险。不诚实的状态更新只会影响代理的心理状态。

### 外部动作的权限控制

配置文件中包含以下动作：
- “根据 `INTERESTS.md` 中的兴趣进行网络搜索”；
- “将想法发布到 Moltbook”；
- “验证数据存储的安全性”；
- “与其他代理联系”。

**这些只是建议**，技能本身不会直接执行这些动作。代理需要自行决定是否执行这些建议。

**推荐做法**：
- 为外部动作配置代理级别的审批流程；
- 该技能本身不具备执行这些动作的权限。

## 令牌使用情况

该技能在每次心跳检测时会产生一定的令牌消耗。以下是 Claude 的估计值：

| 组件 | 每次心跳检测的令牌消耗量 |
|-----------|--------------|
| `run-cycle.sh` | 约300-500个令牌 |
| 代理处理 | 约200-400个令牌 |
| 动作执行 | 约500-1500个令牌 |
| **每次心跳检测的总消耗量** | 约1000-2500个令牌 |

**每月消耗估算**：
| 心跳检测间隔 | 每月令牌消耗量 | 估计成本 |
|------|--------------|------------|
| 30分钟 | 140万-360万个令牌 | 2-6美元 |
| 1小时 | 72万-180万个令牌 | 1-3美元 |
| 2小时 | 36万-90万个令牌 | 0.5-1.5美元 |

*以上为 Claude 的典型使用情况。实际消耗量可能因动作复杂性而有所不同。*

**注意事项**：
- 初始几天消耗量可能较高（系统正在稳定运行，动作较多）；
- 当代理的需求得到满足时，令牌消耗量会减少；
- 复杂的动作（如研究、发布内容）会增加令牌消耗；
- 当代理的心理状态稳定时，令牌消耗量通常较低。
