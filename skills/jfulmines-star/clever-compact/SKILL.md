## Clever Compact 的功能  

Clever Compact 解决了所有使用 OpenClaw 的用户都会遇到的三种内存故障问题：  

- **“/new” 模式下的数据丢失”**：当你开始一个新的会话时，代理会表现得好像从未见过你一样。  
- **数据压缩过程中的信息丢失**：随着上下文数据的增加，代理在执行任务时会忘记你之前的操作。  
- **内存数据漂移**：三个会话之前做出的决策会悄然消失，导致代理重复犯同样的错误。  

**v2.0** 是一个原生的 OpenClaw 插件，包含两个主要部分：  

### 第一部分：会话恢复（自动）  
在每次会话开始时（包括使用 `/new` 命令或数据压缩后），Clever Compact 会检查是否存在最近的状态文件。如果存在（且文件生成时间在 72 小时以内），它会在**第一个回合**将文件内容作为系统上下文数据注入代理。这样，代理就能从上次会话的状态中恢复过来，无需重复注入上下文数据，也不会增加每回合的处理开销。  

### 第二部分：状态数据写入（手动触发）  
OpenClaw 目前没有提供数据压缩前的生命周期钩子，因此状态数据的写入需要**手动触发**。你有以下三种选择：  
1. **手动触发**：告诉代理 “执行 Clever Compact 操作并保存状态”。代理会立即生成 `memory/compact-state-YYYY-MM-DD-HHmm.md` 文件。  
2. **预压缩定时任务**：设置一个定时任务，在数据压缩前自动执行状态数据写入：`“预压缩前清理内存并保存数据”。代理会先写入状态数据，然后再进行压缩。  
3. **心跳机制**：在 `HEARTBEAT.md` 文件中添加一个定期写入状态数据的脚本。这种机制可以按需执行，不会因为时间问题导致数据丢失。  

该插件还提供了 `api.fn("clever-compact:write")` 接口，允许你使用其他工具程序化地触发状态数据的写入。

## 安全性与隐私保护  

状态文件以 **明文 Markdown** 格式存储在 `memory/` 目录中，仅保存在本地机器上，不会被插件传输到任何外部。  

**请勿在状态文件中存储以下内容：**  
- API 密钥、令牌或敏感信息  
- 密码或凭证  
- 任何不应被保存在文本文件中的私人信息  

插件会在每个状态文件中添加提示信息，建议定期检查 `memory/compact-state-*.md` 文件，并删除可能被意外保存的敏感内容。  

提供 `api.fn("clever-compact:write")` 接口是为了方便你使用心跳机制、定时任务或其他工具进行程序化的数据写入。请确保调用此接口的自动化脚本仅写入会话上下文数据，避免泄露敏感信息。

## 为什么需要手动触发数据写入？  
数据压缩操作是在达到内存使用上限**之后**执行的。这意味着在压缩发生之前，代理仍然可以正常运行并响应用户指令。通过设置定时任务或在内存使用达到 75% 时自动触发数据写入，你可以确保每次都会有一个完整的状态备份——而无需依赖目前还不存在的操作系统级钩子。  

当 OpenClaw 提供了数据压缩前的钩子后，Clever Compact 会自动使用该钩子。在此之前，使用定时任务和心跳机制是最佳解决方案。

## 从 v2.0/v2.1 版本迁移的方法：**  
- 数据注入方式保持不变。  
- **注意：** v2.0/v2.1 版本会在每个提示回合都写入状态文件，这会增加每回合的处理开销；v2.2.0 版本仅在会话开始时写入一次状态文件，无需进行其他修改。  

## 从 v1 版本迁移的方法：**  
1. 从 `AGENTS.md` 文件中删除与 Clever Compact 相关的配置。  
2. 删除所有引用预压缩操作的配置。  
3. 按照上述步骤 2 和 3 安装插件。  
4. 系统会自动识别并使用新的配置。
