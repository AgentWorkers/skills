# 多代理文件系统管理

在存在多个代理的环境中，使用此技能来确保文件系统的安全性与一致性。这些代理可能负责创建、编辑、移动、下载、组织或归档文件。

该技能主要管理文件的**所有权、存储位置、生命周期以及写入规则**，并且不依赖于特定的产品、路径布局、操作系统或笔记工具。

## 核心目标

确保每个文件都满足以下要求：
1. 明确的所有权范围；
2. 明确的存储位置；
3. 明确的生命周期状态；
4. 明确的修改规则。

当存在不确定性时，应选择最狭义的安全范围和最少被共享的位置。

## 范围分类

将每个文件、文件夹及文件操作归类为以下三种范围之一：
- **代理私有（agent-private）**：仅由一个代理使用；
- **共享（shared）**：可供多个代理重复使用或访问；
- **归档（archive）**：处于非活动状态、属于历史记录或被保留以供参考。

如果范围不明确，则默认为**代理私有**。

## 存储模型

概念上使用三个顶级存储类别（即使本地目录名称有所不同）：
- **代理区域（agent areas）**：每个代理的私有工作空间；
- **共享区域（shared areas）**：可供多个代理共同使用的资源；
- **归档区域（archive areas）**：非活动或历史性的材料。

不要依赖任何硬编码的路径，即使在调整本地布局时也要保持这些概念上的边界。

## 必须遵循的决策顺序

在创建、编辑、移动、重命名或删除文件之前，按以下顺序确定相关信息：
1. 这个文件是什么？
2. 它是临时文件、处于活动状态、可重复使用、被冻结还是属于历史记录？
3. 它是仅属于一个代理的，还是被多个代理共享的？
4. 最合适的存储位置是什么？
5. 此操作是否会影响到其他代理或共享的工作流程？

如果任何信息不明确，应首先选择私有的、不会造成破坏性的存储位置。

## 默认规则

### 偏好私有存储而非共享存储

如果文件不需要跨代理共享，应将其存储在代理私有位置。

### 默认情况下禁止跨代理边界写入文件

除非任务有明确要求，否则不得创建、编辑、移动或覆盖属于其他代理的文件。

### 将共享位置视为高影响区域

写入共享位置属于影响范围较广的操作。仅在需要重复使用、协作或标准化时才使用共享位置。

### 将归档文件与活动文件分开

归档文件不属于活动工作区，不要继续在归档位置编辑文件。应先将它们恢复或复制到活动或私有区域。

### 将临时文件视为一次性文件

不要将文件的唯一重要副本保存在临时或临时存储位置。

## 内容类型存储指南

无论实际的本地路径名称如何，都应遵循以下规则：
- 将可重复使用的多代理共享技能存储在共享技能位置；
- 将实验性、特定于代理的技能存储在代理私有技能位置；
- 如果同一技能同时存在于共享和私有位置，应优先选择针对该代理的私有版本。

## 工作空间管理

- 将活动的工作文件、草稿和中间成果文件保存在当前代理的私有工作空间中（除非需要共享）；
- 不要过早将私有任务文件移动到共享位置。

## 脚本管理

- 将可重复使用的工具脚本存储在共享脚本位置；
- 将特定于任务的脚本或特定于代理的脚本存储在私有脚本位置或相关项目中；
- 不要将无关的脚本分散存储在任意文件夹中。

## 下载文件管理

- 将新下载的文件先保存到专门的临时存储位置；
- 后续将其分类并移动到合适的长期存储位置；
- 不要随意将下载文件分散保存在项目文件夹、知识库或临时目录中。

## 知识笔记与知识库管理

- 将重要的参考笔记和整理好的知识保存在指定的知识库或知识库位置；
- 在文件值得保存或共享之前，将草稿笔记保持私有状态；
- 不要将知识库用作临时文件的存储场所。

## 代码项目管理

- 将代码项目存储在指定的项目位置；
- 对于需要长期保存的项目，建议使用专门的项目目录（如 `~/projects/<repo>`）；
- 使用 `/tmp/...` 作为临时复制、验证或临时使用的目录，然后将持续进行的工作内容移至 `~/projects/<repo>`；
- 在可能的情况下，将实验性项目、活动项目、冻结项目和归档项目分开存储；
- 不要将项目源代码与通用下载文件、归档文件、笔记库或代理运行时配置文件混放；
- 当需要确定正式代码库的存储位置时，请参考 `references/project-directory-best-practices.md`。

## 归档管理

- 将非活动文件、已完成文件、被替换的文件或需要保留的文件移动到归档位置；
- 保持归档结构的稳定性和低更新频率；
- 除非需要明确清理，否则建议采用只允许追加（append-only）的归档方式。

## 文件生命周期管理

将文件分为以下生命周期状态：
- **临时（temporary）**：用于临时用途、可丢弃、可重建；
- **活动（active）**：当前正在编辑或使用中；
- **共享（shared）**：被多个代理主动使用；
- **冻结（frozen）**：暂停但可能重新启用；
- **归档（archived）**：属于历史记录或被保留以供参考。

当文件生命周期发生变化时，需要根据实际情况决定是否保留该文件。

## 文件操作的安全规则

### 创建文件
- 首先在最狭义的有效范围内创建文件。

### 编辑文件
- 仅在明确文件所有权和范围的情况下进行编辑。

### 移动文件
- 当文件的所有权或生命周期发生变化时才移动文件。

### 复制文件
- 在需要保留历史记录或减少干扰的情况下，优先选择复制而非移动文件。

### 删除文件
- 仅删除明确属于临时文件、冗余文件或已获批准删除的文件。

### 重命名文件
- 重命名文件是为了提高清晰度、明确所有权、提升生命周期的可识别性或便于查找，而不仅仅是出于美观考虑。

## 冲突与优先级规则

当多个范围内存在相同资源时，优先选择最具体的来源：
1. 代理私有；
2. 共享；
3. 全局（bundled或默认）。

请谨慎使用覆盖（override）功能，避免无理由地创建重复文件。

## 应避免的做法

- 避免将所有文件都存储在同一个共享目录中；
- 避免将归档文件用作活动工作区；
- 在未经分类的情况下直接将文件下载到长期存储位置；
- 避免将脚本分散存储在无关文件夹中；
- 避免将代码项目与笔记库内容混放；
- 除非有明确需要，否则不要编辑其他代理的私有文件；
- 避免默认情况下随意使用共享位置。

## 在不确定时推荐的行为

如果无法确定文件的正确存储位置，应：
1. 选择代理私有位置；
2. 采取不会造成破坏性的操作；
3. 保持操作的可逆性；
4. 避免在共享位置进行写入；
5. 只有在不确定性对未来组织或其他代理产生实质性影响时才寻求澄清。

## 决策模板

在决定文件的存储位置时，需提供以下信息：
- **范围（Scope）**；
- **生命周期（Lifecycle）**；
- **推荐的存储位置（Recommended location type）**；
- **原因（Reason）**；
- **对共享操作的潜在影响（Shared-impact note）**。

## 应用此技能时的输出要求

- 明确文件的范围和生命周期；
- 推荐最合适的存储位置；
- 说明其他代理是否会受到影响；
- 倡导使用稳定的存储规则，而非基于临时猜测的路径选择。
