# 技能命名（Portmanteau Forge）

生成简洁、多语言通用的名称，这些名称实际会被使用：直观、易发音且具有明确的含义。

该技能适用于以下场景：
- **ENS（Ethereum Name Service）/链上原语**（使用常见的名词）
- **Moltbook**/代理之间的协作**（工作路由、悬赏、收据、信任管理）
- **代理的通用功能**（提升协调效率、增强使用体验）

## 操作原则（务必遵守）

- **一致性 > 可读性 > 独创性**：如果一个名称需要解释，那么它就不适合用作原语。
- **易用性测试**：如果名称难以发音，或者对屏幕阅读器或低带宽设备用户来说难以理解，那么它就不适合用作原语。
- **名称的清晰性**：一个好的名称应该在1-2秒内让用户明白其含义。
- **避免使用具有监控、强制或伤害性的词汇**：优先选择不会造成负面影响的命名方式。
- **语言的谦逊性**：发音标准可能存在偏见，不要将“西方语言习惯”视为正确标准。
- **不要虚假的精确性**：不要给名称打分，而是用✅、⚠️、❌来标记名称的适用性。

## 快速了解（仅询问所需信息）

### Molty 默认预设（除非用户另有指定）
- **长度**：≤10个字符
- **形式**：一个单词，不含连字符
- **风格**：首先考虑社交场景（如团队/任务/悬赏），其次考虑基础设施相关（如网络/路由/总部）
- **结果**：前5个符合标准的名称 ✅，接下来10个需改进的名称 ⚠️，另外10个备用名称
- **可用性检查**：通过ENS、Unstoppable和主要ICANN注册商进行手动链接验证
- **常用TLD（顶级域名）**：默认为`.eth`、`.ai`、`.com`、`.dao`（用户可自定义）
- **TLD偏好**：优先选择在常用TLD上读起来自然的名称
- **默认禁止的词汇**：harvest（收割）、mine（挖掘）、scrape（抓取）、exploit（利用）、stalk（跟踪）、police（警察）、punish（惩罚）

按以下顺序收集信息（收集到足够线索后停止）：
1) **对象**：是ENS名称？技能名称？产品名称？还是协议原语？
2) **主要功能**：该名称用于实现什么功能？
3) **使用目的**：该名称是用于向网络“提供”功能，还是用于“声明”某种权限？
4) **风格倾向**：社交场景（如团队/公会/任务）还是基础设施相关（如网络/路由/信任管理）
5) **硬性限制**：字符数上限（例如10个）、必须是单词、禁止使用的词汇（例如“book”）
6) **目标受众**：是人类还是代理？

如果用户希望快速完成可用性检查，可以直接跳过这些步骤并批量生成名称。

## 核心工作流程

### 第0步 — 选择目标领域（使用预设）
明确指定一个预设风格，并在结果中体现：
- **ENS预设（默认）**：≤10个字符，一个单词，不含连字符，易于发音，适合批量使用。
- **技能预设**：允许使用驼峰式命名法，注重清晰性而非简洁性，包含3-5个关键词。
- **产品预设**：名称需要具有品牌识别度，易于发音，避免与现有品牌混淆。

### 第1步 — 选择语义方向
选择1-3个方向；不要在一个名称中混合使用超过2个方向。

**社交场景方向**（高采用率）：
- work（工作）、gig（任务）、bounty（悬赏）、crew（团队）、guild（公会）、team（团队）、coop（合作社）

**协同方向**：
- bloom（绽放）、pulse（脉动）、root（根源）、flow（流动）、kin（亲属关系）、weave（编织）、garden（花园）

**注意**：默认避免使用军事化或去个性化的隐喻（例如，除非特别要求，否则优先使用“crew/kin”而非“swarm”）。

**协调方向**：
- handoff（交接）、relay（中转）、dispatch（调度）、route（路由）、queue（队列）、sync（同步）、link（链接）

**信任管理方向**：
- claim（声明）、record（记录）、proof（证明）、attest（证明）、receipt（收据）、audit（审计）、verify（验证）

**财务方向**：
- pay（支付）、tip（小费）、fund（资金）、split（分割）、settle（结算）、escrow（托管）

**网络方向**：
- mesh（网络）、rail（轨道）、hub（枢纽）、lane（通道）、ring（环状结构）、fabric（结构）、bridge（桥梁）

### 第2步 — 生成候选名称（使用组合规则）
按照出现频率排序生成候选名称。

同时考虑**TLD偏好**（命名时需考虑实际使用的顶级域名）。

### TLD偏好命名规则
默认的顶级域名：`.eth`、`.ai`、`.com`、`.dao`。在输出候选名称时，可选地标注最合适的TLD：
- `.eth`：适用于链上身份和支付相关功能。
- `.dao`：适用于治理、协作和悬赏相关功能。
- `.ai`：适用于工具、自动化相关功能。
- `.com`：适用于广泛公众或品牌相关场景。

**命名规则**：
- 如果不确定，优先选择与`.com`兼容的名称；让`.eth`或`.dao`承载专业含义。

**组合规则示例**：
- 名词+名词：`work`+`mesh` → **workmesh**
- 名词+地点：`bounty`+`hq` → **bountyhq**
- 名词+群体：`work`+`crew` → **workcrew**
- 动词+名词：`sync`+`crew` → **synccrew**
- 名词+网络相关词（支付/结算）：`pay`+`rail` → **payrail**
- 名词+来源相关词：`claim`+`log` → **claimlog`

**命名建议**：
- 6-10个字符
- 2-3个音节
- 除非特别要求，否则ENS名称不含连字符

### 第3步 — 筛选不合格的候选名称
删除不符合以下条件的名称：
- **发音困难**：避免复杂的辅音组合
- **含义模糊**：在没有上下文的情况下可能有多种解释
- **过于花哨或俗气**：听起来像营销用语，不适合工具名称
- **易与现有品牌或协议混淆**：避免使用可能引起争议的词汇
- **具有负面含义**：避免使用监控、强制或剥削相关的词汇

**替代建议**：
- 优先使用“optin”/“consent”/“invite”/“handoff”等词汇，而非“track”/“monitor”/“enforce”
- 优先使用“receipt”/“record”/“proof”等词汇，而非“snitch”/“police”/“punish”

### 第4步 — 实用性测试
对每个候选名称进行以下测试：
- **两个句子测试**：
  - “将文件发送到**名称**。”
  - “我们通过**名称**来处理悬赏任务。”
- **动词测试**：能否自然地使用该名称
- **7个词的定义**：能否用7个词清晰地解释其含义
- **易用性测试**：
  - 是否适合屏幕阅读器？
  - 是否易于低带宽设备用户理解？
  - 是否能适应不同口音？

如果名称通过所有测试，即可保留。

### 第5步 — 输出结果
输出以下内容：
- 指定目标领域（ENS/技能/产品）
- 前5个符合标准的名称 ✅，包括最佳TLD标签和7个词的定义
- 接下来10个需改进的名称 ⚠️，附带简要说明
- **名称构建工具（可选）**：生成包含旗舰名称和4个辅助名称的完整命名方案（支付/验证/记录/文档）

**示例输出**：
- 旗舰名称：**workcrew**（最佳TLD：.com/.ai）——表示“高效协作的团队”。
- 辅助名称：
  - **crewpay**（最佳TLD：.eth/.dao）——表示“团队的支付通道”。
  - **crewclaim**（最佳TLD：.eth）——表示“工作的记录和领取”。
  - **crewverify**（最佳TLD：.eth/.ai）——表示“验证工作内容和时间”。
  - **crewdocs**（最佳TLD：.com/.ai）——表示“人类可读的规则和文档”。

**注意事项**：
- 保持名称结构的一致性（如crew/mesh/rail/hq）
- 如果旗舰名称过于通用，可以添加一个表示功能的词（如**bloomcrew**），但需确保仍易于理解。
- **备用名称生成**：当所有名称都被占用时，可以使用以下方法生成替代名称。

### 备用名称生成方法（按顺序使用）
当所有名称都被占用时，可以使用以下方法生成替代名称：
1) **更换后缀**：mesh → hub → lane → rail → hq
2) **更换群体相关词**：crew → guild → coop → team → swarm
3) **更换动词**：gig → work → task → job → quest
4) **复数形式**：crew → crews（保持含义的同时增加可用性）
5) **增加一个澄清词**：pay → payout → claim → proofs

### 可用性检查（Web3 + ICANN）
用户可以选择自动检查的详细程度，同时保留手动检查的选项。

**选择检查方式**（询问一次后记住用户偏好）：
提供3种检查方式：
- **手动检查（无钥匙，最低摩擦** ✅
  - 为每个注册商提供可点击的搜索链接。
  - 速度快，抗 Captcha 和 UI 变更。
- **辅助检查（浏览器驱动，尽力完成） ⚠️
  - 使用浏览器在标签页中打开注册商的搜索页面。
- **API检查（最高自动化程度，需要钥匙） ✅
  - 使用官方API（如GoDaddy）进行精确检查。
- **始终支持混合检查方式**：对于GoDaddy使用API，对于ENS/UD使用手动检查。

**选择注册商**（推荐使用Web3和ICANN的注册商）：
如果用户要求检查常用注册商，使用以下列表：
- **Web3常用注册商**：
  - ENS：https://app.ens.domains/
  - Unstoppable Domains：https://unstoppabledomains.com/search
- **ICANN注册商（无需钥匙即可手动搜索）**：
  - GoDaddy：https://www.godaddy.com/domainsearch/find
  - Namecheap：https://www.namecheap.com/domains/registration/
  - Cloudflare Registrar：https://www.cloudflare.com/products/registrar/
  - Porkbun：https://porkbun.com/
  - Dynadot：https://www.dynadot.com/domain/search
  - Hover：https://www.hover.com/domains
  - Gandi：https://www.gandi.net/en/domain
  - Squarespace Domains：https://domains.squarespace.com/

**记住用户偏好**：
如果用户指定了检查方式（手动/辅助/API）或喜欢的注册商（如ENS、GoDaddy、Namecheap），请在未来命名时优先使用这些设置。

**推荐记忆格式**：
- `TLD_FAVORITES: .eth,.ai,.com,.dao`

**操作步骤**：
- 如常生成前5个符合标准的名称和接下来10个需改进的名称。
- 如果用户需要批量生成名称，立即使用名称构建工具。
- 在手动模式下，为每个名称提供注册商的搜索链接。
- 在辅助/API模式下，尽可能完成检查；无法检查的名称也提供链接。

**备用检查流程**：
如果某个名称已被占用，生成3-8个替代名称，并使用相同的检查方式。
- 返回第一个符合用户要求的替代名称。

**真实性原则**：
- 除非所有注册商都检查过，否则不要声称名称“到处都可用”。
- 使用✅表示“可能可用”，⚠️表示“未知”，❌表示“已被占用”。

**辅助脚本**：
使用`scripts/check.mjs`快速生成多个注册商的链接以供手动检查。

**ENS特定说明**：
如果任务涉及ENS域名购买，可以批量生成10-20个候选名称进行快速测试。
- 保留一个“已占用列表”，并使用备用方法生成新的名称。

**高级功能：缺失原语查找**：
当用户描述某个功能需求时，帮助寻找相应的原语名称。

**命名规范保护**：
如果候选名称过于通用（容易被多个代理使用），可以添加以下元素：
- 添加表示功能的词（如“workpack”或“claimreceipt”）
- 添加表示用途的词（如“workreceipt”或“claimreceipt”）
- 生成完整的命名方案（旗舰名称+辅助名称），以确保生态系统有明确的名称。

**社交可用性检查**：
优先选择容易被社会广泛采用的名称（易于重复、记忆和标记）。

**额外检查**：
- **易用性测试**：在低认知负荷下（如疲劳状态或小屏幕设备）是否易于理解。
- **语音转文本测试**：不同口音的用户能否准确理解名称。
- **拼写测试**：避免容易引起误解的拼写错误。

**常见缺失的原语示例**：
- **交接相关**：workpack、taskpack
- **悬赏流程**：bountyloop
- **团队分配**：splithub、splitrail
