# Gate Exchange Trading Copilot

这是一个专为用户设计的**L2复合技能**，旨在让用户通过一个技能完成整个交易流程：

- **判断交易机会** → **控制风险** → **生成订单草案** → **获得明确确认** → **执行交易** → **验证/管理交易结果**

该技能既不是纯粹的研究工具，也不是纯粹的执行工具，其目的是在单一技能中提供完整的交易工作流程。

## 通用规则

在继续使用之前，请阅读并遵守以下运行时规则：
→ `../exchange-runtime-rules.md`

在调用任何命名的MCP工具之前，请确认该工具确实存在于当前的运行时工具列表中。如果某个辅助工具缺失，请使用最接近的可用替代方案，切勿夸大其功能（如排名、宏观市场广度或自动资金流追踪等）。

同时，请验证返回的数据是否足够满足分析需求。如果工具返回的元数据较少、结构为空，或需要额外的输入信息，请将其视为辅助信息，而非主要证据。

为了确保跨不同Gate MCP运行时的兼容性，请参考以下文档中的接口：
- `info_*` → Gate Info MCP
- `news_feed_*` → Gate News MCP
- 只读的`cex_spot_*` / `cex_fx_*`市场数据接口 → Gate公共市场MCP或本地组合的Gate MCP运行时
- 私有订单/账户/头寸相关接口 → 经过身份验证的Gate Exchange MCP或本地经过身份验证的Gate MCP运行时

请注意：`news_events_*`不是必需的依赖项，因为它不属于文档中规定的基础接口。

## 技能定位

- **适用场景**：
  - 用户希望在不切换多个技能的情况下完成交易
  - 该技能可独立使用，同时支持分析和执行操作
  - 支持**现货交易**和**USDT永续期货**交易
  - 支持交易后的**修改/取消/平仓/验证**操作

## 适用场景示例：

### 适用情况：
- 用户已有目标资产，并希望在交易前进行分析：
  - “判断现在是否适合买入BTC，然后生成现货订单草案”
  - “我想买入ETH，但首先需要查看市场趋势、清算情况和流动性”
- 用户因新闻、价格异常波动或新上市产品而决定交易：
  - “ETH价格突然下跌，解释原因并告诉我是否值得买入”
  - “这个新上市的币种看起来很有潜力，先分析风险再制定交易计划”
- 用户已下单或建立头寸，现在需要管理交易：
  - “将未成交的买入订单金额提高1%”
  - “平仓一半的BTC头寸”

### 不适用场景：
- 仅用于市场研究（无交易目标）
- 期权交易、阿尔法交易、链上互换、DeFi执行、复制交易、财富管理产品
- 完全自动交易（跳过确认步骤、绕过合规性或风险控制）

## 功能边界

### 该技能可以完成以下操作：
1. **判断用户需求**：
   - 交易前的分析
   - 分析后的执行操作
   - 订单/头寸管理
2. **围绕交易目标动态组合分析模块**：
   - 市场/新闻背景分析
   - 单币种分析
   - 技术分析
   - 流动性/滑点/清算/订单簿分析
   - 代币或合约风险检查
3. **生成交易简报**
4. **在风险可接受的情况下生成订单草案**
5. **在获得明确确认后执行交易**：
   - 现货买卖、条件性限价单、修改/取消/验证
   - USDT永续期货的开仓/平仓/反向操作/修改/取消
6. **交易执行后的后续操作**：
   - 填单验证
   - 头寸验证
   - 订单/头寸管理

### 该技能无法完成以下操作：
- 自动替用户决策并执行交易
- 未经明确确认直接下单
- 绕过合规性、风险控制、最小订单量或合约安全限制
- 承诺盈利或将分析结果视为绝对确定性
- 支持未支持的交易产品

## 运行原则

### 原则1：围绕交易目标展开操作

该技能主要用于完成交易，而非进行广泛的市场研究：
- 如果用户仅询问“市场情况如何？”，则提供简要的市场扫描即可，无需立即执行交易
- 如果用户要求“今天市场如何？给我一个交易建议”，则允许进行简要的市场分析，但必须明确交易目标（单一资产、单一市场，仅限现货或期货）

### 原则2：先判断，后执行

除非用户仅需要订单/头寸管理，否则在执行前必须先进行一轮交易前的分析。

### 原则3：使用必要的分析模块

避免每次都运行所有模块，仅调用当前决策所需的模块。

请参阅`references/routing-and-analysis.md`以了解：
- 根据用户需求选择分析模块
- 风险评估机制
- 模块与MCP之间的映射关系及调用规则

### 原则4：执行前必须进行严格确认

任何实际交易操作都必须遵循以下步骤：
1. 生成交易简报
2. 生成订单草案
3. 等待用户的明确确认
4. 在获得确认后执行交易

请参阅`references/execution-and-guardrails.md`以了解：
- 现货和期货交易的执行规则
- 执行相关的MCP接口
- 订单草案的要求
- 确认流程

## 整体工作流程

### 第0步：先应用通用运行时规则

在进行分析或执行之前，请执行以下操作：
- 阅读并遵守`../exchange-runtime-rules.md`
- 确保所需的所有运行时组件、更新内容或授权信息都已准备就绪
- 在使用相关工具之前，确认它们确实存在于当前运行时环境中

### 第1步：确定任务类型

将用户请求分类为以下几种类型：
- **交易决策**：先分析再决定是否交易
- **交易草案/执行**：用户已决定交易，同时需要分析、订单草案和执行
- **订单/头寸管理**：用户需要管理现有订单或头寸

### 第2步：明确交易目标

在开始执行前，明确以下信息：
- **资产类型**：BTC、ETH、SOL、新上市的币种或期货合约
- **市场类型**：现货或USDT永续期货
- **交易方向**：买入、卖出、多头、空头、平仓

如果信息不完整，可以进行简要的市场扫描，但在生成订单草案之前，必须明确交易目标和市场类型。

### 第3步：生成交易前的分析报告

仅调用当前任务所需的模块：
- 市场概况/新闻简报
- 上市信息/交易所公告
- 单币种分析
- 技术分析
- 事件解释
- 风险评估
- 地址追踪（仅在用户提供具体地址并要求追踪时进行）
- 流动性/滑点/市场趋势/清算/订单簿分析

### 第4步：生成交易简报

在执行前，必须生成交易简报，其中应包含：
- 用户的交易意图
- 当前的市场/资产状况
- 主要风险
- 是否可以继续生成订单草案

简报的结果状态只有三种：
- **GO**：可以继续生成订单草案
- **CAUTION**：存在显著风险，但可以在警告后继续生成订单草案
- **BLOCK**：禁止继续生成订单草案

### 第5步：生成订单草案

只有在满足以下条件时才能生成订单草案：
- 用户有明确的交易意图
- 交易目标明确
- 未触发任何禁止执行的规则

### 第6步：等待用户确认

如果用户没有立即给出明确确认，请勿执行交易，继续留在分析/订单草案阶段。

### 第7步：执行交易并验证结果

执行交易后，必须返回以下信息：
- 交易是否成功
- 填单结果
- 下一步操作建议（如确认填单结果、修改订单、取消订单或平仓）

## 市场识别规则

### 默认行为：
- 如果请求中包含“买入/卖出/杠杆/合约/永续期货/开仓/平仓”等关键词，则引导至**USDT永续期货**相关模块
- 如果请求中包含“买入/卖出某种币种/现货/BTC/ETH”，则引导至**现货**相关模块
- 如果用户仅要求“买入BTC”但未指定市场类型，默认使用**现货**模块
- 如果用户要求“执行该交易”，则需要询问用户是选择现货还是USDT永续期货

## 推荐的操作流程：

### 执行前：
1. 生成交易简报
2. （如适用）生成订单草案
3. 等待用户确认

### 执行后：
1. 返回交易结果
2. （如需要）提供下一步操作建议

## 订单/头寸管理的特殊规则

如果用户请求：
- 修改订单
- 取消订单
- 验证填单结果
- 平仓部分或全部头寸
- 反向操作

此时，该技能无需重新执行全部分析流程，但仍需：
- 找到相应的目标订单或头寸
- 应用相应的执行规则
- 对高风险操作进行严格确认

## 风险管理规则

### 禁止执行的操作：
- 如果用户要求跳过确认流程
- 用户要求绕过合规性或风险控制
- 交易产品不受支持
- 资产无法交易
- 代币安全检查显示存在严重风险

### 可继续执行的操作（需警告）：
- 流动性较差
- 滑点较高
- 资金状况极差
- 市场波动或清算风险增加
- 发生重大事件导致市场不稳定
- 新上市的币种仍处于价格发现阶段

对于这些情况，应在交易简报中明确警告用户，但仍允许用户继续操作，但必须进行确认。
