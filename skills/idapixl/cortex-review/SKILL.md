# Cortex 代码审查

通过将代码、设计或提案与你在 Cortex 中积累的知识进行对比，来进行审查。

## 工作流程

### 1. 基于记忆进行审查

在阅读待审查的内容之前，先在 Cortex 中查询相关主题：

```
query("待审查的领域或系统")
````

阅读查询结果。这些结果包含了过去的决策、模式以及经验教训，这些信息将有助于你的审查。

### 2. 根据上下文进行审查

在阅读内容时，将其与 Cortex 返回的信息进行对比：

- **它是否符合**已建立的模式和过去的决策？
- **它是否**与已知的方法有所不同（无论是有意还是无意）？
- **它是否**引入了值得记录的新模式？

### 3. 记录你的发现

**新模式**：如果某项内容引入了值得记住的信息：
```
observe("新的缓存层采用了写通（write-through）机制，并设置了 5 分钟的过期时间（TTL）——对于这种以读取为主的工作负载来说非常有效")
````

**未解决的问题**：如果某些内容不清楚：
```
wonder("为什么他们绕过了内部服务的速率限制器？")
````

**认知更新**：如果某项内容改变了你的理解：
```
believe(concept_id, "更新后的理解", "来自本次审查的证据")
````

### 4. 输出格式

```markdown
## 代码审查 — 基于记忆进行

### 与已知模式一致的情况
- [与 Cortex 中的上下文相符的部分]

### 不一致的情况
- [存在差异的部分，以及关于这些差异是故意还是无意的解释]

### 值得记录的新模式
- [值得使用 `observe()` 方法记录的新方法]

### 未解决的问题
- [需要进一步思考的问题]
````

## 为什么这很重要

如果没有基于记忆的背景知识，每次审查都将从零开始。你可能会忽略诸如“我们三周前尝试过这种方法，结果导致了延迟飙升”或“出于合规性考虑，我们选择了这种模式而非另一种方案”这样的信息。Cortex 会帮助你记住这些背景信息。
```
