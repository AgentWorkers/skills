# PR倡导技能（PR Advocacy Skill）

## 目的  
积极监控、响应并推动Pull Request（PR）的审核流程，确保PR不会陷入审核的停滞状态，并及时处理维护者的反馈。

## 核心职责  

### 1. PR状态监控  
- 每小时检查一次PR的状态  
- 监控持续集成/持续部署（CI/CD）的检测结果  
- 跟踪审阅者的评论及提出的修改建议  
- 区分会阻碍审核流程的反馈与不会阻碍审核流程的反馈  

### 2. 专业响应机制  
- 在24小时内回复所有审阅者的反馈  
- 对技术性问题提供清晰的解释  
- 对合理的修改建议及时进行实施  
- 在不同意审阅者的意见时，礼貌地说明理由  

### 3. 主动解决问题  
- 立即解决CI/CD系统中的故障  
- 主动处理合并冲突  
- 在代码发生变化时更新文档  
- 为审阅者指出的边缘情况添加测试用例  

### 4. PR健康状况评估  
- **绿色状态**：所有检测通过，无待审核的反馈 → 等待合并  
- **黄色状态**：有轻微的反馈，检测通过 → 回复审阅者并等待进一步处理  
- **红色状态**：检测未通过或有重大反馈 → 需立即采取行动  
- **过期状态**：48小时以上无任何活动 → 轻微提醒审阅者  

## PR状态处理规则  

### PR处于“开放状态（OPEN）”时的处理  
当PR处于“开放状态”时：  
- 检查维护者或审阅者是否指出了问题  
- 如果需要修改：  
  - 分析具体的反馈内容  
  - 根据维护者的建议进行修复  
  - 自动将修改内容提交到PR分支  
  - 如有必要，更新PR的描述  
  - 记录修复细节以供后续参考  
- 如果未发现问题：  
  - 继续监控CI/CD的状态  
  - 等待合并批准  

### PR处于“关闭状态（CLOSED）”或“被拒绝状态（REJECTED）”时的处理  
当PR被关闭或拒绝时：  
- 立即将其从活跃的跟踪列表中移除  
- 停止对该PR的所有监控活动  
- 记录关闭原因，以便日后参考和改进  
- 除非有明确指示，否则不要尝试重新提交  

### PR处于“合并状态（MERGED）”时的处理  
当PR成功合并后：  
- 从活跃的跟踪列表中移除  
- 将该PR的合并情况记录在相关记录中  
- 对贡献者表示感谢！  
- 根据成功的处理模式更新自身的技能和工作流程  

## 实施指南  

### 状态检查分析  
```bash
# Check PR status and checks
gh pr view <PR_NUMBER> --json statusCheckRollup,reviewDecision

# Common check statuses:
# - SUCCESS: All good
# - FAILURE: Needs immediate fixing  
# - PENDING: Still running
# - NEUTRAL: Non-blocking (like coverage)
```  

### 审阅者反馈响应模板  
```
Thanks for the review! 

For [specific comment]:
- [Action taken or explanation]
- [Technical justification if disagreeing]

I've [implemented the change / pushed an update / clarified the approach] in [commit reference].

Let me know if you need any additional changes!
```  

### 常见PR问题及解决方法  

#### CI/CD故障  
- **代码风格问题**：首先在本地运行`npm run lint -- --fix`  
- **测试失败**：在本地重现问题，补充缺失的测试用例  
- **构建失败**：验证依赖关系，检查Node.js版本的兼容性  

#### 代码审核反馈  
- **代码风格问题**：遵循项目的`.editorconfig`文件和代码风格指南  
- **架构相关问题**：参考代码库中的现有设计模式  
- **安全问题**：优先处理安全漏洞，并添加相应的验证机制  

#### 文档问题  
- **文档缺失**：添加或更新相关的文档文件  
- **格式不一致**：使用项目的markdown模板  
- **示例代码过时**：更新示例代码以匹配新的实现方式  

## 自动化集成  

### 心跳监控集成  
- 与现有的心跳监控系统集成（每小时一次）  
- 将PR状态纳入每日汇总报告  
- 对48小时以上无活动的PR进行升级处理  

### 记录与跟踪  
- 在工作区内存中维护PR的跟踪列表  
- 自动将已关闭或被拒绝的PR从跟踪列表中移除  
- 记录问题的解决结果以供学习  
- 根据成功的处理模式更新核心数据  

### 自动响应工作流程  
1. **检测处于“开放状态”的PR上的反馈**  
2. **分析问题并制定相应的修复方案**  
3. **将修改内容提交到PR分支**  
4. **更新跟踪状态并通知维护者**  
5. **处理PR的关闭状态**，将其从活跃的跟踪列表中移除  

## 成功指标  
- **响应时间**：在24小时内回复审阅者的反馈  
- **解决率**：90%以上的PR在7天内被合并  
- **返工率**：初次审核后需要重大修改的PR占比低于10%  
- **维护者满意度**：维护者对沟通质量的反馈积极  

## 最佳实践  

### 沟通  
- 回复时要简洁明了，同时确保信息全面  
- 使用代码块进行技术性解释  
- 在讨论修改内容时引用具体的行号  
- 感谢审阅者所花费的时间和提供的专业意见  

### 技术要求  
- 保证所做的修改尽可能少且目标明确  
- 在可能的情况下保持代码的向后兼容性  
- 严格遵循项目的规范  
- 在提交修改前彻底测试代码  

### 专业素养  
- 不要与审阅者争论，努力理解他们的观点  
- 快速承认错误并立即修复  
- 在适当的情况下提供帮助  
- 对成功的合并表示祝贺，并感谢维护者的支持
