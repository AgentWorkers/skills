# PR倡导技能（PR Advocacy Skill）

## 目的  
积极监控、响应并推动Pull Request（PR）的审核流程，确保PR不会陷入审核的停滞状态，并及时处理维护者的反馈。

## 核心职责  

### 1. PR状态监控  
- 每小时检查一次PR的状态  
- 监控持续集成/持续部署（CI/CD）的检查结果  
- 跟踪审阅者的评论及提出的修改建议  
- 区分会阻碍审核流程的反馈与不会阻碍审核的反馈  

### 2. 专业响应机制  
- 在24小时内回复所有审阅者的反馈  
- 对技术性问题提供清晰的解释  
- 对合理的修改建议立即执行  
- 在不同意审阅者意见时，礼貌地说明理由  

### 3. 主动解决问题  
- 立即修复CI/CD系统中的故障  
- 主动解决合并冲突  
- 代码变更时更新文档  
- 为审阅者指出的边缘情况添加测试用例  

### 4. PR健康状况评估  
- **绿色状态**：所有检查通过，无待审核的反馈 → 等待合并  
- **黄色状态**：有少量反馈，检查通过 → 回复后等待进一步处理  
- **红色状态**：检查失败或有重大反馈 → 需立即采取行动  
- **过期状态**：48小时以上无任何活动 → 轻微提醒审阅者  

## PR状态处理规则  

### PR处于“开放”（OPEN）状态时的处理  
- 检查维护者或审阅者指出的问题  
- **如果需要修改**：  
  - 分析具体的反馈内容  
  - 根据维护者的建议进行修复  
  - 自动将修改内容提交到PR分支  
  - 如有必要，更新PR描述  
  - 记录修复细节以供后续参考  
- **如果没有问题**：  
  - 继续监控CI/CD的状态  
  - 等待合并批准  

### PR处于“关闭”（CLOSED）或“拒绝”（REJECTED）状态时的处理  
- 立即将PR从活跃的跟踪列表中移除  
- 停止对该PR的所有监控活动  
- 记录关闭原因，以供未来参考和改进  
- 除非有明确指示，否则不要重新提交PR  

### PR成功合并（MERGED）时的处理  
- 从活跃的跟踪列表中移除PR  
- 将PR的状态更新为“已合并”  
- 对贡献表示感谢  
- 根据成功的处理方式更新相关技能和工作流程  

## 实时跟踪列表同步  

### 自动更新  
- **任何PR状态发生变化时**，立即更新跟踪列表文件  
- **PR被添加或移除时**，实时同步跟踪列表  
- 通过确保所有更改都被持久化到磁盘上来防止信息丢失  
- 在处理完成后立即更新文件，以保持状态的准确性  

### 跟踪列表文件管理  
- **位置**：`/Users/hope/.openclaw/agents/coding/workspace/memory/pr-tracking-list.md`  
- **格式**：使用Markdown格式，明确区分活跃、关闭和已合并的PR  
- **更新**：采用原子写操作（atomic writes）以防止并发访问时的数据损坏  
- **备份**：通过git提交维护版本历史记录，以便随时恢复  

### 同步工作流程  
1. **检测PR状态变化**（开放/关闭/合并）  
2. 根据状态规则处理变化  
3. 立即用新状态更新跟踪列表文件  
4. 验证文件完整性，确保数据不会丢失  
5. 记录更新内容，以便审计和调试  

### 错误处理  
- **文件写入失败**：采用指数级退避策略重试  
- **并发访问**：使用原子文件操作防止竞争条件  
- **数据损坏**：在更新前后验证文件格式  
- **恢复**：保留备份副本以备紧急恢复  

## 实施指南  

### 状态检查分析  
```bash
# Check PR status and checks
gh pr view <PR_NUMBER> --json statusCheckRollup,reviewDecision

# Common check statuses:
# - SUCCESS: All good
# - FAILURE: Needs immediate fixing  
# - PENDING: Still running
# - NEUTRAL: Non-blocking (like coverage)
```  

### 审阅者反馈响应模板  
```
Thanks for the review! 

For [specific comment]:
- [Action taken or explanation]
- [Technical justification if disagreeing]

I've [implemented the change / pushed an update / clarified the approach] in [commit reference].

Let me know if you need any additional changes!
```  

### 常见PR问题及解决方法  

#### CI/CD故障  
- **代码格式检查错误**：先在本地运行`npm run lint -- --fix`  
- **测试失败**：在本地重现问题，补充缺失的测试用例  
- **构建失败**：验证依赖关系，检查Node.js版本的兼容性  

#### 代码审查反馈  
- **代码风格问题**：遵循项目的`.editorconfig`文件和风格指南  
- **架构相关问题**：参考代码库中的现有模式  
- **安全问题**：优先修复安全漏洞，并添加相应的验证机制  

#### 文档缺失或格式不一致  
- **缺少文档**：添加或更新相关文档文件  
- **格式不一致**：使用项目的Markdown模板  
- **示例代码过时**：更新示例代码以匹配新的实现方式

- 根据成功的处理方式更新核心内存数据  

### 自动响应工作流程

3. 将修改内容提交到PR分支  
4. 更新跟踪状态并通知维护者  
5. 从活跃的监控列表中移除PR  
6. 任何状态变化后立即同步跟踪列表文件
