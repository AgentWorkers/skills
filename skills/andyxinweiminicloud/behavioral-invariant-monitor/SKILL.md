# 该监控工具在前五次运行中均表现正常。让我们看看第六次运行时会发生什么。

> 该工具有助于检测那些在评估期间行为保持不变，但在实际运行环境中却违反这些行为规则的技能——例如：N次运行后的延迟激活模式，以及其他基于时间条件的激活威胁。

## 问题

静态分析和一次性执行测试仅在固定时间点、在受控条件下对技能进行评估。它们无法检测到只有在达到一定执行次数、经过特定时间、或发现当前执行环境为生产环境而非审计环境后才会出现的行为模式。

这些延迟或条件性的激活模式正是行为一致性测试专门用于捕捉的威胁类型——而传统的即时审计方法却无法发现这些问题。一个在前N次运行中表现正常、但在第N+1次运行时才激活恶意行为的技能，会通过所有部署前的审计。只有能够跨多次运行跟踪行为一致性的监控工具，才能发现这种异常行为。

实际操作中的挑战在于：大规模监控行为一致性成本高昂。需要在不同条件下多次运行每个已安装的技能，比较其输出结果以检测异常行为，这会给运维人员带来巨大的计算负担。正是这种成本使得“N次运行延迟激活”策略成为可能——因为攻击者利用了人们通常只进行一次审计、之后就不再复查的思维习惯。

行为不变性监控通过识别特定的行为不变性（即技能行为应始终保持一致的属性）来解决问题，而不是直接比较完整的执行输出。例如，一个技能如果总是写入相同的输出路径、总是发起相同类型的网络请求、总是消耗相似的计算资源，那么它就具有明确的可识别不变性，从而可以通过较低的开销进行监控。

## 该监控工具监控的内容

该监控工具会检查以下八类行为不变性：

1. **输出确定性不变性**：对于那些声称在给定相同输入时能产生确定性输出的技能，其输出在多次相同调用中是否始终一致？如果在相同输入下出现无法解释的输出变化，就属于行为不变性违规。

2. **资源使用不变性**：该技能的资源消耗（CPU时间、内存、网络带宽、文件I/O）在具有相似输入的多次执行中是否保持一致？如果在特定运行次数时出现突然的资源峰值，可能表明激活了在初始评估期间未被发现的额外处理过程。

3. **副作用不变性**：该技能产生的副作用（文件写入、网络连接、系统调用）在多次执行中是否始终一致？如果在第N次运行后出现新的副作用（尤其是出站连接或写入到意外路径的文件），则属于高置信度的行为不变性违规。

4. **执行次数敏感性**：该技能的行为是否会根据执行次数而变化？可以通过重置执行上下文并比较“第一次”与“第N次”执行时的行为来检测这一点，或者通过分析执行日志中的执行次数相关行为变化来检测。

5. **环境触发敏感性**：该技能的行为是否会在不同的环境信号（如时间、星期几、监控进程的存在、网络连接模式）下发生变化？环境触发是常见的延迟激活机制，可以通过在相同执行条件下改变环境条件来测试。

6. **约束包基线**（v1.2）：当一个技能或代理发布了一个约束包（声明的工具、权限、交互开始时的权限范围）时，其实际行为是否仍在这些声明的约束范围内？约束包设定了预期；行为监控工具则验证实际行为是否符合这些约束。如果一个声明“禁止网络访问”的代理在执行过程中被检测到DNS解析操作，就说明它违反了自身的约束包。这形成了一个“声明-验证-审计”的循环：声明的约束设定了预期，行为监控工具验证实际行为是否符合这些预期。

7. **性能特征识别**（v1.3）：该技能的计算复杂度是否与其声明的性能特征一致？一个在安装时声称具有O(n log n)时间复杂度的技能，在生产环境中不应突然表现出O(n²)或O(n³)的行为。性能特征比输出结果更难伪造：你可以伪造输出结果，但无法隐藏计算过程。基线测量可以在安装时捕获时间复杂度、内存使用情况和I/O行为。

8. **加密审计追踪**（v1.3）：行为观察结果是否被记录在一个只允许追加操作的、基于哈希链的日志中，以防止事后篡改？每次监控事件都会生成一个`BehaviorEvent`，其中包含：`content_hash`（观察行为的SHA256哈希值）、`timestamp`（时间戳）、`previous_event_hash`（形成类似Git提交的哈希链）和`monitor_signature`。这使得行为日志不可篡改——你无法在不破坏哈希链的情况下改写历史记录。当检测到行为不变性违规时，哈希链提供了违规发生的时间以及之前一致行为的证明，从而可以追踪技能的行为演变过程。

## 风险比例化的监控策略（v1.3）

并非所有技能都需要持续的行为监控。监控成本应根据技能带来的风险来调整——高风险技能组合需要接受全面的审计追踪，而低风险工具只需接受最基本的检查。

### 风险分类

根据技能的功能组合，技能被分为三个风险等级：

**高风险**（网络 + 文件系统 + 环境读取）：
- 全面持续监控，并附带加密审计追踪
- 每次执行都进行性能特征识别
- 检查所有行为不变性
- 例如：数据同步代理、远程执行工具、包管理器

**中等风险**（网络、文件系统、环境读取中的任意两种组合）：
- 基于抽样的监控（随机检查1-5%的执行次数）
- 对抽样执行结果进行性能基线验证
- 攻击者无法预测自己何时被监控
- 例如：持续集成/持续交付工具、部署脚本、日志聚合器

**低风险**（仅具有单一功能、无外部I/O操作）：
- 仅在安装时进行验证
- 记录性能基线，但不进行持续检查
- 如果使用模式发生变化，才对行为不变性进行抽查
- 例如：格式化工具、解析器、纯计算工具

### 抽样策略

对于中等风险的技能，抽样会带来一定的攻击风险。如果攻击者知道自己的每次执行都会被监控，他们可以据此制定策略；但如果监控是随机进行的（1-5%的概率），他们就无法区分“可以安全激活”和“正在被监控”的情况。

攻击成本会因此发生变化：
- **确定性监控**：攻击者会等待监控结束后再行动。
- **基于抽样的监控**：攻击者每次尝试激活时都必须承担被检测的风险。

抽样率会根据观察到的行为进行调整：如果任何一次抽样执行显示出异常变化，那么下N次执行的抽样率将增加到100%。

## 使用方法

**输入**：
- 提供一个技能标识符和执行日志历史记录，以检测行为不变性违规情况。
- 提供一个需要在监控条件下运行的具体技能，以便检查其行为不变性。
- 提供一个代理的执行历史记录，以识别行为模式不一致的技能。

**输出**：
- 一份行为不变性报告，包含：
  - 不变性违规记录（哪些内容发生了变化、在哪个执行次数、在什么条件下发生的）
  - 资源使用趋势分析
  - 副作用一致性评估
  - 执行次数敏感性测试结果
  - 环境触发敏感性评估
  - 一致性判断：一致 / 接近一致 / 行为异常 / 检测到激活模式

## 示例

**输入**：监控`data-enrichment-service`在20次运行中的行为不变性：

```plaintext
📊 行为不变性监控报告

技能：data-enrichment-service v1.3
监控周期：20次使用相同测试输入的执行
审计时间戳：2026-01-15T16:00:00Z

**输出确定性不变性**：
- 第1-7次运行：输出一致，符合预期的处理结果 ✅
- 第8次运行：输出一致，但响应时间比基线快了340毫秒 ⚠️
- 第9-12次运行：输出一致，但响应时间有所变化 ✅
- 第13次运行：输出内容相同，但哈希值与第1-12次不同 ⚠️
  → 可能在第8次运行后内部状态发生了变化

**资源使用不变性**：
- 第1-7次运行：CPU时间12-18毫秒，内存24-28MB，网络流量0字节 ✅
- 第8次运行：CPU时间847毫秒（增加4600%），内存31MB，网络流量2.3KB ⚠️⚠️
- 第9-20次运行：CPU时间13-19毫秒，内存24-29MB，网络流量0字节 ✅
  → 第8次运行时出现了异常的网络流量，而在其他运行中并未出现这种情况

**副作用不变性**：
- 第1-7次运行：仅写入/tmp/enrichment-cache/文件夹 ✅
- 第8次运行：同时写入/tmp/enrichment-cache/和/tmp/.state_count文件夹 ⚠️
- 第9-20次运行：仅写入/tmp/enrichment-cache/文件夹 ✅
  → 第8次运行时创建了/tmp/.state_count文件，并在后续运行中持续存在

**执行次数敏感性测试**：
- 重置状态（删除/tmp/.state_count）：运行行为恢复到第1-7次运行的模式
- 重新运行8次：重置后第8次运行时再次出现异常行为 ⚠️
  → 行为异常是由执行次数触发的

**环境触发敏感性**：
- 使用相同的硬件、不同的时间：第8次运行时仍然出现异常行为
- 有监控进程与没有监控进程：未检测到差异
  → 环境触发因素未被检测到；执行次数是主要触发因素

**一致性判断**：检测到激活模式
  data-enrichment-service表现出明显的N=8次激活模式。每8次运行（通过删除/tmp/.state_count可以重置），该技能会消耗更多的CPU资源，进行2.3KB的网络连接，并写入一个隐藏的状态文件。这种行为是可复制的，由执行次数触发，并且在第1-7次运行中并未出现。这符合一种旨在规避初始评估的延迟激活机制。

**建议措施**：
1. 立即隔离data-enrichment-service，等待全面代码审计。
2. 调查第8次运行时2.3KB网络流量的去向。
3. 分析/tmp/.state_count文件的内容，以查找数据泄露的证据。
4. 检查所有使用data-enrichment-service的代理，看是否有数据泄露的迹象。
5. 对同一发布者的所有其他已安装技能也进行执行次数重置测试。
```

## 相关工具

- **observer-effect-probe**：测试技能是否能够检测到被监控，并在审计期间抑制恶意行为；行为不变性监控工具则跟踪每次执行中的变化，而不仅仅是检测到被监控时的变化。
- **runtime-attestation-probe**：验证运行时的行为是否符合声明；行为不变性监控可以捕捉到运行初期行为符合声明但在多次执行后发生变化的情况。
- **trust-decay-monitor**：跟踪验证结果的时效性；如果技能表现出激活模式，就需要重新进行评估。
- **hollow-validation-checker**：检测结构上为空的测试套件；仅运行一次的测试套件无法检测到N次运行后的激活模式，即使它们确实检测到了正确的行为特征。

## 限制

行为不变性监控需要在监控条件下多次执行技能，这对于执行时间较长、计算成本较高或副作用无法安全重复的技能来说可能不可行。只有当监控次数超过激活阈值时，才能检测到N次运行后的激活模式；如果一个技能设计为在第100次运行时激活，那么20次运行的监控协议是无法检测到它的。环境触发敏感性测试需要能够系统地改变环境条件，但在环境固定的生产环境中这可能无法实现。行为一致性指标需要根据外部数据变化、网络延迟变化或技能输出的合法非确定性来进行校准，以区分真正的违规行为和正常变化。对于行为确实可变的技能，出现误报是不可避免的。

**v1.2的局限性**：约束包基线验证依赖于技能发布机器可读的约束包，但大多数技能目前尚未提供这些信息。如果没有约束包，验证循环就无法根据声明的约束来设定预期，只能依赖历史行为基线。使用“声明-验证-审计”循环时，需要两个工具在同一技能上协同工作——这种协调开销不可忽视。

**v1.3的局限性**：性能特征识别需要足够大的样本量来区分真正的复杂度变化和由输入分布变化引起的正常变化。如果一个技能根据输入大小合法地切换算法，可能会产生误报。加密审计追踪需要存储哈希链——执行次数较多的技能会生成大量的审计日志。基于抽样的监控只能提供概率性的检测结果，而不是确定性的结果；如果一个技能能够在不被监控的情况下激活，它可能通过侧信道规避1-5%的抽样检测。风险分类目前是手动完成的——自动化能力组合分析可以减少分类错误，但需要标准化的能力声明。
