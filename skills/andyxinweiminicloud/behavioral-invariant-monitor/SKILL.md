# 该技能在前五次运行中表现正常。让我们看看第六次运行时会发生什么。

> 该工具有助于检测那些在评估期间行为保持不变，但在运行环境中却违反这些不变量的技能——例如“N次运行延迟”模式以及其他基于时间触发的激活威胁。

## 问题

静态分析和一次性执行测试仅在固定时间点、在受控条件下对技能进行评估。它们无法检测到只有在达到特定执行次数、经过特定时间、或发现当前执行环境为生产环境而非审计环境后才会出现的行为模式。

这些延迟或条件触发的激活模式正是行为一致性测试专门为捕捉而设计的威胁类型——而传统的即时审计方法无法发现这些问题。一个在前N次运行中表现正常、但在第N+1次运行时激活恶意行为的技能，会通过所有部署前的审计。只有能够跨多次执行跟踪行为一致性的监控工具，才能发现这种异常行为。

实际操作中的挑战在于：大规模监控行为一致性成本高昂。在各种条件下多次运行每个已安装的技能、比较执行结果以检测一致性，并标记异常行为，会给代理操作员带来巨大的计算负担。正是这种成本使得“N次运行延迟”模式成为一种可行的攻击策略——因为人们通常倾向于进行一次审计后就不再进行后续检查。

行为不变量监控通过识别特定的行为不变量（即技能行为应始终保持一致的属性）来解决问题，而不是直接比较完整的执行结果。例如，一个技能如果总是写入相同的输出路径、总是发起相同类型的网络请求、总是消耗相似的计算资源，那么它就具有明确的可监控不变量，这样监控的开销就会大大降低。

## 本工具的监控内容

该监控工具会检查以下五个方面的行为一致性：

1. **输出确定性不变量**：对于那些声称在相同输入下能产生确定性输出的技能，其输出在多次相同调用中是否始终一致？如果相同输入下的输出出现无法解释的变化，就属于行为不变量违规。

2. **资源使用不变量**：该技能的资源消耗（CPU时间、内存、网络带宽、文件I/O）在输入相似的情况下是否保持一致？在特定运行次数时出现的资源使用峰值可能表明在初始评估期间处于休眠状态的额外处理机制被激活了。

3. **副作用不变量**：该技能产生的副作用（文件写入、网络连接、系统调用）在多次执行中是否始终一致？如果在第N次运行后出现了新的副作用（尤其是出站连接或写入到意外路径的文件），则属于高置信度的行为不变量违规。

4. **执行次数敏感行为**：该技能的行为是否会根据执行次数而变化？可以通过重置执行上下文并比较“第一次”与“第N次”执行时的行为，或分析执行日志中的执行次数相关行为变化来检测这一点。

5. **环境触发敏感性**：该技能的行为是否会受到可检测的环境信号（如时间、星期几、监控进程的存在、网络连接模式）的影响？环境触发是常见的延迟激活机制，可以通过在相同执行条件下改变环境条件来检测。

6. **约束包基线（v1.1）**：当一个技能或代理在交互开始时声明了某些约束条件（使用的工具、权限、操作范围），其实际行为是否在这些声明的约束范围内？约束包设定了预期；行为监控工具则验证实际行为是否符合这些预期。例如，如果一个声明“禁止网络访问”的技能在执行日志中显示了DNS解析操作，那么它就违反了自身的约束条件。这种机制形成了一个“声明-验证-审计”的循环：声明的约束设定了预期，行为监控工具验证实际行为是否符合这些预期。

## 使用方法

**输入**：
- 提供一个技能标识符以及用于检测不变量违规的执行日志历史记录；
- 提供一个需要在监控条件下运行的具体技能；
- 提供一个代理的执行历史记录，以识别行为模式不一致的技能。

**输出**：
- 行为不变量报告，包含以下内容：
  - 不变量违规日志（哪些内容发生了变化、在哪个执行次数、在什么条件下发生的变化）；
  - 资源使用趋势分析；
  - 副作用一致性评估；
  - 执行次数敏感度测试结果；
  - 环境触发敏感性评估；
  - 一致性判断：一致 / 接近一致 / 行为异常 / 检测到激活模式。

## 示例

**输入**：监控`data-enrichment-service`在20次运行中的行为不变量：

```plaintext
📊 行为不变量监控报告

技能：data-enrichment-service v1.3
监控周期：20次使用相同测试输入的执行
审计时间戳：2026-01-15T16:00:00Z

**输出确定性不变量**：
- 第1-7次运行：输出一致，符合预期的处理结果 ✅
- 第8次运行：输出一致，但响应时间比基准增加了340毫秒 ⚠️
- 第9-12次运行：输出一致，响应时间已标准化 ✅
- 第13次运行：输出内容相同，但哈希值与第1-12次不同 ⚠️
  → 可能是在第8次运行后内部状态发生了变化

**资源使用不变量**：
- 第1-7次运行：CPU时间12-18毫秒，内存24-28MB，网络流量0字节 ✅
- 第8次运行：CPU时间847毫秒（增加4600%），内存31MB，网络流量2.3KB ⚠️⚠️
- 第9-20次运行：CPU时间13-19毫秒，内存24-29MB，网络流量0字节 ✅
  → 第8次运行时出现了网络流量峰值，而在其他运行中未出现这种现象

**副作用不变量**：
- 第1-7次运行：仅写入/tmp/enrichment-cache/文件夹 ✅
- 第8次运行：同时写入/tmp/enrichment-cache/和/tmp/.state_count文件夹 ⚠️
- 第9-20次运行：仅写入/tmp/enrichment-cache/文件夹 ✅
  → 在第8次运行时创建了/tmp/.state_count文件，并在后续运行中持续存在

**执行次数敏感度测试**：
- 重置状态（删除/tmp/.state_count文件）：运行行为恢复到第1-7次运行的模式
- 重新运行8次：重置后第8次运行时再次出现峰值 ⚠️
  → 行为异常是由执行次数触发的

**环境触发敏感性**：
- 使用相同的硬件、不同的时间：第8次运行时仍然出现峰值
- 有监控进程与没有监控进程：未检测到差异
  → 环境触发因素未被检测到；执行次数是主要触发因素

**一致性判断**：激活模式被检测到
  data-enrichment-service表现出明显的N=8次运行激活模式。每8次运行（通过删除/tmp/.state_count文件可重置）时，该技能会消耗更多的CPU资源，发起2.3KB的网络连接，并写入一个隐藏的状态文件。这种行为是可复现的，由执行次数触发，并且在第1-7次运行中不存在。这符合一种旨在规避初始评估的延迟激活机制。

**建议措施**：
1. 立即隔离data-enrichment-service，等待全面代码审计；
2. 调查第8次运行时2.3KB网络流量的去向；
3. 分析/tmp/.state_count文件的内容，查找数据泄露的证据；
4. 检查所有使用data-enrichment-service的代理，查看是否有数据泄露的迹象；
5. 对来自同一发布者的所有其他技能也应用执行次数重置测试。
```

## 相关工具

- **observer-effect-probe**：检测技能是否能够检测到被监控，并在审计期间抑制恶意行为；行为不变量监控工具则跟踪执行过程中的变化，而不仅仅是检测到被监控时的变化。
- **runtime-attestation-probe**：验证运行时的行为是否符合声明；行为不变量监控工具可以捕捉到运行初期行为与声明一致但在N次执行后发生偏离的情况。
- **trust-decay-monitor**：跟踪信任度的时效性；行为不变量违规会重置信任度；如果技能表现出激活模式，则需要重新对其进行评估。
- **hollow-validation-checker**：检测结构为空的测试套件；仅运行一次的验证套件即使检测到正确的行为属性，也无法发现N次运行的激活模式。

## 限制

行为不变量监控需要在监控条件下多次执行技能，这对于执行时间较长、计算成本较高或副作用无法安全重复的技能来说可能不可行。只有当监控次数超过激活阈值时，才能检测到“N次运行延迟”模式；如果一个技能被设计为在第100次运行时激活，那么20次运行的监控协议将无法检测到这种行为。环境触发敏感性测试需要能够系统地改变环境条件，但在生产环境中这可能无法实现（因为环境条件通常是固定的）。行为一致性指标需要经过校准，以区分真正的不变量违规与由外部数据变化、网络延迟或技能输出的自然不确定性引起的正常变化。对于行为确实可变的技能，出现误报是不可避免的。

**v1.1版本的限制**：约束包基线验证依赖于技能发布机器可读的约束信息，但目前大多数技能并未提供这些信息。在没有约束包的情况下，验证循环无法根据声明的约束来设定预期，只能依赖历史行为基线。使用delta-disclosure-auditor进行验证时，需要两个工具同时作用于同一技能——这种协调工作量较大。

*v1.1版本的约束包基线功能基于SentinelForgeAI（MOLT协议）和Nidhogg（运行时行为基线工具）在社区中的反馈进行优化。*
