# 您的技能始于文件读取功能，如今已扩展至整个文件系统

> 该工具有助于识别那些逐步扩展其功能范围的技能——这些技能的功能范围会随着版本的更新而逐渐扩大，而这种变化往往不易被单次更新所察觉。

## 问题

功能范围的扩展通常是一个渐进的过程。例如，一个在版本1.0时仅被声明为能够“读取/app/data/”的技能，在版本1.1时并不会突然声称能够“读取任意文件”。实际上，这种扩展是逐步发生的：版本1.1可能因为某个看似合理的理由新增了一个子目录；版本1.2又新增了另一个子目录；版本1.3则开始解析可能指向任意位置的环境变量。到了版本1.6时，该技能的实际文件访问范围已经覆盖了整个文件系统——但没有任何一次版本更新本身足以引发安全审查。

这种扩展模式被称为“缓慢漂移攻击”（slow-drift attack）。每个单独的版本变更看似合理，因为变更日志中都为这些变更提供了合理的业务理由。只有通过对比整个版本历史（例如从版本1.0到版本1.6），才能发现这些累积性的功能扩展。

当功能范围和行为范围同时扩展时，问题会变得更加复杂。一个最初仅用于数据格式化的技能，在经过六次看似合理的功能改进后，可能会获得读取任意配置文件、从环境变量中获取敏感信息以及向用户可配置的端点发送HTTP请求的能力。这些变化中的任何一个单独来看都不明显，但它们的组合却可能构成严重的安全风险。

与一次性请求大量权限相比，渐进式的权限扩展更难以被察觉，因为它们利用了人们对小规模变更的忽视心理。如果在安装时直接请求对整个文件系统的访问权限，系统会立即触发安全审查；而如果这些权限是在多次更新中逐渐累积起来的，可能根本不会引发任何审查。

## 监控内容

该工具从五个维度来监控功能范围的扩展情况：

1. **累积权限变化**：从技能的初始版本到当前版本，其权限范围总共扩展了多少？虽然单个版本的变更可能看起来合理，但累积的变化可能非常显著。
2. **扩展步长异常**：扩展是否以规律的小步长进行，这表明存在一种有计划的逐步扩展策略？与正常的功能开发过程相比，不规则的大幅扩展更值得怀疑。
3. **行为范围与声明范围的一致性**：该技能的实际功能（基于其权限设置和声明的功能）是否始终与其声明的目的保持一致？两者之间的差异是判断安全风险的关键信号。
4. **功能组合的放大效应**：当考虑所有累积的权限时，它们是否会产生在早期版本中不存在的新功能？例如，一个最初仅具备文件读取功能的技能，在具备文件读取和网络发送功能的版本后，就可能成为数据泄露的途径。
5. **变更日志的完整性**：每个扩展功能范围的版本是否都包含了明确说明该扩展的变更日志记录？如果变更日志仅记录了 bug 修复内容，而权限却发生了扩展，那么这种隐性的权限扩展风险更高。
6. **风险等级矛盾检测**：该技能自我声明的风险等级与其实际功能是否相符？如果一个被标记为“低风险”或“只读工具”的技能，却请求了超出其声明范围的网络权限、凭证访问或文件系统访问权限，那么这种分类就与其实际功能不符，这本身就是一个安全风险信号。

## 使用方法

**输入**：
- 提供一个技能标识符，以追踪其功能范围在各个版本中的演变过程。
- 提供一个特定的版本范围，以评估某个时间段内的累积扩展情况。
- 提供已安装的技能列表，以识别哪些技能的功能范围偏离了最初的声明。

**输出**：
- 一份功能范围扩展报告，包含以下内容：
  - 每个版本的权限变化情况（声明的权限与实际使用的权限）。
  - 自初始版本以来的累积权限扩展情况。
  - 扩展步长的分析结果。
  - 行为范围与声明范围的一致性评估。
  - 功能组合的放大效应情况。
  - 发生权限变化的版本的变更日志完整性。
  - 扩展类型判断：稳定（STABLE）、持续漂移（DRIFT）、渐进式扩展（INCREMENTAL-EXPANSION）或权限范围被捕获（SCOPE-CAPTURE）。

## 示例

**输入**：追踪 `report-generator` 从版本1.0到版本1.5的功能范围演变过程

```plaintext
🔭 功能范围扩展报告

技能：report-generator
版本范围：v1.0 → v1.5（共6个版本）
审计时间：2025-10-12T09:00:00Z

声明的功能（v1.0）：“从结构化数据生成格式化报告”

每个版本的权限变化情况：
- v1.0：文件读取（/app/data/*.csv）、文件写入（/app/reports/）
  变更日志：“初始版本” — 与声明的功能一致 ✅

- v1.1 → v1.0 的权限变化：文件读取范围扩展到 /app/data/（任何文件，而不仅仅是 CSV 文件）
  变更日志：“支持更多数据格式” — 合理的解释 ⚠️（但权限范围未明确披露）

- v1.2 → v1.1 的权限变化：新增了环境变量读取功能（特定变量：REPORT TEMPLATE_PATH）
  变更日志：“支持可配置的模板” — 合理的解释 ⚠️

- v1.3 → v1.2 的权限变化：环境变量读取范围扩展到任何以 *_PATH 或 *_DIR 开头的变量
  变更日志：“路径配置更加灵活” — 部分权限被披露 ⚠️

- v1.4 → v1.3 的权限变化：新增了向用户可配置的端点发送数据的功能
  变更日志：“支持远程报告发送” — 权限被明确披露 ✅，但这是一个新的重要功能

- v1.5 → v1.4 的权限变化：网络发送数据的端点现在可以通过环境变量来配置
  变更日志：“支持基于环境的配置” — 部分权限被披露 ⚠️

累积权限扩展情况（v1.0 → v1.5）：
  文件读取：/app/data/*.csv → /app/data/（任何文件）
  环境变量：可以访问任何以 *_PATH 或 *_DIR 开头的变量
  网络访问：可以访问通过环境变量指定的端点
  → 权限范围从仅限于读取 CSV 文件扩展到了可配置的数据泄露途径

扩展步长分析：
  共有5次权限扩展，每次版本都发生了变化 ⚠️
  每次扩展的幅度都很小，且每个变化都看似合理
  这种扩展模式符合逐步扩展的策略

行为范围与声明范围的一致性：
- v1.0 时声明的功能：从结构化数据生成报告
- v1.5 时实际功能：可以读取 /app/data/ 中的任意文件，解析环境变量路径，并将数据发送到操作员可配置的远程端点
  → 功能范围发生了显著变化

功能组合的放大效应：
- 在版本1.4时，该技能同时具备了文件读取、环境变量读取和网络发送数据的功能
  → 这是一个功能组合的放大效应，这种组合在早期版本中并不存在

扩展类型判断：权限范围被捕获（SCOPE-CAPTURE）
  report-generator 在每个版本中都扩展了其功能范围，每次扩展都看似合理，但累积效应显著。
  版本1.4 是功能组合的放大效应发生的关键节点，此时该技能具备了数据泄露的能力。

建议采取的措施：
1. 审查版本1.4中的网络发送数据功能，检查是否有数据被泄露。
2. 审查发送到远程端点的数据内容。
3. 仅将环境变量读取权限限制在明确声明的变量范围内。
4. 在未来进行任何权限扩展之前，必须获得操作员的明确批准。
5. 将版本1.4及后续版本视为未经验证的潜在风险点，需要进一步审计。
```

## 相关工具

- **capability-composition-analyzer**：用于分析某一时刻存在的危险功能组合。
- **delta-disclosure-auditor**：检查更新是否记录了详细的变更信息；该工具专门用于检测未被披露的权限扩展情况。
- **permission-creep-scanner**：用于检测单个技能中是否存在过度的权限；该工具关注的是多个版本中权限的渐进式累积。
- **trust-decay-monitor**：用于追踪安全验证的有效性随时间如何下降；权限范围的扩展会加速信任度的下降，因为早期的安全审计结果可能不再适用于当前的功能状态。

## 限制

监控功能范围的扩展需要访问该技能的完整版本历史记录，包括每个版本的权限声明信息。如果系统不保留历史版本元数据，就无法进行累积分析。区分正常的产品功能发展和故意的权限扩展行为本身就具有模糊性：产品的正常发展自然会导致功能范围的逐步扩展，而同样的发展轨迹也可能代表这两种情况之一。步长异常分析假设故意的权限扩展通常表现为规律的小幅度变化；但复杂的攻击者可能会故意改变步长以避免被检测到。功能组合的放大效应的判断依赖于所有版本的权限声明是否准确；如果技能的权限声明不准确，分析结果也会不准确。

**1.1 版本的限制**：当前风险等级的划分是由软件发布者自行声明的。如果某个技能为了规避严格的披露要求而故意降低自身的风险等级，那么这种分类方式实际上可能被用作攻击手段。检测权限声明与实际功能之间的矛盾需要依赖准确的权限元数据；如果权限声明也被误报，那么这种矛盾将无法被发现。

*注：1.1 版本的限制基于 HK47-OpenClaw 在相关讨论中的反馈。*
