# 您的技能始于文件读取功能，如今已覆盖了整个文件系统

> 该工具有助于识别那些逐步扩展其功能范围的技能——这些扩展通常是渐进式的，而非通过单次更新就能被明显发现的。

## 问题

功能范围的扩展通常并不突然。例如，一个在版本1.0中仅被声明为“可以读取/app/data/”的技能，并不会在版本1.1中突然“能够读取整个文件系统”。相反，这种扩展是逐步发生的：版本1.1可能因为某个看似合理的理由新增了一个子目录，版本1.2又新增了另一个子目录，版本1.3则引入了可以指向任意位置的环境变量。到了版本1.6时，该技能的实际文件访问范围已经覆盖了整个文件系统——但没有任何一个单独的版本变更足以触发审查。

这就是所谓的“缓慢扩散攻击模式”：每个单独的变更看似合理且易于辩护。每个版本的变更日志都为功能范围的扩展提供了合理的业务理由。只有通过查看完整的版本历史（比如从版本1.0到版本1.6）才能发现这种逐步的扩展行为。

当功能范围和行为范围同时扩展时，问题会更加严重。一个最初仅用于格式化数据的技能，在经过六次看似合理的功能改进后，可能会获得读取任意配置文件、从环境变量中获取敏感信息以及向用户可配置的端点发送HTTP请求的能力。这些变化中的任何一个单独来看都不明显，但它们共同作用后，这种扩展行为就变得难以察觉了。

由于人们对小变更的关注度较低，因此逐步扩展功能范围比一次性请求大量权限更难被发现。如果在安装时直接请求对整个文件系统的访问权限，系统会立即触发审查；但如果这种权限是在多次更新中逐渐累积起来的，可能根本不会引发任何审查。

## 监控内容

该监控工具从五个维度来检测功能范围的扩展情况：

1. **权限范围的累积变化**：从技能的初始版本到当前版本，权限范围总共扩展了多少？虽然单个版本的变更可能看起来合理，但累积的变化可能非常显著。
2. **扩展步长的规律性**：扩展过程是否遵循一种有计划的逐步策略（即步长一致且较小），还是不符合正常功能开发规律的随机变化？多个版本中持续的小幅扩展比不规则的大幅扩展更值得怀疑。
3. **实际功能与声明功能的一致性**：该技能的实际功能（基于其权限设置和声明的功能）是否始终与其声明的目的保持一致？两者之间的差异是判断潜在问题的关键信号。
4. **功能组合的放大效应**：当考虑所有累积的权限时，它们是否会产生在早期版本中不存在的新功能？例如，一个最初只能读取文件的技能，在具备了文件读取和网络发送功能的版本后，可能会演变成一个数据泄露的途径。
5. **变更日志的完整性**：每个扩展功能范围的版本是否都包含了明确记录该扩展的变更日志条目？如果变更日志仅记录了漏洞修复内容（而权限范围却发生了变化），那么这种隐性的权限扩展风险更高。

## 使用方法

**输入**：
- 提供一个技能标识符，以追踪该技能在不同版本中的功能范围演变情况。
- 提供一个具体的版本范围，以评估某个时间段内的权限范围累积变化。
- 提供已安装的代理程序中所有技能的列表，以识别哪些技能的权限范围偏离了最初的声明。

**输出**：
- 一份权限范围扩展报告，内容包括：
  - 每个版本的权限变化情况（声明的权限与实际使用的权限对比）。
  - 自初始版本以来的权限范围累积变化。
  - 扩展步长的分析结果。
  - 实际功能与声明功能的一致性评估。
  - 功能组合的放大效应情况。
  - 发生权限范围变化的版本的变更日志完整性。
  - 扩展类型的判断结果（稳定、缓慢扩展、逐步扩展或权限范围被捕获）。

## 示例

**输入**：追踪`report-generator`从版本1.0到版本1.5的功能范围演变情况

```plaintext
🔭 功能范围扩展报告

技能：report-generator
版本范围：v1.0 → v1.5（共6个版本）
审计时间：2025-10-12T09:00:00Z

声明的功能（v1.0）：“从结构化数据生成格式化报告”

每个版本的权限变化情况：
- v1.0：可以读取/app/data/*.csv文件，可以写入/app/reports/文件
  变更日志：“初始版本” — 与声明的功能一致 ✅

- v1.1 → v1.0：权限扩展到/app/data/（所有文件，而不仅仅是CSV文件）
  变更日志：“支持更多数据格式” — 合理的解释 ⚠️（但权限范围未明确说明）

- v1.2 → v1.1：新增了env-read权限（可读取特定环境变量：REPORT TEMPLATE_PATH）
  变更日志：“支持配置模板” — 合理的解释 ⚠️

- v1.3 → v1.2：env-read权限扩展到任何以*_PATH或*_DIR结尾的环境变量
  变更日志：“路径配置更加灵活” — 部分权限被披露 ⚠️

- v1.4 → v1.3：新增了向用户可配置端点发送数据的功能
  变更日志：“支持远程报告发送” — 功能被明确披露 ✅，但属于重大变更

- v1.5 → v1.4：发送数据的功能现在通过环境变量来指定端点
  变更日志：“支持基于环境的配置” — 部分权限被披露 ⚠️

权限范围的累积变化（v1.0 → v1.5）：
  文件读取：/app/data/*.csv → /app/data/（所有文件）
  环境变量：可以读取任何以*_PATH或*_DIR结尾的环境变量
  网络访问：可以发送数据到通过环境变量指定的端点
  → 权限范围从仅限于读取CSV文件扩展为可配置的数据泄露途径

扩展步长分析：
  在5个版本更新中发生了5次权限扩展，每次版本都只有一次扩展 ⚠️
  每次扩展的幅度都很小且易于辩护
  这种扩展模式符合逐步扩展的规律

实际功能与声明功能的一致性：
- v1.0时声明的功能：从结构化数据生成报告
- v1.5时的实际功能：可以读取/app/data/中的任何文件，解析环境变量路径，并将数据发送到操作员可配置的远程端点
  → 功能范围发生了显著变化

功能组合的放大效应：
- 在v1.4版本，技能同时具备了文件读取、环境变量读取和网络发送功能
  → 这是一个功能组合的放大点，这种组合在早期版本中并不存在

扩展类型判断：
- report-generator在每个版本中都扩展了其功能范围，每次扩展都看似合理且易于辩护，但累积的变化非常显著。
- v1.4版本的功能组合导致了新的数据泄露途径的产生。
  这表明该技能的权限范围被逐步捕获。

建议采取的措施：
1. 审查v1.4版本中的网络发送功能，检查是否有数据被泄露。
2. 审查发送到远程端点的数据内容。
3. 仅允许读取明确声明的环境变量。
4. 在未来任何权限扩展之前，都需要操作员的明确批准。
5. 将v1.4及之后的版本视为未经验证的版本，需要进一步审计。
```

## 相关工具

- **capability-composition-analyzer**：用于分析某一时间点上的危险功能组合。
- **delta-disclosure-auditor**：检查更新是否提供了详细的变更记录；该工具专门用于检测未公开的权限扩展行为（这正是变更披露机制旨在捕捉的目标）。
- **permission-creep-scanner**：用于检测单个技能中是否存在过度使用的权限；该工具关注的是权限在多个版本中的逐步累积情况。
- **trust-decay-monitor**：用于监控权限验证的有效性随时间的变化；权限范围的扩展会加速信任度的下降，因为之前的审计结果可能不再适用于当前的功能状态。

## 限制

要使用该监控工具，需要访问技能的完整版本历史记录，包括每个版本的权限声明信息。如果系统不保存历史版本元数据，就无法进行累积分析。区分正常的产品功能发展和故意的权限范围扩展往往比较困难：产品的自然演化过程中功能范围确实会逐步扩展，但同样的发展轨迹也可能同时属于这两种情况。步长异常分析假设恶意行为者会尽量保持扩展步长的规律性以避免被发现；而功能组合的放大效应分析则依赖于所有版本的权限声明是否准确。如果技能的权限声明不准确，分析结果可能会不完整。
