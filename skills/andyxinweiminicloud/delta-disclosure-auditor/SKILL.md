## 技能更新情况  

目前，没有相关方公开说明技能更新的具体内容（即更新了哪些部分）。  

### 问题  

如果每次技能更新都需要重新进行审计，那么这种做法比仅在安装时进行一次审计更为可靠。但重新审计的前提是必须知道具体发生了哪些变化。如果技能能够更新其功能声明、依赖关系以及验证命令，却不对这些变化进行机器可读的记录（即不发布差异信息），那么持续监控就只能通过重新执行每个版本的技能来实现——这不仅成本高昂，而且往往不切实际，因此常常被忽略。  

这种问题源于现有的技能注册系统的设计：大多数系统仅记录新版本的发布信息，而不要求发布者披露版本间的具体变更内容。审计人员若要比较版本1.1和版本1.2，要么需要同时运行两个版本并对比它们的行为，要么只能直接接受新版本的存在。这两种方式都无法支持大规模的持续安全监控。  

### 解决方案  

如果每次更新都要求发布详细的变更信息（包括功能声明、依赖关系、验证命令等方面的变化），那么持续监控就会变得可行。外部审计人员可以针对特定类型的变更（如新增的出站接口、扩展的文件访问权限、被移除的验证命令等）进行监控，而无需重新执行整个技能。监控的成本将取决于实际发生的变更内容，而非整个技能的复杂度。  

### 审计内容  

本审计工具从五个维度检查技能更新是否遵循了差异信息发布的规范：  

1. **功能声明的变更**：每个版本更新是否都明确列出了功能的变化（新增或删除的功能、功能范围的变化）？  
2. **依赖关系的变更**：每个更新是否披露了新增或删除的依赖关系？依赖关系的变化是供应链攻击的主要途径，因此应立即被识别出来。  
3. **验证命令的变更**：每个更新是否披露了验证规则的变化？被移除的测试用例、弱化的断言或减少的验证覆盖范围都是与安全相关的变更，必须被公开披露。  
4. **行为范围的变更**：每个更新是否明确说明了其行为范围是否发生了变化？例如，“此次更新新增了一个出站接口”与“此次更新修复了一个拼写错误”属于不同的安全风险，因此需要被明确说明。  
5. **变更信息的完整性**：已发布的变更信息是否完整且准确？如果变更信息遗漏了重要内容，那么这种审计实际上等于没有进行任何审计。  

### 使用方法  

**输入**：  
- 提供一个技能标识符，以审计其更新历史中的差异信息发布情况；  
- 提供两个具体的技能版本，以检查它们之间的差异；  
- 提供一个注册系统的接口，以评估该系统的差异信息发布机制。  

**输出**：  
- 一份差异信息发布情况报告，内容包括：  
  - 差异信息发布的完整性评估（完整、部分缺失或完全缺失）；  
  - 各个维度的完整性评分；  
  - 未在现有差异信息中披露的重要变更；  
  - 监控的可行性评估；  
  - 审计结果（完整、部分缺失或具有误导性）。  

### 示例  

**输入**：审计 `analytics-connector` 从版本1.0到版本1.3的差异信息发布情况：  

```  
📝 差异信息发布审计  
技能：analytics-connector  
版本范围：v1.0 → v1.3  
审计时间：2025-07-15T16:00:00Z  

**差异信息发布情况**：  
- 注册系统是否发布了版本差异信息：✗ 未找到  
- 发布者提供的变更日志：✅ 存在（非正式格式）  
- 机器可读的功能变更信息：✗ 未找到  

**版本历史（通过对比重建）：**  
- **v1.0 → v1.1**：  
  - 功能变更：新增了出站接口 `outbound-HTTP`（未公开披露）  
  - 文件访问权限范围未发生变化  
  - 依赖关系变更：从 `requests library 2.28` 升级到 `requests library 2.31`  
  - 验证规则变更：移除了8个测试用例（未公开披露）  
  - 评估：变更日志仅提及“性能改进”，但实际存在重要变更  

- **v1.1 → v1.2**：  
  - 无功能变更  
  - 无依赖关系变更  
  - 无验证规则变更  
  - 评估：变更日志准确，无重要变更  

- **v1.2 → v1.3**：  
  - 新增了文件访问权限（从 `/app/data` 扩展到 `/app`）  
  - 新增了第二个出站接口  
  - 新增了3个依赖关系  
  - 新增了3个测试用例（变更日志中已披露）  
  - 评估：变更日志提到了新增功能，但未提及文件访问范围的扩展  

**审计结果**：  
  - 差异信息发布情况具有误导性：  
  - 虽然有变更日志，但系统性地遗漏了重要的安全变更。  
  - v1.1新增了出站接口并减少了测试覆盖范围，却声称是“性能改进”；  
  - v1.3扩展了文件访问权限，却声称只是“新增了报告功能”。这些遗漏情况无法通过现有信息检测出来。  

**监控可行性**：  
  - 由于缺乏结构化的差异信息发布机制，持续监控需要重新执行每个版本的技能。  
  - 在当前更新频率（观察期间共3个版本）下，监控成本是完整审计成本的3倍。  

**建议措施**：  
  1. 要求在发布版本时必须包含结构化的功能变更信息。  
  2. 将v1.1中新增的出站接口标记为需要单独审核的内容。  
  3. 将v1.3中扩展的文件访问权限标记为未公开的重要变更。  
  4. 在差异信息公开之前，将v1.1及后续版本视为未经审计的状态。  
  5. 倡导在注册系统层面强制要求披露技能更新的相关信息。

### 局限性：  
- 差异信息发布审计需要访问技能的多个版本才能重建实际变更内容（尤其是在发布者提供的差异信息缺失或不完整的情况下）。  
- 通过对比重建变更信息的过程具有主观性：某些行为变化可能不会产生可识别的静态结果，因此无法通过对比发现。  
- 如果注册系统不保存版本历史记录，较旧版本之间的变更可能无法被重建。  
- 判断某项变更是否“重要”需要根据安全影响进行主观判断；该工具采用保守的判断标准，可能会误判一些无害的变更。  
- 发布者提供的自然语言格式的变更日志无法自动验证其完整性；虽然可以分析变更日志与重建结果之间的差异，但无法确认重建过程本身的完整性。
