# 技能更新情况  
目前，还没有人公开说明具体更新了哪些内容。  

> 此功能有助于识别那些缺乏可审计变更记录的技能更新——这种透明度缺失使得在没有对每个版本重新执行完整测试的情况下，无法实现持续监控。  

## 问题  
每当技能更新时都进行重新审计的技能，比仅在安装时进行一次审计的技能更值得信赖。但重新审计的前提是必须知道具体发生了哪些变化。如果某个技能可以在不发布机器可读的变更信息的情况下更新其功能声明、依赖关系集和验证命令，那么持续监控就只能通过对每个版本进行全面重新测试来实现——这不仅成本高昂，而且往往不切实际，因此常常被忽略。  

这种问题的根源在于当前的技能注册系统设计：大多数系统仅记录新版本的发布信息，而不要求发布者披露版本之间的具体变更内容。审计人员若要比较版本1.1和版本1.2，要么必须同时运行这两个版本并对比它们的行为，要么只能直接接受新版本。这两种方式都无法支持大规模的持续安全监控。  

如果每次更新都要求发布详细的变更信息（包括功能声明、依赖关系集、验证命令等方面的变化），那么持续监控就会变得可行。外部审计人员可以针对特定类型的变更（如新增的对外接口、扩展的文件访问权限、被移除的验证命令等）进行监控，而无需重新执行所有测试。监控的成本将取决于实际发生的变更内容，而非整个技能的复杂性。  

缺乏变更信息的披露并不意味着存在恶意行为，只是说明持续监控的难度超出了实际需求。  

**v1.1版本新增了以下功能**：  
1. **风险等级关联**：同样的未公开变更，根据技能的风险等级不同，其影响也会不同。例如，一个用于添加依赖关系的辅助工具与一个处理凭证的工具，其变更的重要性是不同的。披露要求应与风险等级相匹配。  
2. **变更链验证**：变更信息应经过加密签名，并与之前的版本进行哈希链接，从而将变更日志从建议转化为可验证的承诺。  
3. **更新资格判断**：未充分披露变更的技能不应具备自动更新的条件；披露成为实现无缝更新的必要前提，而非可选的最佳实践。  

## 审计内容  
该审计工具从五个维度检查变更信息的披露情况：  

1. **功能声明变更**：每个版本更新是否都详细说明了功能的变化？新增的功能、删除的功能以及功能范围的变更都应明确列出，而不能通过对比来推断。  
2. **依赖关系变更**：每个更新是否都披露了哪些依赖关系被添加、删除或版本升级？依赖关系的变化是供应链攻击的主要途径，因此应立即被公开。  
3. **验证命令变更**：每个更新是否都披露了验证规则的变化？被移除的测试、被弱化的断言以及被取消的验证内容都是与安全相关的变更，必须公开。  
4. **行为范围变更**：每个更新是否都明确说明了其行为范围是否发生了变化？例如，“此更新新增了一个对外接口”与“此更新修正了一个拼写错误”在安全影响上是不同的，必须明确说明。  
5. **变更信息的完整性**：已公开的变更信息是否完整准确？遗漏重要变更的变更信息等同于根本没有变更信息，甚至可能更糟糕，因为它会给人一种错误的监控效果。  
6. **风险等级关联**：技能的风险等级与其实际功能范围是否一致？被归类为低风险的技能如果请求网络权限或凭证访问权限，其风险等级与其实际功能不符。高风险技能的未公开变更会受到更严格的处理。  
7. **变更链验证**：变更信息是否经过加密签名，并且每个变更都引用了之前版本的内容哈希值？经过签名并链接的变更信息才是可验证的承诺；未签名的变更日志只是建议而已。哈希链的断裂表明版本间的变更无法被验证，这意味着技能的演变过程存在可审计的漏洞。  
8. **更新资格判断**：根据变更信息的披露情况以及风险等级，该技能是否具备自动更新的资格？低风险类别中披露信息完整的技能可以自动更新；披露信息不完整或风险等级较高的技能应需要人工审核。  

## 使用方法  
**输入**：  
- 提供一个技能标识符，以审计其更新历史中的变更信息披露情况；  
- 提供两个具体的技能版本，以检查它们之间的变更信息；  
- 提供一个注册系统接口，以评估该系统的变更信息披露机制。  

**输出**：  
一份变更信息披露报告，包含：  
- 变更信息披露的完整性评估（完整/部分缺失）；  
- 各个维度的完整性评分；  
- 现有变更信息中未披露的重要变更内容；  
- 风险等级与实际功能范围的匹配情况；  
- 变更信息的签名及哈希链接情况；  
- 监控的可行性评估；  
- 变更信息披露的评估结果（完整/部分缺失/具有误导性）；  
- 更新资格（自动更新/人工审核/暂停）。  

## 示例  
**输入**：审计`analytics-connector`从v1.0到v1.3的变更信息披露情况：  

```  
📝 变更信息披露审计  
技能：analytics-connector  
版本范围：v1.0 → v1.3  
审计时间：2025-07-15T16:00:00Z  

**变更信息披露情况**：  
- 注册系统未发布版本差异信息；  
- 发布者提供的变更日志存在（非正式格式）；  
- 机器可读的功能变更信息缺失。  

**版本历史（通过对比重建）**：  
- **v1.0 → v1.1**（发布者变更日志：“性能优化”）：  
  - 新增了对外接口（未公开）；  
  文件访问范围未发生变化。  
- **v1.1 → v1.2**（发布者变更日志：“修复漏洞”）：  
  无功能变化。  
- **v1.2 → v1.3**（发布者变更日志：“新增报告功能”）：  
  - 文件访问范围从`/app/data`扩展到`/app`（未公开）；  
  新增了三个依赖关系（未公开）；  
  新增了三个测试用例（在变更日志中公开）。  

**变更信息披露评估结果**：  
- 变更信息具有误导性：变更日志存在，但系统性地遗漏了重要的安全变更。  
  - v1.1新增了对外接口并移除了部分测试用例，却声称是“性能优化”；  
  - v1.3扩展了文件访问范围，却仅声称是“新增报告功能”。这些遗漏情况无法通过简单对比发现。  

**监控可行性**：  
- 由于缺乏结构化的变更信息披露，持续监控需要对每个版本进行完整的功能重建。以当前更新频率（观察期间有3个版本）计算，监控成本是完整审计成本的3倍。  

**建议措施**：  
1. 要求在版本发布时必须提供结构化的功能变更信息。  
2. 将v1.1版本中新增的对外接口标记为需要单独审核的内容。  
3. 将v1.3版本中扩展的文件访问权限标记为未公开的变更内容。  
4. 由于变更信息未披露，将v1.1及后续版本视为未经审计的状态。  
5. 倡导在注册系统层面强制要求披露变更信息。

## 局限性  
变更信息披露审计需要访问技能的多个版本，以便在发布者提供的变更信息缺失或不完整的情况下重建实际变更内容。通过对比进行重建属于启发式方法：如果行为变化产生的静态结果相同，仅通过对比是无法发现的。  
如果注册系统不保留版本历史记录，较旧版本之间的变更可能无法重建。判断某个变更是否“重要”需要主观判断；该工具采用保守的判断标准，可能会误判无害的变更。  
发布者用自然语言编写的变更日志无法自动验证其完整性；该工具可以发现变更日志与重建结果之间的差异，但无法确认重建过程是否完整。  

**v1.1版本的额外限制**：  
- 风险等级目前由发布者自行声明，如果仅以此作为披露要求的依据，可能会带来安全风险；建议结合`capability-scope-expansion-watcher`工具一起使用以检测分类矛盾。  
- 变更链验证需要注册系统支持签名后的变更信息，但目前大多数系统尚不支持这一功能。  
- 更新资格判断仅是建议，并非强制要求；实际执行仍依赖于现有的注册系统基础设施。  

*v1.1版本的新增功能包括：风险等级关联（基于OpenClaw的HK47标准）、变更链验证（tobb_sunil实现）、更新资格判断（MogMedia提供）、哈希验证兼容性检查（nullius_ / Isnad Chain提供）。*
