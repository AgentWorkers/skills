# 那个“已验证”徽章是2024年颁发的。这个技能现在还安全吗？

> 该工具用于追踪技能验证结果的时效性，标记那些时效已过、不再可靠的认证信息。

## 问题背景

某项技能在2025年3月通过了安全审计，并获得了“已验证”的徽章。开发者看到这个徽章后便对其产生了信任。然而18个月后：

- 该技能的4个依赖库共经历了47次更新；
- 有两个新的安全漏洞（CVE）影响了该技能所针对的运行时版本；
- 该技能的API端点指向的域名已变更所有者；
- 市场平台新增了3种在初次审计时并不存在的权限类型。

虽然验证过程是真实有效的，但由此产生的信任度却已不再可靠。安全认证具有“有效期”，但大多数代理市场平台仍将其显示为永久有效的状态。

这种现象被称为“信任衰减”：随着环境的变化，验证结果的可靠性逐渐降低。问题并不在于审计本身有误，而在于审计时的结论已不再适用于当前的情况。

## 该工具的功能  

该工具会为已验证的技能计算一个信任度 freshness 分数：  

1. **自验证以来的时间**：指最后一次审计的时间长度。时间越长，信任度越低（具体衰减程度可配置）；  
2. **依赖库的更新情况**：自审计以来，该技能依赖的库更新了多少次？每次更新都可能使审计时的假设失效；  
3. **生态系统环境的变化**：自审计以来出现了新的安全漏洞、权限类型或攻击模式，审计时评估的威胁环境可能已经发生变化；  
4. **域名和端点的稳定性**：技能中引用的外部URL、API端点或资源链接是否发生了变化；  
5. **重新验证的紧迫性**：自上次审计以来，是否有人（不仅仅是最初的审计人员）对该技能进行了任何形式的安全检查？  

## 使用方法  

**输入**：  
- 提供一个技能的标识符及其验证日期；  
- 提供一个展示已验证技能的市场平台链接；  
- 提供一组技能标识符以进行整体信任度评估。  

**输出**：  
- 每项技能的信任度 freshness 分数（0-100分，100分表示验证结果完全有效）；  
- 常见影响信任度的因素（时间、依赖库更新、生态系统变化等）；  
- 重新验证的紧迫性等级（低/中等/高/危急）；  
- 如果评估多个技能，则提供整体信任度总结。  

## 示例  

**输入**：检查已验证的技能 `api-auth-helper` 的信任度 freshness（验证日期为2025-01-10）：  

```  
⏳ 信任度衰减报告 — 建议重新验证  

技能：api-auth-helper  
验证日期：2025-01-10（408天前）  
审计机构：@seclab-audits  

信任度 freshness 分数：31/100（失效）  

影响信任度的因素：  
- **时间衰减**：-25分（审计后超过12个月）  
- **依赖库更新**：-22分  
  - jsonwebtoken：3次重大更新（9.0.0 → 12.1.2）  
  - node-fetch：2次更新（包含安全补丁）  
  - crypto-utils：1次更新（导致API功能变更）  
- **生态系统变化**：-15分  
  - 自审计以来新增了2个与JWT相关的安全漏洞  
  - 市场平台新增了“credential-store”权限类型（初次审计时未评估）  
- **端点稳定性**：-7分  
  - 技能引用的API端点为 `api.authprovider.example/v2`，现重定向至v3（响应格式不同）  

**重新验证的紧迫性等级：** 高  
- 主要原因：3个依赖库的重大更新以及2个相关安全漏洞；尤其是JWT库的变更可能影响技能的令牌验证功能。  

**建议**：  
- 优先重新审计JWT处理相关部分（受漏洞影响的部分）；  
- 使用当前版本的依赖库进行测试；  
- 确认端点重定向不会影响认证流程；  
- 检查新增的“credential-store”权限是否对技能功能产生影响。
