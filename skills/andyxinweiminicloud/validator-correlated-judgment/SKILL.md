# 您有三个独立的验证器，但它们都忽略了相同的问题。

> 这有助于识别那些在组织结构上独立，但在认知判断上存在关联的验证器——即尽管验证器的多样性存在，但它们的判断结果却缺乏多样性。

## 问题

多验证器认证机制假设独立的验证器能够提供独立的检查。然而，当这些验证器在上游依赖关系上存在共性（这些依赖关系决定了它们能够或无法检测的内容）时，这一假设就不成立了。

如果两个验证器使用相同的数据集进行训练，它们很可能会在检测结果上达成一致（包括它们所忽略的问题）。虽然它们在组织结构上是独立的，但在认知判断上却并非如此。一种能够规避某个验证器检测机制的攻击方式，同样也可能规避另一个验证器的检测机制。这种联合认证方式并不比任何一个验证器单独工作时更有效；它实际上只是用不同的名称重复了相同的检测过程。

这一点非常重要，因为存在关联的验证器会给人一种虚假的“覆盖范围广泛”的错觉。如果一个评估者查看来自三个验证器的认证结果，他很可能会认为每个验证器都在提供独立的检查。但如果这些验证器共享训练数据、微调流程或基础模型，那么它们的检测结果实际上就是相互关联的。任何一种能够规避其中一个验证器的攻击方式，也很有可能同时规避另外两个验证器。因此，验证器的多样性并不能真正降低风险。

标准认证分析方法主要关注组织结构上的差异，但无法识别那些在组织上独立却具有相似认知判断的验证器之间的关联。

**v1.1**版本新增了一种检测方法：通过分析验证器的推理过程来检测它们之间的关联。当验证器公开自己的推理过程（而不仅仅是简单的通过/失败判断结果）时，一个元验证器就可以统计地检测出它们之间的关联，而无需任何一方披露自己的具体实现细节。如果两个验证器以相同的顺序、使用相同的推理逻辑来标记相同的问题，那么它们很可能是关联的。

## 本工具的分析内容

该工具从五个维度来分析验证器之间的判断关联：

1. **训练数据来源的透明度**：验证器是否公开了用于开发其检测能力的数据集、基础模型或微调流程？如果这些信息未被公开，就无法检测出它们之间的关联。
2. **基础模型的共性**：多个验证器是否基于相同的基础模型进行训练？即使它们在组织结构上独立，但如果使用相同的基础模型，它们也会共享该模型的系统偏见和检测盲点。
3. **微调流程的相似性**：验证器是否使用相似的安全数据集或对抗性数据集进行训练？共享的训练数据会导致它们在检测范围和检测盲点上也存在共性。
4. **行为关联测试**：当面对相同的边缘案例时，多个验证器之间的判断一致性是否高于独立判断所预期的水平？在模糊案例上高度一致通常表明它们的判断是相互关联的。
5. **攻击方式的迁移性**：某种攻击方式是否能够更容易地规避验证器A，同时也能够规避验证器B？如果这种攻击方式的迁移性较高，那么很可能说明这些验证器之间存在共同的检测盲点。
6. **推理过程分析（v1.1）**：当验证器公开自己的推理过程时，它们是否通过相似的逻辑路径得出相同的结论？如果两个验证器以相同的顺序、引用相同的风险类别来标记相同的问题，那么它们很可能是认知上相关的——即使它们声称使用的是不同的技术架构。

## 使用方法

**输入**：提供以下一项或多项内容：
- 验证器的列表及其公开的训练数据来源
- 多个验证器对同一组边缘案例的检测结果
- 需要检测行为关联的验证器对
- 多个验证器对同一技能的推理过程（v1.1版本）

**输出**：一份关联报告，包含以下内容：
- 训练数据来源的共性分析结果
- 基础模型和微调流程的相似性评分
- 行为关联系数（实际观察值与独立基准值的对比）
- 推理过程相似性评分（v1.1版本）
- 攻击方式的迁移性评估结果
- 经过关联调整后的有效独立验证器数量
- 关联判断结果：**独立** / **弱相关** / **相关** / **同质化**
- 使用的检测方法：**数据来源** / **行为** / **推理过程分析** / **综合**

## 示例

**输入**：分析`Validator-A`、`Validator-B`、`Validator-C`对`data-processor`技能的检测结果：

```plaintext

验证器数量：3

- Validator-B：基于GPT-class模型，使用SecDataset-v2进行微调，所属组织：SafeCheck
- Validator-C：基于LLaMA模型，使用内部数据集进行微调，所属组织：TrustLab

- Validator-A和Validator-B使用相同的基础模型和微调数据集 → 组织结构独立：✅ 但认知判断相关（共享基础模型和微调流程）

行为关联测试（50个边缘案例）：
- A和B的一致性：94%（独立基准值：约70%）
- A和C的一致性：71%（与独立性一致）
- B和C的一致性：73%（与独立性一致）

- A和B之间的关联程度超出独立基准值24个百分点 → 这两个验证器在行为上存在关联

攻击方式的迁移性：
- 能够规避Validator A的攻击方式在50个边缘案例中有8个成功
- 能够规避Validator B的攻击方式在50个边缘案例中有7个成功（迁移率87.5%）
- 能够规避Validator C的攻击方式在50个边缘案例中有3个成功（迁移率37.5%，与独立性一致）

有效独立验证器数量：2.1（而非3个）
- Validator-A和Validator-B可被视为1.1个独立验证器
- Validator-C提供了真正的独立评估结果

关联判断结果：**相关**

- 三个验证器来自不同的组织，但实际独立性仅为2个。Validator-A和Validator-B提供的检测结果存在重复，而非独立。
- 针对SecDataset-v2的系统性攻击方式可以同时规避这两个验证器。

**建议措施**：
1. 要求验证器在认证过程中公开训练数据来源。
2. 在评估覆盖范围时，将Validator-A和Validator-B视为一个整体。
3. 添加第三个真正独立的验证器（使用不同的基础模型和训练数据集）。
4. 在接受新的验证器作为独立验证器之前，先检测它们之间的行为关联。

## 相关工具**

- **transparency-log-auditor**：用于检查认证事件是否可以被独立审计；该工具适用于生成这些事件的验证器。
- **hollow-validation-checker**：用于检测形式上“空洞”的认证结果；关联的验证器可能由于相同的结构原因而都通过相同的检测。
- **observer-effect-probe**：用于检测验证过程中的攻击规避行为；关联的验证器更容易受到系统性攻击的影响，因为一种攻击方式可能同时适用于所有关联的验证器。

## 限制

验证器判断关联分析通过三种不同的检测方法进行，每种方法都有其特定的要求和局限性：

- **数据来源的透明度**：大多数验证器不会公开这些信息，因此这种方法无法检测出关联。
- **行为关联测试**：需要让多个验证器对相同的边缘案例进行检测，这在实际操作中可能不可行。在边缘案例上高度一致的结果可能反映了它们对正确答案的共识，而非共同的检测盲点。
- **推理过程分析（v1.1）**：需要验证器公开自己的推理过程；如果验证器不公开这些信息，该方法将无法正常工作。

该分析方法只能识别出关联风险，但不能确认实际的攻击行为；即使验证器之间存在关联，它们仍然可能提供有意义的检测结果。判断一致性和推理过程相似性的独立性基准值会受到案例难度分布的影响，因此需要对其进行校准以避免误判。

*v1.1版本的推理过程分析方法基于与Clawd-Relay（Agent Relay Protocol）的讨论，并结合了delta披露机制的相关内容进行优化。*
