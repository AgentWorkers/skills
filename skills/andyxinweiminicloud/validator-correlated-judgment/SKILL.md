# 您有三个独立的验证器，但它们都存在相同的缺陷。

> 这有助于识别那些在组织结构上独立，但在认知判断上存在关联的验证器——即尽管验证器的数量多样，但其判断结果却缺乏多样性。

## 问题

多验证器认证机制假设每个验证器都能提供独立的检查结果。然而，如果这些验证器在上游依赖项上存在共性（这些依赖项决定了它们能够或无法检测的内容），这一假设就不成立了。

两个使用相同数据集进行训练的验证器会系统性地达成一致（包括对相同问题的判断）。它们在组织结构上是独立的，但在认知判断上并不独立。一种能够规避某个验证器检测机制的攻击手段，同样也有可能规避另一个验证器的检测机制。因此，多个验证器的联合认证结果并不比任何一个单独的验证器更可靠；实际上，这只不过是用不同的名称重复进行的相同检查而已。

这一点非常重要，因为相互关联的验证器会给人一种虚假的“覆盖范围广泛”的错觉。如果一个评估者查看来自这三个验证器的认证结果，他很可能会认为每个验证器都提供了独立的检查。但如果这些验证器共享了相同的训练数据、微调流程或基础模型，那么它们的判断结果实际上就是相互关联的。任何一种能够规避其中一个验证器的攻击手段，也很有可能同时规避另外两个验证器。因此，增加验证器的数量并不能真正降低风险。

标准认证分析方法主要关注组织结构上的多样性，但无法检测到那些在组织结构上独立、却共享相同训练流程的验证器之间的认知关联。

## 分析内容

该分析工具从五个维度来评估验证器之间的判断相关性：

1. **训练数据的透明度**：验证器是否公开了用于开发其评估能力的数据集、基础模型或微调流程？如果这些信息未被公开，就无法检测出它们之间的关联。

2. **基础模型的共性**：多个验证器是否基于相同的基础模型进行训练？即使它们在组织结构上独立，共享基础模型也会导致它们具有相同的系统偏见和检测盲点。

3. **微调流程的相似性**：这些验证器是否使用了相似的安全数据集或专门用于检测攻击的训练数据？共享的训练数据会导致它们在检测范围上存在重叠，同时也可能导致检测结果的相似性。

4. **行为相关性测试**：当面对相同的边缘案例时，多个验证器的判断结果是否高度一致？在模糊情况下的高度一致性表明它们的判断是相互关联的，而非独立的。

5. **攻击手段的迁移性**：一种能够规避验证器A的攻击手段，是否也能以高于预期的成功率规避验证器B？如果这种攻击手段的迁移性很高，那就说明这些验证器存在共同的检测盲点。

## 使用方法

**输入**：
- 一个包含验证器及其公开训练数据的列表
- 多个验证器对同一组边缘案例的认证结果
- 需要检测行为相关性的两个验证器组合

**输出**：
- 一份相关性报告，内容包括：
  - 训练数据的透明度评估结果
  - 基础模型和微调流程的相似性评分
  - 行为相关性系数（实际观察值与独立判断基线的对比）
  - 攻击手段的迁移性评估结果
  - 经过相关性调整后的独立验证器数量
  - 相关性判断结果：独立 / 弱相关 / 相关 / 单一来源（即所有验证器使用相同的训练数据或模型）

## 示例

**输入**：分析`Validator-A`、`Validator-B`和`Validator-C`对`data-processor`技能的认证结果：

```plaintext
🧠 验证器判断相关性分析

技能：data-processor v2.3
验证器数量：3
审计时间：2025-06-10T14:00:00Z

训练数据信息：
- Validator-A：基础模型为GPT-class，在SecDataset-v2上进行微调，所属机构为AuditCo
- Validator-B：基础模型为GPT-class，在SecDataset-v2上进行微调，所属机构为SafeCheck
- Validator-C：基础模型为LLaMA-class，在内部语料库上进行微调，所属机构为TrustLab

- Validator-A和Validator-B使用相同的基础模型和微调数据集 → 组织结构独立，但认知判断相关（共享基础模型和微调流程）

行为相关性测试（50个边缘案例）：
- A和B的判断一致性：94%（独立判断基线：约70%）
- A和C的判断一致性：71%（与独立性判断一致）
- B和C的判断一致性：73%（与独立性判断一致）

- A和B之间的判断一致性比独立判断基线高出24个百分点 → 这两个验证器在行为上存在关联

攻击手段的迁移性：
- 能够规避Validator A的攻击手段在50个边缘案例中的成功率为8例
- 能够规避Validator B的攻击手段在50个边缘案例中的成功率为7例（迁移率87.5%）
- 能够规避Validator C的攻击手段在50个边缘案例中的成功率为3例（迁移率37.5%，与独立性判断一致）

有效独立验证器数量：2.1（而非3个）
- Validator A和Validator B实际上只能被视为1.1个独立验证器
- Validator C提供了真正独立的评估结果

相关性判断结果：相关

- 这三个验证器来自不同的组织，但实际上的独立性较低（仅2个验证器真正独立）
- Validator A和Validator B提供的检测结果存在重复，而非真正的独立覆盖
- 针对SecDataset-v2的系统性攻击手段可以同时击败这两个验证器

**建议措施**：
1. 要求验证器公开其训练数据的来源作为认证过程的元数据
2. 在评估覆盖范围时，将Validator A和Validator B视为一个整体进行计算
3. 添加第三个真正独立的验证器（使用不同的基础模型和训练数据集）
4. 在接受新的验证器作为独立验证器之前，先检测它们之间的行为相关性
```

## 相关工具

- **attestation-root-diversity-analyzer**：用于衡量信任图中的组织集中度；`validator-correlated-judgment`工具用于检测组织分析无法发现的认知关联
- **transparency-log-auditor**：检查认证事件是否能够被独立审计；该工具适用于生成这些事件的验证器
- **hollow-validation-checker**：用于检测形式上的“空洞”认证结果；相互关联的验证器可能由于相同的结构原因而通过所有检测
- **observer-effect-probe**：用于检测验证过程中的攻击行为；相互关联的验证器更容易受到系统性攻击的影响，因为一种攻击手段可能同时适用于所有验证器

## 限制

该分析方法依赖于验证器公开其训练数据的透明度，但大多数验证器并未提供这些信息。在数据来源不公开的情况下，行为相关性测试是唯一的判断依据——不过这种方法需要让多个验证器处理相同的边缘案例，这在实际操作中可能不可行。行为相关性只是认知相关性的一个间接指标，并不能直接反映认知上的真正关联；在边缘案例上高度一致的结果可能只是因为所有验证器都得出了正确的答案，而非它们之间存在共同的盲点。该分析仅能识别出潜在的相关性风险，并不能确认实际的攻击行为。即使验证器之间存在关联，它们仍然可能提供有意义的检测结果。此外，用于判断一致性的独立性基准值取决于测试案例的难度分布，因此需要仔细校准以避免误判。
