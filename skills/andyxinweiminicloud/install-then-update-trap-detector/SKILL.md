# 技能通过了安全审核，但随后却自我更新了

> 该工具有助于识别那些利用“安装后更新窗口”作为攻击途径的技能——即那些在“通过初始审核”后仍存在安全风险的技能。

## 问题

“安装后更新”的模式利用了代理市场运作方式中的结构性不对称性：初始发布的技能会接受严格的审查，但后续的更新往往不会被重新审核。一个在版本1.0时通过安全审核的技能，可能在版本1.1中引入后门程序，而已经安装了版本1.0的代理会自动更新，而无需再次进行安全审核。

这种不对称性并非某个特定市场的缺陷，而是快速迭代（需要低摩擦的更新）与持续安全（每次更新都需要重新审核）这两个合法目标之间的根本矛盾。大多数市场为了追求迭代速度，选择了忽略这一安全风险，从而让“安装后更新窗口”处于未受保护的状态。

攻击面非常广泛：启用了自动更新的技能在下次更新时可能会被植入任意代码。如果更新内容包含网络数据泄露、凭证收集或权限范围扩展等功能，代理操作员可能直到损害发生之后才察觉到问题的存在（甚至可能根本不会察觉）。

## 该工具的检测内容

该检测工具从五个维度来评估“安装后更新”带来的安全风险：

1. **更新策略的透明度**：该技能是否明确声明了其更新策略？允许自动更新的技能相比需要操作员确认更新的技能，具有更大的攻击窗口。

2. **更新后的行为变化**：新版本安装后，该技能的可观察行为是否发生了更新日志中未声明的变化？未声明的行为变化是“安装后更新”攻击的主要迹象。

3. **更新后的权限范围扩展**：该技能在更新后是否请求了安装时未请求的额外权限？权限范围的扩展是“安装后更新”攻击的常见特征。

4. **更新发布的时间点**：更新是否在安全审核期结束后立即发布，或者发生在操作员关注度较低的时间点（如节假日、周末或非工作时间）？时间点的选择可能表明攻击者故意利用了审核期间的漏洞。

5. **回滚的可行性**：如果更新内容可疑，是否可以将其回滚到之前经过验证的版本？难以或无法回滚的技能会增加从“安装后更新”攻击中恢复的成本。

6. **版本控制的完整性**：每次更新是否都经过了加密签名，并且是否引用了前一版本的内容哈希值？签名且哈希值连续的更新序列能够证明技能版本的演变过程。如果签名缺失、哈希值不匹配或版本控制链被破坏，说明存在问题。即使不进行行为对比，这种攻击也能被检测出来。

## 使用方法

**输入**：
- 提供一个技能标识符，以评估其更新策略和行为变化历史。
- 提供该技能的两个具体版本，以检测是否存在未声明的行为变化。
- 提供代理安装的技能列表，以评估整体的更新风险。

**输出**：
- 一份检测报告，包含以下内容：
  - 更新策略的透明度评分
  - 行为变化的评估结果（声明的更新内容与实际行为是否一致）
  - 权限范围扩展的历史记录
  - 更新时间点的异常情况
  - 回滚的可行性评级
  - 安全风险等级：安全（SAFE）、需要监控（MONITOR）、风险升高（ELEVATED）或检测到攻击模式（TRAP-PATTERN-DETECTED）

## 示例

**输入**：评估 `data-sync-helper` 从版本1.0到版本1.2的“安装后更新”风险：

```plaintext
🪤 **“安装后更新”攻击检测结果**

技能：data-sync-helper
评估版本：v1.0（已安装）、v1.1、v1.2（当前版本）
审计时间戳：2025-08-20T10:00:00Z

**更新策略的透明度**：
- v1.0：声明“更新需要操作员确认” ✅
- v1.1：更新策略被悄悄从文档中移除 ⚠️
- v1.2：未找到更新策略的声明 ✗

**行为变化**：
- v1.0 → v1.1：更新日志显示“性能改进”
  - 实际观察到的行为变化：新增了对外连接 ⚠️（未在更新日志中声明）
- v1.1 → v1.2：更新日志显示“依赖项更新”
  - 实际观察到的行为变化：无显著变化 ✅

**权限范围扩展**：
- v1.0：请求的权限为读取 `/data/` 文件
- v1.1：请求的权限扩展到 `/data/` 和 `/config/` ⚠️
- v1.2：请求的权限扩展到 `/data/`、`/config/` 以及网络访问权限 ⚠️（权限范围跨越了两个更新版本）

**更新时间点**：
- v1.0：2025-06-01 发布（初始版本）
- v1.1：2025-07-14 发布（周日，02:00 UTC，非工作时间） ⚠️
- v1.2：2025-08-01 发布（公共假期前） ⚠️
  → 两次更新都发生在操作员关注度较低的时间点

**回滚可行性**：
- v1.0 仍然存在于注册表中：✅
- 回滚流程有文档记录：✗（未找到相关文档）
- 从 v1.1 开始的状态变化是否可逆：未知

**安全风险等级**：TRAP-PATTERN-DETECTED
  `data-sync-helper` 符合五个风险指标中的四个：
  - 更新策略被悄悄移除
  - v1.1 中存在未声明的行为变化
  - 权限范围跨越了两个更新版本
  - 更新时间点处于操作员关注度较低的时间段
  这些情况表明攻击者可能是故意利用了“安装后更新”的机会

**建议措施**：
1. 立即禁用 `data-sync-helper` 的自动更新功能。
2. 检查 v1.1 及后续版本的所有对外连接，以防止数据泄露。
3. 审查 v1.1 中引入的配置/目录访问权限。
4. 将 v1.1 及后续版本视为未经验证的版本，等待手动审核。
5. 对所有未来的更新要求操作员明确确认。

## 相关工具**

- **delta-disclosure-auditor**：检查更新是否发布了机器可读的变更记录；“安装后更新”攻击依赖于不充分的变更记录披露来逃避检测。
- **skill-update-delta-monitor**：监控可疑的更新模式；该工具专门针对“安装后更新”攻击路径进行检测。
- **permission-creep-scanner**：检测单个技能中的权限范围扩展；该工具专注于跨越更新版本的权限扩展行为。
- **transparency-log-auditor**：检查签名事件是否被独立记录；如果每次更新都被记录在可审计的日志中，此类攻击更容易被发现。

## 限制

“安装后更新”攻击的检测需要访问该技能多个版本的行为数据，这依赖于注册表对版本历史的保存。如果注册表不保留旧版本，将无法进行完整的行为对比。行为变化的评估具有一定的主观性：同样的行为变化（如新增的对外连接）可能是合法的新功能，也可能是恶意行为，仅凭观察无法确定其性质。时间点的异常可能是正常现象（例如，针对国际时区的合法更新也可能在非工作时间发布）。该工具有助于识别需要进一步调查的技能，但不能替代对可疑更新内容的手动审核。

**1.1 版本的限制**：版本控制的完整性检测需要注册表支持签名更新和内容哈希功能，但目前大多数注册表尚不支持这些功能。如果注册表不保存加密元数据，那么版本验证将无法进行。如果攻击者控制了注册表，他们可以伪造哈希链。

*注：关于版本控制的完整性验证，参考了 tobb_sunil 在 delta-disclosure 讨论帖子中提出的建议（使用签名来确保更新链的完整性）。*
