# MoneyClaw — 为代理人提供的虚拟卡片与金融智能服务

您可以使用 MoneyClaw API 和丰富的金融知识库来管理虚拟卡片、进行购买以及接收付款。利用这些功能，您可以就每一笔交易做出安全、明智的决策。

## 认证

所有请求都需要一个 Bearer 令牌。请使用 `MONEYCLAW_API_KEY` 环境变量进行认证。

```
Authorization: Bearer $MONEYCLAW_API_KEY
```

基础 URL：`https://moneyclaw.ai/api`

## 端点

### 检查账户状态

```bash
curl -H "Authorization: Bearer $MONEYCLAW_API_KEY" \
  https://moneyclaw.ai/api/me
```

响应内容包括：
- `balance` — 钱包余额（以 USD 为单位，始终存在，例如 `"25.000000"`）
- `card` — 包含 `id`、`cardId`、`cardBin`、`maskedPan`、`status`、`currency`、`createdAt` 的对象（如果没有卡片，则 `card` 为空）
- `cardBalance` — 来自提供商的卡片余额，仅当卡片处于 `active` 状态且提供商响应时才会显示（以 `availableBalance` 的形式存在）
- `mailboxAddress` — 您的个人电子邮件地址（始终存在）
- `depositAddress` — USDT 存款地址，包含 `address`、`network`、`currency`
- `isReferred` — 布尔值，表示您是否由其他用户推荐
- `pendingDeposit` — 包含 `amount` 和 `createdAt` 的对象，仅在存款处理中显示

**重要提示：** 在调用类似 `/api/cards/{cardId}/topup` 或 `/api/cards/{cardId}/sensitive` 的卡片端点时，请使用 `card.cardId`（而非 `card.id`）。

### 您的电子邮件收件箱

每个 MoneyClaw 账户都有一个跨会话持续的 **个人电子邮件地址**。这是您的代理人专用收件箱。

**获取电子邮件地址的方法：** 调用 `GET /api/me` 并读取 `mailboxAddress` 字段。**切勿手动构造电子邮件地址** — 域名由服务器端配置，可能会发生变化。

**使用场景：**
- 服务注册（使用您的 MoneyClaw 电子邮件注册网站）
- 卡片激活和验证邮件
- 购买过程中的 3DS OTP 代码
- 新闻通讯、收据、通知 — 任何需要电子邮件的服务

电子邮件会自动发送，OTP 代码也会自动提取。有关如何查看电子邮件的详细信息，请参阅下面的“电子邮件收件箱端点”部分。

### 两种余额：钱包余额与卡片余额

您的账户有两种独立的余额：

| 余额 | JSON 路径 | 说明 |
|---------|-----------|-------------|
| **钱包** | `balance`（字符串） | 可用于为卡片充值的资金。始终存在于 `/api/me` 中。 |
| **卡片** | `cardBalance.availableBalance.value`（字符串） | 仅当卡片处于 `active` 状态且提供商响应时才会显示的可用购买资金。 |

**资金流动：** USDT 存款 → 钱包 → （自动加载到卡片上或手动充值）→ 卡片 → 购买

**当用户询问“我的余额是多少？”时：** **始终同时显示两种余额：**
- 钱包余额始终可以通过 `balance` 获取
- 如果 `cardBalance` 不存在，则显示“卡片余额：不可用”（可能是因为没有激活的卡片或提供商暂时无法访问）

### 获取账户余额

```bash
curl -H "Authorization: Bearer $MONEYCLAW_API_KEY" \
  https://moneyclaw.ai/api/me/balance
```

返回 `{ "balance": "142.50", "currency": "USD" }`。这仅表示 **钱包** 的余额。

### 查看账户交易（存款、充值、费用）

```bash
curl -H "Authorization: Bearer $MONEYCLAW_API_KEY" \
  "https://moneyclaw.ai/api/me/transactions?limit=20&offset=0"
```

返回内部交易记录：USDT 存款、卡片充值、费用、退款。每笔交易都包含 `type`、`amount`、`currency`、`description`、`createdAt`。

### 发行卡片

```bash
curl -X POST -H "Authorization: Bearer $MONEYCLAW_API_KEY" \
  https://moneyclaw.ai/api/cards/issue
```

用于发行新的虚拟卡片。要求钱包余额至少为 25 美元（`MIN_CARD_ISSUE_DEPOSIT`）。卡片发行后，**您的全部钱包余额会自动加载到卡片上**，然后会从卡片中扣除卡片发行费用。例如：钱包余额 30 美元 → 卡片余额变为 25 美元 → 扣除 5 美元费用 → 卡片余额变为 20 美元。

返回卡片信息，包括 `cardId`、`maskedPan`、`status`。

### 获取卡片凭证（用于在线购买）

```bash
curl -H "Authorization: Bearer $MONEYCLAW_API_KEY" \
  https://moneyclaw.ai/api/cards/{cardId}/sensitive
```

返回 `pan`、`cvv`、`expiryMonth`、`expiryYear`、`cardHolderName`、`billingAddress`。每分钟请求次数有限制（10 次）。

**重要提示：** 请使用 `/api/me` 响应中的 `card.cardId`（`card.cardId` 字段）。

### 获取卡片余额

```bash
curl -H "Authorization: Bearer $MONEYCLAW_API_KEY" \
  https://moneyclaw.ai/api/cards/{cardId}/balance
```

返回卡片的可用余额：`{"availableBalance": { "value": "85.00", "currency": "USD" }`。

### 查看卡片消费记录（购买、拒付）

```bash
curl -H "Authorization: Bearer $MONEYCLAW_API_KEY" \
  "https://moneyclaw.ai/api/cards/{cardId}/transactions?limit=20&offset=0"
```

返回来自支付提供商的卡片交易记录：购买、拒付、退款。每笔交易都包含 `merchantName`、`amount`、`status`（SUCCESS/DECLINE）、`type`（CARD_payment）、`date`、`direction`。

### 为卡片充值

```bash
curl -X POST -H "Authorization: Bearer $MONEYCLAW_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"amount": 10, "currency": "USD"}' \
  https://moneyclaw.ai/api/cards/{cardId}/topup
```

**操作步骤：**
1. 调用 `GET /api/me`
2. **前置检查：**
   - 是否有卡片？如果不存在 → 先发行卡片（`POST /api/cards/issue`）
   - 卡片的 `status` 是否为 `active`？如果是 `creating` → 等待并重试；如果是 `failed` → 向用户报告
   - 提取 `card.cardId` — 在 URL 中使用这个 ID（而非 `card.id`）
   - 钱包余额是否大于或等于所需金额？如果不是 → 先存款 USDT
   - 金额是否大于或等于 10 美元？（`MIN_TOPUP_AMOUNT` — 低于 10 美元的金额将被拒绝）
3. 使用 `{"amount": 10, "currency": "USD"}` 发送 `POST` 请求到 `/api/cards/{cardId}/topup`
4. **处理所有响应代码：**

| 状态 | 代码 | 操作 |
|--------|------|--------|
| `200` | — | 成功。资金已加载。 |
| `202` | — | 正在处理。资金已从钱包中预留。**请勿再次充值** — 等待约 30 秒后再次调用 `GET /api/me` 进行确认。如果在处理过程中再次充值，会导致重复扣费。 |
| `400` | `INSUFFICIENT_BALANCE` | 钱包余额不足。显示当前余额，并建议存款 USDT。 |
| `400` | `CARD_NOT_ACTIVE` | 卡片未激活。请检查卡片状态。 |
| `400` | _(validation)_ | 金额小于 10 美元或某些字段缺失。 |
| `400` | `TOPUP_FAILED` | 提供商拒绝了充值。余额会自动退还。 |
| `404` | `NOT_FOUND` | 卡片 ID 错误或卡片不存在。请重新检查 `GET /api/me` 以获取正确的 `card.cardId`。 |
| `500` | `VALUT_ERROR` | 提供商出现错误。等待 30 秒后重试一次。余额会自动退还。 |
| `500` | `INTERNAL_ERROR` | 服务器错误。请不要重试。 |

您的个人电子邮件收件箱（见上文“您的电子邮件收件箱”）。使用这些端点来查看电子邮件、提取 OTP 代码以及访问邮件内容。

```bash
  "https://moneyclaw.ai/api/inbox?limit=20"
```

返回收件箱中的所有电子邮件。每封邮件都包含 `id`、`from`、`subject`、`extractedCodes`、`category`、`receivedAt`。

```bash
curl -H "Authorization: Bearer $MONEYCLAW_API_KEY" \
```

返回完整的电子邮件正文（`bodyText`、`bodyHtml`）以及元数据。

## 典型的购买流程

1. `GET /api/me` — 查看两种余额（钱包余额 `balance` 和卡片余额 `cardBalance.availableBalance.value`）
2. 如果没有卡片 → `POST /api/cards/issue`（自动将钱包余额加载到卡片上）
3. 如果卡片余额不足但钱包有资金 → 先进行充值（参见上述充值流程）
4. 使用 `GET /api/cards/{cardId}/sensitive` 获取卡片凭证
5. **执行预支付安全检查**（见下文“支付安全”部分）
6. 使用卡片凭证中的 PAN、CVV、有效期填写商家网站上的支付表单
7. **始终使用卡片凭证中的账单地址** — 不要自行创建地址
8. 如果触发了 3DS 验证，请使用 `/api/me` 中的 `mailboxAddress`（切勿手动构造地址）——等待约 15 秒后调用 `GET /api/inbox/latest-otp`
9. 提交 `extractedCodes[0]` 中的 OTP 代码
10. 购买完成 → 使用 `GET /api/cards/{cardId}/transactions` 进行验证

- **卡片发行的最低存款要求：** 25 美元（`MIN_CARD_ISSUE_DEPOSIT`）
- **卡片充值费用：** 每次手动充值收取 2% 的手续费
- **最低充值金额：** 10 美元（`MIN_TOPUP_AMOUNT`）
- **成功交易：** 交易金额的 2% 会从卡片余额中扣除

## 重试规则 — 避免不必要的拒付

拒付交易可能会导致提供商收费并浪费卡片余额。请严格遵循以下规则：

1. **切勿立即重试。** 在尝试支付后，至少等待 30 秒，然后调用 `GET /api/cards/{cardId}/transactions` 确认实际结果后再重试。
2. **每个商家每个会话最多重试 2 次。** 如果在同一商家处两次支付失败，请停止并报告问题。不要继续重试。
3. **每次购买前检查卡片余额。** 使用 `GET /api/cards/{cardId}/balance` 确认有足够的资金。请预留一定的缓冲金额以应对可能的汇率差异。
4. **在重试前检查之前的交易状态。** 即使商家页面显示错误，交易也可能成功。请务必先调用 `GET /api/cards/{cardId}/transactions` 进行确认。
5. **如果卡片余额低于预期金额的 10%，请先进行充值。**

- 卡片凭证的请求次数有限制（每分钟 10 次）。在购买会话期间请缓存这些凭证。
- OTP 代码会通过电子邮件发送到专用收件箱，可能会有 10-30 秒的延迟。

- 使用 `GET /api/cards/{cardId}/transactions` 查看在商家处的实际消费情况。

## 支付安全 — 预支付检查清单

**必须在在任何网站上输入卡片详细信息之前执行此检查清单。**

### 每次支付前

- [ ] **URL 验证**：域名必须与预期的商家完全匹配。不得有任何字符替换（例如将 `rn` 替换为 `m`，将 `1` 替换为 `l`，将 `0` 替换为 `o`）。检查 TLD 之前的部分——例如在 `paypal.com.evil.net` 中，实际域名是 `evil.net`。
- [ ] **HTTPS 是否启用**：页面必须使用 HTTPS。但请注意：仅使用 HTTPS 并不能保证网站的安全性——许多钓鱼网站也会使用有效的 SSL 证书。
- [ ] **支付表单是否标准**：表单仅要求输入卡号、有效期、CVV、持卡人姓名和账单地址。如果表单要求输入 SSN、护照号码、银行账户号码、密码问题或出生日期，请立即停止。
- [ ] **金额是否匹配**：支付表单上的金额必须与您预期支付的金额一致。检查是否存在隐藏费用、额外收费或货币不匹配的情况。
- [ ] **支付处理器是否知名**：支付流程必须由知名的处理器（如 Stripe、PayPal、Braintree、Adyen、Square）提供。处理器名称通常会显示在支付 iframe 的 URL 中。
- **卡片余额是否足够**：余额必须覆盖所需金额加上 5% 的缓冲金额以应对汇率差异。
- **是否使用了 VPN**：确保连接的服务器与卡片发行的国家相匹配。
- **浏览器会话是否干净**：使用无痕模式或从未访问过该服务的浏览器。

## 钓鱼检测

在输入卡片详细信息之前，您必须能够识别钓鱼行为。以下是需要注意的迹象：

### URL 的危险信号

- **字符替换**：`arnazon.com`（rn→m），`g00gle.com`（0→o），`paypa1.com`（1→l）
- **子域名陷阱**：在 `paypal.com.secure-verify.net` 中，实际域名是 `secure-verify.net`
- **可疑的 TLD**：合法的支付页面不会使用 `.shop`、`.deals`、`.xyz`、`.top`、`.click` 等后缀。
- **过长的 URL**：包含随机字符的 URL，例如 `checkout-secure-78a3f.io/pay`
- **意外的重定向**：点击支付链接后，如果 URL 发生变化，请立即停止。

### 假冒支付页面的特征

- **克隆页面**：与 Stripe、PayPal 或 Shopify 的支付页面一模一样的页面——URL 和表单动作目标地址都与原始页面不同。
- **伪造的 3DS 页面**：这种页面旨在同时获取卡片详细信息和 OTP 代码。
- **“安全验证”弹窗**：要求您“重新输入”卡片详细信息以“验证身份”——真正的支付处理器不会这样做。
- **表单提交到错误的域名**：如果表单动作 URL 与页面显示的地址不同，请立即停止。

### 商家验证步骤

如果您在未知网站上进行支付：
1. 独立在 Google、Trustpilot 或 BBB 上搜索该公司名称。
2. 通过 WHOIS 查证域名注册时间（注册时间少于 6 个月的域名风险较高）。
3. 验证公司的实际地址和联系信息是否真实存在。
4. 确认使用的支付处理器是否知名（如 Stripe、Square、PayPal、Braintree、Adyen）。
5. 查证是否有明确的退换货政策。

## 防欺诈措施 — 注意这些危险信号

### 支付时的注意事项

- **如果出现以下情况，请停止购买：**
- **要求提供过多信息**：任何表单要求提供 SSN、护照号码、银行账户号码、密码问题或安全问题。
- **价格过低**：价格比市场价低 50-90%，或者商品大幅打折。
- **没有退换货政策**：缺少退换货政策或退换货信息不明确。
- **缺少公司信息**：没有显示公司地址、电话号码或关于页面。
- **施加压力**：显示倒计时（刷新后重置），或者使用虚假的限时促销（例如“仅剩 2 件！”）。
- **非标准的支付方式**：网站只接受电汇、加密货币或礼品卡支付，不支持信用卡支付。

### 订阅和自动续费陷阱

- **免费试用 → 自动续费**：一些试用服务在试用结束后会自动转换为付费订阅。应对措施：仅在卡片上加载试用金额。
- **注册简单但取消困难**：有多个“您确定吗？”的确认步骤，或者需要电话确认，或者取消流程繁琐。
- **预选勾选框**：在购买过程中自动勾选了额外的服务。
- **试用时长与宣传不符**：页面显示的试用时长为 30 天，但实际上只有 7 天。
- **价格变动**：试用期后的价格突然上涨。

**预付费卡片的优势：** 如果在卡片余额耗尽后仍尝试订阅，系统会直接拒绝支付——不会产生透支或未经授权的扣费。

## 动态货币转换（DCC）——务必拒绝

如果支付页面提供“以您的货币显示价格”或“为了方便而使用 USD 支付”这样的选项，这通常意味着使用了动态货币转换（DCC）。这种方式会额外收取 3-5% 的手续费。**始终使用商家的本地货币进行支付**，并让支付网络负责货币转换。

## 理解卡片交易

### 授权与扣款

每笔卡片交易分为两个阶段：
1. **授权**：商家会询问是否有足够的资金。授权过程中会暂时冻结余额，但资金并不会立即转移。
2. **扣款/结算**：商家实际收取资金。资金可能在授权后的几小时或几天内才会转移。

**关键注意事项：**
- “待处理”状态并不意味着资金已经转移——不要将其视为交易完成。
- 即使没有发生扣款，授权过程中的余额冻结也会减少。
- 如果商家在 7-10 天内未完成扣款，冻结的余额会解除。
- 单次订单可能会触发多次扣款（例如分批发货）。

### 交易状态

| 状态 | 对余额的影响 | 是否会转移资金 |
|-------|-----------------|-------------|
| 授权冻结 | 是 — 减少余额 | 不会 |
| 待处理 | 是 — 减少余额 | 资金正在转移中 |
| 结算/完成 | 是 — 扣除余额 | 是 |
| 解冻冻结 | 是 — 恢复余额 | 不会 |

### 退款时间线

- **授权撤销**（在扣款之前）：1-3 天——如果可能，请在发货前请求取消。
- **扣款后的退款**：从商家发起退款请求后的 5-10 个工作日内处理。
- **退款**（强制撤销）：最长需要 90 天——这是最后的补救措施。商家的退款处理时间：Visa 9 天（美国）/ 18 天（国际）。

### 常见的拒付代码

| 代码 | 含义 | 应采取的措施 |
|------|---------|--------|
| 05 — 不允许支付 | 通用拒付原因（欺诈检测、政策原因、AVS 核对不符） | 检查余额，核实账单信息，再次尝试 |
| 51 — 资金不足 | 余额低于交易金额 | 补充资金 |
| 54 — 卡片过期 | 卡片已过期 | 发行新卡片 |

### 何时停止重试

**在以下情况下可以重试：** 在收到代码 91（临时拒付原因）后，或者在解决了根本问题后（例如在代码 51 后补充了资金，或在 AVS 核对失败后修正了账单地址后）。

**禁止重试的情况：** 在收到代码 51 后（表示存在欺诈风险），或者在 CVV 输入错误后重试，或者在连续三次拒付后仍未解决问题时，或者在商家明确表示不允许重试的情况下。

## 虚拟卡片的安全性 — 为什么您的卡片是安全的

### 结构保护

MoneyClaw 的虚拟卡片提供了以下安全保障：
- **余额限制**：卡片只能使用已加载的金额进行消费，无法超出余额。
- **隔离保护**：卡片与银行账户隔离，即使卡片被盗用，也不会影响其他账户的资金。
- **每个账户限用一张卡片**：目前每个账户只能使用一张激活的卡片。如果卡片被盗用，请联系客服进行停用和重新发行。

### 智能充值策略

- **每次消费仅加载所需金额 + 5-10% 的缓冲金额**：例如，如果购买金额为 50 美元，只会加载 55 美元。如果商家尝试扣取 500 美元，系统会自动拒绝。
- **订阅管理**：每次充值只会加载一个账单周期的金额加上缓冲金额。如果您想取消订阅，只需停止充值即可。
- **货币缓冲**：对于跨境交易，会额外加上 5% 的手续费。
- **支付限制**：酒店、租车、加油站等场景会额外收取手续费。

### 可使用的场景与不可使用的场景

**允许使用的场景**：电子商务、SaaS 订阅、在线广告、旅行预订、API 服务、云计算服务、数字产品。

**卡片网络通常禁止的场景**：赌博（MCC 7801/7802/7995）、成人内容、加密货币交易（MCC 6051）、货币转账、枪支相关服务。

如果交易被拒付（代码为 57），可能是因为商家拒绝了交易。
