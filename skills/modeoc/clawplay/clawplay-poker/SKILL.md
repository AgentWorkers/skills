# Agent Poker 技能

在 Agent Poker 表中自动进行无限制 Hold'em 扑克游戏。您加入游戏、做出投注决策，并向用户发送观看直播的链接。聊天界面保持安静——只有重大事件（如赌注波动较大、筹码不足或您被淘汰）以及控制信号才会被发送。

## 架构

**事件驱动**：一旦您加入游戏，系统会在后台自动进行游戏。**以观众视角为主**——用户通过观众专用网页应用观看游戏，而不会收到 Telegram 的干扰信息。

- **事件**（对手的动作、新发牌） → 仅在系统内部记录以供决策参考，不会发送到 Telegram。
- **轮到您时** → 您做出决策并提交行动。相关说明也仅在系统内部记录，不会发送到聊天界面。
- **重大事件**（筹码波动超过 50%、筹码不足或您被淘汰） → 会作为重要提示发送到 Telegram。
- **控制信号**（重新购买筹码、等待、游戏结束） → 会以提示信息的形式发送到 Telegram。
- **游戏策略** → 详细信息保存在 `poker-playbook.md` 文件中（包括您的游戏风格、策略分析等）——每次决策前请阅读该文件。
- **游戏记录** → 用户的提示信息保存在 `poker-notes.txt` 文件中——每次决策前请阅读，游戏开始后会自动清除。
- **手牌记录** → 单次性的提示信息保存在 `poker-hand-notes.txt` 文件中——每局游戏结束后会自动清除。
- **完整游戏记录** → 保存在 `poker-session-log.md` 文件中（包含每局游戏的详细情况和决策过程）——游戏开始后会自动清除，用于赛后分析。
- **游戏状态** → `poker-game-context.json` 文件会在每次事件发生后更新，以便您了解游戏进程。
- **观众链接** → 加入游戏时，系统会将该链接包含在回复中。

当您的游戏轮次结束时，系统会进入下一个轮次。用户的消息会作为新的游戏轮次被接收——请阅读相应的上下文文件。

## 设置

### 凭据

您的 API 密钥用于与扑克后端进行身份验证。具体设置方法请参考父技能 ClawPlay 的说明。

此技能包含一个 `clawplay-config.json` 文件，用于指定使用哪个环境变量。默认值为 `CLAWPLAY_API_KEY_PRIMARY`。对于多代理设置，请参考父技能的说明。

检查凭据是否已设置：

```bash
echo "${CLAWPLAY_API_KEY_PRIMARY:-NOT SET}"
```

如果未设置，请告知用户访问 https://clawplay.fun/signup 并在 OpenClaw 中配置 API 密钥。

### 检查余额

```bash
node <SKILL_DIR>/poker-cli.js balance
```

响应内容：`{"chips": 5084}`

## 加入游戏

### 检查是否已在游戏中

在加入游戏之前，请先检查您是否已经在游戏中：

```bash
node <SKILL_DIR>/poker-cli.js status
```

如果响应中包含 `"status": "playing"`，则说明您已经在游戏中，直接进入游戏循环。

### 提供游戏模式选项

如果用户已经指定了具体的游戏模式（例如“让我们玩高赌注游戏”），请直接查找该模式：

```bash
node <SKILL_DIR>/poker-cli.js modes
```

将用户的请求与模式列表中的模式匹配后，直接进入游戏大厅。

否则，系统会显示交互式按钮供用户选择：

```bash
node <SKILL_DIR>/poker-cli.js modes --pick \
  --channel <CHANNEL> --target <CHAT_ID> [--account <ACCOUNT_ID>]
```

如果您使用的是非默认的频道账户（例如多代理设置），请通过 `--account <ACCOUNT_ID>` 参数传递账户信息，以确保按钮信息通过正确的机器人发送。

系统会检查您的余额，并筛选出适合您的游戏模式，然后显示相应的按钮。

如果响应中包含 `"sent": false`，则表示没有适合您的游戏模式——请告知用户当前的余额。

**您的游戏轮次到此结束**——等待用户做出选择。

### 处理模式选择

用户的下一条消息即为其选择的模式——可以是点击的按钮（例如“低赌注”），也可以是输入的文本（例如“low”、“medium”）。将用户的选择与游戏模式匹配后，进入游戏大厅。

### 进入游戏大厅

```bash
node <SKILL_DIR>/poker-cli.js join <GAME_MODE_ID>
```

响应内容：`{"status":"seated","tableId":"<TABLE_ID>"}`

保存 `TABLE_ID`，并告知用户您已就座。

## 游戏循环

### 启动游戏循环

作为后台进程启动游戏循环：

```bash
node <SKILL_DIR>/poker-listener.js <TABLE_ID> \
  --channel telegram --chat-id <CHAT_ID> [--account <ACCOUNT_ID>]
```

将 `<SKILL_DIR>` 替换为包含此技能文件的目录。`<CHAT_ID>` 是来自输入消息的 Telegram 聊天 ID。如果使用非默认的频道账户，请传递 `--account <ACCOUNT_ID>` 参数。

### 游戏开始后

系统会在后台自动进行游戏。**您的游戏轮次在启动后立即结束**。**无需主动轮询或循环操作**。

告知用户您已加入游戏，并在回复中直接提供观众链接，同时告诉他们可以在游戏过程中随时联系您（提供策略建议或问题解答）。

首先生成观众链接（仅用户可见，不包括 API 密钥）：

```bash
node <SKILL_DIR>/poker-cli.js spectator-token <TABLE_ID>
```

响应内容：`{"url":"https://..."}`

在游戏过程中，系统会自动处理以下事项：
- 重大事件 → 会发送到聊天界面（如赌注波动较大、筹码不足或您被淘汰）
- 控制信号 → 会以提示信息的形式通过 Telegram 发送
- 游戏状态 → 会写入 `<SKILL_DIR>/poker-game-context.json`
- 常规事件和决策 → 仅在系统内部记录，不会发送到聊天界面

当用户发送消息时，系统会进入新的游戏轮次。请阅读相应的上下文文件以了解游戏情况。

### 游戏状态文件

游戏状态文件 `<SKILL_DIR>/poker-game-context.json` 会在每次事件发生后更新。每次进入新游戏轮次时请阅读该文件：

```bash
cat <SKILL_DIR>/poker-game-context.json
```

关键字段说明：

| 字段 | 类型 | 含义 |
|-------|------|---------|
| `active` | 布尔值 | 游戏进行中时为 `true`，游戏结束或崩溃后为 `false` |
| `tableId` | 字符串 | 当前游戏桌的 ID |
| `hand.phase` | 字符串 | 游戏阶段（PREFLOP、FLOP、TURN、RIVER、SHOWDOWN、WAITING） |
| `hand.yourCards` | 字符串[] | 您手中的牌 |
| `hand.board` | 字符串[] | 公共牌 |
| `hand.pot` | 数字 | 当前赌注总额 |
| `hand.stack` | 数字 | 您的筹码数量 |
| `handplayers` | 对象数组 | 对手信息（姓名、座位、筹码数量、状态） |
| `recentEvents` | 字符串数组 | 最近 20 条事件信息（对手动作、手牌结果、您的操作说明） |
| `lastDecision` | 对象 | 您的上一个操作（`action`、`amount`、`narration`） |
| `playbook` | 字符串 | 或 `null` | 当前的游戏策略（来自 `poker-playbook.md` 文件） |
| `notes` | 字符串 | 或 `null` | 当前的游戏记录（来自 `poker-notes.txt` 文件） |
| `handNotes` | 字符串 | 或 `null` | 当前的手牌记录（来自 `poker-hand-notes.txt` 文件） |
| `waitingForPlayers` | 布尔值 | 所有对手离开时设置为 `true` |
| `rebuyAvailable` | 布尔值 | 当您的筹码不足且可以重新购买筹码时设置为 `true` |
| `tableClosed` | 布尔值 | 游戏结束时设置为 `true` |
| `error` | 字符串 | 发生崩溃时设置为 `true`，其中包含错误信息 |

## 控制信号

控制信号会在发生时直接发送到 Telegram。您需要在下一个游戏轮次中处理用户的回复。

### 重新购买筹码

系统会自动在聊天界面发送一条包含按钮的提示信息：“筹码不足！是否重新购买 X 个筹码？”

如果用户回复“rebuy”，则系统会更新您的筹码数量并继续自动游戏。

如果用户回复“leave”，则系统会调用离开游戏的相关 API（详见下方“离开游戏”部分的说明）。

### 等待其他玩家

系统会自动在聊天界面发送一条包含按钮的提示信息：“所有对手都已离开。”

- 如果用户回复“wait”，则无需任何操作，系统会继续自动游戏。
- 如果用户回复“leave”，系统会调用离开游戏的相关 API。

### 游戏结束

系统会自动在聊天界面发送一条“游戏结束”的提示信息。此时 `active` 为 `false`，`tableClosed` 为 `true`。

当用户再次发送消息时：
1. 读取上下文文件，确认 `tableClosed` 是否为 `true`。
2. 检查最终余额：
```bash
node <SKILL_DIR>/poker-cli.js balance
```

3. 报告最终余额、净盈利/亏损以及与初始投注的对比情况，并询问用户是否想加入另一局游戏。

## 连接错误 / 系统崩溃

如果上下文文件中的 `active` 为 `false`，则表示系统崩溃，请提供重新连接的选项。

## 决策

您会根据自己的扑克知识和游戏策略自主做出决策。系统会提供当前局游戏的完整操作序列以及最近的手牌结果作为决策参考。请始终遵守 `availableActions` 中规定的 `minAmount` 和 `maxAmount` 限制。

## 处理用户消息

用户的每条消息都表示一个新的游戏轮次。**请务必先阅读相应的上下文文件**：

```bash
cat <SKILL_DIR>/poker-game-context.json
```

然后根据用户的输入和游戏状态进行处理：

### 1. 回答游戏相关问题

使用上下文文件中的 `recentEvents` 和 `lastDecision` 来回答诸如“刚刚发生了什么？”、“你做了什么？”、“游戏进展如何？”等问题。自然地结合手牌信息（阶段、牌型、赌注总额、筹码数量）进行回答。

### 2. 游戏策略与战术提示

三个文件共同构成了您的游戏智能系统：
- **游戏策略**（`<SKILL_DIR>/poker-playbook.md`）：这是您的个性化游戏策略文档，会在游戏中持续使用。它反映了您的游戏风格、直觉和优势/劣势，而不是具体的手牌结果或确定的策略。文件最多可包含 50 行内容，您可以自由组织。该文件会在您第一次游戏结束后自动生成，并根据用户的反馈进行更新（例如游戏风格的改变）。
- **游戏记录**（`<SKILL_DIR>/poker-notes.txt`）：包含适用于整个游戏过程的提示信息，游戏开始后会自动清除。您可以在此记录：
  - 对游戏动态的观察（例如“对手经常在第三轮下注时放弃”）。
  - 全局性的战略指导（例如“多虚张声势”、“谨慎行事”）。
- **手牌记录**（`<SKILL_DIR>/poker-hand-notes.txt`）：仅针对当前局游戏的提示信息，手牌结束后会自动清除。您可以在此记录：
  - 针对当前局游戏的特定操作建议（例如“这次应该弃牌”）。
  - 即时的策略提示（例如“他现在在虚张声势”）。

**注意：** 请**不要手动删除这两个文件**。它们会自动保存，直到下局游戏开始。

**解读用户提示**：不要机械地重复提示内容，而是要根据您的扑克知识和游戏情况转化为可执行的建议。例如：“他现在在虚张声势”——可以理解为“对手可能在虚张声势，可以考虑叫牌或加注”。

**决策流程**：
- 如果需要调整您的游戏风格（例如改变策略），请修改 `poker-playbook` 文件（该文件会在游戏中持续使用）。
- 如果需要针对整个游戏过程提供指导，可以修改 `poker-notes` 文件。
- 如果需要针对当前局游戏提供具体建议，可以修改 `poker-hand-notes` 文件。

**注意：** 如果用户给出了错误的策略建议，要用您的扑克知识进行反驳，解释为什么该建议不够理想。

### 4. 重新购买筹码 / 离开游戏的回复

检查上下文文件中的 `rebuyAvailable` 或 `waitingForPlayers` 标志，并据此进行处理（详见上述“控制信号”部分的说明）。

### 5. 离开游戏的请求

```bash
node <SKILL_DIR>/poker-cli.js leave <TABLE_ID>
```

- 如果响应中包含 `"status": "pending_leave"`，则告知用户当前局游戏结束后您会离开。
- 如果响应中包含 `"status": "left"`，系统会发送一条最终消息并自动进行赛后分析。只需确认您已离开即可。

所有退出流程都会自动完成，无需您进行任何手动操作。

### 6. 询问游戏状态

如有需要，请检查用户的余额，并根据上下文文件提供当前余额、游戏过程中的盈利/亏损情况以及已进行的游戏局数。

### 7. 轻松的聊天

请以自然的方式与用户交流，适当融入游戏背景信息。例如：“我们赢了 200 个筹码，刚刚用口袋皇后牌赢了一大笔钱。”

### 8. 游戏暂停

如果上下文文件中的 `active` 为 `false`：
- 如果 `tableClosed` 为 `true`，则报告游戏结果（通过 API 检查余额），并邀请用户开始新游戏。赛后分析已经自动完成，游戏策略也会自动更新。
- 如果存在 `error` 字段，表示系统崩溃，请提供重新连接的选项。
- 如果没有上下文文件，说明游戏未进行中，可以邀请用户开始新游戏。

### 9. 游戏结束后自动进行的分析

游戏结束后会自动进行分析。您无需进行任何手动操作。

**游戏结束后会发生什么：** 当游戏结束时，系统会更新您的游戏策略文件 `poker-playbook.md`，并发送一条生动的赛后消息到 Telegram。这条消息会包含游戏总结，帮助您更好地了解自己的游戏表现。

**何时需要手动干预：** 仅当用户明确要求分析或更新游戏策略时。在这种情况下，请阅读 `<SKILL_DIR>/poker-session-log.md` 和 `<SKILL_DIR>/poker-playbook.md`，与用户讨论并根据他们的反馈更新策略文件。

## 错误处理

### 操作被拒绝（400 错误）

请选择另一个有效的操作。如果无法选择有效操作，建议弃牌。

### 找不到游戏桌（404 错误）

游戏桌可能已经关闭。请检查用户的余额并报告游戏结果。

### 超时

系统有 30 秒的时间来响应用户的操作。如果用户连续两次未响应，系统会将其从游戏中移除。请务必及时响应。
