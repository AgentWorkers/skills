# 跟踪视频分析（Tracked Video Analysis）

此技能适用于处理**时长较长、背景噪音较大或操作过程较为复杂的视频**，在这些情况下，可靠的分析结果与最终的总结内容同样重要。

核心步骤如下：

1. **首先提取视频内容**。
2. **对提取的内容进行结构化处理**。
3. **明确记录整个分析过程的每个阶段**。

除非有操作系统进程在运行或存在最新的状态文件证明，否则切勿声称某个后台进程仍在运行。

## 核心工作流程

### 1) 确保视频数据的安全获取

优先选择以下数据来源：
- 直接访问本地文件
- 直接下载的视频链接
- 上传的文档文件
- 备用的外部文件存储服务

如果无法访问聊天中的媒体文件，请直接请求视频的下载链接，而不要无限次地尝试重新获取。

工作目录设置为 `tmp/video_analysis/`。

### 2) 在非root权限环境下准备所需工具

建议使用工作区内的本地软件包，而非系统安装的软件包。

常用的本地工具包括：
- `ffmpeg-static`
- `ffprobe-static`
- `@xenova/transformers`
- `wavefile`

如果无法使用root权限或高级权限安装软件包，请**不要**延迟任务进度——尽可能在工作区内进行本地安装。

### 3) 运行视频内容提取

提取过程应生成以下文件：
- `tmp/video_analysis/status.json`
- `tmp/video_analysis/progress.log`
- `tmp/video_analysis/transcript.jsonl`
- `tmp/video_analysis/analysis.md`

**注意事项**：
- 尽量采用分块提取的方式，而非一次性处理整个视频。
- 为了保证稳定性，优先选择性能较快的自然语言处理（ASR）算法。
- 每提取完一部分内容后，立即更新状态信息。
- 如果提取过程失败，应从已提取的数据中继续处理，而不是重新开始。

### 4) 进行视频内容的最终结构化处理

结构化处理应生成以下文件：
- `tmp/video_analysis/final_status.json`
- `tmp/video_analysis/final_progress.log`
- `tmp/video_analysis/final_analysis.md`

此阶段需要完成以下任务：
- 删除冗余内容及重复的短语
- 将相关内容分组
- 判断内容的类别
- 规范化文本表达
- 将原始的文字记录转换为用户要求的格式。

### 5) 如实报告分析进度

遵循以下规则：
- **提取过程正在进行中** → 报告 `status.json` 的更新情况。
- **提取完成，且没有后台处理进程在运行** → 直接说明这一点。
- **最终的结构化处理正在进行中** → 报告 `final_status.json` 的生成情况。
- **分析结果已准备好** → 阅读 `final_analysis.md` 并提供相应的反馈。

## 标准输出格式

常见的输出格式包括：
- **类别 → 功能 → 描述 → 益处**
- **类别 → 功能 → 简短描述**
- **功能列表及时间戳**
- **包含不确定性说明的清晰总结**

对于背景噪音较大的视频，优先选择易于阅读的文本规范化处理方式，而非追求高精度。

## 状态报告规范

除非满足以下条件之一，否则切勿声称“分析过程正在运行”：
- 操作系统进程仍在运行。
- 相关的状态文件正在被实时更新。

如果提取任务已完成，请明确说明：
- 提取任务已经完成。
- 没有任何后台处理进程在运行。
- 仅剩下结构化处理步骤（如果确实如此）。

## 需要时查看相关文件

- 查阅 `references/pipeline.md` 以了解标准的分析流程和故障处理方法。
- 使用 `scripts/transcribe_tracked_light.mjs` 作为提取任务的起点。
- 使用 `scripts/final_structurer.py` 作为初始结构化处理的起点。

## 交付文档的风格

建议使用简洁、易于阅读的文档格式。

当用户需要一份格式规范的交付成果时：
1. 创建一个格式清晰的 `.md` 文件。
2. 保持文档结构的整洁美观。
3. 如有需要，可将文档以文件形式发送给用户。

## 实用提示

此技能主要侧重于**操作的可靠性**，而非完美的文本转录准确性。如果自然语言处理结果存在误差，应提供有用的结构化摘要，并明确说明其中的不确定性，而不是假装原始文本是准确的。
