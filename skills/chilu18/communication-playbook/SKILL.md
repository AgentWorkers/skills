# HeySalad 沟通手册

本手册汇集了全球顶尖的沟通标准，适用于八个不同领域。在 HeySalad 中生成、审查或指导任何形式的沟通时，请务必遵循这些原则。

---

## 快速决策树

在生成任何沟通内容之前，首先确定沟通的领域：

| 领域 | 适用场景 |
|--------|--------------|
| **内部沟通** | Slack、团队邮件、异步更新、文档 |
| **外部沟通** | 投资者更新、监管机构沟通、客户/合作伙伴邮件 |
| **演示文稿** | 演示文稿、全员会议、利益相关者评审 |
| **书面沟通** | 文档、维基、变更日志、提案、报告 |
| **跨职能沟通** | 多团队项目、RACI 规范、项目启动会议 |
| **会议** | 会议议程、会议引导、会议记录、决策制定 |
| **透明度** | 应分享的内容、分享方式及对象 |
| **积极倾听** | 一对一辅导、冲突解决、谈判、反馈 |

通常会同时涉及多个领域——请使用所有相关的内容。

## 1. 内部沟通

### 核心原则：先说结论（BLUF）

每条内部信息都应以结论开头，明确说明你的需求，再提供支持性背景信息。切勿隐藏请求内容。

**错误示例：**“嘿，我在查看 MTN Mobile Money 的集成情况时发现了一些延迟问题。日志显示昨天 14:00 UTC 有错误记录。这可能会影响结算。你能查一下吗？”

**正确示例：**“**需要采取行动**：MTN Mobile Money 的结算可能出现问题。日志显示 14:00 UTC 有错误记录，请在今天结束前调查并反馈。详细信息如下。”

### 四大要素
- **清晰性**：每条信息只有一个明确的目的，避免使用专业术语或含糊其辞的表述。
- **节奏性**：更新信息要有规律，团队能知道信息何时到达。
- **背景信息**：不仅要说明“发生了什么”，还要说明“为什么发生”。
- **确认**：确保信息被接收、被理解，并明确下一步行动。

### 通道选择
| 通道 | 适用场景 | 应避免的场景 |
|---------|---------|-----------|
| Slack/即时通讯 | 快速提问、实时协调 | 敏感话题、复杂决策 |
| 邮件 | 正式更新、需要保留书面记录 | 紧急问题、需要来回沟通 |
| 视频通话 | 复杂话题、建立关系 | 简单的状态更新 |
| 异步视频 | 演示文稿、操作指南、一对多沟通 | 需要双向交流 |
| 书面文档 | 决策性内容、流程说明、长期参考 | 对时间敏感的沟通 |
| 面对面沟通 | 战略讨论、头脑风暴、冲突解决 | 团队分散时使用 |

### 异步信息发送前的检查清单
在发送任何异步信息之前，请确认：
- [ ] 第一行是否明确了目的？
- [ ] 是否指定了回复负责人（仅用于告知信息）？
- [ ] 是否包含了截止日期或预期回复时间？
- [ ] 选择的通道是否合适？
- [ ] 是否只提出一个请求，而不是多个请求？

### 文化规范（在沟通辅导中融入）
- 在不确定的情况下，倾向于过度沟通。
- 始终假设信息发送者的意图是积极的。
- 公开表扬，私下提出批评——切勿在团队通道中指责个人。
- 随时记录决策内容——记忆不可靠。
- 如果有不同意见，要明确表达并执行相应的决策。

## 2. 外部沟通

### 五大原则
1. **统一口径**：所有外部信息在定位、数字和语气上要保持一致。
2. **精确性**：避免使用模糊的语言（如“很快”、“也许”），使用具体的日期和数字。
3. **专业性**：所有内容都要经过拼写检查，并根据受众的正式程度调整语言。
4. **及时性**：在 24 小时内回复。即使无法立即给出完整答案，也要确认收到信息。
5. **保密性**：假设所有外部沟通内容都可能被转发。

### 投资者更新（每月一次）
每份投资者更新的内容结构如下：
1. **标题**：本月发生的最重要事项。
2. **进展**：与上月相比的关键指标和里程碑。
3. **挑战**：存在的问题及应对措施。
4. **优先事项**：下个月的三大重点任务。
5. **请求**：投资者可以提供的具体帮助方式。

> 定期收到一致且诚实的投资者更新（包括负面信息）的投资者，比那些只在情况良好时才收到信息的投资者，更有可能在危机中支持你。

### 监管机构沟通（FCA / 赞比亚银行 / EFIU）
- 始终使用官方头衔和参考编号。
- 绝不错过监管机构的截止日期。
- 记录所有沟通内容：日期、联系人姓名、内容摘要、回复截止日期。
- 除非被明确要求，否则不要进行猜测或做出前瞻性陈述。
- 在发送前，务必由第二个人审核每份监管机构的回复。
- 如果不确定，应咨询法律顾问——切勿自行揣测监管规定。

### 客户/合作伙伴沟通——三步回应流程
1. **确认接收**：确认收到信息并理解内容。
2. **采取行动或升级处理**：说明解决方案或下一步的具体安排及时间表。
3. **给予肯定**：以自信的态度结束沟通：“我们已经处理好了。”

## 3. 演示文稿技巧

### 金字塔原则（适用于所有演示文稿结构）
每张幻灯片只传达一个主要观点。观众不应对你要表达的内容感到困惑。

### 幻灯片设计规则
| 规则 | 原则 |
|------|-----------|
| 每张幻灯片只展示一个观点 | 如果你说了“还有……”，则需要新建一张幻灯片。 |
| 每个要点不超过六个字 | 每个要点最多六个字，优先使用视觉元素。 |
| 标题要能传达核心信息 | 例如：“收入环比增长 40%”而非“收入”。 |
| 保持视觉语言的一致性 | 使用统一的字体、配色方案和图标集。 |
| “空白测试” | 核心信息是否能在 5 秒内被理解？ |
| 去除无意义的装饰元素 |

### 演示文稿交付检查清单
- [ ] 开场要有吸引力——通过故事、数据点或引人深思的问题引起注意。
- [ ] 在前 60 秒内明确主旨思想。
- [ ] 使用引导语：“首先……其次……最后……”
- [ ] 重要观点之后要有短暂的停顿。
- [ ] 语速要有所变化——关键观点时语速慢一些，介绍背景时语速快一些。
- [ ] 结尾要有明确的行动号召。
- [ ] 至少练习三次演讲。

### 处理问题
- 回答问题前先重复问题（这有助于思考并确认理解）。
- “我想给出准确的答案——明天会以书面形式回复你。”
- 绝不要敷衍了事。坦诚表达不确定性比给出错误答案更可信。

## 4. 书面沟通

### 五大原则
1. **简洁性**：用最简洁的语言表达完整的意思。
2. **清晰性**：使用简单、具体的语言。尽可能避免使用专业术语。
3. **具体性**：不要说“很快”，而要说“周五 17:00 前完成”；不要说“一些用户”，而要说“34%”。
4. **主动语态**：使用主动语态，例如“团队部署了更新”，而非“更新被部署了”。
5. **以读者为中心**：思考读者阅读后需要做什么或相信什么。

### 邮件结构
- **主题**：具体、具有行动导向，开头明确说明目的：“**需要采取行动**：FCA 提交材料——3 月 14 日”。
- **开头**：如有必要，提供一段背景信息。
- **结论**：用 1-2 句话概括请求或核心信息。
- **正文**：仅包含必要的支持性细节。
- **行动部分**：明确下一步行动的负责人、截止日期和格式要求。
- **结尾**：使用“最佳”或“谢谢”等礼貌用语。

> 在确定邮件内容后，再填写主题。

### 根据受众调整沟通方式
| 受众 | 语气和风格 | 应避免的内容 |
|----------|-------------|-------|
| 投资者 | 直接、数据驱动、自信的态度，以结果为导向 | 过度推销、模糊的里程碑、隐藏负面信息。 |
| 监管机构 | 正式、精确、尊重对方的语言。使用他们的术语。 | 推测性语言、非正式的表达方式。 |
| 客户 | 温暖、人性化、富有同理心。用最简洁的方式解决问题 | 使用专业术语、拖延责任、推卸责任。 |
| 团队内部 | 清晰、直接、具有激励性。提供背景信息并保持诚实。 | 使用公司官方语言、表述含糊不清。 |
| 媒体和公众 | 控制信息量、具有策略性，便于引用。 | 避免使用不可靠的信息。 |
| 文档编写**：假设读者对背景一无所知（他们可能在 6 个月后阅读文档）。 |
- 为每份文档标注日期，并指定负责人负责维护其更新。 |
- 链接相关文档，避免重复内容。 |
- 归档文档，不要删除——过去的决策也有参考价值。

## 5. 跨职能协作

### RACI 框架（在每个多团队项目开始时使用）
| 角色 | 定义 | 规则 |
|------|-----------|------|
| **负责人** | 负责执行具体工作 | 可以是多人。 |
| **责任人** | 对最终结果负责 | 只有一个人。 |
| **咨询者** | 在决策前需要他们的意见 | 双向沟通。 |
| **知情者** | 仅被告知结果 | 单向沟通。 |

> 警告：如果多个角色都被标记为“责任人”，则实际上没有人真正负责。这是 RACI 模型最常见的失败原因。

### 项目启动会议——六个必备问题
1. 我们为什么要做这个项目？（战略背景和紧迫性）
2. 成功的标准是什么？（具体、可衡量的结果。）
3. 谁负责什么？（明确责任分配。）
4. 面临哪些限制？（时间表、预算、监管要求、技术限制。）
5. 我们将如何沟通？（沟通的节奏、渠道和升级流程。）
6. 已知的风险有哪些？（障碍和应对措施。）

### 单一信息来源（SSOT）
每个跨职能项目都必须有一个指定的信息来源——一个文档、一个平台或工具，用于记录所有决策、进度更新和变更。如果某个信息没有记录在 SSOT 中，那么这个信息就不存在。

### 冲突解决流程
1. 清晰表达分歧，避免指责：“我的看法不同。”
2. 明确各方实际的需求（而不仅仅是他们提出的要求）。
3. 寻找能满足双方需求的解决方案。
4. 如果问题无法解决，将问题上报给负责人，并附上双方的立场摘要。
5. 负责人做出决定并执行。

## 6. 会议引导

### 会议类型及用途 | 会议时长 | 会议频率 |
|------|---------|----------|-----------|
| 每日站会 | 完成的工作/下一步计划/障碍处理 | 每日 |
| 每周团队同步会议 | 进展情况、优先事项、障碍处理 | 每周/每两周 |
| 一对一会议 | 建立关系、提供辅导、收集反馈 | 每周/每两周 |
| 决策会议 | 审议特定决策 | 根据需要安排时间 |
| 回顾会议 | 团队流程反思 | 每月/每个冲刺阶段 |
| 全员会议 | 公司更新、文化分享、问答 | 每月 |

### 会议准备
- [ ] 这次会议真的有必要吗？或者可以异步解决吗？
- [ ] 会议议程是否提前 24 小时以上编写并分享？
- [ ] 所有受邀者都必不可少吗？需要删除“仅被告知情况”的人吗？
- [ ] 是否明确了会议决策或结果？
- [ ] 是否指定了会议记录员和计时员？
- [ ] 会议材料是否已经提前分发给了所有参与者？ |

### 会议进行中
- **开场时明确目的**：在会议开始前 60 秒内说明会议目的和预期结果。
- **记录会议中提到的非主题性内容**：将重要的但与会议主题无关的事项记录下来。
- **鼓励沉默者发言**：在会议结束前，询问“[某人]，你的看法是什么？”
- **推动决策**：明确讨论方向：“我们已经从多个角度讨论过这个问题——我们最终会选择什么？”

### 会议结束后（24 小时内）
会议记录必须包括：
- 会议日期和参与者名单。
- 会议做出的决策。
- 分配的行动任务及负责人和截止日期。
- 记录在会议中提到的非主题性内容及负责人。

会议记录需与所有参与者和相关利益相关者共享。行动任务需在共享系统中跟踪，而不仅仅是通过邮件。

## 7. 透明度

### 透明度的层次
| 透明度层次 | 含义 | 适用场景 |
|-------|--------------|---------|
| 默认公开**：所有信息无需请求即可获取 | 文档、维基、路线图。 |
| 应请求提供**：任何请求者均可获取 | 敏感但非机密的信息。 |
| 仅限相关人员知晓**：仅限于直接相关的人员 | 人员信息、法律策略相关内容。 |
| 保密信息**：需要正式授权才能查看 | 董事会相关内容、监管机构要求的文件。 |

### 领导层透明度的规范
- 公开解释决策背后的原因——背景信息有助于自主决策。
- 及时公开承认错误——在错误成为谣言之前主动承认。
- 如果无法分享某些信息： “我无法在这里透露详细信息，但会尽快更新您。”
- 绝不要让员工通过非正式渠道提前得知重要信息。

### 心理安全（透明度的基础）
- 对负面信息要表现出好奇心：“请告诉我更多细节。”
- 表扬那些提前发现问题的人。
- 正常化不确定性：“我目前还不知道。”
- 绝不要利用私下分享的信息在后续的争论中攻击他人。

### 积极倾听的技巧
| 听力层次 | 描述 | 目的 |
|-------|-------------|------|
| 第一级**：表面倾听**：说话者说话时只考虑自己的回应。 | 应避免这种倾听方式。 |
| 第二级**：专注倾听**：全神贯注地听，注意说话者的言辞和肢体语言。 | 这是基本的倾听方式。 |
| 第三级**：全面倾听**：关注说话者的所有表达，包括沉默和未说出口的内容。 | 在高风险的对话中尤为重要。 |

### 六种核心倾听技巧
1. **暂停**：在回答前等待 2-3 秒，表示你在思考。 |
2. **反思性倾听**：用自己的话复述对方的话：“我听到的内容是……，是这样吗？” |
3. **澄清性提问**：“那么实际情况会怎样？” / “你为什么会有这样的担忧？” |
4. **表示理解但不表示同意**：“我明白你的感受。” |
5. **非语言性倾听**：保持眼神交流、舒适的姿势；在视频通话中不要把摄像头对着屏幕。 |
6. **关注未说出口的信息**：注意遗漏的参与者、被忽略的信息点或被回避的议题。

### 常见的倾听失误
| 失误类型 | 表现形式 |
|---------|--------------------|
| 准备好回答**：说话者说话时眼睛呆滞。 |
| 选择性倾听**：只听符合自己观点的内容。 |
| 打断**：在说话者说完之前就插话。 |
| 过早提供建议**：在未完全理解问题之前就给出解决方案。 |
| 分心**：使用手机、开启通知或环境干扰。 |

### 根据不同场景调整倾听方式
- **一对一沟通**：团队成员应占谈话时间的 60%以上。提问时不要主动提供建议。 |
- **谈判**：对方分享的每一条信息都应被视为重要数据。不要在对方说完之前就打断。 |
- **冲突解决**：在达成解决方案之前，要确保双方的意见都被充分听取。 |

## 输出标准

在生成任何沟通内容时，请遵循以下不可妥协的原则：

### 在写作之前
1. 目标受众是谁？
2. 他们阅读后需要相信什么或采取什么行动？
3. 关键信息是什么？
4. 适合的沟通渠道和格式是什么？
5. 截止日期或期望的回复时间是什么？

### 普遍的质量标准
- 每条信息都有明确的目的和明确的行动号召。
- 全文使用主动语态。
- 表达具体明确——使用具体的日期、数字和人名。
- 阅读内容时先理解内容，再关注字数（尽量减少字数，减少 20%）。
- 如果写作耗时超过 5 分钟，要让第二个人审核。

### HeySalad 的特殊背景
