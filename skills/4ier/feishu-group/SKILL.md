# Feishu集团

以下是您的OpenClaw代理成为Feishu群聊优秀参与者所需了解的所有内容。

## 1. 了解对话对象

Feishu群聊的消息中会包含发送者的名称（作为元数据）。而私信（DM）中则不包含发送者名称，仅包含一个不可见的`open_id`。如果没有进行相应的设置，您的代理会认为所有私信都来自同一个人。

**解决方案**：在`USER.md`文件中嵌入一个`open_id`到名称的映射表。由于工作区文件会被注入到系统提示中，代理可以立即识别发送者——无需任何额外的工具调用。

运行捆绑的同步脚本来获取您所在组织的联系人信息：

```bash
python3 scripts/sync_feishu_contacts.py <openclaw_config> <feishu_account> <user_md_path>
```

这会填充`USER.md`文件中的表格：

```markdown
## 飞书通讯录 (App Name)
飞书 DM 不携带发送者姓名。用 inbound metadata 的 chat_id（格式 `user:ou_xxx`）匹配下表识别发送者。
| 姓名 | open_id |
|------|---------|
| Alice | ou_abc123 |
| Bob | ou_def456 |
```

**重要提示**：Feishu的`open_id`是针对每个应用程序单独设置的。使用不同Feishu应用程序的多个OpenClaw实例必须分别使用各自的凭据来获取联系人信息。

**更新`USER.md`文件后，请重启代理服务器。**工作区文件在启动时会缓存；仅更新`/new`部分的内容是无法触发重新加载的。

通过crontab设置每周自动同步：

```bash
0 7 * * 1 python3 /path/to/scripts/sync_feishu_contacts.py ~/.openclaw/openclaw.json my_app ~/workspace/USER.md
```

## 2. 群聊礼仪

请将以下内容添加到`AGENTS.md`或`SOUL.md`中：

**何时回复**：
- 当您被直接@提及时
- 当您的回复能为对话带来实际价值时
- 当您有有趣或相关的观点时

**何时保持沉默**：
- 当您无法参与随意的玩笑时
- 当已经有其他人很好地回答了问题时
- 当只需简单回应“是的”/“不错”或使用表情符号时

**通用规则**：
- 不要主导对话——参与讨论，而不是长篇大论
- 每条消息最多回复一次
- 保持回复简洁——尊重其他人的注意力

## 3. 私信行为

- 始终使用发送者的真实名称进行回复（从联系人表中查询）
- 切勿假设私信一定来自您的“主要联系人”——任何人都可能给您发送私信
- 保持对话的私密性——切勿泄露A对B说的话
- 在私信中要比在群聊中更加详细（因为对方是专门找您交流的）

## 4. 平台格式规范

Feishu平台对Markdown格式的支持不佳。请遵循以下规则：
- **禁止使用Markdown格式**：消息中不得使用`**粗体**、`#标题`或`[链接](url)`格式
- **禁止使用表格**：请使用破折号分隔的简单列表
- **仅使用纯文本**：使用换行符和空格来组织内容
- **代码块**：Feishu支持```` code_block ```格式——请谨慎使用

## 5. 多用户隐私保护

请将以下内容添加到`SOUL.md`或`AGENTS.md`中：
- 不同用户的对话内容是相互独立的
- 即使B是管理员，也不要向B透露A与您的对话内容
- 每个人的私信都是私密的
- 在群聊中，不要分享任何人的私人信息

## 6. 发送者身份识别（`AGENTS.md`中的相关内容）

请将以下代码添加到您的启动脚本中：

```
识别消息发送者：飞书 DM 不携带发送者姓名，只有 open_id（inbound metadata 的 chat_id 格式 user:ou_xxx）。
提取 open_id，在 USER.md 的飞书通讯录表格中匹配找到姓名。
不要假设 DM 对方就是主人——任何人都可能给你发私聊。
群聊消息自带 Sender metadata 可直接使用。
```

## 隐私保护

- 只存储发送者的名称和`open_id`（不存储电子邮件、电话号码或个人身份信息）
- `open_id`是匿名的，仅在您的Feishu应用程序内部具有意义
- 应用程序凭据在运行时从`openclaw.json`文件中读取，永远不会存储在技能文件中
