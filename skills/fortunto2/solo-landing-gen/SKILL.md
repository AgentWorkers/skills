# /landing-gen

根据项目的 PRD（产品需求文档）生成 landing 页面内容，包括首页、产品特性、用户评价、呼叫行动（CTA）信息、SEO 元标签以及 A/B 测试用的标题变体。如果检测到使用 astro-static 框架，可以自动生成页面文件。

## MCP 工具（如有可用）

- `kb_search(query)` — 查找相关的方法论（如转化策略、文案撰写）
- `project_info(name)` — 获取项目的技术栈和详细信息
- `web_search(query)` — 分析竞争对手的 landing 页面

如果 MCP 工具不可用，可改用 Glob、Grep 和 Read 工具来获取所需信息。

## 步骤

1. **从 `$ARGUMENTS` 中解析项目信息**：
   - 阅读 PRD、README 或 research.md 文件以获取产品相关信息。
   - 如果信息缺失，可通过 AskUserQuestion 功能请求用户提供。

2. **检测项目使用的技术栈**：
   - 检查是否存在 `astro.config.*` 文件：如果存在，则使用 astro-static 框架（可自动生成页面结构）。
   - 检查是否存在 `next.config.*` 文件：如果存在，则使用 Next.js 框架（可自动生成页面路由）。
   - 如果以上文件都不存在，仅生成纯 Markdown 格式的文本内容。

3. **从 PRD/README 中提取 landing 页面所需的信息**：
   - **问题描述**：用一句话概括用户面临的主要痛点。
   - **解决方案**：用一句话描述产品的核心功能。
   - **目标用户群体**（ICP）：明确产品的目标用户群体。
   **产品特性**：列出 3-4 个最具竞争力的产品特性，并附上详细说明。
   **竞争对手**：参考 research.md 文件中的信息，以便了解竞争对手的情况。
   - **定价信息**：如果有的话，也一并列出。

4. **竞争对手 landing 页面分析**（可选，如果 MCP 或 web_search 工具可用）：
   - 分析排名前三的竞争对手的 landing 页面。
   - 注意观察他们的标题格式、呼叫行动语言以及使用的用户评价类型。
   - 找出产品在定位上的优势或不足。

5. **制定转化策略**：
   在生成页面内容之前，先明确以下内容：
   - **主要转化目标**：用户应该执行的动作（注册、下载、购买等）。
   - **潜在障碍**：用户可能遇到的三大障碍。
   **信任信号**：如何克服这些障碍。
   **页面展示内容**：仅展示问题描述、解决方案和呼叫行动信息，避免其他复杂内容。

6. **生成 landing 页面内容**：

   ### 6a. 首页（Hero Section）
   - **标题**：以问题为核心，强调产品带来的好处（8-12 个词）。
   - **副标题**：产品如何解决用户的问题（15-25 个词）。
   - **呼叫行动按钮**：明确用户应该执行的动作（如“立即免费试用”、“立即下载”）。
   - **视觉元素**：确定需要展示的图片、截图或演示内容。

   ### 6b. A/B 测试标题变体（5 个选项）：
   - 设计 5 个不同的标题版本：
     1. 以问题为核心：“厌倦了 {问题} 吗？”
     2. 以产品好处为核心：“{结果}，无需 {麻烦}”
     3. 以用户好奇心为核心：“{形容词} 的 {动作} 方式”
     4. 以用户评价为核心：“加入 {N} + {用户} 的行列，体验 {结果}”
     5. 直接介绍产品优势：“{产品}：{一句话概括的产品价值”）

   ### 6c. 产品特性部分（3-4 个特性）：
   - 为每个特性提供：
     - 图标建议（使用表情符号或图标名称）
     - 特性名称（3-5 个词）
     - 特性描述（1-2 句）
     - 强调用户能获得的好处（而不是产品本身的功能）

   ### 6d. 用户评价部分：
   - 提供用户评价的占位内容（模板包括评价者姓名、职位和评价内容）。
   - 使用占位符显示相关数据（如“{X} + 用户”、“速度提升 {Y}%”、“覆盖 {Z} 个国家”）。
   - 提供信任背书（如应用商店评分、奖项、媒体报道等占位内容）。

   ### 6e. 呼叫行动部分：
   - 重复标题或使用不同的标题变体。
   - 呼叫行动按钮与首页保持一致。
   - 提供风险缓解信息（如“永久免费”、“无需信用卡”、“随时取消”）。

   ### 6f. SEO 元标签
   ```html
   <title>{title — 50-60 chars}</title>
   <meta name="description" content="{description — 150-160 chars}">
   <meta property="og:title" content="{og title}">
   <meta property="og:description" content="{og description}">
   <meta property="og:image" content="{suggest image dimensions 1200x630}">
   <meta property="og:type" content="website">
   ```

7. **输出结果**：
   - 如果检测到使用 astro-static 框架，将生成的页面文件保存在 `src/pages/index.astro` 文件中；否则，保存在 `docs/landing-content.md` 文件中：
   ```markdown
   # Landing Page Content: {Project Name}

   **Generated:** {YYYY-MM-DD}
   **Primary CTA:** {action}
   **Target ICP:** {persona}

   ## Hero
   **Headline:** {headline}
   **Subheadline:** {subheadline}
   **CTA:** {button text}
   **Visual:** {description}

   ## A/B Headlines
   1. {variant 1}
   2. {variant 2}
   3. {variant 3}
   4. {variant 4}
   5. {variant 5}

   ## Features
   {features with icons and descriptions}

   ## Social Proof
   {templates and placeholders}

   ## Final CTA
   {closing section}

   ## SEO Meta Tags
   {html meta tags}

   ---
   *Generated by /landing-gen. Fill in social proof, add visuals, then publish.*
   ```

8. **输出总结**：包括最终的标题、呼叫行动内容以及后续建议。

## 注意事项：

- 标题需要经过 A/B 测试，至少测试 2-3 个不同的版本。
- 页面的主要展示区域（“Above the fold”）应仅包含标题和呼叫行动信息，避免信息过载。
- 用户评价部分为占位内容，实际使用时需替换为真实数据。
- 在测试阶段，只需展示首页、呼叫行动按钮和收集用户邮箱即可（例如在产品发布前 2 小时内）。
- 该工具支持与 /seo-audit 工具配合使用：先生成内容，再对部署后的页面进行 SEO 审计。

## 常见问题及解决方法：

### 问题：找不到 PRD 或产品信息
**原因**：项目缺少 `docs/prd.md` 文件或没有包含产品描述的 README 文件。
**解决方法**：运行 `/validate` 命令生成 PRD 文件，或提供包含问题描述、解决方案和产品特性的 README 文件。

### 标题过于泛化
**原因**：问题描述不够具体，或缺乏对竞争对手的分析。
**解决方法**：添加 `research.md` 文件以了解竞争对手的情况，并根据具体问题定制标题。

### 无法检测到适合生成页面的技术栈
**原因**：项目中没有 `astro.config.*` 或 `next.config.*` 文件。
**解决方法**：默认情况下，该工具会生成纯 Markdown 格式的文本内容。如果需要生成实际页面文件，请确保项目使用了 astro-static 或 Next.js + supabase 框架。
