# /landing-gen

根据项目的 PRD（产品需求文档）生成 landing 页面内容。该工具会生成页面的首页部分、产品特性介绍、用户评价、呼叫行动（CTA）信息、SEO 元标签以及多种 A/B 测试用的标题选项。如果检测到项目使用了 astro-static 框架，还可以自动生成实际的页面文件。

## MCP 工具（如有可用）

- `kb_search(query)` — 查找相关的方法论或写作技巧
- `project_info(name)` — 获取项目的技术栈和详细信息
- `web_search(query)` — 分析竞争对手的 landing 页面

如果 MCP 工具不可用，则需要手动使用 Glob、Grep 和 Read 工具来获取所需信息。

## 步骤

1. **从 `$ARGUMENTS` 中解析项目信息**：
   - 读取 PRD、README 或 research.md 文件以获取产品相关信息。
   - 如果这些文件为空，可以通过 AskUserQuestion 提问用户获取所需信息。

2. **检测项目使用的技术栈**：
   - 查找 `astro.config.*` 文件：如果存在，则表示项目使用了 astro-static 框架，可以自动生成页面结构。
   - 查找 `next.config.*` 文件：如果存在，则表示项目使用了 Next.js 框架，可以自动生成页面路由。
   - 如果没有找到相应的配置文件，则仅生成纯 Markdown 格式的页面内容。

3. **从 PRD/README 中提取 landing 页面所需的信息**：
   - **问题描述**：用一句话概括用户面临的问题。
   - **解决方案**：用一句话描述产品的功能。
   - **目标用户群体**（ICP）：明确产品的目标用户群体。
   **产品特性**：列出 3-4 个最具竞争力的产品特性，并附上简要描述。
   **竞争对手**：如果项目有 research.md 文件，可以从中获取竞争对手的信息，以便进行市场定位。
   **定价信息**：如果有的话，也请一并提取。

4. **分析竞争对手的 landing 页面**（可选，前提是 MCP 或 web_search 可用）：
   - 查找竞争对手的前 3 个 landing 页面。
   - 注意观察这些页面的标题格式、呼叫行动语句以及用户评价的内容。
   - 分析竞争对手在哪些方面存在不足。

5. **制定转化策略**：
   - 在生成页面内容之前，先思考以下问题：
     - **主要转化目标**：用户应该执行的具体操作（注册、下载、购买等）。
     - **用户可能的顾虑**：用户可能不愿意采取行动的三个主要原因。
     - **信任信号**：如何消除这些顾虑。
     - **页面的主要展示内容**：仅包含问题描述、解决方案和呼叫行动语句。

6. **生成 landing 页面内容**：

   ### 6a. 首页部分
   - **标题**：以问题为核心，强调产品带来的好处（8-12 个字）。
   - **副标题**：产品如何解决用户的问题（15-25 个字）。
   - **呼叫行动按钮**：明确用户应该执行的操作（如“立即免费试用”或“立即下载”）。
   - **视觉元素**：确定需要展示的图片、截图或演示视频。

   ### 6b. 多种标题选项（5 种）
   生成 5 种不同的标题格式：
   1. 以问题为核心：例如：“厌倦了 {问题} 吗？”
   2. 以产品好处为核心：例如：“{结果}，无需 {麻烦}”
   3. 以用户好奇心为核心：例如：“{形容词} 的 {动作} 方式”
   4. 以用户评价为核心：例如：“{N} + 名用户正在使用 {产品}，获得 {结果}”
   5. 直接介绍产品优势：例如：“{产品}：{一句话概括的产品优势}”

   ### 6c. 产品特性部分（3-4 个特性）**
   - 为每个特性设计一个图标（建议使用表情符号或图标名称）。
   - 特性标题（3-5 个字）。
   - 特性描述（1-2 句）。
   - 强调用户能从中获得的好处（而不是产品本身能做什么）。

   ### 6d. 用户评价部分**
   - 提供用户评价的占位符（模板包括评价者的姓名、职位和评价内容）。
   - 提供数据指标的占位符（例如：“{X} + 名用户”、“速度提升 {Y}%”、“覆盖 {Z} 个国家”）。
   - 提供信任标志的占位符（如应用商店评分、奖项信息等）。

   ### 6e. 呼叫行动部分**
   - 重复标题或使用另一种标题变体。
   - 呼叫行动按钮与首页部分保持一致。
   - 提供额外的安全保障信息（例如“永久免费试用”或“无需信用卡”）。

   ### 6f. SEO 元标签**
   ```html
   <title>{title — 50-60 chars}</title>
   <meta name="description" content="{description — 150-160 chars}">
   <meta property="og:title" content="{og title}">
   <meta property="og:description" content="{og description}">
   <meta property="og:image" content="{suggest image dimensions 1200x630}">
   <meta property="og:type" content="website">
   ```

7. **输出结果**：
   - 如果检测到项目使用了 astro-static 框架，将生成的页面文件保存在 `src/pages/index.astro` 文件中；否则，将内容保存在 `docs/landing-content.md` 文件中。
   ```markdown
   # Landing Page Content: {Project Name}

   **Generated:** {YYYY-MM-DD}
   **Primary CTA:** {action}
   **Target ICP:** {persona}

   ## Hero
   **Headline:** {headline}
   **Subheadline:** {subheadline}
   **CTA:** {button text}
   **Visual:** {description}

   ## A/B Headlines
   1. {variant 1}
   2. {variant 2}
   3. {variant 3}
   4. {variant 4}
   5. {variant 5}

   ## Features
   {features with icons and descriptions}

   ## Social Proof
   {templates and placeholders}

   ## Final CTA
   {closing section}

   ## SEO Meta Tags
   {html meta tags}

   ---
   *Generated by /landing-gen. Fill in social proof, add visuals, then publish.*
   ```

8. **输出总结**：包括最终的标题、呼叫行动语句以及后续建议。

## 注意事项

- 标题需要经过 A/B 测试，至少测试 2-3 种不同的版本。
- 页面的主要展示内容应简洁明了（仅包含问题描述、解决方案和呼叫行动语句）。
- 用户评价部分使用占位符，待实际数据准备好后再填充。
- 在测试阶段，只需展示首页部分和呼叫行动按钮即可（例如用于测试目的）。
- 该工具可以与 /seo-audit 工具配合使用：先生成页面内容，然后再对实际部署的页面进行审核。

## 常见问题及解决方法

### 问题：找不到 PRD 或产品信息
**原因**：项目缺少 `docs/prd.md` 文件或 README 文件中未包含产品描述。
**解决方法**：运行 `/validate` 命令生成 PRD 文件，或者提供包含问题描述、解决方案和产品特性的 README 文件。

### 标题过于泛化
**原因**：问题描述不够具体，或缺乏对竞争对手的分析。
**解决方法**：添加 `research.md` 文件以进行竞争分析。明确的问题描述有助于生成更具针对性的标题。

### 无法检测到适合生成页面的技术栈
**原因**：项目中没有找到 `astro.config.*` 或 `next.config.*` 文件。
**解决方法**：此时工具会默认生成纯 Markdown 格式的页面内容。如果需要生成实际页面文件，请确保项目使用了 astro-static 或 Next.js 框架。
