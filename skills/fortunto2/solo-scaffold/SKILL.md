# /scaffold

本脚本用于根据产品需求文档（PRD）和指定的技术栈模板，快速搭建一个完整的项目框架。它会创建项目目录结构、配置文件、`CLAUEDE.md`文档、Git仓库，并将项目推送到GitHub。同时，脚本会通过SoloGraph分析现有项目的最佳实践，以便在新项目中采用这些成熟的模式和库版本。

## 步骤

1. **解析参数**：从 `$ARGUMENTS` 中提取 `<project-name>` 和 `<stack-name>`。  
   - 如果参数缺失或不完整，会通过 `AskUserQuestion` 功能提示用户输入缺失的信息。  
   - 从 `templates/stacks/*.yaml` 文件中列出所有可用的技术栈；如果项目使用了MCP（MCP项目信息可用），还会显示相关项目的具体技术栈信息。  
   - 项目名称应使用驼峰式命名法（kebab-case）。

2. **加载组织默认配置**：从 `~/.solo-factory/defaults.yaml` 文件中读取组织相关的配置信息（如 `org_domain`、`apple_dev_team`、`github_org`、`projects_dir`）。  
   - 如果该文件不存在，会通过 `AskUserQuestion` 提示用户输入这些信息，并将这些信息保存到 `~/.solo-factory/defaults.yaml` 中以供后续使用。  
   - 所有生成的文件中都会使用这些默认配置值。

3. **加载技术栈、产品需求文档（PRD）及相关开发原则**：  
   - 查找对应的技术栈配置文件（`stacks/<stack>.yaml`）。  
   - 如果找不到该文件，会使用内置的默认信息来配置项目。  
   - 检查是否存在产品需求文档（`docs/prd.md`），如果不存在，则生成一个基本的PRD模板。  
   - 查找项目的开发原则文件（`dev-principles.md`），或者使用预设的SOLID/DRY/KISS/TDD开发原则。

4. **通过SoloGraph分析现有项目**：在生成代码之前，先研究使用相同技术栈的现有项目。  
   - 使用 `project_info()` 函数列出所有使用该技术栈的项目，并筛选出相关的项目。  
   - 分析这些项目的架构和常用模式，避免盲目复制可能存在问题的旧代码。  
   - 了解可复用的代码结构和库版本。

5. **使用Context7获取最新库版本和最佳实践**：  
   - 对于技术栈中的每个关键库，使用 `mcp__context7__resolve-library-id` 和 `mcp__context7__query-docs` 获取最新版本、推荐的使用方式及配置信息。  
   - 重点关注3-4个最重要的库，以便更好地进行项目配置。

6. **向用户展示项目计划并获取确认**：  
   - 显示项目路径（`<projects_dir>/<name>`）、技术栈名称及对应的库版本信息，以及建议的目录结构。  
   - 在创建文件之前，让用户确认或调整这些信息。

7. **创建项目目录**：  
   ```bash
   mkdir -p <projects_dir>/<name>
   ```

8. **根据技术栈生成文件结构**：所有项目都会包含一些通用文件，例如用于定义领域模型（Domain Schemas/Models）的文件。  
   ```
   <projects_dir>/<name>/
   ├── CLAUDE.md          # AI-friendly project docs (map, not manual — see Harness Engineering)
   ├── Makefile           # Common commands (run, test, build, lint, deploy, integration)
   ├── README.md          # Human-friendly project docs
   ├── docs/
   │   ├── prd.md         # Copy of PRD
   │   ├── QUALITY_SCORE.md  # Domain quality grades (harness: garbage collection)
   │   └── ARCHITECTURE.md   # Module boundaries and dependency rules
   ├── cli/               # CLI utility — mirrors core business logic (CLI-First Testing principle)
   │   └── main.ts|py     # Deterministic pipeline entry point (no LLM required)
   ├── .claude/
   │   └── skills/        # Product-specific workflow skills
   │       └── dev/
   │           └── SKILL.md  # Dev workflow skill (run, test, deploy)
   └── .gitignore         # Stack-specific ignores
   ```

9. **优先考虑CLI测试**：生成一个 `cli/` 目录，其中包含从 `lib/` 目录导入的核心业务逻辑的测试代码。这样可以在不依赖大型语言模型（LLM）、网络或用户界面的情况下运行测试。  
   参见 `dev-principles.md` 中的“CLI-First Testing”原则。

### `.claude/skills/dev/SKILL.md` — 产品开发工作流程技能文档  
该文档用于指导Claude如何使用新搭建的项目。文档内容应包括：  
- 技术栈的详细信息（关键库、版本及配置文件的位置）；  
- 常用的命令（`make dev`、`make test`、`make build`、`make deploy`）；  
- 项目架构和命名规范；  
- 测试方法及测试文件的存放位置；  
- 常见任务（如添加新页面、API接口或模型）。  
   这些内容确保新项目能够快速被Claude理解和使用。

10. **（可选）MCP服务器配置**：如果产品需求文档中涉及数据/AI/开发相关功能，还会生成MCP服务器的配置文件。  
   参见 `templates/mcp-skills-bundle.md` 了解完整的MCP配置模板和使用规则。

11. **生成Makefile**：根据技术栈的特点生成相应的Makefile，包含各种构建命令（`help`、`dev`、`test`、`lint`、`format`、`build`、`clean`、`deploy`）。  
   如果项目需要集成测试，还需添加 `integration` 命令。

12. **生成`CLAUEDE.md`文档**：包含项目概述、技术栈信息、可用技能列表及使用方法、目录结构等。

13. **生成`.gitignore`文件**：根据技术栈的特定需求配置忽略规则。

14. **初始化Git仓库并提交第一个版本**：  
    ```bash
    cd <projects_dir>/<name>
    git init && git add . && git commit -m "Initial project scaffold

    Stack: <stack-name>
    Generated by /scaffold"
    ```

15. **创建GitHub私有仓库并推送代码**：  
    ```bash
    cd <projects_dir>/<name>
    gh repo create <name> --private --source=. --push
    ```

16. **（可选）注册项目并索引代码**：如果项目使用了MCP工具，还需将项目添加到代码索引中。  
    ```
      mcp__solograph__project_code_reindex(project="<name>")
      ```

17. **输出项目搭建结果**：  
    ```
    Project scaffolded!

      Path:   <projects_dir>/<name>
      GitHub: https://github.com/<user>/<name>
      Stack:  <stack-name>
      PRD:    docs/prd.md
      CLAUDE: configured
      Skills: .claude/skills/dev/ (project workflow)

    Next steps:
      cd <projects_dir>/<name>
      <install command>    # pnpm install / uv sync / etc.
      <run command>        # pnpm dev / uv run ... / etc.

    Then: /setup → /plan "First feature" → /build
    ```

## 验证步骤  
在确认项目搭建完成之前，需要执行以下验证：  
- 确保所有生成的文件都存在。  
- 运行安装命令（如 `pnpm install`、`uv sync` 等），确保一切正常。  
- 运行开发/构建命令，确保没有错误。  
- 确认Git仓库的创建成功（`git log --oneline -1`）。  
- 确认GitHub仓库已成功创建。

**注意事项**：  
在宣布项目搭建完成之前，必须先完成上述所有验证步骤。

## 常见问题及解决方法：  
- **技术栈配置文件找不到**：  
  **原因**：`templates/stacks/` 目录中缺少相应的模板文件，或者模板文件没有创建符号链接。  
  **解决方法**：脚本会使用内置的默认配置信息；请确保 `solo-factory/templates/stacks/<stack>.yaml` 文件存在。  

- **GitHub仓库创建失败**：  
  **原因**：未登录GitHub或仓库名称已被占用。  
  **解决方法**：先使用 `gh auth login` 登录；如果仓库名称已被占用，选择其他名称。  

- **Context7查询失败**：  
  **原因**：MCP服务器未运行或Context7服务受到速率限制。  
  **解决方法**：脚本会使用默认的技术栈配置信息作为备用方案。  

- **组织默认配置缺失**：  
  **原因**：`~/.solo-factory/defaults.yaml` 文件未创建。  
  **解决方法**：运行 `init` 命令进行一次性配置；如果需要，脚本会提示用户输入项目ID和团队ID。
