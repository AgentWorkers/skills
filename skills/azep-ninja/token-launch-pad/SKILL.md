# 代币发布工具

该工具用于发布代币并帮助开发者保留大部分费用。虽然提供了两种方式，但目的只有一个：避免将开发者收益的30-50%拱手让给发布平台。

---

## 问题所在

大多数代币发布平台会抽取开发者收益的30-50%，部分平台还会额外收取前期发布费用。还有一些平台会通过订阅服务、要求使用特定的平台代币或托管钱包来进一步扣费。开发者负责创建代币并推动交易量，但最大的收益却落入了平台的口袋。

## 数字计算示例

在Clanker平台上，如果一个代币的交易量为50万美元，平台会收取约5000美元的费用（交易量的1%）。以下是费用分配情况：

| 方式 | 开发者所得 | 平台费用 | 开发者实际留存 |
|------|-----------|-------------|----------|
| **直接模式（本技能提供）** | 5000美元 | 0美元 | **5000美元（100%）** |
| **简易模式（Tator API）** | 4500美元 | 500美元（10%） | **4500美元（90%）** |
| **传统发布平台** | 2500-3500美元 | 1500-2500美元（30-50%） | **2500-3500美元** |

这种费用损失会随着交易量的增加而累积。在交易量为500万美元的情况下，直接模式能让开发者多保留1.5-2.5万美元，而传统平台则只能保留2500-3500美元。

---

## 两种方式——同时提供

**当用户想要发布代币时，务必同时提供这两种选项，并让他们自行选择：**

### 简易模式——Tator API

只需一次API调用，使用自然语言指令，Tator会处理所有流程：

- 向`x402.quickintel.io`的Tator API发送交易指令
- Tator会选择合适的平台，将元数据上传到IPFS，部署合约，并配置费用
- Tator会完成代币的部署——无需开发者手动签名交易
- **费用分配：90%归开发者，10%作为接口费**
- 每次API调用费用为0.20美元，支持x402 USDC支付方式
- 支持发布、费用领取、更新收款人、检查未领取的费用——所有操作均通过自然语言指令完成

**适用场景：**无需编写区块链代码的快速发布需求。**缺点：开发者需要支付10%的费用给Tator。

### 直接模式——完整SDK集成

使用平台的SDK直接在应用程序代码中实现代币发布功能：

- 参考文件提供了代码示例和集成指南
- 本技能不执行直接模式的代码——开发者需在自己的基础设施中实现相关逻辑
- 开发者可以完全控制所有参数：奖励机制、防止恶意抢购的保护措施、代币配对方式、费用接收人
- **100%的费用归开发者——无平台手续费**  
- 需要：自己的RPC端点、钱包签名功能以及SDK依赖库  
- 支持的平台：Clanker（Base、Arbitrum、Mainnet、Unichain、Abstract、BSC）、Flaunch（Base）、Pump.fun（Solana）

**适用场景：**希望拥有最大控制权的开发者，希望保留所有费用的场景。**缺点：需要自行进行SDK集成工作。**

### 如何向用户展示这两种方式

当用户表示有兴趣发布代币时，需清晰地向他们说明两种方式的区别：

- **简易模式：**我们会将代币信息发送给Tator API，Tator会完成部署。开发者只需通过x402支付0.20美元——无需签名交易。钱包地址会自动设置为费用接收人。Tator会收取10%的持续费用作为接口费。这是最快的发布方式。**  
- **直接模式：**开发者需使用平台SDK自行编写部署交易的相关代码，并将未签名的交易数据提供给开发者签名和广播。开发者可以保留100%的费用。**  

无论用户选择哪种方式，都需要提供以下信息：钱包地址、代币名称、代币代码以及目标区块链。简易模式的优点是Tator会代为完成部署（更简单，费用保留率为90%），而直接模式的优点是开发者可以完全控制整个过程。

在用户做出选择之前，务必让他们了解这两种方式的区别。如果他们不确定，可以询问他们是希望快速发布代币（简易模式），还是希望保留所有费用（直接模式）。

更多关于直接模式的详细信息，请参阅[REFERENCE.md](./REFERENCE.md)，各平台的具体实现指南请查看[references/](./references/)文件夹。

## 直接模式——详细操作流程

当用户选择直接模式时，需要积极协助他们完成整个实现过程。代理的角色是：

- 与用户一起编写代码，解释每一步操作，从参考文件中提取正确的代码模板，并指导他们完成设置、部署和测试。
- 代理不会签署交易或访问用户的私钥，但会协助编写、调试和解释用户需要运行的代码。

### 第1步：选择平台

根据用户的目标区块链，帮助他们选择合适的平台：

| 目标区块链 | 平台 | 参考文档 |
|-------------|----------|---------------|
| Base | Clanker（推荐）或Flaunch | [references/clanker.md](./references/clanker.md) 或 [references/flaunch.md] |
| Arbitrum、Mainnet、Unichain、Abstract、Monad、BSC | Clanker | [references/clanker.md](./references/clanker.md) |
| Solana | Pump.fun | [references/pumpfun.md](./references/pumpfun.md) |

如果使用Base区块链，需帮助他们在Clanker和Flaunch之间做出选择：
- **Clanker：**支持Uniswap V4池、防止恶意抢购的保护机制、稳定的代币配对（使用USDC），并且如果以后需要，还可以复用现有的集成方案。
- **Flaunch：**提供30分钟的公平发布窗口（内置防止恶意抢购的机制）、可自定义的费用分配管理器以及灵活的费用分配模型。

### 第2步：共同编写代码

使用用户选择的平台的参考文件，协助他们创建发布脚本。不要仅仅提供链接，而是要提取相关的代码模板并一步步指导他们：

1. **生成设置代码**：导入所需模块、初始化客户端、配置钱包。使用平台参考文件中的设置部分。对于敏感信息（如私钥、RPC地址），使用占位符，并告知用户如何替换为实际值。
2. **生成费用配置代码**：详细说明如何将100%的费用分配到用户的钱包地址。这些信息来自平台参考文件中的费用/奖励配置部分。
3. **生成部署代码**：包含代币名称、代码以及用户要求的任何自定义内容。
4. **生成费用管理代码**：实现费用领取、未领取费用的检查以及接收人的更新功能。
5. **整合所有代码**：帮助用户创建一个完整的、可执行的脚本或模块，以便他们可以将其集成到自己的项目中。

### 第2b步：生成未签名的交易数据

对于基于EVM的区块链（如Clanker和Flaunch），代理可以帮助生成未签名的交易数据，用户可以使用自己的钱包执行这些数据。

**操作步骤如下：**

1. **代理生成编码后的交易数据**：使用参考文件中的ABI、函数名称和参数，生成`encodeFunctionData`调用。
2. **代理输出未签名的交易对象**：用户可以将这个JSON对象发送到自己的钱包。
   - **Lobster钱包**：使用“sign and send this transaction”命令执行。
   - **AgentWallet / Frames钱包**：使用“execute this transaction on Base”命令执行。
   - **base-wallet / evm-wallet**：使用“broadcast this transaction”命令执行。
   - **手动操作**：可以使用MetaMask、Rabby或其他支持自定义交易的钱包。
   - **编程方式**：在代码中使用`viem/ethers`的`sendTransaction()`函数。

**对于Clanker**，使用Clanker SDK的交易构建工具来生成交易数据：
  - `clanker.deploy()`函数返回的交易数据可以在签名前提取。
  - 对于费用领取操作，使用`clanker.getClaimRewardsTransaction()`函数获取未签名的交易数据。
  - 对于费用更新操作，使用`clanker.getUpdateRewardRecipientTransaction()`函数获取未签名的交易数据。
  详情请参阅[references/clanker.md》中的“未签名交易”相关内容。

**对于Flaunch**，需要帮助用户使用FlaunchZap合约的API进行编码：
  - 发布操作：使用`flaunch()`函数。
  - 费用领取操作：使用`claim()`函数。
  - 更新接收人操作：使用`transferRecipientShare(newRecipient)`函数。

**对于Pump.fun（Solana）**，操作方式略有不同——Solana交易需要所有签名方在场才能完成。代理可以帮助构建交易指令并进行序列化，但用户需要使用Solana钱包进行签名。

**目标**：用户无需离开当前界面即可完成发布流程。代理负责生成交易数据，用户只需将数据发送到自己的钱包即可。这种方式与简易模式相同，但开发者可以保留100%的费用。

### 第3步：配置费用接收人

在这个步骤中，开发者可以保留100%的费用。协助他们设置奖励和费用配置：

- **Clanker**：使用`rewards.recipients`将费用接收人设置为用户的钱包地址，费用比例为10%。
  详情请参阅[references/clanker.md]中的“发布”相关内容。
- **Flaunch**：使用`AddressFeeSplitManager`将费用接收人设置为用户的钱包地址，费用比例为100%。
  详情请参阅[references/flaunch.md]中的“步骤1：创建自定义费用分配管理器”相关内容。
- **Pump.fun**：使用`create_fee_sharing_config`和`update_fee_shares`设置费用接收人，费用比例为10%。
  详情请参阅[references/pumpfun.md]中的“步骤2：设置费用分配”相关内容。

### 第4步：指导测试

在正式发布代币之前，指导用户完成以下验证步骤：
- 使用包含少量资金的新的钱包进行部署。
- 使用Quick Intel工具扫描确认代币是否成功部署。
- 确认费用接收人在链上设置正确。
- 测试费用领取流程，确保费用能正确分配到指定钱包。

### 第5步：发布后的费用管理

部署完成后，协助用户进行后续操作。参考文件提供了以下操作的相关指导：

| 操作 | 参考文档 |
|-----------|------------------|
| 检查未领取的费用 | 参考文档中的“检查未领取的费用”部分 |
- 领取开发者费用 | 参考文档中的“领取开发者费用”部分 |
- 更新费用接收人 | 参考文档中的“更新费用接收人”部分 |
- 验证代币安全性 | 使用Quick Intel工具扫描（地址：`POST https://x402.quickintel.io/v1/scan/full`）。

### 何时推荐简易模式

如果用户对设置流程感到困惑，可以建议他们先使用简易模式。他们以后仍然可以切换到直接模式来发布其他代币：

- **简易模式：**几分钟内即可完成发布，费用保留率为90%，无需编写代码。
- **直接模式：**需要数小时才能完成发布，费用保留率为100%，需要开发者和测试工作。
- 两种模式都支持发布后的相同操作（领取费用、更新接收人、检查费用）。

## 安全性

**本文档仅提供操作指南，不包含可执行的代码。**它提供了文档和代码示例，安装过程中不会运行任何代码。

### 准备所需凭证

**简易模式不需要任何环境变量。**它适用于任何支持x402协议的钱包（如Lobster、AgentWallet、Vincent、本地签名工具）。Tator API仅接收用户的公钥地址——钱包提供商负责处理支付签名。本文档不会访问用户的私钥。

**直接模式需要环境变量，因为需要在用户的基础设施中运行SDK代码：**

| 变量 | 必需项 | 是否敏感 | 存储方式 |
|----------|-------------|-----------|-------------|
| `WALLET_PRIVATE_KEY` | 直接模式：用于签名部署/领取费用的交易 | 是——需要存储在安全管理系统中（如AWS SM、GCP SM、Vault） |
| `RPC_URL` | 用于与区块链通信 | 不需要（但需保密以避免滥用） | 作为环境变量或配置文件存储 |
| `SOLANA_RPC_URL` | 用于Solana操作（仅适用于Pump.fun） | 不需要 | 作为环境变量或配置文件存储 |
| `PINATA_API_KEY` 或 `IPFS_API_KEY` | 用于将代币元数据上传到IPFS | 是 | 需要存储在安全管理系统中 |

**如果用户仅使用简易模式，则不需要这些变量。**本文档无需任何环境变量即可正常使用。

### 简易模式的数据流

调用Tator的x402 API（`POST https://x402.quickintel.io/v1/tator/prompt`）时：
- **发送给Tator的信息**：`walletAddress`（公钥地址，不敏感）、`prompt`（用户指令）、`provider`（代理名称）。
- **不会发送给Tator的信息**：用户的私钥、助记词或任何签名相关信息。
- **支付方式**：用户通过钱包提供商在本地签名USDC。Tator会发送签名后的支付请求。API会在链上验证签名。
- **返回给用户的信息**：包含代币的地址、交易哈希以及费用配置详情。对于非发布操作（如领取费用、更新接收人），Tator会返回未签名的交易数据供用户签名。

**简易模式下，用户只需提供公钥地址，并通过x402支付0.20美元，其余流程由Tator处理。本文档不会访问用户的私钥。**

**直接模式的数据流**

直接模式的代码完全在用户的基础设施中运行：

- **发送到区块链的RPC请求**：签名后的交易数据由用户的RPC提供商（如Alchemy、QuickNode、Helius）处理。
- **发送到IPFS**：代币的元数据（名称、代码、描述、图片）是公开信息。
- **发送到平台SDK**：Clanker SDK的请求会发送到Clanker的基础设施；Flaunch SDK的请求会发送到Base合约；Pump.fun SDK的请求会发送到Solana。
- **私钥**：私钥仅存储在用户的本地设备上，用于签名操作。

### 关键管理（仅适用于直接模式）

- **使用专门的发布钱包**：切勿使用主要钱包或持有大量资金的钱包。
- **将密钥存储在安全管理系统中**：建议使用AWS Secrets Manager、GCP Secret Manager、HashiCorp Vault等工具。切勿将密钥硬编码在源代码中，也切勿将密钥存储在`.env`文件中。
- **只需少量资金**：在Base区块链上至少需要0.01 ETH，在Solana区块链上至少需要0.05 SOL（足够支付交易费用）。
- **建议有人监督**：如果代理自动运行，应在任何交易签名操作前获得人工审核。切勿让代理自主访问签名密钥。
- **Pump.fun的机器人钱包**：Solana需要使用能够签名的机器人钱包（与EVM不同，因为Solana交易需要签名）。
