# Pi 优化

针对资源受限的 ARM64 硬件（Raspberry Pi 5，8GB 内存，microSD 存储，Debian Bookworm）环境，对 Node.js 代码库进行以安全为先的逐文件性能审计。

## 哲学

在资源受限的硬件上，性能问题不仅仅是便利性的问题，更是安全问题。例如：

- 当事件循环在同步文件系统调用上阻塞时，网关将停止处理身份验证、工具确认和速率限制操作；
- 当内存压力触发频繁的垃圾回收（GC）时，延迟峰值可能导致超时，从而绕过预定的控制流程；
- 当 microSD 卡的 I/O 操作达到饱和时，凭证写入可能会无声地失败。

所有提交的优化方案都应从安全角度进行评估：这样的优化是否能让系统在资源受限的情况下变得更加可预测，并降低被攻击的风险？

## 审计方法

逐个文件地审查代码库。使用轻量级的 shell 命令（如 grep、find、wc）来识别需要重点优化的文件，然后深入分析这些文件。不要批量扫描整个代码库。每个文件都应得到专注的审查。

优先审查以下文件类型：
1. 与安全密切相关的路径：身份验证、会话隔离、工具控制、输入验证；
2. 网关的核心处理路径：HTTP 处理程序、消息处理、WebSocket 处理；
3. 资源管理：连接池、缓存、速率限制器；
4. 会话和状态管理：凭证、配置数据的文件 I/O 操作以及内存使用；
5. 命令行接口（CLI）和初始化代码（优先级最低）。

审计进度记录在 `memory/openclaw-audit.md` 文件中。

## 目标硬件规格

- CPU：4 个 Cortex-A76 核心，2.4GHz，aarch64 架构；
- 内存：8GB；
- 存储：microSD 卡（随机 I/O 速度慢，写入耐久性有限）；
- 操作系统：Debian 12 Bookworm，内核版本 6.12.x；
- Node.js 版本：v22.x。

## 审计检查项

### 1. 与安全相关的性能问题

- 标记身份验证/会话控制路径中的同步操作（事件循环阻塞可能导致安全风险）；
- 检查速率限制器的实现是否在低速硬件上仍能保持准确的计时精度；
- 查找可能导致资源耗尽的无限制内存分配（潜在的 DoS 攻击手段）；
- 确认凭证/令牌操作是否为异步执行（在 microSD 卡上的同步操作可能导致身份验证超时）；
- 检查是否存在泄露时间信息的错误路径；
- 标记那些因 I/O 操作缓慢而可能导致安全检查被跳过或超时的路径。

### 2. I/O 模式

- microSD 卡的顺序读写速度约为 25MB/s，随机读写性能极差；
- 标记核心处理路径中的同步文件操作；
- 查找过度创建临时文件或日志记录的行为，这些操作会加重 microSD 卡的负担；
- 确保在数据完整性至关重要的情况下使用了 `fsync` 函数；
- 寻找将数据缓存到内存中以减少磁盘读写的时机；
- 确保与安全相关的文件权限操作是原子的（防止数据损坏）。

### 3. 内存和资源管理

- 在实际的内存限制（512MB–1GB）下运行程序；
- 注意大字符串缓冲区、无限制的缓存以及事件监听器的泄漏问题；
- 检查是否有有助于在资源受限环境下优化 V8 JavaScript 运行的配置选项；
- 查找与输入数据大小不成比例的内存分配操作（潜在的 DoS 攻击手段）；
- 确认连接池和资源池的配置是否合理。

### 4. CPU 和计算性能

- 识别计算密集型函数：大负载数据的 JSON 解析、大字符串的正则表达式处理、加密操作；
- 检查是否存在依赖 x86 架构的代码（如 SIMD 指令、特定的缓冲区对齐要求）；
- 查找可能阻塞事件循环的同步加密或压缩操作；
- 标记请求路径中使用的 `child_process.execSync` 函数；
- 检查正则表达式模式是否存在 DoS 漏洞（在性能较低的硬件上问题会更加严重）。

### 5. 网络和并发处理

- 检查连接池的大小是否适合 Raspberry Pi 的硬件配置；
- 标记那些假设硬件性能较高的硬编码超时设置；
- 查找可能超出 4 核心处理器和有限内存处理能力的并发操作；
- 检查流处理中的背压控制机制；
- 检查在大量连接情况下 WebSocket 处理程序的内存使用情况。

### 6. 构建和启动过程

- 测量程序的冷启动时间；
- 检查是否有不必要的预加载操作，这些操作实际上可以延迟加载；
- 标记那些在安装后编译本机代码但未支持 ARM64 架构的脚本。

### 7. 配置默认值

- 查找硬编码的缓冲区大小、缓存限制、连接池大小等配置值；
- 建议使用适合 Raspberry Pi 硬件的默认值或基于环境的动态配置方式；
- 检查配置项是否能够根据 `os.totalmem()` 或 `os.cpus().length` 进行动态调整。

## 提交代码修改的流程

当找到优化方案后，请按照以下步骤操作：

1. 创建一个名为 `pi/short-description` 的分支；
2. 保持每次提交的代码修改都是原子性的（即一次只修改一个功能）；
3. 提交请求（PR）的标题格式为：`perf: 优化内容的简短描述` 或 `security: ...`（如果主要涉及安全问题）；
4. 在 PR 正文中说明问题、具体的 Raspberry Pi 环境背景、安全影响以及可用的基准测试结果；
- 引用相关的审计报告内容；
- 如适用，引用 OpenClaw 的安全文档（SECURITY.md、trust.openclaw.ai）。

## 在 X 平台上发布审计结果

遵循 `VOICE.md` 的规定。在审计结果中标记 @openclaw；对于符合 `VISION.md` 中优先级的优化（性能、安全、测试基础设施）相关的 PR，同时标记 @steipete。
