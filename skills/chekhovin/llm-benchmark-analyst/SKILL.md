# LLM基准分析专家

## 概述  
使用此技能来研究基准测试结果，并撰写结构化的报告，内容包括：  
1. 单个模型的优点和缺点；  
2. 某一能力领域内表现最佳的模型；  
3. 基准测试所测量的内容及其可信度；  
4. 前代模型与当前模型的进步情况。  

默认使用用户的语言。切勿自行编造分数、排名、日期、基准测试版本或表格中的缺失数据。  

## 核心约束条件  
- 将基准测试的范围限制在 `references/benchmark-source.md` 文件中；如果某个基准测试不在该文件中，则应将其排除。  
- 使用 `references/core-dimensions.md` 将分散的基准测试整合为少数几个报告维度。  
- 遵循 `references/search-playbook.md` 中的规则进行路由、数据重叠处理、证据收集和比较。  
- 按照 `references/report-template.md` 的格式生成报告内容。  
- 对每个基准测试应用 `references/data-defect-warnings.md` 中的警告信息（无论是内嵌的还是在报告的限制部分）。  
- 优先使用官方的基准测试页面或基准测试作者的页面；主要通过聚合器来获取链接和背景信息。  
- 确切记录评估方式，包括基准测试的版本、分割方式、难度级别（公开/私有）、验证状态、是否使用工具等详细信息。  

## 必要的工作流程  
1. **在搜索前对模型信息进行标准化处理**  
   - 确定模型的提供者、系列、生成版本、版本后缀以及发布标签。  
   - 明确指定时间点和版本信息；对于模糊的别名（如 `claude`、`gemini pro`、`gpt latest` 或 `qwen max`），在找到具体相关的模型名称后才能使用。  
   - 记录每个关键分数的评估时间点或访问日期。  

2. **在网络爬取前根据核心维度筛选数据**  
   - 首先参考 `references/core-dimensions.md` 选择主要评估维度。  
   - 然后列出属于这些维度的候选基准测试。  
   - 最后再逐个网站进行数据收集。  
   - 保持搜索范围狭窄且高效：先从目标领域内表现最佳的3-6个基准测试开始，必要时再逐步扩展。  

3. **深入挖掘更多相关信息**  
   - 不要被源文档的标题所迷惑；在选择主要维度后，仔细阅读基准测试的描述和相关标签，以找到其他章节中的相关测试内容。  
   - 例如：代码分析可能需要包含代码相关的基准测试、代理编程相关的基准测试，以及包含代码组件的通用基准测试；多模态分析可能需要视觉处理、OCR、GUI/计算机使用、多模态深度研究等相关的基准测试。  

4. **按以下顺序收集证据**  
   - 官方排行榜或基准测试网站  
   - 基准测试的官方文档或 README 文件  
   - 基准测试作者的博客或发布说明  
   - 可信赖的聚合器  
   - 仅将供应商的博客作为次要证据来源（若没有独立的排行榜记录，则需明确标注为“供应商提供的信息”）。  

5. **在排行榜难以解析时使用多模态提取方法**  
   - 如果页面包含图片、画布、截图或仅使用图表展示内容，而纯文本提取无法获取完整数据，需检查截图或页面图像。  
   - 仅提取清晰可见的信息，并标注数据来源为“图片提取”。  
   - 如果图片无法阅读或部分被遮挡，应如实说明。  

6. **进行基准测试之间的比较**  
   - 对于代码或代理编程相关的基准测试，与最新的 Claude Opus、Claude Sonnet 和 GPT 系列模型进行比较。  
   - 对于多模态分析，与最新的 Gemini 模型进行比较（如适用，还需考虑最新的 GPT 多模态模型）。  
   - 对于智能或推理分析，与最新的 GPT 系列模型进行比较。  
   - 不要随意假设某个模型就是“最新版本”，务必先进行搜索确认。  

7. **进行前后代模型比较**  
   - 如果有数据可用，将目标模型与其直接的前代模型或同一提供者/系列中的上一个可比较的模型进行对比。  
   - 仅比较同类基准测试；如果前代模型仅以不同的测试模式出现，应说明比较结果可能不准确。  

8. **添加质量警告**  
   - 任何存在已知质量或方法论问题的基准测试都应附带来自 `references/data-defect-warnings.md` 的警告信息。  
   - 如果报告的结论依赖于这些存在问题的基准测试，应明确说明其可信度较低。  

## 决策规则  
- 当用户询问“某个领域内表现最佳的模型”时，不要仅依赖一个基准测试结果，而应综合多个相关基准测试的结果并解释每个测试的意义。  
- 当用户询问某个模型的优势或劣势时，先从核心维度进行分析，再通过基准测试数据加以佐证。  
- 当不同基准测试的分数存在冲突时，优先考虑数据的新鲜度、版本匹配度、官方来源的质量以及支持该结论的基准测试数量，而非单一的高分结果。  
- 当基准测试数据不稳定、通过图片提取获得或存在明显误差时，应认为这些数据不具有决定性。  
- 必须包含一段简短的说明，说明每个基准测试实际测试的内容。  

## 需要收集的最低限度证据  
对于引用的每个基准测试，需记录以下信息：  
- 基准测试名称  
- 它所测试的具体内容（用简短语句描述）  
- 模型的完整名称  
- 准确的分数及单位  
- 排名或相对位置（如果有的话）  
- 基准测试的版本、分割方式或测试模式  
- 数据的获取日期或访问时间  
- 如果数据来源非官方，需注明其质量情况  
- 如有数据质量问题，需添加相应的警告信息  

## 报告格式要求  
使用 `references/report-template.md` 中提供的模板。  
每份报告至少应包含以下内容：  
- 报告的范围和主题  
- 简短的执行摘要  
- 模型的优点和缺点  
- 证据表格  
- 对比分析结果  
- 数据质量问题及可信度说明  
- 测试方法或排除标准
