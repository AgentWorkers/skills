# 技能文档编写指南

请根据宿主的期望结构来编写或更新技能文档。请使用以下结构和检查清单。

## 输入信息

1. **目标技能** – 技能文件夹的路径（例如：项目中的技能目录 `skills/example/`）或技能名称。
2. **来源** – 需要转化为技能的草稿、用户说明或笔记。

**如果未提供其中任何一项**：则使用当前上下文或提供的信息（例如：正在讨论的技能或路径）。

## 输出结果

更新后的 SKILL.md 文件（如需要，还包括相关支持文件）。

## 编写流程

### 1. 技能文档结构

- 每个技能对应一个文件夹。主要文件始终为 `SKILL.md`（文件名需严格匹配）。
- **文件夹名称**：必须使用 **kebab-case** 格式（小写字母、连字符；禁止使用空格、下划线或大写字母）。前言部分的 `name` 必须与文件夹名称一致。
- **技能存放位置**：在项目中，通常位于专门的技能目录中（例如：`skills/<名称>/`）。即使技能位于嵌套目录中（如包内），宿主也能从该位置找到它。
- **可选的子文件夹**：可以使用 `scripts/**`（存放可执行代码）、`references/**`（存放按需加载的文档）和 `assets/**`（存放模板、图标等）。不要将 `README.md` 放在技能文件夹内；所有文档都应放在 `SKILL.md` 或 `references/` 中。长篇参考资料应放在 `references/` 中，并在 `SKILL.md` 中链接（实现逐步披露机制）。
- **长度限制**：`SKILL.md` 的内容应控制在 5,000 字以内。长篇参考资料可放在 `references/` 中，并在 `SKILL.md` 中链接。
- **逐步披露机制**：技能文档分为三个层级：（1）YAML 前言部分始终会被加载，以便助手知道何时使用该技能；（2）当技能相关时，`SKILL.md` 的主体内容会被加载；（3）`references/` 或 `assets/` 中的文件仅在需要时才会被打开。这样可以减少不必要的信息显示，同时仍能提供所需的专业知识。
- **可组合性**：技能文档可以与其他技能协同使用。编写时请考虑与其他技能的兼容性；不要假设该技能是唯一被使用的技能。

### 2. SKILL.md 文件格式

#### 前言部分（位于首行 `---` 之间的 YAML 代码）

前言部分用于配置技能的运行时机和方式。以下内容遵循 [官方技能文档规范](https://code.claude.com/docs/en/skills.md#frontmatter-reference)。

`name` 和 `description` 是必填项。`name` 必须与文件夹名称一致（使用 kebab-case 格式）。`description` 应包含以下内容：`[功能描述] + [使用场景] + [核心能力]`，长度不超过 1024 字；禁止使用 XML 标签 `<` 或 `>`。避免使用模糊的描述（如 “帮助处理项目”），请包含具体的触发语句以及相关文件类型（例如：“.fig 文件”、“PDF 合同审核”）。

| 字段                          | 是否必填    | 说明                                                                                                                         |
| :------------------------- | :---------- | :------------------------------------------------------------------------------------------------------------------------- |
| `name`                     | 是         | 必须使用 kebab-case 格式；文件夹名称由小写字母、数字和连字符组成（最多 64 个字符）。                                   |
| `description`              | 是         | `[功能描述] + [使用场景] + [核心能力]`。包含触发语句（例如：“当用户输入 ‘save’ 或使用 `/save` 时使用该技能”）。如有必要，可提及文件类型。长度不超过 1024 字；禁止使用 `<` 或 `>`。     |
| `argument-hint`            | 否          | 自动完成时显示的提示信息，用于提示用户输入参数的格式。例如：`[issue-number]` 或 `[filename] [format]`。                   |
| `disable-model-invocation` | 否          | 设置为 `true` 可禁用自动加载；适用于需要手动通过 `/name` 调用的工作流程。默认值：`false`。                   |
| `user-invocable`           | 否          | 设置为 `false` 可隐藏该技能，避免用户直接通过 `/` 菜单调用。默认值：`true`。                             |
| `allowed-tools`            | 否          | 该技能激活时助手可以使用的工具列表。                                               |
| `model`                    | 否          | 激活该技能时使用的模型。                                                     |
| `context`                  | 否          | 设置为 `fork` 时，在分支子代理环境中运行该技能。                                           |
| `agent`                    | 否          | 当 `context: fork` 时使用的子代理类型。                                               |
| `hooks`                    | 否          | 与该技能生命周期相关的钩子。具体配置方法请参考 [Skills 和 Agents 中的钩子](/en/hooks#hooks-in-skills-and-agents)。           |

**可选字段**：`license`（例如开源项目的 MIT 许可证）、`compatibility`（1–500 字：产品兼容性、系统依赖等）、`metadata`（例如作者、版本号、mcp-server 等）。详细信息请参阅 [完整的技能构建指南](https://resources.anthropic.com/hubfs/The-Complete-Guide-to-Building-Skill-for-Claude.pdf)。

**安全提示**：前言部分禁止使用 XML 标签 `<` 或 `>`。技能名称中避免使用保留字（如 “host” 或 “vendor”）。

**YAML 注意事项**：值中不要使用冒号，否则可能被解析为新的键。例如，使用 “Scope is” 而不是 “Scope:”。

#### 文档内容类型

- **参考内容**：包含约定、模式和样式信息。助手会直接将这些信息应用到用户的当前操作中。无需控制内容的调用方式。
- **任务内容**：针对特定操作的详细步骤说明（如部署、提交、代码生成等）。通常通过 `/skill-name` 直接调用该技能。建议设置 `disable-model-invocation: true` 以禁用自动加载。

#### 文档主体结构（所有技能均采用相同结构）

为了便于查看，请按照以下顺序组织内容：

1. **# 标题** – 以 H1 标题显示技能名称，然后简要介绍技能的功能。
2. **## 输入信息** – 技能所需的输入内容（用户输入、路径、选项等）。对每个输入项进行编号；如果不需要输入内容，可标注为 “None” 或 “Optional”。
3. **## 输出结果** – 技能执行后的结果（生成的文件、产生的行为或传递的结果）。
4. **## 执行步骤** – 操作的具体步骤。使用编号的步骤或子章节进行说明（例如：“### 1. 步骤名称”）。请提供具体且可操作的指导（例如：“运行 `python scripts/validate.py --input {filename}`”，而非 “验证数据”）。对于常见的错误情况，请提供错误处理方式；如需引用相关文档，请注明 “在查询前，请参考 `references/api-patterns.md`”）。
5. **## 示例** – 可选但推荐。提供常见使用场景的示例（例如：“用户输入 X → 执行操作 → 结果”）。
6. **## 故障排除** – 可选但推荐。列出错误或问题现象 → 原因 → 解决方案。
7. **## 参考资料** – 可选。提供相关技能或文档的链接。

如果某个部分确实不适用，请跳过。不确定如何处理时，至少保留输入信息、输出结果、执行步骤和参考资料；如果适用，可添加示例和故障排除信息。保持与仓库中其他技能的格式一致。

#### 文本中的占位符

根据 [官方文档](https://code.claude.com/docs/en/skills.md#available-string-substitutions)：

- `$ARGUMENTS` – 调用技能时传递的所有参数。如果文档中未提及参数，这些参数会以 `ARGUMENTS:` 的形式添加。
- `$ARGUMENTS[N]` 或 `$N` – 第 N 个参数（从 0 开始计数）。
- `${CLAUDE_SESSION_ID}` – 当前会话 ID（由宿主定义）。
- `${CLAUDE_SKILL_DIR}` – 包含技能 `SKILL.md` 文件的目录（由宿主定义）。

关于在技能执行前注入 shell 输出的信息，请参阅 [动态上下文注入](https://code.claude.com/docs/en/skills.md#inject-dynamic-context)。

### 3. 编写或更新技能文档时的检查清单

1. **前言部分**：`description` 应遵循 `[功能描述] + [使用场景] + [核心能力]` 的格式，并提供具体的使用场景（例如：“当用户输入 ‘save’ 或使用 `/save` 时使用该技能”）。`name` 必须与文件夹名称一致。对于需要用户操作的技能，建议设置 `disable-model-invocation: true`。
2. **文档结构**：按照输入信息、输出结果、执行步骤的顺序组织内容；如果需要，可添加示例和故障排除部分；最后是参考资料。
3. **文档长度**：内容长度控制在 500 行以内；长篇参考资料应放在其他文件中，并在 `SKILL.md` 中链接。
4. **支持文件**：如果使用了支持文件，请在 `SKILL.md` 中说明文件的位置及加载方式。确保 `SKILL.md` 保持简洁；将详细文档放在 `references/` 中并添加链接（实现逐步披露机制）。
5. **参数处理**：如果技能需要输入参数，请使用 `$ARGUMENTS` 或 `$N`，并根据需要添加 `argument-hint`。
6. **调用控制**：使用 `disable-model-invocation` 和/或 `user-invocable` 来控制技能的调用时机（详见 [控制技能调用者](https://code.claude.com/docs/en/skills.md#control-who-invokes-a-skill)）。

### 4. 编写步骤

1. 阅读目标技能的路径及任何相关源文件。
2. 按照上述结构和检查清单进行编写或更新文档。除非用户另有要求，否则不要更改技能的功能。
3. 编写或更新 `SKILL.md` 文件（如需要，还包括相关支持文件）。仅记录技能的实际功能，不要添加不存在的功能。

## 示例

**用户请求：** “创建一个用于验证 CSV 上传的技能。”

**操作步骤：** 创建文件夹 `skills/validate-csv/`（或项目中的技能路径，使用 kebab-case 格式）。编写 `SKILL.md` 文件，包含必要的前言内容（`name: validate-csv`，`description` 中包含 “验证 CSV 上传。当用户输入 ‘validate CSV’、‘check upload’ 或使用 `/validate-csv’ 时使用该技能。核心能力：模式验证、编码检测”）。添加输入信息、输出结果、执行步骤以及可选的参考资料。如果技能需要执行脚本，请添加 `scripts/validate.sh` 并在文档中引用该脚本。

**结果：** 创建一个新的技能文档，用户可以通过指定语句触发该技能；如果该技能是流程的一部分，协调器和相关流程也会相应更新。

**用户请求：** “更新 save 技能，使其包含 ‘update-gitignore’ 的功能。”

**操作步骤：** 打开 `skills/save/SKILL.md` 文件，根据需要添加新的步骤或参考资料。如果触发语句发生变化，请更新文档描述。如果相关流程（如 `references/coordinator-flows.md`）也需要更新，请进行同步。

## 常见问题及解决方法

**前言部分无效或上传失败：**
- **原因**：YAML 格式错误（缺少 `---` 分隔符、引号未闭合，或者值中包含冒号）。
- **解决方法**：确保前言部分位于两行 `---` 之间；描述内容中避免使用冒号；使用 “When” 而不是 “Use”。

**技能无法触发：**
- **原因**：描述过于模糊或缺少完整的信息（功能描述、使用场景或核心能力）。
- **解决方法**：使用 `[功能描述] + [使用场景] + [核心能力]` 的格式，并明确触发语句（例如：“当用户输入 ‘X’ 或使用 `/skill-name` 时使用该技能”）。如果该技能是流程的一部分，请确保这些描述也出现在流程表中。**调试方法**：询问助手 “你会在什么情况下使用 [技能名称]？” 并根据实际情况调整描述。

**技能被频繁触发：**
- **原因**：描述过于宽泛或与其他技能重复。
- **解决方法**：添加明确的限制条件（例如：“不适用于简单的数据探索；请使用 data-viz 技能”）。缩小技能的应用范围（例如：“仅用于 PDF 合同审核”）。明确说明不适用的场景（例如：“仅适用于在线支付流程，不适用于通用财务查询”）。

**文件夹名称错误：**
- **原因**：文件夹名称中包含空格、下划线或大写字母。
- **解决方法**：将文件夹名称改为 kebab-case 格式。更新 `SKILL.md` 中的 `name` 以及所有相关引用文件（协调器、代理程序、README、package.json、检查清单脚本等）。

## 编写完成后

- **协调器同步**：如果修改了技能的描述（例如使用场景或核心能力），请更新协调器代理程序和流程文件（如 `references/coordinator-flows.md`），以确保流程表和代理程序描述保持一致。
- **重命名/移动**：如果技能被重命名或移动（例如：`generate-figma` 更改为 `designer-figma`），请更新所有相关引用文件：协调器、代理程序、README、package.json、其他引用文件（如 `verify-task/scripts/checklist.ts`、`agents/references/` 等）。
