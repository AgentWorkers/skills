# Apollo 问题审核

请按照以下工作流程来审核 Apollo 问题，并生成简洁的维护者回复。

## 核心原则

- 首先进行分类：行为/回归问题还是咨询/支持问题。
- 对于行为/回归问题：先尝试复现问题，再进行分析。
- 对于咨询/支持问题（例如“是否有官方脚本/文档”）：先核实证据，然后直接回答；不要强制使用“已复现/未复现”的表述。
- 解决用户的问题，而不是争论用户是对还是错。
- 如果问题已经复现且结论明确，就不要再要求额外的信息。
- 除非用户明确要求版本对比或这会改变建议，否则不要默认进行“版本回归”分析。
- 使用与问题语言相匹配的回复格式：英文问题用英文回复，中文问题用中文回复。
- 使用仓库中规范的 Apollo 模块名称（来自 `pom.xml` 文件中的 AGENTS/module 结构），并在需要时纠正错误的名称。
- 如果已有评论已经回答了相同的问题（包括机器人的回复），避免重复冗长的回复；最好添加简短的补充内容，仅说明修正内容或差异。
- 不要将 GitHub 的 @mention 处理方式用反引号或代码格式括起来；使用普通的 @handle，以确保通知能够正常触发。
- 如果社区用户自愿承担实现任务（“认领”问题），首先要表示感谢并给予鼓励，然后对提案进行评估，并明确可行性边界和具体的改进建议。
- 对于与 OpenAPI 相关的请求，要区分 Portal Web API（例如 `/users`）和 OpenAPI 端点（例如 `/openapi/v1/*`）；只有在验证了基于令牌的 OpenAPI 路径后，才能声称“OpenAPI 支持 X”。
- 在得出“功能不可用”的结论之前，要核对代码、文档/脚本以及 `pom.xml` 中的模块/依赖项信息，以避免因路径假设导致的错误判断。

## 输入信息

在审核之前收集或推断以下字段：

- `repo`：`<owner>/<repo>`
- `issue_number`：问题的唯一编号
- `issue_context`：问题的标题、正文和评论
- `publish_mode`：`draft-only`（默认）或 `post-after-confirm`
- `output_mode`：`human`（默认）或 `pipeline`

可选但推荐的信息：

- `known_labels`：问题已有的标签
- `desired_outcome`：用户是仅希望进行问题分类还是希望问题分类后还有实现帮助

如果缺少 `issue_number` 或 `issue_context`，在继续之前请询问用户以获取相关信息。

## 工作流程

1. 收集问题事实和用户请求
- 在得出结论之前，阅读问题的正文和评论。
- 提取主要问题、症状、预期行为、实际行为，以及用户请求的是哪种具体路径。
- 明确用户的请求内容（例如“需要更好的解析 API 还是原始文本 API”：两者都要回答）。
- 检测问题讨论中是否包含关于承担实现任务的请求（例如“我可以接手这个问题吗？”），并将其视为指导性回复，而不仅仅是功能是否可用的回复。
- 从问题的标题、正文或最近的评论中确定主要使用的语言，并在起草回复时使用相应的语言。
- 预先判断问题类型：
  - 行为/回归问题（需要复现验证）
  - 咨询/支持问题（需要证据核实）
- 将名称规范化为 Apollo 仓库中使用的规范模块/服务名称（例如使用 `apollo-portal`，而不是自创的服务名称）。
- 如果 GitHub API 访问不稳定，请使用以下代码块：
```bash
curl -L -s https://api.github.com/repos/<owner>/<repo>/issues/<id>
curl -L -s https://api.github.com/repos/<owner>/<repo>/issues/<id>/comments
```

2. 运行相应的验证步骤（强制要求）
- 对于行为/回归问题：
  - 为报告的问题构建一个最小化的、可在本地运行的复现环境。
  - 优先使用仓库自带的单元测试或简单的临时脚本，而不是进行推测。
  - 记录观察到的确切输出和类型，而不仅仅是解释。
- 对于咨询/支持问题：
  - 通过仓库中的文档/脚本/代码路径进行验证，而不是通过推测性的复现操作。
  - 对于 API 可用性的请求，在得出结论之前在三个地方进行验证：
    1) 实际的控制器路径
    2) 文档/OpenAPI 脚本
    3) `pom.xml` 中的模块/依赖项信息
  - 记录搜索的文件/路径以及实际存在的内容和不存在的内容。
- 示例验证步骤：
```bash
rg -n "<api_or_path_related_to_issue>" -S
go test ./... -run <target_test_name>
# or a minimal go run script under /tmp for one-off validation
# consultative evidence scan example:
rg --files | rg -i "<keyword1|keyword2>"
rg -n "<keyword>" docs scripts apollo-* -S
```

3. 根据验证结果划分处理方式
- 行为/回归问题：
  - 如果问题可以复现：
    - 明确说明问题已经得到确认。
    - 判断这是正常支持的功能、使用方式不匹配，还是当前的功能缺口。
    - 然后直接回答用户的问题（是否可以使用现有 API、是否有变通方案或是否不支持）。
  - 如果问题无法复现：
    - 仅要求提供最基本的缺失证据：
      - 输入样本
      - 读取/访问代码
      - 预期输出与实际输出
    - 保持回复简洁明了。
- 咨询/支持问题：
  - 如果所需的功能/脚本/文档存在：提供具体的路径/链接和使用方法。
  - 如果不存在：直接说明“目前不可用”，并提供一个实际的替代方案。
  - 如果已有评论已经给出了相同的结论：只需发布简短的补充内容，避免重复完整的回答。

4. 起草维护者回复（重点说明具体操作）
- 用问题语言写一段总结：
  - 行为/回归问题：复现情况的总结
  - 咨询/支持问题：直接给出结论
- 然后包括以下内容：
  - 当前的功能支持情况
  - 可行的解决方案：用户现在可以使用的具体 API/命令/变通方案
  - 后续步骤：邀请用户提交 PR 并提供具体的文件/测试数据，或者说明维护者可能会在未来安排处理，但不要过度承诺时间表。
- 如果问题讨论中包含关于承担实现的提案，回复内容应包括：
  1) 表示感谢和鼓励
  2) 对提案的可行性评估
  3) 具体的实现改进建议（哪些内容可以重用，哪些不能直接使用）
- 如果用户的请求是二选一的情况，要明确回答两者。
- 如果已经确认存在功能缺口，通常不需要再要求额外的日志或步骤。
- 保持用词客观简洁。
- 在最终回复中使用规范的模块名称；如果问题使用了非规范的名称，要简要纠正。
- 如果已有正确的评论，最好采用“参考 + 最小补充”的格式。
- 提到用户或机器人时，使用纯文本格式（例如 @dosubot），而不是代码格式的引用。
- 根据问题语言使用本地化的部分标题和用词（例如：中文线程中使用“Reproduction Result”/“Current Support Boundary”/“Practical Path”/“Next Step”）。

5. 请求发布确认（强制要求）
- 默认情况下，仅生成草稿；不要自动发布。
- 先展示评论内容，然后在同一条评论中请求用户的确认。
- 使用与问题语言相同的语言提问，例如：
  - 中文：`是否直接发布到 issue #<id>？回复“发布”或“先不发”。`
  - 英文：`Post this to issue #<id> now? Reply "post" or "hold".`
- 如果没有回复或回复含糊不清，视为“未批准”。

6. 只有在获得明确确认后才能发布回复
- 允许的确认回复示例：`发布` / `帮我发` / `直接回复上去`。
- 如果用户的意图不明确，在发布前请询问一个简短的确认问题。
- 当 `gh` 传输不稳定时，使用以下代码块：
```bash
TOKEN=$(gh auth token)
curl --http1.1 -sS -X POST \
  -H "Authorization: token $TOKEN" \
  -H "Accept: application/vnd.github+json" \
  -d '{"body":"<reply>"}' \
  https://api.github.com/repos/<owner>/<repo>/issues/<id>/comments
```
- 发布后，将评论的 URL 作为验证依据提供。

## 输出格式

默认的输出格式（`output_mode=human`）应便于人类阅读：

1. **问题总结**
  - 问题类型 + 验证结果（是否已复现）
  - 是否准备好进行实现帮助

2. **问题分类建议**
  - 需要添加的标签
  - 缺失的信息（如果有的话）
  - 是否可以开始实施

3. **维护者回复草稿**
  - 第一句话必须与问题类型相匹配：
    - 行为/回归问题：复现状态（已复现/暂未复现）
    - 咨询/支持问题：功能的可用性结论
  - 包括至少一个具体的 API/代码路径/文件引用
  - 如果今天不支持该功能：说明支持边界、可行的变通方案和下一步行动
  - 如果问题可以复现且结论明确：不要请求额外的数据
  - 如果问题无法复现：仅请求最基本的复现所需信息
  - 如果已有评论已经解决了问题：仅提供简短的补充内容
  - 不要将未经验证的根本原因作为事实呈现
  - 保持语言与问题语言一致，除非用户另有要求

4. **发布控制**
  - 如果没有明确的发布确认，结尾处应写：
  - 中文：`是否直接发布到 issue #<id>？回复“发布”或“先不发”。`
  - 英文：`Post this to issue #<id> now? Reply "post" or "hold".`

如果 `output_mode=pipeline`，在人类可读的输出后面添加一个机器可读的代码块：

```yaml
handoff:
  issue_classification:
    type: "功能咨询|问题排查|技术讨论|Bug 反馈|Feature request"
    validation_path: "behavior-regression|consultative-support"
    confidence: "high|medium|low"
  triage_decision:
    labels_to_add: []
    missing_info_fields: []
    ready_for_issue_to_pr: false
    ready_reason: ""
  implementation_handoff:
    goal: ""
    acceptance_criteria: []
    suggested_modules: []
    risk_hints: []
```
