# CallMyCall（OpenClaw技能）

此技能可帮助您通过聊天界面操作CallMyCall服务。该服务采用pull模式（不使用Webhook回调）：您发起呼叫后，会将返回的呼叫ID存储在系统状态中，之后可根据需要获取呼叫的状态和结果。

## API密钥的获取方式（OpenClaw最佳实践）

按照以下顺序获取API密钥：

1. 环境变量：`CALLMYCALL_API_KEY`（优先推荐）
2. OpenClaw用户配置文件：`~/.openclaw/openclaw.json`中的`skills.openclaw-phone.apiKey`字段
3. 如果仍未获取到密钥，先询问用户是否希望将其保存到`~/.openclaw/openclaw.json`中以供将来使用：
   - 如果用户同意，获取密钥并保存；
   - 如果用户不同意，仅获取密钥用于本次调用，不进行保存。

**密钥保存规则：**
- 绝不要将API密钥保存在`SKILL.md`文件、示例代码、参考资料或内存/状态文件中。
- 不要将API密钥写入`recent_calls`列表或任何对用户可见的输出内容中；也不要向用户透露“我不会将密钥显示出来”。
- 如果用户拒绝保存密钥，仅使用本次调用所需的密钥。

## 该技能的工作原理

1. 按照“API密钥的获取方式（OpenClaw最佳实践）”中的顺序获取API密钥。
2. 通过分层验证流程收集所需的呼叫信息。
3. 提供简要的呼叫概要并请求用户确认。
4. 确认后，发送`POST /v1/start-call`请求。
5. 将返回的`sid`值存储在系统状态中，作为最近一次的呼叫记录。

## 必需的认证方式

在请求头中包含API密钥：

```
Authorization: Bearer YOUR_API_KEY
```

**注意：**切勿将API密钥显示给用户，也不要将其记录在日志或总结信息中。

## 状态管理（必需）

系统需要维护最近10次呼叫的记录：

- `recent_calls`：一个对象数组，包含以下字段：
  - `id`（呼叫ID）
  - `phone_number`（电话号码）
  - `task`（任务信息）
  - `started_at`（呼叫开始时间）
  - `status`（可选，可在后续请求时更新）

利用此列表，用户可以执行如“结束呼叫1”或“显示呼叫2的结果”等操作。

## 分层验证流程（借鉴自Web应用程序）

不要依赖单一的验证步骤，而是要使用以下所有验证层级：

### 第1层：结构化信息收集

在所有必填字段都齐全之前，不要完成呼叫请求：
- `phone_number`（电话号码）
- `language`（语言）
- `call_brief`（呼叫背景及目标）

### 第2层：任务信息补充

当用户首次发起请求时，分析缺失的信息，然后仅请求缺失的部分。如果用户部分回答，重新进行分析并继续请求剩余的字段。

### 第3层：提示性验证

只要还有缺失的信息，就继续收集所需的字段。在所有必填字段都齐全之前，不要进行呼叫。

### 第4层：运行时验证

在发送呼叫请求之前：
- 确认电话号码存在且格式符合E.164标准；
- 阻止拨打紧急或高级服务号码；
- 确保`from_number`（来电号码）与`phone_number`不同；
- 对`language`字段进行规范化处理；如果提供了`genderVoice`或`openaiVoice`字段，也进行规范化处理；
- 如果有调度信息，需解析并确保时间有效。

### 第5层：人工审核

向用户展示简要的呼叫概要：
- 电话号码
- 呼叫背景及目标
- 语言（以及提供的声音信息）
- 任何调度安排

询问用户：“确认后开始呼叫吗？”未经明确确认，不要继续进行呼叫。

## 工作流程

### 发起呼叫

1. 使用分层验证流程收集所需字段。
2. 如果用户请求了具体的调度时间，按照“预定呼叫”的流程操作，而不是立即发送API请求。
3. 否则，发送`POST /v1/start-call`请求。
4. 将返回的`sid`值存储在`recent_calls`列表中。
5. 回复用户呼叫已发起的信息及呼叫ID。

### 预定呼叫（OpenClaw端处理）

由于API不支持调度功能，因此需通过OpenClaw进行调度：
1. 现在收集所有必填字段。
2. 将呼叫信息（电话号码、背景信息、语言/声音设置等）保存到技能状态中。
3. 在预定时间创建一个`cron`作业来执行呼叫。
4. 在预定时间触发作业，发送`POST /v1/start-call`请求，存储`sid`值，并将呼叫ID返回给用户。

### 查看最近通话记录

1. 从系统状态中读取`recent_calls`列表。
2. 如有需要，通过`GET /v1/calls/:callId`获取每条呼叫的详细状态。
3. 以编号列表的形式显示所有通话记录。

### 重复尝试直到接通

当用户要求反复尝试直到接通时：
1. 发送`POST /v1/start-call`请求。
2. 定期通过`GET /v1/calls/:callId`检查呼叫状态。
3. 根据返回的状态（`status`、`duration`）处理结果：
   - 如果状态为`busy`、`no-answer`、`failed`或`canceled`，则等待指定时间后再次尝试。
4. 当状态变为`in-progress`或`completed`（且`duration`大于0）时，停止尝试循环。
5. 向用户报告每次尝试的详细信息（包括呼叫ID和状态）。

**实现提示：**每次请求都使用相同的基URL（推荐使用`https://call-my-call-backend.fly.dev`），并在启动呼叫和查询状态时保持一致。

### 结束呼叫

如果用户要求结束呼叫但未指定具体哪次呼叫，列出所有最近的通话记录并询问用户想要结束哪次呼叫。
- 使用`POST /v1/end-call`请求，并传入`callSid`参数。

### 获取通话结果

当用户请求查看通话结果时：
1. 通过`GET /v1/calls/:callId`获取通话状态。
2. 如果有通话记录，通过`GET /v1/calls/:callId/transcripts/stream`获取通话录音。
3. 如果有录音文件，通过`GET /v1/calls/:callId/recording`获取录音URL。

**返回内容：**
- 呼叫状态（已完成、失败、已取消）
- 简要总结（1-3条要点）
- 通话记录摘录（前几行）
- 录音URL（如有）

## 安全性与用户体验

- 如果用户输入不明确，询问用户以获取更详细的信息。
- 绝不要在通话记录中显示API密钥。
- 如果请求失败，显示HTTP错误信息并提示用户下一步该怎么做。
