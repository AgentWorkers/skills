# 客户入职流程 — 自由职业项目入职指南

## 该流程存在的必要性  
当 Kevin 完成客户对接后，他需要在 24 小时内展现出专业形象并启动项目。手动准备入职资料需要 2 小时的时间（主要用于编写文档）。而这个工具能够将项目描述一键转换为完整的入职资料包。

## 使用场景  
适用于以下情况：  
“为新客户入职做准备”、“我刚刚接到了一个项目”、“为 [客户名称] 设置项目文件”，或任何新自由职业项目启动的场合。  

**调用方式**：`/client-onboard [项目描述]`  

---

## 流程  

### 第 1 步：解析项目信息  
从 `$ARGUMENTS` 中提取以下信息：  
- **项目类型**：Web 应用、API、自动化、审计、移动应用、数据管道等  
- **客户行业**：SaaS、电子商务、医疗保健、金融、创作者经济等  
- **时间线要求**：“需在 2 周内完成”、“没有紧迫时间”、“尽快”或具体日期  
- **预算信息**：提及的小时费率、固定价格、“预算有限”或企业级需求  
- **技术限制**：“必须使用 React”、“我们在 AWS 上运行”、“不允许绑定供应商”  
- **团队情况**：“单人创始人”、“我们有一个开发团队”、“替换现有承包商”  

**未指定时的默认假设**：  
- **时间线**：4-6 周（适用于大多数 Web 项目）  
- **预算**：每小时 75 美元或 2,000-5,000 美元的固定费用（Kevin 的标准范围）  
- **技术栈**：Next.js + Supabase + Stripe + Python（Kevin 的默认配置）  
- **部署方式**：Vercel（前端）+ Railway 或 Supabase（后端）

## 第 1 里程碑：基础建设 — 第 1-2 周  
**目标**：建立项目结构、实现认证功能及核心数据模型。虽然各部分尚未完全集成，但框架已搭建完成。  

**交付物**：  
- [ ] 设置完代码仓库并配置 CI/CD（Vercel 自动部署）  
- [ ] 安装 Supabase 及数据库架构（包括表格和访问控制策略）  
- [ ] 实现登录、注册和会话管理功能  
- [ ] 建立基本导航界面  
- [ ] 更新 `CLAUEDE.md` 以符合项目规范  

**交接标准**：Kevin 演示可用的登录界面及仪表板雏形；客户确认数据库架构无误。

## 第 2 里程碑：核心功能开发 — 第 2-4 周  
**目标**：实现客户付费的核心功能并确保其正常运行。  

**交付物**：  
- [ ] [核心功能 1] — 开发并测试完成  
- [ ] [核心功能 2] — 开发并测试完成  
- [ ] [核心功能 3] — 开发并测试完成  
- [ ] 实现错误提示和加载状态提示  
- [ ] 保证应用在移动设备上正常显示  

**交接标准**：客户能够按照主要使用场景使用该应用。  

---

## 第 3 里程碑：优化与集成 — 第 4-5 周  
**目标**：完成第三方集成和处理边缘情况，优化用户界面。  

**交付物**：  
- [ ] 完成支付、邮件和数据分析功能的集成  
- [ ] 处理所有测试中的边缘情况  
- [ ] 应用性能达到 Lighthouse 评分 85 分以上  
- [ ] 完成安全审查（包括访问控制策略和输入验证）  

**交接标准**：应用具备生产环境所需的所有功能，且无已知漏洞。  

## 第 4 里程碑：上线与交接 — 第 5-6 周  
**目标**：应用正式上线，客户可开始使用。  

**交付物**：  
- [ ] 完成生产环境部署（自定义域名、SSL 证书配置）  
- [ ] 设置监控系统（例如 Sentry 或基本错误日志记录工具）  
- [ ] 录制客户使用指南  
- [ ] 提供管理员文档（包括用户管理、内容管理等内容）  
- [ ] 将代码仓库权限转移给客户  

**付款计划（建议）**：  
- 项目启动时支付 25% 的费用  
- 第 2 里程碑完成后支付 25%  
- 第 3 里程碑完成后支付 25%  
- 最终交付完成后支付剩余 25%  

根据项目规模调整里程碑数量和时间线：小型项目（1-2 周）设置 2 个里程碑；大型项目（8 周以上）设置 5-6 个里程碑。

│   ├── app/                    // Next.js 应用代码

│   │   ├── dashboard/                // 仪表板代码  
│   │   ├── settings/                  // 配置文件  
│   │   ├── public/                  // 公共页面代码  
│   │   ├── page.tsx                  // 登录页代码  
│   │   ├── api/                    // API 代码  
│   │   ├── hooks/                  // 自定义 React 钩子  
│   │   ├── types/                  // 共享 TypeScript 类型定义  
│   ├── supabase/                  // 数据库相关代码  
│   ├── functions/                  // 边缘函数代码  
│   ├── scripts/                  // 一次性脚本  
│   ├── docs/                    // 客户文档

根据实际使用的技术栈进行调整。仅使用 Python 的项目可能需要调整目录结构。

## 错误处理建议：  
- 如果客户要求“开发一个应用程序”这样的模糊需求，使用默认配置并添加 5 个问题到 `PROJECT_BRIEF.md` 的“未解决的问题”部分，并明确说明需要客户在开始前提供答案。  
- 如果客户对技术栈有特殊要求，添加“考虑的替代方案”部分并解释原因。  
- 如果预算信息不明确，使用占位符百分比，并注明“请在发送邮件前与客户确认费率”。  
- 如果 `CLIENT_ONBOARD/` 目录已存在，请在目录名后添加时间戳后缀（例如 `CLIENT_ONBOARD_[YYYY-MM-DD]/` 以避免覆盖现有文件）。  
- 如果缺少技术细节，根据推测填写相关信息，并用 `[ASSUMED]` 标明。  

## Kevin 的沟通风格：**  
- 表达直接、自信，避免使用正式的商务语言  
- 例如：“我会使用 Y 技术来实现 X 功能”，而不是“我建议可以考虑使用……”
