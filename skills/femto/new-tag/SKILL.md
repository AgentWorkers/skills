# 新标签的创建流程

## 工作流程

### 检查发布规范

- 在更改版本之前，先查看 `git status --short` 的输出、当前分支以及最近的标签信息。
- 查看 `.github/workflows/` 目录，了解标签的命名规则（如 `v*`）、包过滤规则、构建任务以及发布任务的相关信息。
- 查看包的清单文件（manifest）和发布脚本，以确定应使用哪个包版本来创建新的标签。

### 选择版本计划

- 除非用户有明确的要求，否则遵循仓库现有的标签命名规则。
- 仅更新受到此次发布影响的包的版本号。
- 当某个共享包的更改会被其他将通过该标签发布的包使用时，也需要更新该共享包的版本号。
- 除非新版本确实需要更高的最低版本要求，否则保持兼容性检查机制不变。
- 如果工作目录中包含与此次发布无关的更改，请在将其纳入发布流程之前停止并征得确认。

### 应用版本变更

- 更新包清单文件中的版本号。
- 当包名或发布的工件发生变化时，更新发布工作流程、工作区过滤规则以及包引用信息。
- 在准备发布过程中，修复任何出现的打包或 TypeScript 解析问题。

### 标签创建前的验证

- 运行发布工作流程所要求的全部构建任务，而不仅仅是那些在本地能够成功通过的构建任务。
- 至少要构建所有会被标签触发的工作流程所涉及的包。
- 在任何必要的构建任务失败的情况下，不要创建或推送标签。

### 提交并推送发布准备工作

- 只提交与发布准备工作相关的文件。
- 使用清晰的提交信息，例如 `Prepare vX.Y.Z release`。
- 在创建标签之前先推送分支，确保标签指向的是远程仓库中已存在的提交。

### 创建并推送标签

- 创建一个带有注释的标签，其名称应与工作流程的触发条件相匹配（通常为 `vX.Y.Z`）。
- 显式地推送该标签。
- 报告提交的哈希值（commit SHA）、标签名称以及所有应被触发的工作流程。

## 命令模式

请使用以下命令序列作为默认模式，并根据实际情况进行调整：

```bash
git status --short
git tag --sort=-creatordate | head
rg -n "tags:|push:|workflow_dispatch|publish|--filter" .github/workflows -g '*.yml' -g '*.yaml'

# edit package versions and release metadata

pnpm --filter <package-a> build
pnpm --filter <package-b> build
pnpm run <release-build-if-defined>

git add <release-files>
git commit -m "Prepare vX.Y.Z release"
git push origin <branch>
git tag -a vX.Y.Z -m "vX.Y.Z"
git push origin vX.Y.Z
```

## 启发式规则

- 优先选择主要发布的包的版本作为标签版本。
- 如果仓库已经使用了混合版本的发布策略，请继续使用最近使用的版本命名规则。
- 如果标签触发的工作流程仍然引用旧的包名，请在创建标签之前进行修正。
- 即使某个包的发布任务可以跳过已经发布的版本，也要确保构建任务能够成功完成，以避免 GitHub 发布页面显示错误信息。
- 如果发布内容包括浏览器扩展程序和原生应用程序，请验证打包或 ID 的兼容性变化不会影响已发布的扩展程序的正常使用。

## 输出结果

- 被更新的包及其新版本号。
- 执行的验证命令列表。
- 提交的哈希值（commit SHA）。
- 被推送的标签名称。
- 如果有某些内容被故意保留为不变状态，还需说明这些内容及其潜在的风险。
