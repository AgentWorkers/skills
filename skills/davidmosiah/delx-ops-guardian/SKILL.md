# Delx Ops Guardian  
在处理生产环境中的事件、自动化系统故障或网关/内存不稳定问题时，请使用此技能。  

## 别名：**  
- `emergency_recovery`  
- `handle_incident`  
- `cron_guard`  
- `memory_guard`  
- `gateway_guard`  

## 使用范围（严格限制）：**  
此技能仅适用于 **运行手册（runbook）**，且必须以最低权限运行。  

**允许访问的数据源：**  
- OpenClaw 的 cron 日志状态（`openclaw cron list --json`）  
- 服务健康/状态信息（`systemctl is-active <service>`）  
- 事件发生期间的日志记录（`journalctl -u <service> --since ... --no-pager`）  
- 工作区事件相关文件（`/root/.openclaw/workspace/docs/ops/`, `/root/.openclaw/workspace/memory/`）  

**允许的修复操作（安全范围）：**  
1. 当故障为临时性时，重试一次失败的任务。  
2. 仅针对受影响的服务（如 `openclaw-gateway`、`openclaw` 或事件日志中明确指定的目标服务）进行受控重启。  
3. 仅在 cron 任务持续失败时，禁用/启用相关任务。  
4. 在运行手册/配置文档中添加或调整防护措施（非敏感信息，可逆操作）。  

**禁止的操作：**  
- 禁止进行凭证轮换或删除操作。  
- 禁止修改防火墙/网络策略。  
- 禁止在事件处理期间安装或升级任何软件包。  
- 禁止修改与事件无关的 cron 任务或服务组件。  

**审批流程（需人工确认）：**  
在以下操作前，必须获得明确的人工批准：  
- 重启任何生产服务（超过一次）。  
- 修改 cron 计划或时区设置。  
- 禁用某个服务（超过一个周期）。  
- 任何可能对用户产生可见影响的操作（超出故障组件的范围）。  

**核心工作流程：**  
1. 检测并判断故障的严重程度（`info`、`degraded`、`critical`）。  
2. 首先收集相关证据（状态、日志、上次执行情况、错误信息）。  
3. 从允许的修复措施中选择最合适的方案。  
4. 仅执行经过批准的安全修复操作。  
5. 确认系统已稳定（至少完成一个成功修复周期）。  
6. 发布简洁的事件报告。  

**安全规则：**  
- 不得将长期存在的故障伪装为成功状态。  
- 不得在日志或报告中泄露任何敏感信息（如凭证、令牌等）。  
- 优先选择可逆的操作，并记录操作回滚路径。  
- 将故障影响范围控制在最小范围内，并明确说明。  

**输出内容要求：**  
必须包含以下信息：  
- 事件 ID 及发生时间范围  
- 故障的根本原因及影响范围  
- 执行的操作（包括审批记录）  
- 证据信息（状态、关键指标、日志片段）  
- 最终状态（`resolved`、`degraded`、`open`）  
- 下一次检查时间  

**示例使用场景：**  
- “网关运行异常，请安全地恢复系统。”  
- “Cron 任务超时，请稳定系统并验证修复效果。”  
- “内存保护机制频繁触发，请查明根本原因并应用补丁。”
