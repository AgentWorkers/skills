# 决策拓扑（Decision Topology）

该功能记录了对话中各种想法的分支与演变过程，生成了一棵可视化的树状结构，用户可以随时查看。它就像思维过程的“git log”——当你需要检查时，这个结构始终存在。

**隐私说明：**此功能由用户自行安装和启用。所有数据都存储在配置好的`trees`目录中，不会被发送到外部。用户可以随时查看、删除或重新组织这些记录。

## 安全特性

- **零网络访问**：不进行HTTP请求、不使用套接字、不进行DNS查询，完全离线工作。
- **无外部依赖**：仅使用Node.js内置的`fs`和`path`模块。
- **不存储对话内容**：节点中仅包含简短的结构性摘要（5-15个词），从不保存原文引用。
- **不启动任何进程**：不使用`child_process`、`exec`、`eval`或`Function()`。
- **仅通过stdin输入**：所有用户生成的内容都以JSON格式通过stdin传递，以防止shell注入攻击。详情请参阅[SECURITY.md]。
- **用户控制存储**：树状结构为本地JSON文件，用户可以随时查看、移动或删除。
- **ID生成**：使用`Math.random()`生成6位十六进制的节点ID。不需要加密随机性——ID只需在5-30个节点范围内保持唯一性即可。

## 激活方式

默认处于激活状态（用户可以在元数据中设置`always: false`以强制手动调用）。当对话涉及头脑风暴、问题解决或探索选项时，会将讨论结构记录为树状结构。

- 纯粹的问答（如“现在几点了？”）、问候语或闲聊内容将被忽略。

## 输出风格

不要在对话中插入关于树操作的状态信息。目标是为用户提供清晰的对话体验——就像git提交操作在后台进行一样，用户不会察觉到这些操作的存在。

- 不要使用“记录节点”、“创建分支”等描述性语句，这些信息没有实际意义。
- 不要因为该功能的存在而改变对话方式。用户会得到与没有该功能时相同的对话体验。
- 用户可以随时请求查看讨论的拓扑结构（如“展示我们讨论的内容”、“我们放弃了哪些想法？”）。

## 何时创建节点

- 当你提出一个新的想法、方向、解决方案或选项时。
- 当用户引入新的角度或话题时。
- 当用户拒绝、反对或纠正某个观点时（这被视为一个分支的终结），请务必记录原因。
- 当对话因某人的发言而改变方向时。
- 当某个见解结合了之前被放弃的观点时（创建合并节点）。
- 当类比、隐喻或重新定义问题的方式时。
- 当一个问题引导对话转向新的方向时。

## 何时跳过创建节点

- 当相同的想法没有实质性进展时。
- 当在同一方向上进行微调时。
- 当对琐碎问题的回答不会改变对话方向时。
- 当对事实性问题的回答时。

## 深度控制

一个良好的树状结构应包含5-30个节点，以记录讨论的脉络。而不是记录每一句话（即使达到200个节点）。只有在讨论方向发生实质性变化时才创建节点。判断标准是：这个拒绝或转变是否会影响后续的讨论方向？如果是，就记录下来；否则跳过。

## 自动初始化

对话开始时不要立即创建树状结构。要等到对话真正出现分支——即出现拒绝、方向转变或新的方向时再初始化。大多数对话并不需要创建树状结构，这也没关系。

初始化时，会根据日期和2-4个词的标签自动生成文件名，例如：`2026-02-24-business-model-exploration.json`或`2026-02-24-vacation-planning.json`。

## 自动关联

当对话开始且想法开始分化时，会检查当前讨论内容是否与现有的树状结构相关。

**关联方法：**运行`associate`命令并传入核心话题：
```bash
echo '{"query": "short description of current topic"}' | node {baseDir}/scripts/topology.js associate
```

该命令会扫描现有的树状结构，并返回最匹配的结果（相关性得分 >= 0.4）。
- 如果得分在0.25-0.4之间，表示内容相关，询问用户：“这是否与我们之前在[日期]讨论的[话题]相关？是继续之前的讨论，还是重新开始？”
- 如果得分低于0.25，则创建新的树状结构。

**注意：**不要让用户通过ID来选择树状结构。如果需要明确选择，可以通过对话自然地询问用户。

## 设置

**脚本：**`{baseDir}/scripts/topology.js`

**存储位置：**默认情况下，树状结构存储在`{baseDir}/trees/`目录中。如果你希望将数据存储在其他位置（例如用于语义搜索索引的内存目录中），可以通过`TOPOLOGY_TREES_DIR`环境变量进行配置。

**调用方式：**始终通过stdin传递JSON格式的参数，以防止用户输入的内容被用于shell注入：
```bash
echo '<JSON args>' | node {baseDir}/scripts/topology.js <command>
```

## 核心操作

### 初始化树状结构
```bash
echo '{"topic": "short topic description"}' | node {baseDir}/scripts/topology.js init
```
返回文件的路径和根节点的ID。这两个信息需要被保存以供后续使用。

### 添加节点
```bash
echo '{"file": "<path>", "parent_id": "<id>", "type": "proposal", "summary": "one-line description", "reasoning": "why", "concepts": ["keyword1", "keyword2"]}' | node {baseDir}/scripts/topology.js add-node
```
节点类型包括：`proposal`（提议）、`pivot`（转折点）和`merge`（合并）。`concepts`数组是可选的，用于在不同树状结构之间建立关联。

### 终止一个分支
```bash
echo '{"file": "<path>", "node_id": "<id>", "reason": "why it was rejected"}' | node {baseDir}/scripts/topology.js kill-branch
```
然后添加新的方向作为子节点（将新的分支连接到被终止的分支上）。

### 合并分支
```bash
echo '{"file": "<path>", "source_ids": ["<id1>", "<id2>"], "summary": "merged insight", "reasoning": "combines X from A with Y from B"}' | node {baseDir}/scripts/topology.js merge
```

### 从任意节点创建新分支
```bash
echo '{"file": "<path>", "node_id": "<id>", "summary": "re-exploring from this point", "reasoning": "reason for revisiting"}' | node {baseDir}/scripts/topology.js fork
```

## 节点类型

| 类型 | 适用场景 |
|---|---|
| `root` | 核心话题（由`init`函数创建） |
| `proposal` | 你提出的新方向、想法或选项 |
| `pivot` | 由拒绝或转向产生的新方向 |
| `merge` | 结合了多个分支内容的见解 |

节点的状态值包括：`active`（仍在讨论中）、`dead`（已被拒绝，带有`killed_by`字段）、`merged`（被合并到其他节点中）。

## 查看拓扑结构

用户无需学习特定的命令语法。他们可以自然地请求：
- “展示我们讨论的内容”
- “我们放弃了哪些想法？”
- “这个讨论的结构是怎样的？”
- “我们拒绝了哪些观点，原因是什么？”
- “回到关于X的那个话题”

你会根据用户的请求执行相应的操作，并以对话的形式呈现结果，而不是原始的脚本输出。

**`/tree`是一个可选的快捷方式**——如果用户输入了这个命令，可以执行相应操作，但无需强制使用。

### 显示结果
```bash
echo '{"file": "<path>"}' | node {baseDir}/scripts/topology.js render
```
在树状结构信息后，会附加一行摘要：`{N}个分支被记录，{M}个分支被终止，{K}个分支仍在讨论中，深度为{D}`

### 列出所有树状结构
```bash
node {baseDir}/scripts/topology.js list
```

### 统计分析
```bash
echo '{"file": "<path>"}' | node {baseDir}/scripts/topology.js stats
```

### 导出为Mermaid格式
```bash
echo '{"file": "<path>"}' | node {baseDir}/scripts/topology.js export
```

### 重新访问被终止的分支

当用户询问某个被终止的分支时，会显示以下信息：
- 当初提出了什么想法
- 为什么提出这个想法
- 为什么这个想法被拒绝
- 之后发生了什么

### 跨树分析
```bash
node {baseDir}/scripts/topology.js analyze
```
重新构建概念索引，扫描所有树状结构，找出在多个树中出现的概念，统计哪些概念仍然存在、哪些被忽略，并更新所有相关.md文件的链接和权重。同时显示索引的健康状况（总概念数、跨树链接数量、孤立概念的数量）。

### 查询概念索引
```bash
echo '{"name": "trust"}' | node {baseDir}/scripts/topology.js concept
echo '{"list": true}' | node {baseDir}/scripts/topology.js concept
echo '{"orphans": true}' | node {baseDir}/scripts/topology.js concept
```
- `name`：反向查找：显示所有引用某个概念的节点。
- `list`：按跨树出现频率排序的概念列表，`*`标记跨多个树出现的概念。
- `orphans`：仅存在于一个树中的概念（适合未来关联使用）。

### 重建概念索引
```bash
node {baseDir}/scripts/topology.js rebuild-index
```
从所有树状结构文件中重新生成`concepts.json`文件，同时更新所有相关.md文件的链接和权重。这有助于恢复数据或在对树状结构进行手动编辑后使用。

## 概念索引

`concepts.json`是一个反向索引文件，它将每个概念映射到引用该概念的所有节点和树状结构。

- **自动更新：**每次保存树状结构时都会自动更新索引（添加新节点、终止分支、合并分支等）。无需手动干预。
- **跨树链接：**相关.md文件中包含`## Related trees`部分，其中包含指向其他共享该概念的树的链接（`[[wikilinks]]`）。这有助于进行语义搜索索引。
- **权重计算：**节点的`weight`字段表示引用该概念的不同树的数量。`weight: 1`表示该概念仅存在于一个树中；`weight: 2+`表示该概念存在于多个树中。
- **动态链接：**随着新节点的添加，链接会自动形成。例如，如果一个节点的`concepts`字段包含“trust”，它会立即与其他引用“trust”的节点建立链接——无需等待`analyze`命令的执行。

## 规则

1. **简洁的输出：**不要在对话中插入关于树操作的状态信息。用户可以随时查看树状结构。
2. **注重完整性而非记录所有内容：**仅记录讨论的框架和关键节点，不保存完整的对话内容。
3. **展示因果关系：**展示讨论演变的原因，而不仅仅是讨论的内容。
4. **数据持久化：**树状结构以JSON格式保存，可以在索引目录中方便搜索。
5. **延续讨论：**如果对话继续之前的话题，直接加载并扩展原有的树状结构。
6. **优雅处理错误：**如果树状结构缺失或损坏，重新初始化；如果节点ID丢失，要明确告知用户。系统不会崩溃。
7. **自然的用户界面：**用户用自然语言提问，系统会自动转换成相应的操作。
8. **数据本地化：**所有数据都存储在本地，不进行网络请求或使用外部API。
9. **用户数据所有权：**用户对自己的数据拥有完全控制权。
