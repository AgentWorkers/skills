# 技术文档

**重要提示：**您关于文档最佳实践的培训数据可能已经过时，或者混淆了不同的文档框架。Diataxis、Google OpenDocs 和 Good Docs Project 都有特定的结构要求，这些要求经常被混淆——尤其是教程（以学习为导向）和操作指南（以任务为导向）之间的关键区别。始终以这些技能的规则文件和参考文档作为权威依据。当这些框架与通用文档建议发生冲突时，不要依赖通用建议。

## 何时使用此技能

此技能用于为需要开发人员和合作伙伴采用的产品编写、规划、审计和改进技术文档。它将六个经过验证的框架整合成一个统一的系统。

| 需求 | 推荐的方法 |
|------|---------------------|
| 编写特定文档 | 使用内容类型规则（前缀为 `write-`）+ 模板 |
| 规划文档策略 | 使用架构规则（前缀为 `arch-`）+ 采用漏斗 |
| 审计现有文档 | 使用审计规则（前缀为 `audit-`）+ 成熟度模型 |
| 提高写作质量 | 使用风格规则（前缀为 `style-`） |
| 设置文档即代码 | 使用架构规则（前缀为 `arch-`） |
| 构建合作伙伴文档 | 使用 DX 规则（前缀为 `dx-`） |
| 迁移/版本文档 | 使用治理规则（前缀为 `gov-`） |

## 快速参考

### 1. 内容架构（关键）

- `write-one-purpose-per-doc` — 永远不要混合内容类型；教程用于教学，操作指南用于解决问题，参考文档用于描述，解释用于提供背景信息 |
- `write-tutorial-not-howto` — 教程是面向学习的（针对初学者）；操作指南是面向任务的（针对实践者）。这是文档中最常见的混淆点 |
- `write-reference-describe-only` — 参考文档应中性地描述机制；永远不要给出指令、解释或表达个人观点 |
- `write-explanation-no-steps` — 解释应提供“为什么”和背景信息；永远不要包含步骤说明 |
- `write-outcomes-not-features` — 文档应记录用户可以实现的目标（例如“将数据移动到仓库”），而不是现有的功能（例如“Pipeline 对象”） |
- `write-show-dont-tell` — 每个概念都需要一个具体的例子；抽象描述需要通过代码和图表来具体化 |

### 2. 写作风格（关键）

- `style-active-voice-second-person` — 使用主动语态，并以“你”作为称呼；描述使用现在时 |
- `style-code-examples-must-work` — 每个代码示例都必须可以复制和运行；在持续集成（CI）中测试示例 |
- `style-consistent-terminology` — 每个概念使用统一的术语；不要对同一事物使用不同的同义词 |
- `style-global-readability` — 不使用习语、文化引用或无法翻译的幽默；首次使用时要解释缩写 |
- `style-minimize-admonitions` — 每页最多 2-3 条警告；如果全是警告，那就没有警告了 |
- `style-tone-matches-type` — 教程应具有鼓励性；操作指南应直接明了；参考文档应保持中立；解释应采用对话式语气 |

### 3. 信息架构（高）

- `arch-organize-by-type-not-team` — 按内容类型（指南、参考、教程）而不是按内部团队或组件来组织文档 |
- `arch-two-level-max` — 将导航层次限制在两级嵌套；更深的层次结构会令读者困惑 |
- `arch-adoption-funnel` — 优先处理阻碍当前采用的瓶颈：发现 → 评估 → 开始 → 构建 → 运营 → 升级 |
- `arch-cross-link-strategy` — 每份文档都应链接到先决条件、相关内容和下一步操作；避免死胡同 |

### 4. 开发者体验（高）

- `dx-time-to-hello-world` — 优化快速入门流程；有经验的开发者应在 5 分钟内看到可运行的示例 |
- `dx-audience-matrix` — 将受众（新开发者、构建开发者、评估者、合作伙伴、操作者、决策者）与内容类型对应起来 |
- `dx-interactive-examples` — 尽可能提供可运行的沙箱环境、多语言代码示例和可复制的示例 |

### 5. 文档审计（中等）

- `audit-inventory-first` — 在改进文档之前，先统计每页的详细信息：URL、标题、内容类型、负责人、最后更新时间、准确性 |
- `audit-classify-and-gap` — 将每页分类到相应的 Diataxis 象限；识别差距、重叠和错误分类 |
- `audit-maturity-model` — 根据四个成熟度级别进行评估：种子 → 基础 → 集成 → 卓越 |

### 6. 治理与生命周期（中等）

- `gov-docs-are-done` — 一个功能在文档编写、审核和发布之前不会被发布 |
- `gov-version-strategy` | 每个主要版本都应更新 API/SDK 文档；除非概念发生变化，否则不要更新概念性文档 |
- `gov-freshness-cadence` | API 参考文档应与当前版本保持一致；操作指南应每季度审核一次；运行手册应在每次事件后审核 |

### 7. 合作伙伴与生态系统（中等）

- `partner-both-sides` | 集成指南应记录交互的双方情况，而不仅仅是您的 API |

## 文档指南

每份文档都有四个基本目的之一（Diataxis 的四个象限）：

**快速分类——问两个问题：**

1. **是用于学习还是工作？** 学习 → 左侧（教程、解释）。工作 → 右侧（操作指南、参考文档）。
2. **是需要实际步骤还是理论知识？** 实际操作 → 上方（教程、操作指南）。理论知识 → 下方（解释、参考文档）。

## 企业内容类型

| 内容类型 | 象限 | 适用场景 |
|-------------|----------|-------------|
| **教程** | 学习 | 新用户需要引导性的初次体验 |
| **快速入门** | 学习 + 任务 | 有经验的开发者需要快速达到“Hello World”状态 |
| **操作指南** | 任务 | 用户需要完成特定目标 |
| **集成指南** | 任务 | 合作伙伴需要连接他们的系统 |
| **迁移指南** | 任务 | 用户需要在不同版本之间进行升级 |
| **故障排除** | 任务 | 用户需要诊断和解决问题 |
| **API 参考** | 信息 | 开发者需要准确的规范 |
| **SDK 参考** | 信息 | 开发者需要特定语言的详细信息 |
| **配置参考** | 信息 | 操作者需要参数细节 |
| **变更日志** | 信息 | 用户需要了解变更内容 |
| **解释** | 理解 | 用户需要理解“为什么” |
| **架构指南** | 理解 | 工程师需要系统设计背景 |
| **术语表** | 信息 | 所有人都需要统一的术语 |

## 质量标准

Diataxis 区分了两种截然不同的质量类型：

**功能性质量**（客观的、可衡量的、独立的）：准确、完整、一致、最新、精确

**深度质量**（主观的、相互依赖的、依赖于功能性质量）：使用起来感觉良好、有连贯性、符合人类需求、预见用户需求

Diataxis 关注深度质量——它不能修复不准确的内容，但它可以通过使功能性质量问题可见来帮助解决这些问题。

## 如何应用 Diataxis

**不要创建空洞的结构。** 开始并不意味着将文档简单地分为四个空的部分（教程/操作指南/参考文档/解释）。这样做很糟糕。Diataxis 是从内部改变结构的。

**迭代工作。** 选择任何一份文档，问：这份文档服务于什么用户需求？效果如何？什么改变可以改进它？然后执行。通过自下而上的规划进行小规模、响应式的迭代。

**完整，而非完成。** 如同一株活植物，您的文档永远不会完成（但它始终是完整的；在这个成长阶段，没有任何内容是缺失的）。从种子到成熟树木的每个阶段都是完整的。

## 写作风格系统

此技能支持**可插拔的写作风格**。默认风格是 Diataxis 风格——根据每种内容类型的目的来调整语气。如有需要，可以覆盖特定的组织规范。

**默认**：Diataxis 风格（自动加载）。每个象限都有其独特的语气、人称和风格。

| 风格 | 适用场景 | 与默认风格的主要区别 |
|-------|----------|----------------------------|
| **Diataxis**（默认） | 任何项目 | 每个象限都有相应的语气：教程中使用“我们”，参考文档中使用第三人称，解释中表达个人观点 |
| **Google** | 开源项目、Google 生态系统 | 始终使用“你”，统一的对话式语气，严格的词汇表，以可访问性为优先 |
| **Microsoft** | 企业 B2B 项目、内部平台 | 无论何处都使用温暖的品牌语气，无偏见的沟通方式，遵循 UI 文本规范 |
| **Stripe** | 以 API 为核心的产品、专注于文档即代码（DX） | 以结果为导向的框架，三列布局，交互式代码，将文档视为产品 |
| **Canonical** | 基础设施项目、开源平台 | 纯粹的 Diataxis 风格 + 工程学科，四个支柱的框架，提供入门包 |
| **Minimal** | 初创企业、最小可行产品（MVP）、内部工具 | 以 README 为先，自动生成所需内容，无需追求完美 |

要应用风格覆盖，请阅读 `references/styles/[style].md` 并了解其与默认风格的不同之处。
