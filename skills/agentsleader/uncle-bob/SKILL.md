# Uncle Bob — 清洁代码与架构原则

在编写、审查或重构代码时，请遵循这些原则。它们并非需要盲目遵守的规则——请运用判断力，但默认情况下应追求代码的简洁性。

## “童子军规则”

将代码写得比你接收它时更整洁。每次提交都应使代码库得到改进，哪怕只是微小的改进。

## 清洁代码的基础原则

### 命名

- 命名应能清晰地表达代码的意图。如果一个名称需要注释来解释，那么这个名称就是错误的。
- 使用易于发音且可搜索的名称。避免使用缩写、单个字母（循环计数器除外）以及前缀。
- 类/类型：使用名词或名词短语（例如 `AccountManager`、`OrderRepository`）。
- 函数/方法：使用动词或动词短语（例如 `calculateTotal`、`fetchUser`、`isValid`）。
- 布尔值：应能像问句一样阅读（例如 `isActive`、`hasPermission`、`canExecute`）。
- 避免使用难以理解的命名。例如，`r` 并不代表 URL，应该将其命名为 `url`。

### 函数

- 函数应尽可能简洁。一个函数应该只完成**一个任务**。
- 理想情况下，函数应该接受 0 到 2 个参数。超过 3 个参数可能表明设计有问题——可以考虑提取相关参数到单独的对象中或重新设计函数。
- 函数不应产生副作用。例如，名为 `checkPassword` 的函数不应同时负责初始化会话。
- 命令与查询分离：函数要么执行某个操作（命令），要么返回某个结果（查询），不能同时兼具两者。
- 避免代码重复（DRY 原则）——但不要过早地抽象化代码。当出现三次重复时才需要抽象。
- 可以提取子功能时，请务必提取出来。

### 注释

- 良好的代码本身就能说明其功能。注释用于补充代码中未能表达的信息。
- 注释可以说明代码的用途、提供必要的信息、警告潜在问题或记录待办事项。
- 应删除被注释掉的代码。版本控制系统会自动记录这些更改。
- 切勿编写重复代码本身的注释（例如，在 `i++` 之前写 `// increment i`）。

### 格式

- 垂直布局：类似报纸的排版方式——将高层次的总结放在顶部，详细信息放在下方。
- 相关的函数应放在一起。调用函数的代码应位于被调用函数的上方。
- 水平布局：避免滚动阅读。保持代码行长度适中。
- 整个团队应使用统一的格式，个人偏好可以适当调整。

### 错误处理

- 尽量使用异常或结果类型来处理错误，而不是错误代码。
- 不要返回 `null`，也不要传递 `null` 值。
- 将错误处理代码放在函数的最上层，而不是分散在函数各处。
- 错误处理应该专注于处理错误本身，不应承担其他功能。
- 定义异常类时，应考虑调用者的需求，而不是代码的实现细节。

### 对象与数据结构

- 对象用于隐藏数据并暴露其行为；数据结构则直接暴露数据，本身没有行为。
- 不要将两者混用。同时具有公共字段和业务方法的类是最糟糕的设计。
- 德米特法则（Demeter’s Law）：一个方法应该只调用其自身对象的方法、其参数的方法、它创建的对象的方法，或者其直接依赖的对象的方法。例如，不要写 `a.getB().getC().doThing()`。

## SOLID 原则

有关详细解释和示例，请参阅 [references/solid.md](references/solid.md)。

- **S — 单一职责原则（Single Responsibility）**：一个类应该只有一个改变的理由。一个类只负责一个功能。
- **O — 开放封闭原则（Open/Closed）**：类应该易于扩展，但不应容易被修改。使用多态性而非条件语句来实现扩展。
- **L — 里氏替换原则（Liskov Substitution）**：子类型应该能够替换其基类型而不改变原有代码的行为。
- **I — 接口隔离原则（Interface Segregation）**：多个具体的接口比一个通用的接口更好。客户端不应依赖它们不使用的方法。
- **D — 依赖倒置原则（Dependency Inversion）**：应该依赖抽象层，而不是具体实现层。高层模块不应依赖于底层模块。

## 清洁架构

有关完整的架构指南，请参阅 [references/clean-architecture.md](references/clean-architecture.md)。

### 依赖关系原则

源代码中的依赖关系应该**指向更高层次的组件或规则**。

```
Frameworks & Drivers → Interface Adapters → Use Cases → Entities
(outer)                                                  (inner)
```

- **实体（Entities）**：代表企业业务规则的纯领域对象。
- **用例（Use Cases）**：特定于应用程序的业务规则，用于协调实体的操作。
- **接口适配器（Interface Adapters）**：负责将用例格式与外部格式进行转换（如控制器、展示层、网关）。
- **框架与驱动程序（Frameworks & Drivers）**：最外层的组件，负责处理数据库、Web 框架、用户界面等细节。这些组件是可以替换的。

### 关键规则

- 内部组件不应了解外部组件的实现细节。
- 跨界传递的数据应使用通用的数据传输对象（DTO）或值对象，避免使用特定于框架的类型。
- 数据库和 Web 技术属于细节部分，框架也是细节。
- 框架本身也是可替换的组件。

## 组件设计原则

- **共同封闭原则（Common Closure Principle, CCP）**：经常一起变化的类应该被放在同一个模块中。
- **共同复用原则（Common Reuse Principle, CRP）**：不要强迫用户依赖他们不需要的组件。
- **稳定依赖原则（Stable Dependencies）**：组件之间的依赖关系应遵循稳定性原则。
- **稳定抽象原则（Stable Abstractions）**：稳定的组件应该具有抽象性。

## 代码问题（警示信号）

- **代码僵化（Rigidity）**：对代码进行微小修改会导致其他地方出现连锁反应。
- **代码脆弱性（Fragility）**：对某个部分的修改可能会破坏其他无关部分的代码。
- **代码不可重用（Immobility）**：无法在不引入额外依赖的情况下重用某个模块。
- **不必要的复杂性（Unnecessary Complexity）**：过度泛化或过早抽象化。
- **代码重复（Redundancy）**：重复的代码。
- **代码可读性差（Opacity）**：代码难以理解。
- **代码冗长（Long Code）**：函数过长、类过大、参数列表过长、使用布尔标志或基于类型进行判断。

## 测试（Test-Driven Development, TDD）

- **TDD 的三大原则**：(1) 先编写一个无法通过的测试用例；(2) 只编写足够的测试用例来覆盖需要测试的功能；(3) 只编写足够的代码来使测试通过。
- 测试用例应被视为与生产代码同等重要的代码。保持测试用例的简洁性、可读性和高效性。
- 每个测试用例应该只验证一个功能点。
- 测试用例应满足 F.I.R.S.T. 的标准：快速（Fast）、独立（Independent）、可重复（Repeatable）、自我验证（Self-Validating）、及时（Timely）。
- 测试应关注功能行为，而不是具体的实现细节。

## 如何应用这些原则

在审查或编写代码时，请按以下顺序进行检查：

1. **可读性（Readability）**：其他人能在 30 秒内理解这段代码吗？
2. **命名（Naming）**：名称是否能够清晰地表达代码的意图？
3. **函数规模（Function Size）**：代码是否可以进一步拆分或优化？
4. **单一职责原则（Single Responsibility）**：每个代码单元是否只负责一个功能？
5. **依赖关系（Dependencies）**：代码的依赖关系是否指向稳定的组件或抽象层？
6. **耦合度（Coupling）**：代码之间的耦合是否合理？
7. **错误处理（Error Handling）**：错误处理是否清晰且一致？
8. **测试（Tests）**：是否已经编写了足够的测试用例？测试用例是否能够有效地验证代码的功能？
