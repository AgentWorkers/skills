# 技能：持久化用户记忆  
**版本：** 1.0.0  
**作者：** community  
**标签：** `memory`、`personalization`、`context`、`learning`、`stateful`  
**要求：** 具备文件系统访问权限；可选：支持向量存储或 SQLite  

---

## 概述  
该技能为 OpenClaw 提供了对用户信息的长期、结构化的存储机制。与会话级上下文不同，这些信息在系统重启后仍会保留，并会随着时间的推移学习用户的偏好和行为模式；在用户执行任何重要操作前，系统会主动参考这些信息。  
**目标：** 使 OpenClaw 更像是一个了解用户的可靠助手，而非一个“陌生的工具”。  

---

## 记忆存储位置  
所有用户数据都存储在一个本地文件中：  
```
~/.openclaw/memory/user_profile.json
```  

**注意：**  
- **切勿将数据存储在临时目录中。**  
- **除非用户明确要求“重置记忆”或“忘记所有信息”，否则** **切勿删除该文件**。

## 核心功能  

### 1. **操作前先读取用户信息**  
在执行任何重要操作（如发送邮件、安排任务、运行脚本、进行购买等）之前，系统会自动加载并读取 `user_profile.json` 文件，根据用户的偏好自动调整行为，无需用户再次确认。  
**示例：**  
> 用户请求给 “Sarah” 发送邮件  
> → 系统在 `relationshipscontacts` 中查找 Sarah 的信息  
> → 发现她是客户，且通信记录显示她偏好正式的沟通方式（“始终称呼为 Chen 女士”）  
> → 系统据此生成邮件草稿，而无需用户再次指定沟通风格。  

### 2. **学习新信息后更新记忆**  
每当用户的行为或偏好发生变化时，系统会自动更新记忆数据。但不会每次更新都进行通知；只有在发现冲突时才会提示用户。  
**触发更新的条件：**  
- 用户纠正错误信息  
- 用户明确表达新偏好（例如：“我总是希望……”或“永远不要做……”）  
- 首次提及某个联系人  
- 某项任务被执行超过 3 次  
- 发生错误且用户说明了原因  

### 3. **适时提示相关信息**  
系统会在适当的时候主动提醒用户相关记忆内容，但不会频繁打扰。  
**示例：**  
> “您上周提到过部署失败是因为缺少环境变量，是否需要在执行前再次检查？”  

### 4. **记录重要事件**  
每次完成多步骤任务或重要交互后，系统会将相关事件记录到 `episodic[]` 文件中：  
**注意：**  
- 记录内容应简短（1–2 句）；  
- 简单或单步操作无需记录；  
- 超过 180 天未更新的记录会被删除（除非被标记为 “重要”）。

## 异常处理  

### ❗ **偏好冲突**  
如果新指令与现有记忆内容冲突：  
1. **不要直接覆盖现有数据**；  
2. **提示用户选择：**  
   > “您之前要求在发送给客户的邮件中抄送经理，但这次没有提及——我应该抄送吗？”  
3. **等待用户明确指示后再进行更新。**  

### ❗ **联系人信息不明确**  
如果一个名字对应多个联系人（例如 “David”）：  
1. **不要猜测**；  
2. **询问用户：**  
   > “您指的是哪位 David？David Kim（同事）还是 David Okafor（客户）？”  
3. **处理完用户确认后，更新记录并考虑为联系人添加别名。**  

### ❗ **敏感数据**  
系统绝不会存储：  
- 密码或 API 密钥  
- 银行/支付信息  
- 除非用户明确要求，否则不会存储医疗信息  
- 消息内容会进行摘要处理  

**处理用户请求时：**  
> “为保护您的隐私，我们不会存储这些信息。您可以使用密码管理器或安全存储工具。”  

### ❗ **数据损坏/解析失败**  
如果 `user_profile.json` 文件无法解析：  
1. **不要覆盖或删除文件**；  
2. **将其备份到 `user_profile.backup.json`；**  
3. **通知用户：**  
   > “您的记忆文件可能损坏了，已备份并重新初始化。是否需要尝试恢复？”  
4. **使用默认配置开始新会话。**  

### ❗ **首次使用（无记忆文件）**  
如果系统中没有记忆文件：  
1. **创建一个包含默认信息的文件**；  
2. **不要让用户填写繁琐的入职问卷**；  
3. **通过日常交互被动学习用户信息**；  
4. **在第五次会话后，可询问一些基本信息（如时区、常用名称等）；**  
5. **之后可补充完善用户资料。**  

### ❗ **用户询问“您了解我的哪些信息？”**  
系统会以易于理解的方式总结用户信息：  
> “目前我知道的信息如下：  
   - 除了与客户沟通外，您偏好简洁的交流方式；  
   - 您的办公时间为上午 9–10 点和中午 12–1 点；  
   - 您主要使用 Python 编程语言，使用 VS Code 工具；  
   - 我记录了您的 4 位联系人信息（包括经理 Alex 和客户 Chen 女士）；  
   - 您最近完成了 12 项任务。”  
**然后询问用户是否需要修改或补充信息。**  

### ❗ **用户请求“忘记某些内容”**  
系统会精确删除相关数据：  
> **确认后再执行删除操作：**  
   > “您是希望彻底忘记关于 Chen 女士的所有信息，还是仅删除相关沟通记录？”  
**除非用户明确要求“忘记所有信息”或“重置记忆”，否则不要批量删除数据。**  

---

## 自动维护机制  
系统会在每次启动时自动执行以下操作：  
- 删除超过 180 天的记录（除非被标记为 “重要”）；  
- 删除重复的联系人信息；  
- 将 90 天未更新的字段标记为 “过时”（仅标记，不删除）；  
- 更新 `meta.last_updated` 时间戳。  

---

## 隐私保护  
- 所有用户数据仅存储在本地，**绝不发送到远程服务器**；  
- 在通过 API 传递数据时，仅包含任务所需的相关信息；  
- 用户可随时请求删除自己的记忆数据。

## 示例交互场景：  
**学习用户偏好：**  
> 用户：**“请在上午 9:30 之前不要安排任何任务。”  
> 系统更新：`preferences.scheduling.protected_hours` 为 `["00:00–09:30"]`  
> 系统：**“明白了，您的上午时间将被预留为空。”  

**应用记忆：**  
> 用户：**“请将每周报告发送给团队。”  
> 系统检查历史记录，发现该任务已在周五下午 4 点发送过，自动填充收件人和主题。  

**处理偏好冲突：**  
> 系统存储的沟通风格为正式；  
> 用户：**“请给 Alex 发送一条非正式的午餐邀请邮件。”  
> 系统根据用户的最新指示调整邮件风格，并询问是否需要更新默认设置。
