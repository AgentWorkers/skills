# TokenKiller（通用节流工具）

## 目标

在不显著降低成功率的前提下，系统性地减少令牌消耗。适用于具备多种功能的代理（如搜索、编码、调试、测试、文档生成等）。

## 任务复杂性评估

在设定预算之前，请先评估任务的复杂性：

| 复杂度 | 评估标准 | 工具使用次数 | 输出内容长度 |
|------------|----------|-------------|---------------|
| 简单 | 单个文件修改、单点定位、需求明确 | ≤3次调用 | ≤50行 |
| 中等 | 涉及2-3个文件、需要简单探索、需求相对明确 | ≤6次调用 | ≤120行 |
| 复杂 | 模块间重构、多步骤调试、需求不明确 | ≤10次调用 | ≤200行 |

**扩展机制（软警告）**：当预算即将耗尽但任务尚未完成时：
1. 输出警告：`[TokenKiller] 预算即将用完，当前进度：X/Y，剩余工作：...`
2. 继续执行，但切换为更保守的策略
3. 用户可以随时中断或请求更详细的输出

## 默认工作模式（平衡模式）

### 全局硬性规则（必须遵守）

- **目标优先，证据其次**：首先用一句话陈述目标（L0），然后再决定是否需要证据（L2/L3）。
- **三个问题限制**：需要澄清时，每次最多提问3个问题；否则根据“默认假设”继续操作，并标记需要替换的部分。
- **逐步披露信息**：默认情况下，仅获取“最低限度必要的信息”；切勿直接将大文件或完整日志放入上下文中。
- **优先输出差异内容**：优先输出补丁/变更内容及结果摘要；避免重新上传整个文件。
- **避免重复引用**：已查看过的信息只需简要提及，不要再次粘贴。

### 预算限制（预算 + 控制机制）

在每个任务开始时，评估复杂性并设定相应的预算（参见上述“任务复杂性评估”），然后执行相应的控制机制：

- **工具使用次数限制**：根据复杂性设定（简单 ≤3次，中等 ≤6次，复杂 ≤10次）。
- **文件读取限制**：默认情况下，单个文件会完整读取；超过200行的文件只读取部分内容。
- **输出内容长度限制**：根据复杂性设定（简单 ≤50行，中等 ≤120行，复杂 ≤200行）。

如果超出任何限制：

- 首先缩小搜索范围（路径/文件/模块）→ 然后调整搜索策略 → 最后逐步扩展读取和输出范围。

## 令牌消耗自查

### 应避免的高消耗行为

- 完整读取超过500行的文件
- 输出整个文件内容（应仅输出差异部分）
- 重复粘贴相同的代码或日志
- 列出整个目录结构
- 输出冗长的解释性文本

## 自我检查频率

每使用3次工具后，快速进行自我检查：

- 我当前处于L0-L2阶段吗？
- 有重复的信息吗？
- 输出内容是否超过了必要的长度？

## 信息层次结构（L0-L3）

- **L0**：一句话的目标（必须提供）
- **L1**：最多3条硬性约束（必须提供）
- **L2**：证据摘要（文件路径 + 行号 / 关键命令的输出行数 / 关键配置项）
- **L3**：完整内容（仅在特定情况下获取，详见下文“L3获取场景”）

默认的输出和上下文应保持在L0-L2层次。

### L3获取场景（特定情况）

仅在以下情况下获取L3（完整内容）：

1. **代码修改**：当需要精确的缩进/格式匹配时，读取目标函数的完整代码。
2. **配置调试**：当配置项相互依赖时，需要查看完整的配置块。
3. **错误分析**：当错误信息不完整时，需要完整的堆栈跟踪或上下文。
4. **用户明确请求**：用户要求查看完整内容。

**决策流程**：
- L2阶段获取证据 → 尝试继续执行 → 失败 → 判断是否需要L3阶段的信息 → 获取最低限度必要的内容

## 多技能协作

当此技能与其他技能同时使用时：

### 优先级规则

- **功能技能优先**：如`pdf`、`xlsx`等特定技能的规则具有优先权。
- **TokenKiller作为约束层**：在其他技能执行过程中，持续应用预算和规则。
- **用户需求优先**：用户的明确输出格式/内容需求优先于节流规则。

### 协作模式

```
[User Request] → [Functional Skill Processing] → [TokenKiller Constrains Output]
```

## 工作流程（通用）

### 1) 任务启动（适用于任何领域）

1. 提供L0和L1级别的信息（如果用户未提供，则快速推断）。
2. 选择策略（搜索/直接修改/验证）。
3. 执行最小范围的操作。
4. 立即验证结果（选择最简单的验证方法）。
5. 总结：仅提供关键结论及下一步操作。

### 2) 搜索/探索（优先领域）

优先顺序：

1. **文件名/路径**（全局搜索）
2. **精确字符串**（正则匹配）
3. **语义搜索**（SemanticSearch）
4. **文件读取**（按段落或行范围读取）

规则：

- 仅读取接近匹配的结果（±20行）或与目标函数/组件相关的段落。
- 不要未经处理就阅读整个仓库的内容。

### 3) 编码/重构

规则：

- 先进行最小范围的修改：如果可以修改1个文件，就不要修改5个文件。
- 避免“全部重写”；优先使用现有结构。
- 修改后，立即进行最简单的验证（如`tsc`/`build`/`lint`）。
- 仅显示关键的差异内容（最多3处代码引用）。

### 4) 调试/故障排除

规则：

- 首先列出3个最有可能的故障原因（按信息价值排序）。
- 每次只验证1个假设，并仅收集必要的证据。
- 日志记录应包括：错误行号、堆栈跟踪顶部信息、相关配置项及重现命令（其余内容简化处理）。

### 5) 测试/验证

优先顺序（从简单到复杂）：

1. 代码检查（lint / 类型检查）
2. 构建代码
3. 单元测试
4. 环境测试（e2e）/ 浏览器自动化测试

如果测试失败，仅添加差异信息，不要重新上传全部输出内容。

### 6) 文档生成/总结

规则：

- 默认输出为“简短总结 + 下一步操作”。
- 不要重复用户原有的表述；使用结构化的信息呈现方式。
- 当需要生成文档时，采用逐步披露的方式：先提供大纲/要点，再详细说明。

## 输出模板（默认）

使用以下结构，除非用户另有要求：

- **结论**：一句话总结。
- **证据**：2-5条信息（文件路径/行号/关键命令的输出结果）。
- **变更内容/操作**：说明了哪些操作（最多5条）。
- **下一步操作**：最重要的下一步行动。
