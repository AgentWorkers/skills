# X增长自动化

为用户构建一个**独立且可重用的X自动化项目**。请不要假设他们当前的生产项目需要修改；除非用户明确指向现有的代码库，否则建议创建一个新的文件夹。

## 核心工作流程

1. 首先使用`references/setup-questionnaire.md`询问相关设置问题。
2. 选择合适的发布模式，参考`references/rollout-modes.md`。
3. 使用`scripts/scaffold_x_growth_project.py`搭建一个干净的项目框架。
4. 根据用户的回答填写配置文件。
5. 默认情况下禁用发布功能，除非用户明确要求进行实时发布。
6. 如果启用实时发布，请确保明确指定发布频率、时间槽安排及备用方案。

## 该技能的目标

构建一个可重用的X自动化系统，包含以下功能：
- 通过Bird CLI进行内容发现
- 内容选择与评分机制
- 草稿生成功能
- 可选的基于大型语言模型（LLM）的审核流程
- 可选的从外部编辑源获取内容的机制
- 可选的社区互动提示（CTA）逻辑
- 通过X API进行基于时间槽的发布
- 先进行模拟运行（dry-run）后再正式发布

## 不可妥协的安全性默认设置

- 除非用户另有说明，否则将项目创建在**新文件夹**中。
- 将Bird工具视为主要用于内容发现（读取数据）的工具。
- 将X API视为用于内容发布（写入数据）的工具。
- 新安装默认设置为**模拟运行**模式。
- 即使用户希望自动化程度更高，也应在配置文件中设置每日/每月的发布上限。
- 绝不要从其他项目中默默复制私密令牌、链接、处理方式或特定假设。
- 将回复自动化视为比普通帖子更高风险的操作。
- 在实时发布模式下，必须记录所有发布的、跳过的或失败的操作。

## 在搭建项目框架之前请询问以下问题

阅读`references/setup-questionnaire.md`，仅询问未回答的问题。如果用户已经回答了部分问题，请不要一次性提出所有问题。

**解锁项目设置的最低要求**：
- 专注的领域/主题
- 发布语言
- 是进行模拟运行还是实时发布
- 每日的发布目标范围
- 是否可以使用Bird工具？
- 是否可以使用X API？
- 是否需要从外部来源获取内容？
- 是否需要包含社区互动提示（CTA）？

## 收集到明确答案后

运行搭建脚本：

```bash
python3 scripts/scaffold_x_growth_project.py --path <target-dir> --profile-json '<json>'
```

JSON文件可能包含以下字段：
- `project_name`（项目名称）
- `language`（语言）
- `niche_summary`（领域概述）
- `daily_min`（每日最低发布量）
- `daily_max`（每日最大发布量）
- `monthly_cap`（每月发布上限）
- `community_enabled`（是否启用社区功能）
- `community_link`（社区链接）
- `reply_cta_enabled`（是否启用回复CTA）
- `reply_cta_style`（回复CTA样式）
- `source_branching_enabled`（是否启用内容分支）
- `source_branching_label`（内容分支标签）
- `bird_enabled`（是否启用Bird工具）
- `x_api_enabled`（是否启用X API）
- `live_publish`（是否启用实时发布）

## 推荐的设置流程

### 如果用户表示“请为我安装这个系统”
- 询问相关设置问题。
- 搭建一个新的项目框架。
- 填写配置文件。
- 除非用户明确同意实时发布，否则禁用发布功能。
- 向用户说明应如何存放凭据。

### 如果用户表示“能否根据我的领域进行定制？”
- 询问用户的领域、目标受众、语气及内容方向。
- 更新相关配置（如主题设置、搜索设置或关注账户设置）。
- 如有需要，更新跨平台发布/回复策略。

### 如果用户表示“希望自动化程度更高”
- 明确提高发布频率和发布上限。
- 确保发布上限被记录在配置文件中。
- 确认回复是否需要包含CTA。
- 除非用户要求，否则不要取消质量检查机制。
- 即使在自动化程度较高的模式下，也要保留针对回复的防护措施（如目标验证、失败时跳过发布、每日发布上限等）。

### 如果用户表示“希望集成社区功能”
- 询问外部社区/内容源是否：
  - 是内容的真实来源
  - 仅用于发布CTA
  - 两者皆是
- 分别配置内容分支和回复CTA规则。
- 除非用户明确说明，否则不要默认使用Telegram作为社区平台。

## 文件定制顺序

搭建项目框架后，按照以下顺序进行定制：
1. `config/publish-policy.json`（发布策略配置）
2. `config/topics.json`（主题配置）
3. `config/budget-policy.json`（预算策略配置）
4. `.env.example`（环境变量配置）
5. `docs/operator-notes.md`（操作员指南）
6. 如果用户需要特定的语气风格，可相应调整相关配置文件。

## 何时启用实时发布模式

只有在满足以下所有条件时才能启用实时发布模式：
- 凭据已准备好
- 用户明确要求实时发布
- 发布频率和上限已设定
- 回复策略已定义
- 内容分支行为已明确
- 用户了解自动发布的规则

## 发布策略建议

推荐的实时发布内容结构：
- 如果配置了每日新闻/来源内容，发布1条
- 如果配置了每日特殊内容，发布1条
- 每天发布0-1条带有CTA的回复
- 每天从主要草稿流程中发布0-2条核心内容

如果用户希望调整发布内容组合，请将其明确写入配置文件，而非通过文字描述。
