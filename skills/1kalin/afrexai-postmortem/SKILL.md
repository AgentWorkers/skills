# 事故事后分析工具

该工具能够从原始的笔记、Slack聊天记录或项目列表中自动生成无责备性的事故事后分析报告。

## 功能概述
该工具能够将杂乱无章的事故详细信息整理成符合 Google/Atlassian SRE 最佳实践的结构化报告。

## 使用方法
只需以任何形式提供事故详细信息（如时间线列表、Slack 转发的内容或口头记录），工具便会生成如下报告：
1. **执行摘要**：事故经过、影响范围、持续时间、严重程度
2. **时间线**：从发现事故到解决问题的详细过程（按分钟记录）
3. **根本原因分析**：采用“5个为什么”（5 Whys）的方法进行分析，避免互相指责
4. **影响评估**：受影响的用户数量、对收入的影响、是否违反了服务水平协议（SLA）
5. **待处理事项**：列出需要解决的优先级问题及其负责人和截止日期
6. **经验教训**：总结哪些措施有效、哪些无效、哪些是运气好的因素
7. **预防措施**：提出系统性改进措施以防止类似事故再次发生

## 使用说明
当用户提供事故详细信息时：
- 仅在关键信息缺失（如事故严重程度、持续时间或解决时间）时才提出澄清问题
- 生成完整的 markdown 格式报告
- 指出报告中需要团队补充的部分
- 建议 3-5 项具体的、可执行的预防措施，并按实施难度和影响程度进行排序

### 格式规范
- 时间线使用 ISO 标准时间戳
- 严重程度用粗体显示（SEV1-SEV4）
- 待处理事项应包含：描述、负责人、截止日期、优先级（P0-P3）
- 语言表述需保持中立，避免使用指责性语言（例如：使用“部署流程”而非“Bob 负责部署”）

### 严重程度分级标准
- **SEV1**：对收入造成影响，所有用户均受影响，持续时间超过 1 小时
- **SEV2**：核心功能失效，超过 30% 的用户受到影响，持续时间超过 30 分钟
- **SEV3**：系统性能下降，受影响用户比例低于 30%
- **SEV4**：小问题，已有临时解决方案

## 输入示例
```
prod went down at 2pm, bad deploy, rolled back at 2:45, ~500 users couldn't checkout, lost maybe $12k revenue
```

```markdown
# Incident Postmortem: Checkout Service Outage
**Date:** 2026-02-22 | **Severity:** SEV1 | **Duration:** 45 minutes
**Author:** [Team Lead] | **Status:** Draft

## Executive Summary
...
```

## 使用建议
即使是小事故，也建议在每次事故发生后使用该工具进行分析。通过对比多次事故报告中的规律，团队能够更快地发现系统性问题。对于严重程度为 SEV3 以上的事故，使用该工具后，发现问题速度可提高 3 倍。
