# 你知道吗？

维基百科的[“你知道吗？”](https://en.wikipedia.org/wiki/Wikipedia:Did_you_know)板块会展示来自最近创建或扩展的文章中的、来源可靠的事实。这些内容由志愿者每天至少更新一次。

当用户想要了解有趣的事实、希望自动接收每日琐事信息，或者想要自定义他们看到的事实类型时，可以使用这个功能。

> **机器人提示：** 除非用户明确要求，否则不要向用户显示bash命令或技术细节。所有命令都应在后台默默执行。用户体验应该是对话式的——包括事实的提供、问题的提出以及自然的后续交流。在回应用户的偏好时，避免使用诸如“品味真棒！”或“选择很棒！”之类的空洞赞美语。

## 用户可能的需求

| 用户的需求 | 应采取的措施 |
|---|---|
| 需要一个事实 | 提供一个事实（参见“提供事实”部分） |
| 自定义看到的事实类型 | 帮助用户设置偏好（参见“管理偏好”部分） |
| 自动接收事实 | 帮助用户设置事实推送的日程（参见“安排事实推送”部分） |
| 更新事实队列 | 执行更新流程 |

## 是否需要任何设置？

**事实功能可以立即使用——无需任何设置。** 只需要提供一个事实即可。

**偏好设置是可选的，但推荐使用。** 如果用户希望调整显示的主题（例如更多历史类内容、减少科学类内容等），你可以在一两分钟内完成设置。偏好设置需要启用更新流程，以便新事实在被评分之前能够被正确标记。

**只有当偏好设置激活时，才需要安排自动更新。** 如果没有偏好设置，该功能会根据需要自动获取并缓存事实。如果用户设置了偏好，就需要同时设置推送日程和更新日程（以便对新事实进行评分）。

如果用户询问是否需要设置，可以这样回答：“您现在就可以获取事实——无需任何设置。如果您希望调整显示的主题，或者希望每天自动接收一个事实，我们可以在一分钟内为您设置好。需要吗？”

## 提供事实

> **机器人提示：** 在后台默默执行此操作——不要向用户显示命令内容。

```bash
python3 {baseDir}/scripts/dyk.py
```

将获取到的事实原样返回给用户。

**提供的事实：**
```
Did you know that heavy-metal guitarist Kiki Wong played drums for Taylor Swift before joining the Smashing Pumpkins?

https://en.wikipedia.org/wiki/Kiki_Wong
```

**没有更多事实可提供：**
```
No more facts to share today; check back tomorrow!
```

**错误：**
```
Something went wrong with the fact-fetching; please try again later.
```

提供事实后，检查`~/.openclaw/dyk-prefs.json`文件是否存在。如果该文件不存在，说明用户是第一次使用该功能——可以接着问：“您是否希望调整看到的事实类型（例如更多关于音乐的内容、减少科学类内容等），或者希望每天自动接收一个事实？” 如果文件已经存在，或者用户之前已经设置了偏好或推送设置，就无需再做任何操作。

## 管理偏好

事实的评分和排名是根据用户的偏好来决定的。用户喜欢的标签会提高其得分；不喜欢的标签会降低得分。同时，事实的新鲜度和多样性也会对评分产生正面影响。

### 先进行交流

在执行任何命令之前，先了解用户的喜好。如果用户已经告诉了你他们的喜好（例如“我喜欢黑暗题材的历史故事，但不喜欢体育相关的内容”，请直接将他们的偏好映射到相应的标签上——不要让他们重复说明。只有当用户的意图不明确时，再提出后续问题。有两个主要的偏好维度：

- **领域**：主题类别（例如历史、科学、音乐）
- **风格**：内容风格或情绪（例如离奇、励志、黑暗）

不要一开始就列出所有的标签选项——只需用自然的语言询问用户他们的喜好，然后根据他们的回答进行匹配。例如：“我喜欢黑暗题材的历史故事”可以映射为`domain: history`（喜欢）和`tone: dark`（喜欢）。

在开始讨论偏好设置时，要告诉用户你还会同时设置自动更新功能——这样新事实到达时就能根据他们的偏好进行评分。语气要轻松自然，例如：“我还会在后台设置自动更新功能，以便新事实能够根据您的偏好进行评分。” 用户不需要了解具体的设置机制，但他们应该知道整个设置过程正在进行中。

### 设置偏好

了解用户的偏好后：
1. 检查`prefs.py`文件是否存在。如果不存在，先初始化它（使用`prefs.py init`命令）。
2. 为每个偏好设置相应的值（使用`prefs.py set`命令）。
3. 用简单的语言向用户解释已经设置的偏好以及这些偏好将如何影响他们看到的事实。

如果用户想查看当前的偏好设置，可以运行`prefs.py list`命令，并以易于理解的方式呈现结果——不要以原始的命令输出形式呈现。

> **机器人提示：** 在后台默默执行偏好设置命令——不要向用户显示命令内容。

```bash
python3 {baseDir}/scripts/prefs.py init                     # Create prefs file with neutral defaults. Fails if already exists.
python3 {baseDir}/scripts/prefs.py list                     # Show all current preferences
python3 {baseDir}/scripts/prefs.py get domain science       # Get a single preference
python3 {baseDir}/scripts/prefs.py set domain science like  # Set a preference: like | neutral | dislike
```

**领域：** 动物 · 经济与商业 · 电影 · 历史 · 新闻业 · 语言与语言学 · 文学 · 医学与健康 · 军事历史 · 音乐 · 神话与民间传说 · 自然 · 表演艺术 · 地点 · 宗教 · 科学 · 体育 · 科技 · 电视 · 视觉艺术

### 设置偏好后

偏好设置完成后，立即执行更新流程——因为现有的缓存内容还没有被标记，所以在更新之前无法进行评分。不要等待预定的定时任务，现在就直接执行更新操作。然后设置自动更新日程，以便新事实到达时能够自动被标记。可以这样告诉用户：“我现在会为您标记现有的事实，以便它们立即根据您的偏好进行评分，并在后台设置自动更新功能。”

如果用户还没有设置自动推送功能，也可以建议他们设置。

## 安排事实推送

当用户希望自动接收事实时，在进行任何设置之前，先与用户进行一次交流，了解他们希望接收事实的频率：
- 他们希望每天接收一次事实吗？比如在早餐时间、通勤途中或一天结束时接收都很好。一天中分几次接收也是可行的。
- 需要注意的是：事实推送的频率越高，事实在队列中的优先级就越低。

用户选择好推送频率后，使用OpenClaw的定时任务来安排推送。使用`--session isolated`和`--announce`选项，以便将推送结果发送回用户的聊天窗口。**不要**使用操作系统的定时任务——因为操作系统的定时任务没有推送上下文，推送结果可能会丢失。

> **机器人提示：** 在执行此命令之前，从会话上下文中获取当前的频道平台和聊天ID，然后使用这些信息设置`--channel`和`--to`参数。命令要在后台默默执行——不要向用户显示命令内容。

```bash
openclaw cron add \
  --name "DYK delivery" \
  --cron "<schedule>" \
  --session isolated \
  --message "Share a Did You Know fact" \
  --announce \
  --channel <platform> \
  --to <platform>:<id>
```

将`<schedule>`替换为一个符合用户选择的推送频率的五字段cron表达式（例如`57 7 * * *`表示每天早上8点左右）。避免使用精确的`:00`分钟时间——时间可以稍微提前或推迟几分钟。将`<platform>`和`<id>`替换为当前频道的平台和聊天ID（例如`--channel telegram --to telegram:1234567890`）。

如果用户已经设置了偏好，还需要设置自动更新功能——否则新事实将不会被标记，也无法根据偏好进行评分。可以这样告诉用户：“我还会设置自动更新功能，以确保您的信息队列始终保持最新状态。” 然后按照“设置自动更新”部分的说明进行操作。

如果用户没有设置偏好，或者不希望设置自动更新功能，可以跳过这一步——因为该功能会自动处理基本的缓存更新。

## 设置自动更新

维基百科的“你知道吗？”模板由志愿者在大约**00:00 UTC**和**12:00 UTC**更新。因此，可以在这些时间点之后立即安排更新任务，以便新事实能够及时被获取和标记。使用`--session isolated`和`--no-deliver`选项——这个任务对用户是不可见的，也不会在聊天窗口中显示任何信息。

> **机器人提示：** 在后台默默执行此命令——不要向用户显示命令内容。

```bash
openclaw cron add \
  --name "DYK refresh" \
  --cron "7 0,12 * * *" \
  --session isolated \
  --message "Refresh the DYK cache and tag new hooks" \
  --no-deliver
```

定时任务会按照设定的时间表执行更新操作。

## 更新流程

当用户请求更新“你知道吗？”事实的缓存并标记新内容时，执行以下步骤：
1. 运行`python3 {baseDir}/scripts/fetch_hooks.py`命令：
   这个命令会获取最新的内容，并将它们存储在`~/.openclaw/dyk-facts.json`文件中。对于新添加的内容，`tags`字段会被设置为`null`。
   如果命令执行失败（返回非零状态），则停止并报告错误，不要继续执行后续操作。

2. 读取`~/.openclaw/dyk-facts.json`文件，找出所有`tags`字段为`null`的条目。
   如果没有这样的条目，说明没有新的内容需要标记，直接结束流程。

3. 一次性为所有未标记的内容分配标签——不要多次循环或重复调用工具。对于每个内容，使用以下规则进行标签分配：
   - 标签分配指南：`{baseDir}/references/tagging-guide.md`
   - 标签词汇表：`{baseDir}/references/tags.csv`

   输出要求：
   - 仅使用`tags.csv`文件中定义的标签值
   - 如果标签分配的可靠性较低，设置`"low_confidence": true`
   - 生成的JSON格式必须有效，不要包含任何注释或解释
   - 将结果收集到一个JSON数组中
   - 将数组写入临时文件`/tmp/dyk-tags.json`

   ```bash
   # Example array written to /tmp/dyk-tags.json
   [
     {"url": "https://en.wikipedia.org/wiki/Kiki_Wong",
      "domain": ["music"], "tone": "surprising", "low_confidence": false}
   ]
   ```

4. 运行`python3 {baseDir}/scripts/write_tags.py --json-file /tmp/dyk-tags.json`命令：
   如果这个命令执行失败（返回非零状态），报告错误。

5. 除非出现错误，否则不要向用户发送任何信息。

有关完整的命令参考，请参阅`{baseDir}/references/commands.md`文件。
