# 改善人际关系

本技能用于检测与他人的联系机会，对这些机会进行排序，并准备合适语气和语言的沟通草稿。**所有草稿在用户明确批准之前都仅处于草稿状态。**

**核心流程：**检测 → 评分 → 排序 → 草拟 → 等待批准。

## 何时激活该技能

- 用户询问应联系谁或请求发送信息摘要
- 用户希望在生日、节日、重要时刻或重新建立联系时收到草稿
- 用户共享浏览器标签页（任何平台）以供关系评估
- 用户询问近期未通过任何渠道联系过的联系人
- 用户提及文化相关场合并请求协助进行联系

**请勿**将此技能用于销售流程、潜在客户开发或CRM数据录入。

## 信号来源

信号可以来自用户提供的或可访问的任何信息：

- **日历**（Google日历、Outlook、Apple日历）——生日、纪念日、事件
- **电子邮件**（Gmail、Outlook等）——最后一次联系时间、对话的最近性
- **即时通讯**（Slack、WhatsApp、iMessage、Teams）——最后一次联系时间、对话内容
- **共享的浏览器标签页**——用户共享的任何平台（LinkedIn、Instagram、Twitter/X、Facebook、公司页面、新闻文章）
- **用户提供的信息**——对话中提及的内容、联系人备注或上传的文件
- **新闻和网站**——文章、新闻稿、公开公告

该技能支持所有这些来源，且不依赖于任何特定平台。

## 需要检测的信号

| 信号 | 典型来源 | 证据 |
|--------|----------------|----------|
| 生日 | 日历、用户提及 | 如果有具体日期，则为高信号 |
| 工作纪念日 | LinkedIn、电子邮件签名、用户提及 | 中等信号 |
| 职位变动/晋升 | LinkedIn、Twitter/X、新闻、电子邮件、用户提及 | 中等偏高的信号 |
| 长时间未联系（60-90天以上） | 最后一次联系时间、即时通讯记录、用户声明 | 中等信号 |
| 节日/重要场合 | 日历日期、用户请求 | 高信号 |
| 公共成就 | 任何共享的标签页、新闻、社交帖子 | 中等信号 |
| 公司新闻 | 新闻文章、共享的标签页、用户提及 | 低信号 |
| 生活事件（婚礼、婴儿出生、搬家） | 用户提及 | 如果得到确认，则为高信号 |
- 表示慰问 | 用户提及 | 如果得到确认，则为高信号 |
- 轻度重新建立联系 | 信号较弱 + 关系重要性较低 | 低信号 |

**请仅引用实际存在的信息来源。**切勿捏造访问权限或证据。

## 证据等级

| 等级 | 含义 | 例子 |
|-------|---------|---------|
| **高** | 用户确认或日历验证 | 日历中记录的生日 |
| **中等** | 在共享的标签页或对话中可见 | 在LinkedIn、Instagram帖子或Slack消息中可见的晋升信息 |
| **低** | 推测性证据、单一间接来源 | 关于联系人的公司新闻文章 |

对于每个建议，都需要展示证据等级和来源。如果证据等级较低，建议用户在发送前进行核实。

## 工作联系与个人联系的区分

将每个联系人标记为**工作联系**、**个人联系**或**需要询问**。在信息摘要中自动区分这两类联系：

- **工作联系**：电子邮件、LinkedIn、Slack。使用专业语气。
- **个人联系**：短信、WhatsApp、私人电子邮件、电话。使用亲切的语气。
- 除非用户特别要求，否则不要混合这两类联系的信息。

## 评分标准

五个维度（1-5分）：

| 维度 | 含义 |
|-----------|---------|
| 信号强度 | 信息是否明确？（生日 = 5分，模糊的新闻 = 2分） |
| 最近性 | 信息是否紧急？（今天的生日 = 5分，3个月未联系 = 3分） |
| 关系重要性 | 该联系的重要性如何？（如果是询问，则需要评估） |
| 联系方式的适宜性 | 这种联系方式是否合适？（晋升通知 = 5分，裁员传闻 = 1分） |
| 证据可靠性 | 信息的可靠性如何？（已确认 = 5分，推测 = 2分 |

**综合评分** = 五个维度的平均值。评分最高的信息优先显示。

### 排序标准

| 综合评分 | 推荐行动 |
|-----------|--------|
| **4.2分及以上** | 立即推荐 — 在当前页面显示 |
| **3.2分–4.1分** | 建议稍后处理 — 放入待处理列表 |
| **低于3.2分** | 从默认信息摘要中省略 |

**默认显示前5条信息。**始终注明总数：“显示{N}条机会中的前5条。”当用户点击“下一个”或“更多”时，再显示接下来的5条。

## 为什么现在推荐

每个推荐都会回答以下问题：

1. **为什么选择这个人？**——基于什么信号？
2. **为什么现在推荐？**——为什么现在推荐，而不是以后？
3. **通过哪种方式联系最合适？**——哪种方式最能有效地与他们取得联系？

如果无法清晰回答这三个问题，建议将信息降级为“待处理”状态。

## 联系方式的选择逻辑

| 信号 | 工作联系 | 个人联系 |
|--------|------|----------|
| 晋升/职位变动 | LinkedIn或电子邮件 | 短信或WhatsApp |
| 生日 | 电子邮件（与同事关系密切） | 短信、WhatsApp或电话 |
| 长时间未联系 | 电子邮件 | 短信、WhatsApp或电话 |
| 重要场合/节日 | 电子邮件（简短问候） | WhatsApp、短信或电话 |
| 成就 | LinkedIn或电子邮件 | 短信或Instagram私信 |
- 表示慰问 | 电子邮件（谨慎处理） | 电话、短信或当面沟通 |

如果用户设置了**首选联系方式**，请尊重其选择。否则默认使用该联系人的常用联系方式。

## 语气选择

| 语气 | 适用场景 |
|------|---------|
| 亲切的私人语气 | 与亲密朋友、家人或个人重要场合的联系 |
- 尊重的专业语气 | 与工作联系人或正式场合的联系 |
- 庆祝语气 | 晋升、成就、婚礼、生日等场合 |
- 轻度重新建立联系 | 长时间未联系时的简单问候 |
- 节日场合 | 斋月、开斋节、排灯节、诺鲁孜节 |
- 表示慰问 | 用于表达哀悼或支持时 | 语气要温和、谨慎 |

对于亲密关系，参考之前的交流记录。切勿捏造回忆。

## 多语言支持

草稿应使用用户和联系人实际使用的沟通语言。本技能支持**任何语言**，不限于预设的语言列表。

**语言优先级**：

1. **沟通历史**：查看用户与该联系人的实际沟通语言。如果他们的邮件、消息或聊天记录是葡萄牙语，那么草稿也应使用葡萄牙语。
2. **联系人资料**：如果没有沟通历史，可以查看联系人的个人资料、社交帖子或简介中的语言信息。
3. **联系人的语言偏好**：如果用户为该联系人设置了语言偏好，请使用该语言。
4. **用户的默认语言**：如果以上信息均不可用，使用用户自己的语言，并注明：“我使用[语言]起草——您希望我更换语言吗？”

**规则**：

- 用户或联系人使用的任何语言都可以作为草稿语言。例如阿拉伯语、法语、西班牙语、普通话、印地语、德语、日语、斯瓦希里语等，根据关系情况选择合适的语言。
- 根据场合使用合适的问候语（例如：“Ramadan Mubarak”/“Ramadan Kareem”、“新年快乐”、“Feliz Navidad”）。
- 如果联系人使用多种语言，工作沟通时使用专业语言，私人沟通时使用日常语言。
- 对于语言可靠性较低的草稿，请特别标注。
- 如果语言偏好不明确，在起草前请询问用户。

## 文化场合

本技能支持全球通用的节日和场合——不仅限于西方节日：

- 斋月（Ramadan）、开斋节（Eid al-Fitr）、古尔邦节（Eid al-Adha）、农历新年、排灯节（Diwali）、诺鲁孜节（Nowruz）、圣诞节、光明节（Hanukkah）、新年、感恩节。

- 不要假设联系人的文化或宗教背景。如果不确定，请询问。
- 核对日历日期。在斋月期间不要发送节日祝福。
- 如果联系人的背景不明确，使用通用、非宗教性的问候语。

## 输出格式

```markdown
### [Name]
- **Stream**: Work / Personal
- **Signal**: [trigger]
- **Why now**: [timing + appropriateness + channel fit]
- **Score**: [composite] — [High / Medium / Low] evidence ([source])
- **Channel**: [channel]
- **Language**: [language] (detected from [history / profile / field / default])
- **Tone**: [tone]
- **Draft**: > [1-3 sentences]
- **Action**: Approve to send / Edit / Snooze / Skip
```

默认按联系类型分组显示前5条信息，然后显示待处理列表。始终注明总条数。

每条信息摘要的结尾都会显示：“这些是草稿——尚未发送。当前显示{X}条机会中的{N}条。点击‘下一个’查看更多，或告诉我哪些信息需要最终确定。”

## 隐私与安全

1. **最小权限**：仅访问用户提供的数据。严禁访问与用户无关的平台。
2. **仅生成草稿**：本技能仅负责准备草稿，不会发送、发布或传递任何信息。所有信息都必须由用户亲自批准后才能发送。
3. **数据保留**：除非另有配置，否则数据仅在会话期间保存。
4. **证据限制**：明确说明所查看的内容和来源。不得跨来源进行推断。
5. **跨场景隐私保护**：在处理私人联系时不得使用工作相关的信息；在处理工作联系时，除非用户明确同意，否则不得使用私人信息。
6. **敏感情况**：在涉及裁员、健康问题、分手或死亡等敏感话题时，必须在起草前征求用户同意。
7 **安全起草**：除非用户明确要求，否则不得包含敏感信息（如健康状况、财务信息或法律相关内容）。
8 **谨慎处理**：默认采取谨慎态度。在信息不明确时，请务必询问用户。

## 信息摘要的工作流程

- 默认显示**本周**、**两类联系**、**前5条信息**。
- 从**所有可用信息来源**（日历、电子邮件、即时通讯、共享标签页、对话记录、联系人备注）收集信号。
- 进行评分、排序并应用评分标准。
- 以指定的格式呈现结果。
- 提供用户选择的草稿准备服务。**切勿发送草稿，仅提供草稿并等待用户确认**。

**定期更新信息摘要**：可以设置定时任务或定期自动更新摘要。

**浏览器标签页审查**

当用户共享标签页（LinkedIn、Instagram、Twitter/X、Facebook、公司页面、新闻文章或其他平台）时：

1. 识别平台及页面内容。
2. 提取相关信息：职位变动、帖子、重要事件、成就等。
3. 评估重新建立联系的必要性。
4. 说明证据来源：“根据此[平台]标签页上的信息……”
5. 严禁声称拥有API访问权限。你仅查看用户共享的内容，不得超出权限范围。

**示例交互方式**：

- “这周我应该联系谁？” → 显示排名前5条信息及总数。用户可点击“下一个”查看更多。
- “为亲密联系人起草节日祝福信息。” → 用合适的语言和语气起草草稿并提交给用户审批。
- “查看这个LinkedIn标签页——有什么可以重新建立联系的理由吗？” → 从标签页中提取信息并进行评分，并说明证据来源。
- “查看这个Instagram页面——有什么值得联系的内容吗？” → 同样的处理流程，适用于任何共享的标签页。
- “查找90天内未联系过的人。” → 按重要性排序并建议发送重新联系的信息。
