## **Instruction Anchor Guard**  
**功能说明：**  
在数据压缩、会话重启或长时间的多轮任务中，防止用户的关键约束信息丢失或发生偏离。  

****触发条件：**  
- 用户将某条指令标记为“重要”、“关键”、“必须执行”、“始终执行”、“绝不能执行”、“最高优先级”或“规则”；  
- 任务包含超过3个步骤且跨越多个执行轮次；  
- 发生数据压缩（或可能发生压缩），且任务具有不可协商的要求；  
- 代理行为与用户之前的明确约束存在差异。  

****锚点存储机制：**  
锚点信息存储在以下路径中（按优先级和创建时间排序）：  
1. `memory/anchors.md`  
2. `memory/instruction-anchors.md`  
3. （仅作为工作区根目录的备用存储）`.anchors.md`  

****工作流程：**  
### 1. 捕获用户指令**  
- 解析用户的最新消息，提取可能的锚点内容；  
- 仅保留指令/约束信息，删除示例和无关内容；  
- 为每个锚点分配默认值：  
  - 优先级：P1（除非用户明确指定为P0或P2）  
  - 作用范围：当前会话（除非用户指定为全局或任务范围）  
  - 失效时间：会话结束（除非用户指定为“永远有效”或具体日期）。  

### 2. 确认锚点有效性**  
- 如果锚点的作用范围为“全局”或优先级为P0，执行确认操作后再进行持久化存储；  
- 未经用户明确许可，不得将优先级从P2或P1提升为P0。  

### 3. 持久化锚点信息**  
- 将锚点信息添加到存储系统中；  
- 如果新生成的锚点与已存在的相同作用范围的锚点冲突，标记旧锚点为“已被替换”；  
- 禁止默默地修改历史记录，确保审计追踪的完整性。  

### 4. 每轮执行及压缩后更新**  
- 加载当前有效的锚点信息（状态为“active”且未过期）；  
- 在工具执行前，根据优先级和创建时间对锚点信息进行排序；  
- 将排序后的锚点信息用于计划制定过程。  

### 5. 检查计划偏离情况**  
- 在给出最终答案或执行破坏性操作之前，对比当前计划与有效锚点信息：  
  - 如果存在冲突：  
    - 优先级为P0的锚点冲突：立即停止并修正计划；  
    - 优先级为P1的锚点冲突：自动修正计划并记录调整内容；  
    - 优先级为P2的锚点冲突：只有在没有用户明确反对的情况下才继续执行当前计划；  
    - 在响应中输出“DRIFT_CHECK”提示信息。  

****冲突解决规则：**  
- 决策顺序：系统安全策略 > 用户设定的锚点（P0 > P1 > P2）；  
- 在优先级相同的情况下，作用范围更具体的锚点优先；  
- 如果优先级和作用范围都相同，新生成的锚点优先；  
- 仅在不违反安全规则的情况下，才允许用户手动覆盖现有锚点设置。  

****输出格式：**  
当存在锚点信息时，会在输出中包含以下格式的提示块：  
```markdown
ANCHORS_ACTIVE
| ID | P | Scope | Expires | Instruction |
|----|---|-------|---------|-------------|
| anchor-001 | P0 | global | never | Never perform destructive data deletion without confirmation |

PLAN_GUARD
- Current action: <action>
- Conflicts: none | <anchor ids>
- Decision: clear | corrected | paused-awaiting-user

DRIFT_CHECK
- Last anchor sync: <timestamp>
- Drift: no | yes
- Fix applied: <none|what changed>
```  

****安全注意事项：**  
- **数据保护：**  
  - 绝不存储令牌、API密钥、密码、Cookie或认证信息；  
  - 除非用户明确要求，否则不存储任何个人敏感数据；  
  - 敏感内容应替换为`[REDACTED]`；  
  - 仅存储约束条件，而非完整的数据集；  
  - 不能仅凭锚点信息的存在就执行破坏性操作，仍需用户明确确认。  

****过期与维护：**  
- 当锚点的失效时间早于当前时间时，将其标记为“已过期”；  
- 支持以下控制命令：  
  - `/anchors list`：列出所有锚点信息  
  - `/anchors pause <id>`：暂停指定锚点的生效  
  - `/anchors resume <id>`：恢复指定锚点的生效  
  - `/anchors delete <id>`：删除指定锚点  
  - `/anchors pin <id> never`：永久固定指定锚点的生效状态  

****集成说明：**  
- 该功能需与`memory-self-heal`配合使用，以便在计划偏离后进行自动重试或回退；  
- 与`task-execution-guard`配合使用，确保在每个执行阶段都执行锚点检查；  
- 保持功能的确定性和简洁性；在存在冲突时避免自由解释。
