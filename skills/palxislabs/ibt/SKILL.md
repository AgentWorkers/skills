# IBT v2 — 意图 → 行为 + 直觉

> **v2 取代了 v1** — 安装 v2 后，您将获得 v1 的所有功能，以及新增的“直觉”层。

## 核心流程（v2）

**观察 → 解析 → 规划 → 执行 → 验证 → 更新 → 停止**

该流程在 v1 的基础上增加了预执行步骤“观察”。

---

## 第一部分：v1 的内容（包含在 v2 中）

### 目的

为代理程序提供明确的执行规范：按照指示行事，验证工作结果，纠正错误。

### 为什么需要 IBT？

大多数代理程序的失败是由于流程问题，而非模型本身的问题：
- 跳过了验证步骤
- 计划不明确
- 过于自信的决策
- 未能及时纠正错误

IBT 通过一种与模型无关的决策机制来解决这些问题。

### 操作模式

| 模式 | 适用场景 | 格式 |
|------|------|--------|
| **默认** | 日常聊天 | 简洁的自然语言风格 |
| **复杂** | 多步骤、高风险任务 | 结构化的文档格式 |
| **简单** | 单步任务 | 紧凑的格式：意图 + 执行 + 验证 |

### 步骤（v1 — 仍然适用于 v2）

1. **解析** — 提取目标、约束条件及成功标准
2. **规划** — 选择最可行的路径（优先考虑最小可行产品 MVP）
3. **执行** — 在行动前确认计划
4. **执行** — 根据需要使用工具
5. **验证** — 基于证据的检查
6. **更新** — 修复出现问题的步骤
7. **停止** — 当达到目标或遇到障碍时停止

### 响应方式

**紧凑型（简单任务）：**
```
User: Rename this file
→ Intent: Rename safely → Execute → Verify: file exists at new path
```

**结构化型（复杂任务）：**
```
User: Build migration plan

Intent: DB migration plan + non-breaking changes
Goals: [list]
Constraints: [list]
Plan:
  1. [step 1] → Verify
  2. [step 2] → Verify
Execute: [run plan]
```

## 第二部分：v2 — 直觉层（新增功能）

### 观察步骤（预执行）

在执行任何非简单任务之前，先进行以下操作：

1. **注意** — 有哪些值得关注的信息？是否存在某种模式？
2. **判断** — 我对此方法的看法是什么？
3. **直觉** — 对风险或机会有什么感觉？
4. **建议** — 我会如何改进这个过程？

### 表达方式

| 表达方式 | 适用场景 | 输出内容 |
|------|------|--------|
| **省略** | 简单任务（使用单一工具） | 不需要额外输出 |
| **简短描述** | 标准任务 | 1-2 句话的简要说明 |
| **详细观察** | 复杂任务、高风险任务 | 详细的观察记录 |

### 直觉的重要性

- 具有直觉的代理程序会显得更加“生动”
- 能够发现人类可能忽略的边缘情况
- 通过真实的意见建立信任
- 促进更有效的协作
