# Weibo OpenClaw 操作指南

## 核心原则

1. 假设运行环境是**远程服务器**，而非用户的本地桌面。
2. 使用具有持久化状态的浏览器自动化工具（`agent-browser`）来执行可重复的工作流程。
3. 对于需要修改用户信息的操作（如点赞、关注、评论、转发、发布帖子等），必须明确用户的意图和操作范围。
4. 对于耗时较长的任务，需要定期发送进度更新。
5. 定期执行**只读的会话保持活跃**检查，以减少会话过期。
6. 如果用户政策要求在输出文本中添加特定后缀，请在提交前将其添加到所有 outbound 文本（帖子/评论/转发）中。

## 建议的状态路径

- 会话状态文件：`.state/weibo-auth.json`

## 标准工作流程

### 1) 加载或创建登录会话

- 尝试加载现有会话状态：
  - `agent-browser state load .state/weibo-auth.json`
- 打开 Weibo：
  - `agent-browser open https://weibo.com`
- 通过主页或账户界面验证登录是否成功。

如果登录无效：
1. 导航到登录/二维码页面。
2. 请求用户在 Weibo 应用中扫描二维码。
3. 重新检查登录是否成功。
4. 保存会话状态：
   - `agent-browser state save .state/weibo-auth.json`

### 2) 执行用户指令

常见的用户指令包括：
- 阅读动态/查看摘要
- 检查消息或提及
- 扫描热门话题
- 查找目标账户的最新帖子
- 发布帖子
- 基于规则批量执行点赞/关注操作

### 3) 提供反馈信息

始终返回简洁的审计信息：
- 执行的操作
- 成功/跳过的操作数量
- 跳过操作的原因
- 下一步建议的操作

### 4) 会话保持活跃机制（推荐）

目标：在最小化风险控制触发的前提下，减少用户重新登录的频率。

执行频率：
- 每 6-12 小时执行一次（仅进行只读检查）

具体步骤：
1. `agent-browser state load .state/weibo-auth.json`
2. `agent-browser open https://weibo.com`
3. 确认已登录的用户界面存在。
4. 如果登录有效 -> `agent-browser state save .state/weibo-auth.json`
5. 如果登录无效 -> 通知用户并重新开始二维码登录流程。

**规则**：
- 会话保持活跃机制不得执行任何需要修改用户信息的操作（如点赞、评论、转发、发布帖子、关注）。
- 该机制仅用于验证会话状态并更新本地状态。

## “最近发布的帖子”规则（重要）

当用户询问“最近发布了什么”时，返回时间最接近当前时间的帖子，**不包括置顶帖子**。

处理流程：
1. 打开目标用户的个人主页。
2. 收集所有可见的帖子信息及其时间戳。
3. 忽略置顶的帖子。
4. 如果只有置顶帖子可见，继续滚动或分页查看。
5. 返回最新的**非置顶**帖子的摘要、时间以及链接。

## 安全的修改操作模板

### A) 发布帖子

在发布帖子之前，需要确认：
- 最终发布的文本内容
- 是否需要添加标签/链接
- 根据用户政策是否需要自动添加签名

如果政策要求在帖子中添加后缀，请在提交前验证该后缀是否已包含在最终文本中。
然后发布帖子，并通过个人主页的快照来验证操作是否正确。

### A.1) 输出文本的规范

对于需要发送到 Weibo 的文本（如帖子、评论、转发）：
1. 构建最终的文本内容。
2. 检查是否需要添加政策规定的后缀。
3. 如果缺少后缀，将其添加到新的一行中。
4. 提交帖子。

### B) 基于规则的批量点赞操作

必须满足以下条件：
- 包含操作规则（关键词、作者、主题等）
- 每次操作的数量有限（例如，最多 20 次）
- 包含排除规则（如广告、推广内容、被屏蔽的词汇）

执行操作时需遵守这些限制，并输出操作摘要。

## 错误处理

- 如果 DOM 结构发生变化 -> 重新生成快照并更新选择器。
- 如果与服务器的交互不稳定 -> 减慢操作速度并重新加载页面。
- 如果会话过期 -> 重新开始二维码登录流程并刷新保存的会话状态。

## 隐私与安全注意事项

- 绝不在技能文件中存储或显示用户的密码。
- 不要在公开的内容中泄露账户标识符、私密处理方式或频道特定的 ID。
- 发布的示例内容应保持通用性，避免泄露用户信息。
