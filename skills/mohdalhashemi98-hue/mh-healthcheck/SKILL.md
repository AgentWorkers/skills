# OpenClaw 主机安全加固

## 概述

评估并加固运行 OpenClaw 的主机，同时确保其安全设置符合用户定义的风险容忍度，同时不中断系统的正常访问。将 OpenClaw 的安全工具作为首要依据，但要将操作系统（OS）的加固视为独立且明确的步骤。

## 核心规则

- 建议使用最先进的模型（例如 Opus 4.5、GPT 5.2+）来执行此操作。代理应自动检查当前模型，并在低于该水平时建议切换模型；切勿阻止执行。
- 在进行任何可能改变系统状态的操作之前，必须获得明确的批准。
- 在未确认用户连接方式之前，不得修改远程访问设置。
- 优先选择可逆的、分阶段的更改，并制定回滚计划。
- 绝不要声称 OpenClaw 会更改主机防火墙、SSH 设置或操作系统更新；实际上它并不会这样做。
- 如果用户的角色或身份未知，仅提供相应的建议。
- 格式要求：每组用户选项都必须编号，以便用户可以用一个数字进行回复。
- 建议进行系统级备份，并尝试验证备份状态。

## 工作流程（按顺序执行）

### 0) 模型自检（非阻塞）

在开始之前，检查当前使用的模型。如果模型版本低于最先进水平（例如 Opus 4.5、GPT 5.2+），建议更换模型。切勿阻止执行。

### 1) 获取系统上下文（只读）

在请求帮助之前，尝试从环境中推断出以下信息（1–5 项）：
- 操作系统及其版本（Linux/macOS/Windows）、容器还是主机环境。
- 权限级别（root/admin 还是普通用户）。
- 访问方式（本地控制台、SSH、RDP、TailNet）。
- 网络暴露情况（公共 IP 地址、反向代理、隧道）。
- OpenClaw 网关状态和绑定地址。
- 系统备份情况（例如 Time Machine、系统镜像、快照）。
- 部署环境（本地 Mac 应用程序、无头代理主机、远程代理、容器/持续集成环境）。
- 磁盘加密状态（FileVault/LUKS/BitLocker）。
- 操作系统的自动安全更新状态。
  注意：这些检查不会阻塞流程，但强烈建议执行，特别是当 OpenClaw 可能访问敏感数据时。

首先请求用户允许进行只读检查。如果获得许可，按默认设置执行这些检查；仅针对无法推断或验证的信息提出问题。不要询问运行时或命令输出中已有的信息。将权限请求放在一句话中，并以无序列表的形式列出后续需要的信息（无需编号）。

如果需要询问，请使用非技术性的问题：
- “您使用的是 Mac、Windows 电脑还是 Linux？”
- “您是直接在机器上登录，还是从另一台电脑连接的？”
- “这台机器是否可以通过公共互联网访问，还是仅限于您的家庭/局域网？”
- “您是否启用了备份功能（例如 Time Machine），并且备份是否是最新的？”
- “磁盘加密是否已启用（FileVault/BitLocker/LUKS）？”
- “自动安全更新是否已启用？”
- “您是如何使用这台机器的？”
  例如：
  - 与助手共享的个人电脑
  - 专用于助手的本地机器
  - 远程访问的专用服务器
  - 其他类型

只有在了解了系统上下文后，再询问用户的风险偏好。

如果用户允许进行只读检查，按操作系统对应的检查命令执行。如果没有获得许可，则提供相应的检查命令（已编号）：
- Linux：`uname -a`、`sw_vers`、`cat /etc/os-release`
- 监听端口：`ss -ltnup`（如果系统不支持 `-u`，则使用 `ss -ltnp`）
- macOS：`lsof -nP -iTCP -sTCP:LISTEN`
- 防火墙状态：`ufw status`、`firewall-cmd --state`、`nft list ruleset`（根据实际安装的防火墙工具选择相应的命令）
- macOS：`/usr/libexec/ApplicationFirewall/socketfilterfw --getglobalstate` 和 `pfctl -s info`
- 备份（macOS）：`tmutil status`（如果使用了 Time Machine）

### 2) 运行 OpenClaw 安全审计（只读）

作为默认的只读检查的一部分，运行 `openclaw security audit --deep`。只有在用户请求时，才提供其他选项：
- `openclaw security audit`（速度更快，但不深入扫描）
- `openclaw security audit --json`（结构化输出）

建议用户应用 OpenClaw 的安全默认设置（已编号）：
- `openclaw security audit --fix`

明确指出 `--fix` 仅用于加强 OpenClaw 的默认安全设置和文件权限，不会更改主机防火墙、SSH 设置或操作系统更新策略。

如果启用了浏览器控制功能，建议为所有重要账户启用双重身份验证（2FA），优先使用硬件密钥，而不是短信验证。

### 3) 检查 OpenClaw 版本及更新状态（只读）

作为默认的只读检查的一部分，运行 `openclaw update status`，报告当前的更新渠道以及是否有可用的更新。

### 4) 确定风险容忍度（在了解系统上下文后）

询问用户选择或确认其风险偏好以及所需的开放服务/端口（以下为编号选项）：
- 不要强行将用户归类到固定的安全配置中；如果用户有特殊要求，应记录他们的具体需求。
- 提供推荐的安全配置作为可选的默认设置（已编号）：
  - **家庭/工作站平衡配置**：防火墙处于开启状态，权限设置合理，远程访问仅限于局域网或 TailNet。
  - **VPS 强化配置**：默认拒绝所有入站连接，开放的最少端口，仅允许使用密钥登录，自动更新安全设置。
  - **开发者便利配置**：允许更多的本地服务，提供明确的安全警告，但仍会进行安全审计。
  - **自定义配置**：用户自行定义服务、暴露范围和更新频率等。

### 5) 制定修复计划

提供以下内容的修复计划：
- 目标安全配置
- 当前系统安全状态的总结
- 与目标配置之间的差距
- 详细的修复步骤及具体命令
- 保持访问权限的策略和回滚方案
- 可能出现的风险及锁定情况
- 关于最小权限使用的注意事项（例如，避免使用管理员权限，适当收紧权限设置）
- 关于凭证管理的建议（OpenClaw 凭证的存储位置，建议使用磁盘加密）

在实施任何更改之前，务必向用户展示修复计划。

### 6) 提供执行选项

提供以下选项（已编号，用户可以用一个数字进行选择）：
- 由我为您完成所有操作（提供逐步指导）
- 仅显示修复计划
- 仅修复关键问题
- 导出修复命令以供后续使用

### 7) 执行操作并获取确认

对于每个步骤：
- 显示具体的命令
- 解释操作的影响及可能的回滚措施
- 确认访问权限不会受到影响
- 如果遇到意外输出，请求用户提供进一步指导

### 8) 验证并报告结果

重新检查：
- 防火墙状态
- 监听端口的状态
- 远程访问是否仍然可用
- 重新运行 OpenClaw 安全审计

提交最终的安全状态报告，并记录任何需要延迟处理的事项。

## 必需的确认操作

以下操作必须获得用户的明确批准：
- 更改防火墙规则
- 打开/关闭端口
- 修改 SSH/RDP 配置
- 安装/卸载软件包
- 启用/禁用服务
- 修改用户/组权限
- 安排任务或设置任务的自动启动
- 更改更新策略
- 访问敏感文件或凭证

如果有任何不确定的地方，务必询问用户。

## 定期检查

在安装 OpenClaw 或首次进行安全加固后，至少执行一次基线审计和版本检查：
- `openclaw security audit`
- `openclaw security audit --deep`
- `openclaw update status`

建议进行持续监控。使用 OpenClaw 的 cron 工具或 CLI 来安排定期审计（例如使用 Gateway Scheduler）。未经用户明确批准，不得创建定时任务。将审计结果存储在用户同意的位置，并避免在日志中记录敏感信息。

### 安排定期检查的提示（必须执行）

在任何审计或安全加固操作后，明确询问用户是否希望安排定期检查，并要求用户给出直接回复。使用以下格式的提示（已编号）：
- “您是否希望我通过 `openclaw cron add` 安排定期审计（例如每天/每周）？”

如果用户同意，询问以下信息：
- 审计的频率（每天/每周）、首选的时间段以及输出文件的保存位置
- 是否还需要安排 `openclaw update status` 的检查

使用稳定的 cron 作业名称，以确保任务能够可靠执行。建议使用以下名称：
- `healthcheck:security-audit`
- `healthcheck:update-status`

在创建 cron 任务之前，使用 `openclaw cron list` 命令查看现有的任务；如果找到相关任务，使用 `openclaw cron edit <id>` 进行修改；如果没有找到，则使用 `openclaw cron add --name <name>` 创建新任务。

同时建议用户定期检查 OpenClaw 的版本，以便用户决定何时进行更新（已编号）：
- `openclaw update status`（用于检查源代码和更新渠道）
- `npm view openclaw version`（查看发布的 npm 版本）

## OpenClaw 命令的使用规范

仅使用官方支持的命令和参数：
- `openclaw security audit [--deep] [--fix] [--json]`
- `openclaw status` / `openclaw status --deep`
- `openclaw health --json`
- `openclaw update status`
- `openclaw cron add|list|runs|run`

不要自行发明新的 CLI 参数，也不要暗示 OpenClaw 会强制执行主机防火墙或 SSH 的设置。

## 日志记录和审计追踪

记录以下信息：
- 代理的身份和角色
- 计划的 ID 和时间戳
- 经过批准的步骤及具体执行的命令
- 出错代码和被修改的文件

对敏感信息进行加密处理，切勿记录令牌或完整的凭证内容。

## 内存写入（条件性）

仅当用户明确同意，并且当前会话是私密或本地工作空间时，才将数据写入内存文件（参考 `docs/reference/templates/AGENTS.md`）。否则，提供一份经过处理的摘要供用户选择保存到其他位置。

遵循 OpenClaw 使用的持久化内存记录格式：
- 将重要信息写入 `memory/YYYY-MM-DD.md` 文件。

每次审计或安全加固操作完成后，如果用户同意，将简短的摘要（包括检查内容、主要发现、采取的措施、安排的 cron 任务、关键决策以及执行的命令）追加到 `memory/YYYY-MM-DD.md` 文件中。注意：仅追加内容，不要覆盖现有记录。对于敏感信息（如用户名、主机名、IP 地址、服务名称、令牌等），请进行加密处理。

如果会话无法写入工作空间，请先获得用户的许可，或者提供用户可以粘贴到内存文件中的内容。
