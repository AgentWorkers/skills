# 代码质量与评审

## 清晰代码的原则

| 原则 | 描述 |
|-----------|-------------|
| SRP（单一职责原则）| 每个函数/类只负责一个职责 |
| DRY（避免重复）| 不要重复自己编写代码 |
| KISS（保持简单）| 保持代码简单易懂 |
| YAGNI（你不会需要它）| 避免编写将来可能用不到的代码 |
| 命名规则 | 使用清晰、描述性的变量名/函数名 |

## 代码评审检查清单

### 提交前
- [ ] 代码能够无错误地编译
- [ ] 所有测试都通过
- [ ] 代码中不存在 `console.log` 或调试代码
- [ ] 类型使用正确
- [ ] 代码中不存在硬编码的敏感信息（如密码等）

### 评审要点
- [ ] 逻辑正确无误
- [ ] 错误处理得当
- [ ] 性能表现可接受
- [ ] 充分考虑了安全性
- [ ] 代码易于阅读

## 提交代码（Pull Request, PR）的标准

```markdown
## Description
Brief description of changes

## Type
- [ ] Bug fix
- [ ] Feature
- [ ] Refactor

## Testing
- [ ] Unit tests added
- [ ] Integration tests pass

## Screenshots (if UI)
```

## 检查清单
- [ ] 遵循命名规范
- [ ] 编写具有自文档功能的代码
- [ ] 保持函数规模适中（避免函数过长）
- [ ] 正确处理错误
- [ ] 对复杂的逻辑添加注释
- [ ] 提交前先自行审核你的代码
