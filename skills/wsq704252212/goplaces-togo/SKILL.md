# goplaces-togo

## 目标

通过使用 `goplaces` CLI 获取实时信息（评分、营业时间、评论），并结合用户的个人备注、饮食偏好和位置偏好以及访问历史记录，帮助用户从他们保存的 Google Places 列表中选择一个目的地进行参观，并给出明确的推荐。

## 先决条件

- `goplaces` 可执行文件已安装并添加到 PATH 环境变量中。
- `GOOGLE_PLACES_API_KEY` 环境变量已设置。
- 从 [https://takeout.google.com](https://takeout.google.com) 导出 **Google Takeout 保存的地标 CSV 文件**：
  1. 点击“取消选择所有”，然后仅选中“已保存”。
  2. 点击“下一步”→“导出一次”→“创建导出文件”。
  3. 下载 zip 文件，找到其中的 `Saved Places.csv` 文件。
  4. 将文件路径提供给机器人，它将自动导入并保存该文件。
- 本地数据文件 `skills/goplaces-togo/goplaces-visits.json` 用于保存导入的地标列表和访问历史记录（首次使用时自动生成）。

## 数据文件结构

`skills/goplaces-togo/goplaces-visits.json` 是所有持久化数据的唯一来源：

```json
{
  "savedList": [],  // 用户保存的地标列表
  "places": {
    // 地标 ID 为键，存储访问历史记录和最新评分信息
  }
}
```

### 步骤

### 1. 加载或请求保存的地标列表

读取 `skills/goplaces-togo/goplaces-visits.json`。检查 `savedList` 是否存在且至少包含一个条目。

- **如果存在保存的地标列表**，向用户显示列表并询问：
  > 这是您上次保存的地标列表：
    > 1. <地点名称> — <用户备注> 或 "(无备注)"
    > 2. ...
  >
  > 使用此列表，还是提供一个新的 CSV 文件来替换它？（按 Enter 键使用现有列表。）
  
- 如果用户按 Enter 键或回答“使用它”/“是”等类似肯定答复，则保持 `savedList` 不变并继续执行。
- 如果用户提供新的 CSV 文件路径或粘贴 CSV 内容，则用新解析的条目替换 `savedList`（见步骤 2），然后写入磁盘并继续执行。
- 如果用户回答“添加”或“更新”并提供了新的 CSV 文件，则合并新条目，保留不重复的现有条目（根据 `name`（不区分大小写）或 `mapsUrl` 进行匹配）。将合并后的列表写入磁盘。

- **如果不存在保存的地标列表**，则明确告知用户：
  > 我需要您的 Google 保存的地标列表。操作方法如下：
    >
    > 1. 访问 https://takeout.google.com
    > 2. 点击“取消选择所有”，然后向下滚动并仅选中“已保存”
    > 3. 点击“下一步”→“导出一次”→“创建导出文件”
    > 4. 下载 zip 文件，打开它，在 **Saved/** 文件夹中找到 CSV 文件（例如 `Saved Places.csv`）
    > 5. 分享文件路径或在此处粘贴其内容
  >
  等待用户提供 CSV 文件后再继续执行步骤 2。

### 2. 解析 CSV 文件并提取用户备注

从 Google Takeout 导出的 CSV 文件具有以下头部和格式：

对于每一行非空数据（跳过头部和空行）：
- `Title` → 地点名称
- `Note` → 用户备注（为空时使用 `null`）
- `URL` → `mapsUrl` — 保持原样以供参考；**不要** 试图从 URL 路径中提取地点 ID，因为 `data=!4m2!3m1!1s<hex>` 是 Google 的内部 CID 格式，而非 Places API ID
- `Tags`、`Comment` — 目前忽略
- 将 `addedAt` 设置为当前日期
- 将数据存储为 `{ name, mapsUrl, userComment, addedAt, placeId: null }`

解析完成后，立即将结果数组写入 `skills/goplaces-togo/goplaces-visits.json` 的 `savedList` 中，然后再进行任何 API 调用。这样可以确保即使会话提前结束，列表也不会丢失。

同时更新数据文件结构以包含 `mapsUrl`：

```json
{
  "savedList": [
    // ...
  ],
  "places": {
    // ...
  }
}
```

### 3. 询问用户的饮食偏好和位置偏好

明确询问用户：
- 今天您想吃哪种类型的美食？您有特定的社区或区域偏好吗？（按 Enter 键跳过）

等待用户的回答。解析两个可选值：
- `cuisinePreference` — 例如 "Japanese"（日式）、"Italian"（意大利式）、"anything spicy"（任何辛辣口味）、`null`（跳过）
- `locationPreference` — 例如 "Shibuya"（涩谷）、"within 2km of Shinjuku Station"（新宿站 2 公里范围内）、`null`（跳过）

如果用户跳过了这两个选项，则不进行偏好筛选。

### 4. 将每个地点名称解析为地点 ID

对于每个没有 `placeId` 的条目，执行以下操作：

解析 JSON 数据，获取 `candidates[0].place_id`。如果结果为空，则将该条目标记为无法解析并跳过（在最后报告）。

### 5. 获取每个地点的详细信息

收集每个地点的以下信息：
- `displayName.text` — 人类可读的地点名称
- `currentOpeningHours.openNow` — 该地点现在是否营业？
- `rating` — 总评分（0–5 分）
- `userRatingCount` — 评论数量
- `priceLevel` — 0（免费）到 4（非常昂贵）
- `primaryType` 或 `types[]` — 食物类别标签
- `location` — `{ latitude, longitude }` 用于计算距离
- `reviews[0].text.text` — 最佳评论的前 150 个字符
- `editorialSummary.text` — 如果有的话，显示一条简短的评论摘要

### 6. 加载访问历史记录

在步骤 1 中已经读取了 `skills/goplaces-togo/goplaces-visits.json` 中的 `places` 部分。对于工作列表中的每个地点，查找其 `place_id`，并获取以下信息：
- `visitCount` — `visits` 数组的长度（如果键不存在则为 0）
- `daysSinceLastVisit` — 从今天到 `visits[last].date` 之间的天数（如果从未访问过则为 `null`）

### 7. 计算评分和排名

为每个可解析的地点计算评分。所有加分项都是累加的。

根据评分降序排列地点。将无法解析的条目排除在排名之外，但仍然在列表中显示。

### 8. 推荐一个地点

使用以下格式呈现排名最高的地点作为推荐：

---

**推荐：<地点名称>**

- 现在营业：是 / 否
- 评分：X.X / 5（N 条评论）
- 价格：$ / $$ / $$$ / $$$$（如果不可用则省略）
- 您的备注："<用户备注>"（如果为空则省略）
- 访问次数：N 次（最后一次访问：YYYY-MM-DD）/ 从未访问
- 推荐理由：<结合评论摘要、最佳评论、用户备注和用户偏好的一句话或两句话>`

---

然后以排名表格的形式列出其余可解析的地点：

| 排名 | 地点 | 评分 | 是否营业 | 访问次数 | 评分 |
|------|-------|--------|------|--------|-------|
| 2 | ... | | | | |

如果只推荐了一个地点，仍需确认该地点是否合适（或者如果该地点已关闭、评分很低或最近访问过，请进行提示）。

### 9. 确认选择并记录访问

显示推荐后，询问用户：
> 您打算去 <地点名称> 吗？回答“是”以记录这次访问，或者告诉我您选择了列表中的哪个地点。

等待用户的回答。

- 如果用户确认了地点（通过回答“是”或说出地点名称），则记录访问信息：
  1. 读取 `skills/goplaces-togo/goplaces-visits.json`（已加载；使用内存中的副本）。
  2. 确保 `places["<place_id>"]` 存在；如果不存在，则创建该条目，内容为 `{ name, visits: [] }`。
  3. 添加新的访问记录：
    ```json
    {
      "name": "<地点名称>",
      "visits": [newVisitDate]
    }
    ```
    使用当前日期和本地时间（24 小时格式）进行记录。不要修改 `savedList`。
  4. 将更新后的完整对象（`savedList` 和 `places`）写回 `skills/goplaces-togo/goplaces-visits.json`。
  5. 确认：“已记录您在 <日期> <时间> 的访问记录。祝您愉快！”

- 如果用户回答“不”或跳过，则说“没关系——祝您一天愉快！”

### 10. 处理特殊情况

- **没有解析出的地点**：告诉用户没有匹配到的地点，并要求他们重新核对名称或粘贴 Google 地图 URL。
- **所有地点都关闭**：说明所有选项目前都处于关闭状态，但仍按评分进行排名，并在推荐中加以说明。
- **位置解析失败**：显示错误信息，并要求用户验证 `GOOGLE_PLACES_API_KEY`。
- **历史文件损坏**：警告用户，将 `savedList` 和 `places` 视为为空，直到用户提供新的列表或确认访问记录。
- **用户希望清空列表**：如果用户说“清除我的列表”或“忘记我的地点”，将 `savedList` 设置为 `[]`（保持 `places` 中的历史记录不变），并确认：“您的保存列表已清除。准备好后请重新添加列表。”
- **用户想要删除某个条目**：如果用户说“删除 <地点名称>”，则根据名称从 `savedList` 中删除相应的条目（不区分大小写），然后写入磁盘并确认删除。保留该地点 ID 的访问历史记录。

## 检测用户行为

这些语句可以在任何时候说出——不仅仅是在推荐过程中。检测用户的意图并立即采取行动，无需用户处于特定步骤。

### 回溯性访问记录

| 用户的回答 | 应采取的操作 |
|-----------|--------|
| “我昨晚去了 Nobu” | 在 `savedList` 中查找“Nobu”（或运行 `goplaces resolve`）。询问：“明白了——您是什么时候去的？（或按 Enter 键跳过）。”使用昨天的日期和提供的时间记录访问记录（或使用 `null`）。|
| “我们最后去了那家拉面店” | 确定地点（如果名称不明确则进行澄清）。询问时间，然后使用周六的日期记录访问记录。|
| “刚从 Trattoria Roma 回来” | 使用今天的日期和当前时间记录访问记录。|
| “这周去了 3 个地方：X、Y、Z” | 依次记录这三个地点。对于每个地点，询问“X 是在什么时间和什么地点去的？”然后进行记录。|

### 访问后的反馈

| 用户的回答 | 应采取的操作 |
|-----------|--------|
| “太棒了” / “很喜欢” | 找到最近记录的地点（今天或上次访问的记录），并将该访问记录的 `note` 设置为正面评价。更新 `savedList` 中的 `userComment` 以反映正面评价。|
| “一般般” / “没什么特别的” | 同上，但记录中保持中立评价。|
| “令人失望，不会再去了” | 在该访问记录中记录负面评价。更新 `savedList` 中的 `userComment` 为“令人失望——避免再去”。这将在未来的评分中扣分（-1.5 分）。|
| “午餐不错，但晚餐时太吵了” | 作为条件性评论记录。更新 `userComment` 为“午餐不错，但晚餐时太吵了”。未来的评分会考虑时间因素。|
| “Omakase 很棒，适合在工作日去” | 更新 `savedList` 中的 `userComment` 为实际建议。确认：“已更新您对 <地点名称> 的评价。” |

### 列表管理

| 用户的回答 | 应采取的操作 |
|-----------|--------|
| “把 Sukiyabashi Jiro 加到我的列表里” | 将 `{ name: "Sukiyabashi Jiro", mapsUrl: null, placeId: null, userComment: null, addedAt: today }` 添加到 `savedList` 中。确认：“已将 Sukiyabashi Jiro 添加到您的列表中。”|
| “添加 Pizza Pilgrims — 适合团队用餐” | 从用户的话中解析名称和备注，然后添加到 `savedList` 中：`userComment: "适合团队用餐"`，`mapsUrl: null`。|
| “这是我的更新后的 Takeout CSV 文件：<路径>" | 重新解析 CSV 文件并合并到 `savedList` 中：添加新条目，更新现有名称的 `userComment`，保留访问历史记录。|
| “从我的列表中删除 Shake Shack” | 从 `savedList` 中删除相应的条目（不区分大小写）。确认删除。|
| “显示我的列表” | 打印 `savedList` 中的所有条目：索引、地点名称、用户备注和访问次数。|
| “我去过 Nobu 多少次？” | 在 `places` 中查找该地点的访问次数。回答：“您去过 Nobu 4 次。最后一次访问：2025-11-03。”|
| “我对 Trattoria Roma 有什么印象？” | 查找该条目的 `userComment` 和 `note` 字段，并总结评价。|

### 偏好设置快捷方式

| 用户的回答 | 应采取的操作 |
|-----------|--------|
| “给我一个惊喜” | 完全跳过步骤 3（不设置偏好），进行完整评分并推荐排名最高的地点。|
| “在我附近的地方” | 询问“您当前的社区或地标是什么？”然后使用该信息作为 `locationPreference`。|
| “我想吃日式料理，随便什么地方都行” | 将 `cuisinePreference` 设置为 “Japanese”，`locationPreference` 设置为 `null`，直接进行评分。|
| “和上次一样” | 如果之前有保存的偏好设置，则重新使用；否则再次询问。将上次使用的偏好存储在 `skills/goplaces-togo/goplaces-visits.json` 的 `"lastPreferences": { cuisine, location }` 中。|
