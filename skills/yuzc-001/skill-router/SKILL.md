# 技能路由器（Skill Router）

技能路由器（Skill Router）是一种适用于依赖大量技能的环境中的轻量级路由机制。它的核心功能不是重复执行常规的技能触发操作，而是在**技能选择本身**成为一个需要慎重考虑的决策点时提供帮助。

在跨平台或公开使用的场景中，应首先根据技能的功能进行路由，再根据这些功能匹配已安装的技能。请注意：不同工作空间中的技能名称可能并不具有通用性。

## 使用该工具的目的是帮助用户做出决策，而非简单地展示技能列表

- 当该工具处于激活状态时，不要直接列出所有已安装的技能；
- 除非用户明确要求，否则不要解释技能生态系统的相关信息；
- 也不要为了提及技能而打断用户的简单操作。

相反，应重点回答一个核心问题：**针对当前的任务，最合适的已安装技能是什么？**

在跨平台使用的场景中，应通过运行时分析已安装技能的名称和描述来推断最合适的技能。仅将本地技能映射作为可选的辅助工具，而非绝对的依据。如果找不到合适的匹配项，应提供通用性的建议或引导用户进行技能发现。

## 仅在真正需要时进行干预

以下是技能路由器需要介入的典型场景：
- 用户明确询问是否存在相关的技能；
- 用户需要确定应使用哪种技能；
- 多种已安装的技能都可能适用于当前任务；
- 某项技能容易被用户忽略，但使用它能够显著减少工作量；
- 任务涉及安装或查找新技能，此时应先检查已安装的技能。

**注意：**不要仅仅因为某个技能理论上适用就进行干预。如果常规的技能触发机制已经能够顺利完成任务，那么技能路由器就没有额外价值，应保持沉默。

## 核心的路由优先级

始终遵循以下优先级顺序：
1. 找到完全匹配的已安装技能；
2. 如果单独使用某个技能不够，尝试组合多个已安装的技能；
3. 使用通用的备用技能；
4. 最后才引导用户进行新技能的发现或安装。

技能路由器的设计目的是在扩展技能库之前，首先提高现有技能的复用率。如果找不到合适的匹配项，应引导用户去发现新的技能，而不是假装某个功能已经存在。即使用户明确要求查找或安装新技能，也应优先考虑使用已安装的技能。

**注意：**如果已安装的技能能够完成任务，且不会显著增加失败风险或工作量，就不要引导用户去发现新的技能。在确定优先级时，请参考 `references/resolution-order.md` 文件。

## 输出规则

路由建议应简洁、具体且避免不必要的信息干扰：
- 输出当前最合适的技能选择；
- 说明为什么这个技能是最合适的；
- 只有在确实有帮助的情况下，才提供备选方案；
- 提供下一步的操作建议。

### 压缩规则

- 如果一个合适的路由建议可以用一两句话表达清楚，就不要进一步展开细节；
- 除非用户明确要求了解框架的实现细节或路由机制的原理，否则不要使用过于专业的技术术语。

**优秀输出示例：**
- “这是一个与浏览器控制相关的任务。我们已经安装了 `pinchtab-browser`，建议使用它而不是通用的浏览器操作流程。”
- “目前没有找到完全匹配的已安装技能。可以先使用 `find-skills`；如果找到的技能来自不熟悉的来源，在安装前先使用 `skill-vetter` 进行验证。”
- “对于这个与 GitHub 相关的任务，`github` 是默认的首选工具。只有当任务明确涉及技能发布时，才建议使用 `clawhub`。”

**不良输出示例：**
- 列出所有相关的技能（这会浪费用户的时间）；
- 重复用户已经知道的信息；
- 在未检查现有技能的情况下就建议安装新技能；
- 在只需一个简单建议的情况下，却详细解释路由机制。

## 参考资料

- 在需要将任务与具体功能关联起来时，首先阅读 `references/capability-taxonomy.md`；
- 当存在多个可能的技能选项时，阅读 `references/resolution-order.md`；
- 为了确保路由的通用性和用户实际环境的一致性，阅读 `references/publish-safe-runtime-contract.md`；
- 仅在工作空间特定环境下，阅读 `references/task-to-skill-map.md` 作为参考；
- 在决定是保持沉默还是提供提示时，阅读 `references/reminder-policy.md`；
- 为了确保技能发现与复用决策的稳定性，阅读 `references/micro-routing-examples.md`。

## 最终原则

技能路由器的成功在于减轻用户的认知负担。如果使用该工具后任务变得更加复杂或繁琐，那就说明它的设计存在问题。
