# 网页内容获取工具

本工具用于获取可读的网页内容，并采用可靠的备用获取方式作为首选。

## 核心原则

**不要**承诺能够直接访问所有网站**。部分网站会使用 Cloudflare、登录验证、机器人检测或法律限制等机制。在这些情况下，应切换到其他备用方式，而不要强制使用最初尝试的方法。

## 首选获取顺序

### 1) 直接获取可读内容

首先尝试使用轻量级的转换服务：

1. **r.jina.ai**
   ```
   https://r.jina.ai/http://example.com
   ```

2. **markdown.new**
   ```
   https://markdown.new/https://example.com
   ```

3. **defuddle**
   ```
   https://defuddle.md/https://example.com
   ```

对于需要多次重试的情况，可以使用以下脚本：

```bash
python {baseDir}/scripts/fetch_url.py "https://example.com/article"
```

该脚本会返回以下信息：
- 选择的获取方式
- 重试历史记录
- 内容被阻止或内容过少的情况
- 成功获取的最终内容

当用户需要文章文本、页面摘要或从普通公开页面中提取结构化数据时，可以使用这些方法。

### 2) 及时检测失败情况

如果出现以下提示，应认为获取失败或不可靠：

- “稍等……”
- “正在进行安全验证”
- “需要启用 JavaScript 和 Cookie”
- 出现验证码或挑战页面
- 显示登录验证界面而非目标内容
- 文章内容明显被截断或缺失

在这种情况下，**不要将返回的结果视为页面的完整内容**。

### 3) 对于受保护的网站（需要浏览器访问的）

对于被 Cloudflare 阻止或需要通过浏览器访问的网站：

- 如果可以使用 OpenClaw 浏览器工具，优先选择通过浏览器访问。
- 如果用户使用了 Chrome 中继/扩展程序，请让用户将相关标签页添加进来，然后检查实际渲染的页面内容。
- 对页面进行快照处理，并仅提取所需的信息。

适用于以下情况：
- 含有大量 JavaScript 代码的页面
- 被 Cloudflare 保护的页面
- 需要在加载后才能显示关键内容的页面
- 直接使用 markdown 服务时返回验证页面的网站

### 4) 搜索/索引备用方式

如果直接获取或浏览器获取均不可用或仍然失败：

- 搜索目标页面、期刊或文章的标题
- 使用搜索结果、出版商的镜像页面、缓存摘要或其他备用来源
- 如果搜索能够找到所需信息，优先使用官方出版商的页面
- 如果数据不是直接从目标页面获取的，应明确标注其为备用来源

这些方法通常足以获取元数据信息，例如：
- 主编姓名
- 期刊的影响因子
- 出版频率
- ISSN
- 机构隶属关系

### 5) 部分完成模式

如果某个网站的获取结果不稳定，应返回混合结果而非停滞不前：

- 填充可以直接验证的信息
- 明确标记被阻止或未解决的信息
- 说明失败的原因以及使用了哪种备用方式

## 实用提取策略

### 对于单个页面

1. 首先尝试使用 `r.jina.ai`
2. 如果被阻止，尝试使用 `markdown.new`
3. 如果仍然被阻止，尝试使用 `defuddle`
4. 如果还是无法获取，使用浏览器工具
5. 如果浏览器不可用，使用搜索/索引备用方式
6. 提供结果的可靠性等级

### 对于多个相似的页面

1. 首先获取索引/列表页面
2. 提取所有目标页面的 URL 或代码
3. 分批处理这些页面
4. 记录每个页面的获取结果（成功/失败）
5. 仅对那些使用更可靠备用方式仍无法获取的页面进行重试
6. 最终提供尽可能完整的数据表

## 输出指南

在提取结构化数据时，建议使用以下字段：

- 来源 URL
- 提取方式（`direct`、`browser`、`search`、`secondary`）
- 可靠性等级（`high`、`medium`、`low`）
- 对于被阻止或未验证的页面的备注

## 示例

- 用户：“阅读这篇文章” → 首先尝试直接获取
- 用户：“这个页面内容是什么？” → 先尝试直接获取，如果被阻止则使用浏览器备用方式
- 用户：“爬取这个期刊网站” → 先获取索引页面，然后使用备用方式批量提取内容
- 用户：“Cloudflare 阻止了访问” → 切换到浏览器或搜索备用方式，避免重复使用相同的失败方法
