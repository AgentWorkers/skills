# 幽灵依赖（Phantom Dependencies）

> “最危险的依赖，往往是那些曾经存在过的依赖。”

## 功能介绍

幽灵依赖（Phantom Dependencies）能够检测到**“幽灵引用”**——即那些试图访问已不存在资源的代码。这些引用不会导致程序崩溃（因为代码检查工具（linter）通常能够捕获这类错误），但它们会隐藏潜在的问题。例如：未被设置的环境变量、在三个开发周期前就被重命名的配置键、虽然已被弃用但仍未从客户端代码中移除的API接口、或者指向仅存在于原始开发者机器上的文件路径。

所有代码库中都存在这类“幽灵引用”。它们本身不会引发错误，但会导致各种问题：比如某个功能在开发环境中可以正常运行，但在生产环境中却出问题；新员工入职需要两周时间才能熟悉系统，而不是原本的两天。

## 为何需要这种工具

静态分析工具可以检测代码中的**错误**，代码检查工具可以发现代码中的**不规范之处**，而幽灵依赖工具则能发现代码与实际环境之间的**不匹配之处**。

| 传统工具能检测到的问题 | 幽灵依赖能检测到的问题 |
|-------------------|-------------------|
| 无效的导入语句       | 仍然引用了已删除代码的导入语句       |
| 语法错误         | 引用了已删除概念的代码             |
| 未使用的变量       | 仍然引用了不存在状态的变量           |
| 不存在的文件       | 文件存在，但其内容基于旧架构设计         |
| 类型不匹配       | 类型看似匹配，但实际上引用的内容已不存在     |

## 六种类型的幽灵依赖

### 1. **环境幽灵依赖（Environmental Phantoms）**
这些引用指向的环境变量、配置文件或系统状态，实际上从未被任何程序设置过。

**检测方法：** 将 `process.env`、`os.environ`、`ENV[]` 中的引用与实际的 `.env` 文件、CI/CD 配置文件以及部署配置进行交叉比对。

### 2. **引用幽灵依赖（Referential Phantoms）**
代码引用了被移动、重命名或删除的函数、类或模块，但由于存在替代方案（如 shim、重新导出的函数等），这些引用仍然能够“正常工作”。

**检测方法：** 追踪每个导入语句的来源，标记那些引用链超过两层的引用，并标记路径中包含 “legacy”（旧版本）、”compat”（兼容性）、”old”（旧版）或 “deprecated”（已弃用）等标记的引用。

### 3. **时间依赖幽灵依赖（Temporal Phantoms）**
代码依赖于某些在旧架构下有效的时序、顺序或执行流程，但在新架构中已不再成立。

**检测方法：** 分析所有隐含的时序假设，标记那些在没有显式等待或防护机制的情况下进行的操作。

### 4. **契约依赖幽灵依赖（Contractual Phantoms）**
代码依赖于某些API接口、数据库模式或数据传输格式，但对方不再遵守这些约定。

**检测方法：** 将代码生成的输出与最新的API规范或数据库模式进行比对，标记那些生成后从未被使用的字段。

### 5. **意图幽灵依赖（Intentional Phantoms）**
注释、待办事项（TODO）或文档中描述的代码行为，与实际代码行为不一致。这些描述已经过时。

**检测方法：** 解析文档注释，并将文档中提到的功能与实际代码实现进行对比，标记那些在代码中并未实现的特性（如重试、缓存、队列处理等）。

### 6. **名称幽灵依赖（Identity Phantoms）**
变量、函数或模块的名称与其实际功能不符。

**检测方法：** 通过语义分析来检查变量名与其实际功能之间的矛盾（例如，名称为 “temp” 但代码中没有任何过期处理逻辑，或名称为 “async” 但代码却是同步执行的）。

### 严重性评分

| 严重性 | 描述 | 示例                |
|-------|-----------------|-------------------|
| **严重** | 幽灵依赖导致数据丢失或损坏       | API字段被忽略，数据未被保存           |
| **较高** | 幽灵依赖导致间歇性故障       | 由于时序问题导致的错误             |
| **中等** | 幽灵依赖引起混淆，但不会引发运行时错误 | 名称误导导致的误解             |
| **较低** | 幽灵依赖虽然无害，但会增加开发负担   | 无效的代码引用或过时的配置文件         |
| **遗留** | 幽灵依赖无害，但表明架构存在问题   | 两年前留下的待办事项注释           |

### 集成方式

- 在新员工入职时使用，帮助他们了解系统中存在的“幽灵依赖”。
- 在进行重大重构后使用，找出重构过程中遗留的问题。
- 在生产环境部署前使用，提前发现潜在问题。
- 在架构审查过程中使用，找出设计意图与实际实现之间的差异。

## 重要性

所有代码库都存在一种“幽灵架构”——即系统自认为的架构，它建立在实际运行的架构之上。这种差异是隐藏漏洞、导致新员工入职困难以及技术债务逐渐累积的根源。

幽灵依赖工具本身不会直接发现错误，而是揭示那些使得错误不可避免的条件。

该工具完全不依赖外部依赖或API调用，仅通过结构分析来检测问题。
