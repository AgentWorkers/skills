# Ariadne 线性文档系统

该系统旨在帮助 AI 代理高效地浏览代码库，仅展示在需要时才需要的信息。

## 适用场景

当用户遇到以下情况时，应使用此方法：
- 新项目启动，需要为 AI 设计友好的结构；
- 重构或索引现有代码库；
- 添加项目导航结构（如 `AGENTS.md`、`INDEX.md`、`llms.txt` 文件）；
- 设计模块边界和依赖关系。

**语言无关性**：该方法适用于任何编程语言。具体语言的实现细节请参考 [language-adaptation.md](references/language-adaptation.md)。

**不适用场景**：
- 非常小的单文件项目；
- 快速原型开发；
- 索引开销大于收益的代码库。如果项目源文件数量少于 5 个，可以考虑使用简化的 `AGENTS.md` 文件。

## 核心理念

就像阿丽亚德涅的线一样，一个索引完善的代码库能为 AI 代理提供从项目概览到实现细节的清晰路径。**代理应能够理解项目结构，定位任何函数，并在无需阅读整个代码库的情况下评估修改的影响。**

**两个优化目标**：
1. **导航效率**：最小化查找目标代码所需的资源消耗；
2. **修改安全性**：通过明确的契约和依赖关系可见性，提高修改的正确性。

## 四层索引架构

AI 代理从第 0 层（L0）开始，根据需要逐步深入。

### L0：项目根目录索引

在项目根目录创建一个 `AGENTS.md` 文件，内容包括：
- 项目描述（一行）；
- 目录结构图（ASCII 树形结构，附带功能说明）；
- 快速导航表（按功能分类的入口点）；
- 构建和测试命令；
- 代码规范总结；
- 架构约束（不允许的依赖关系方向、全局不变量）；
- 跨模块通用行为模式（错误处理、日志记录、并发模型）；
- 代理工作流程（文件末尾的部分，采用清单格式）：编辑前必须执行的步骤——通过快速导航定位模块，打开 `INDEX.md` 查看依赖关系，确认修改公共 API 时的风险，完成修改前运行测试；
- 索引排除项（可选的 `.cursorignore` 文件，语法与 `.gitignore` 相同），用于排除构建产物、依赖文件和生成的代码。常见示例：`node_modules/`、`dist/`、`vendor/`、`__pycache__/`、`target/`、`*.generated.*`——根据实际情况进行调整。

**跨模块通用行为模式** 非常重要：它防止 AI 为了理解“如何处理错误”而阅读大量不必要的文件。一条 30 行的声明可以替代大约 1500 行的推理代码。

**所有 9 个部分都是必需的**——它们决定了 `AGENTS.md` 文件是真正指导 AI 行为的文档，还是仅仅用于人类阅读的文档。最常被忽略的部分（架构约束、跨模块通用行为模式、代理工作流程清单）对 AI 来说价值最高。

**导航优先级**：建议优先通过 `INDEX.md` 和依赖关系进行导航，而不是使用语义搜索或关键词搜索。结构化的依赖关系路径能揭示出常规搜索可能遗漏的关键文件。修改模块的公共 API 时，务必先查看其依赖关系列表以评估修改的影响范围。

**信息布局**：大型语言模型（LLMs）更关注文档的开始和结束部分。将高优先级的内容（构建信息、快速导航、代理工作流程）放在 `AGENTS.md` 的开头或结尾；次要内容则放在中间部分。

**契约注释说明**：
| 注释类型 | 用途 | 使用场景 |
|---------|---------|-------------|
| `@pre` | 调用前必须满足的条件 | 输入验证 |
| `@post` | 返回值之后的保证 | 返回值约束 |
| `@throws` | 可能抛出的错误 | 所有错误条件 |
| `@sideeffect` | 返回值之外会改变的状态 | 数据库写入、事件、缓存失效 |
| `@threadsafe` | 并发安全性 | 多线程系统 |
| `@performance` | 延迟/吞吐量约束 | 高频执行的函数 |
| `@lines` / `@range` | 大文件的代码行范围 | 例如：`@lines 45-89 main logic` |

**路径约定**：在 L1 层，使用相对于项目根目录的路径（如 `src/auth/authenticate.ts`），以避免使用 `./`、`../` 以确保路径的清晰性和与语音编码系统的兼容性。

完整的 `INDEX.md` 模板请参考 [index-templates.md](references/index-templates.md)。

## 模块设计原则

### 单一职责原则

- 每个模块只有一个明确的功能（在 `INDEX.md` 中说明）；
- 每个文件只包含一个概念，代码行数不超过 300 行（对于语法较复杂的语言，不超过 500 行）；
- 如果无法清晰地命名模块，应将其拆分。

### 明确的依赖关系方向

典型的层结构（示例；可根据项目实际情况调整名称，例如 `app/`、`packages/`、`layers/`）：

- 依赖关系应从内部指向外部（从展示层到业务逻辑层，再到基础设施层），绝不能反向；
- 在 `AGENTS.md` 的架构约束部分记录实际的依赖方向。

### 小而专注的文件

| 规范 | 目标 | 原因 |
|--------|--------|--------|
| 每文件的代码行数 | < 300 行（语法复杂的语言 < 500 行） | AI 会读取整个文件；文件越小，浪费的上下文信息越少 |
| 每文件的公共符号数量 | 1-5 个 | 使公共接口更清晰 |
| 每文件的依赖关系数量 | < 5-8 个导入/包含项 | 限制修改的影响范围 |

### 库与应用程序项目

该索引结构适用于两者，但库在某些方面有所不同：

| 方面 | 应用程序 | 库 |
|--------|-------------|---------|
| **构建与测试** | 包含构建和运行命令 | 可能没有构建步骤（只有头文件）；测试通常在单独的项目中进行 |
| **公共 API** | 模块之间相互调用 | 消费者是外部的；公共 API 是导出的头文件或包接口 |
| **依赖关系（依赖方向）** | 同项目内的其他模块 | 主要为外部消费者（在依赖关系中标记为“外部”） |
| **文件大小目标** | 有严格要求（你可以控制代码） | 仅适用于你无法控制的上下游/第三方代码 |

**仅包含头文件的库**（C++）：不需要 `include/` 和 `src/` 文件夹——所有代码都放在头文件中。入口头文件（如 `library.hpp`）充当模块的入口文件。没有独立的构建步骤；需要记录消费者如何包含和链接这些头文件。

**对于你无法维护的库（例如第三方库）**：在索引中创建 L0 和 L1 层的索引以方便导航，但不要修改源文件（不要修改头文件中的内容，也不要重构文件）。在索引中用 `⚠ UPSTREAM` 标记问题文件。

## 以意图为导向的代码编写

编写代码时，应明确说明代码**实现了什么功能**，而不是**实现方式**。

### 命名原则（通用规则）

| 层级 | 原则 | C++ 示例 | TypeScript 示例 | Python 示例 |
|-------|-----------|-------------|-------------------|----------------|
| 文件 | 使用描述性名称 | `invoice_calculator.cpp` | `calculate-invoice.ts` | `invoice_calculator.py` |
| 目录 | 按功能分类 | `billing/`, `auth/` | `billing/`, `auth/` | `billing/`, `auth/` |
| 函数 | 动词在前，结果在后 | `calculate_total()` | `calculateTotal()` | `calculate_total()` |
| 类 | 使用 PascalCase 命名 | `InvoiceCalculator` | `InvoiceCalculator` | `InvoiceCalculator` |
| 常量 | 使用大写驼峰式命名 | `MAX_RETRY_COUNT` | `MAX_RETRY_COUNT` | `MAX_RETRY_COUNT` |
| 布尔值 | 使用明确的语义 | `isAuthenticated` | `isAuthenticated` | `isAuthenticated` |

### API 设计

适用于**任何外部接口**（REST、gRPC、库头文件、命令行接口）：
- 使用资源/概念来命名 API；
- 保持输入/输出契约的一致性；
- 为 API 分配版本号（URL 路径、头文件名或命名空间）；
- 明确记录错误条件。

详细的设计规范请参考 [naming-api-conventions.md](references/naming-api-conventions.md)。

## 文档分层

并非所有文档都面向相同的受众，也不需要相同的更新频率。本文档将文档分为两个层级：

| 层级 | 文件类型 | 面向的受众 | 更新频率 | 过时对系统的影响 |
|------|-------|----------|---------------|-----------------|
| **A 层：AI 运行时索引** | `AGENTS.md`、`INDEX.md`、文件头部（L2）、内联注释（L3） | AI 代理 | **实时更新**——与代码变更同步 | 如果过时，会导致导航错误、依赖关系遗漏或契约失效 |
| **B 层：人类交流文档** | `README.md`、`architecture.md`、API 文档、变更日志（ADRs） | 人类开发者 | **批量更新**——在迭代结束时更新 | 过时可能会导致人类困惑，但不会影响 AI 的使用 |

**关键要点**：A 层索引是 AI 的“工作记忆”——必须与代码变更同时更新。B 层文档是从 A 层派生的“报告”，可以在迭代结束时重新生成。

**在开发过程中**：AI 代理仅维护 A 层文档。B 层文档对 AI 是透明的——在迭代结束前不会被读取或修改。

**资源优化**：这种分层方式可以将每次代码变更导致的 AI 处理开销降低约 60%，同时还能提高 B 层文档的质量（批量更新比增量更新更一致）。

## 项目文档（B 层）

面向人类的文档遵循相同的渐进式披露结构，但在迭代结束时更新，而不是实时更新。常见结构如下（可根据项目实际情况调整；ADR 可选——如果不需要使用 ADR，可以省略 `adr/` 文件）：

**规则**：
1. **每个文档只关注一个主题**；
2. 每个文档目录中都包含 `INDEX.md`；
3. 使用链接引用 A 层索引（`INDEX.md`）作为权威信息来源；
4. 每个文档的行数不超过 500 行；
5. 在迭代结束时更新。

文档模板和 ADR 格式的详细信息请参考 [doc-standards.md](references/doc-standards.md)。

## 语言适配

四层索引和设计原则是通用的。针对不同语言（C++、TypeScript、Python、Go、Rust、Java/Kotlin）的特定约定（目录结构、文件命名、L2 层的契约注释、构建命令等）请参考 [language-adaptation.md](references/language-adaptation.md)。

## 工作流程：新项目

1. **确定项目使用的语言和生态系统**（参考 [language-adaptation.md]）；
2. 在编写代码之前设计目录结构；
3. 创建 `AGENTS.md`——包含项目目录结构、跨模块通用行为模式和 AI 代理使用说明；
4. 在创建每个模块目录时生成对应的 `INDEX.md` 文件（包含所有相关内容）；
5. 为文件添加意图头部信息——内部文件使用简化的头部信息，公共 API 文件添加完整的契约注释；
6. 从项目开始就遵循命名规范；
7. 延迟更新 B 层文档——先创建 `docs/` 目录结构，待第一次迭代结束时再填充内容。

## 工作流程：重构现有项目

**原则**：先创建文档，再重构代码。根据现有代码库生成索引。在索引中标记问题文件（如文件过大或存在违规情况），但不要因此推迟重构工作。

**单仓库/子项目**：如果目标是一个大型仓库中的子目录，将 `AGENTS.md` 文件放在子目录的根目录（例如 `lib/AGENTS.md`）。构建命令应仅针对该子目录生效（或注明使用的父目录命令）。具体操作请参考 [monorepo--sub-project-variant](references/index-templates.md#monorepo--sub-project-variant)。

**优先级**：从依赖关系最复杂的模块开始重构——这样可以获得最大的导航效率提升。

**部分迁移**：如果只迁移单个模块，直接为该模块生成 L1 层的索引（`INDEX.md`）。跳过 L0 层；假设父目录的 `AGENTS.md` 会随后生成或已经存在。

**库项目**：如果目标是一个库（而非应用程序），请参考 [Library vs Application Projects](#library-vs-application-projects) 了解构建与测试、公共 API 和依赖关系的差异。对于仅包含头文件的 C++ 库，入口头文件即视为模块的入口文件；在 `AGENTS.md` 中记录这些文件，并将主头文件目录视为一个独立的模块。

**对于你无法控制的代码（例如第三方库）**：
- 为库创建 L0 和 L1 层的索引以方便导航，但不要修改源文件头部或内联注释；
- 对于文件过大或结构复杂的文件，用 `⚠ UPSTREAM` 标记问题文件。

1. **创建 L0 层文档**：编写 `AGENTS.md`，包含目录结构、导航表和 AI 代理使用说明。在架构约束部分标记任何发现的架构问题；
2. **创建 L1 层文档**：为每个主要模块生成 `INDEX.md`，包括公共 API、依赖关系和依赖文件。使用 `grep -rn "from [module]"` 或 IDE 搜索工具来查找依赖关系。对于文件较多的目录，使用 `INDEX.md` 中的章节标题来组织文件结构；
3. **完善 L1 层文档**：添加接口契约、任务路由、修改风险和测试信息。在文件列表中标记文件过大或结构复杂的文件；
4. **验证 L2 层文档**：确保公共 API 文件包含契约头部；内部文件添加意图头部（上游代码可忽略）；
5. **审核 L3 层文档**：删除不必要的注释，为复杂逻辑添加解释（上游代码可忽略）；
6. **验证每个模块的索引**：在完成每个模块的索引后，使用 AI 测试功能定位和修改的影响。如果发现文件被多次引用或遗漏依赖关系，先修复索引后再进行下一步操作；
7. **同步 B 层文档**：所有 A 层索引完成后，执行迭代结束时的同步操作，生成/更新所有面向人类的文档。
