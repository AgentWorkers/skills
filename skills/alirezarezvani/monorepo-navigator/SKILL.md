# 单仓库导航器（Monorepo Navigator）

**级别：** 强大（Powerful）  
**类别：** 工程（Engineering）  
**领域：** 单仓库架构 / 构建系统（Monorepo Architecture / Build Systems）  

---

## 概述  
该工具用于导航、管理和优化单仓库（monorepo）系统。支持 Turborepo、Nx、pnpm 工作区（workspaces）以及 Lerna 等技术。具备跨包影响分析（cross-package impact analysis）功能，能够仅对受影响的包执行构建或测试操作，支持远程缓存（remote caching），提供依赖关系图可视化（dependency graph visualization），并支持从多仓库架构逐步迁移到单仓库架构（structured migrations）。同时支持 Claude Code 配置，以实现与工作区相关的开发流程（workspace-aware development）。  

---

## 核心功能  
- **跨包影响分析**：确定共享包更改时哪些应用程序会受到影响。  
- **选择性命令**：仅对受影响的包执行构建或测试操作，避免不必要的资源浪费。  
- **依赖关系图**：以 Mermaid 图表形式展示包之间的依赖关系。  
- **构建优化**：支持远程缓存、增量构建（incremental builds）和并行执行（parallel execution）。  
- **迁移**：支持逐步将多仓库系统迁移到单仓库系统，且迁移过程中不会丢失任何历史记录（step-by-step migrations with zero history loss）。  
- **发布功能**：支持版本控制（versioning）和 npm 发布流程（publishing via npm）。  
- **Claude Code 配置**：提供与工作区相关的 CLAUDE.md 文件，其中包含针对每个包的详细使用说明。  

---

## 使用场景  
- 当多个包或应用程序共享代码（如 UI 组件、工具函数、类型定义、API 客户端等）时。  
- 当代码构建速度较慢（因为每次修改都需要重新构建所有包时）。  
- 当需要将多个仓库合并为单个单仓库时。  
- 当需要协调发布多个包到 npm 并进行版本控制时。  
- 当团队需要在多个包之间协作且需要统一开发工具时。  

## 不适用场景  
- 当项目为单一应用程序且没有共享代码时。  
- 当团队/项目之间的代码隔离度较高（此时多仓库架构更合适）。  
- 当共享代码量较少且可以接受复制粘贴操作带来的效率损失时。

## 工具选择  
| 工具        | 适用场景                | 主要特点                                      |  
|-------------|------------------|-----------------------------------------|  
| **Turborepo**     | JS/TS 类型的单仓库系统，配置简单            | 领先的远程缓存技术，配置需求低                          |  
| **Nx**        | 大型企业，丰富的插件生态系统             | 支持项目结构可视化、代码自动生成等功能                   |  
| **pnpm workspaces** | 支持工作区协议，提高磁盘使用效率        | 提供 `workspace:*` 语法以引用本地包                   |  
| **Lerna**       | 支持 npm 发布和版本控制             | 支持批量发布、常规提交流程                           |  
| **Changesets**    | 现代化的版本控制系统（优于 Lerna）         | 支持变更日志生成和预发布通道                         |  

**推荐组合**：**pnpm workspaces + Turborepo + Changesets**

## 常见问题及解决方法  
| 问题            | 解决方案                                      |                                                                                   |  
|----------------|--------------------------------------------------|                                                                                  |  
| 在每次提交（PR）时都运行 `turbo run build` 而未使用 `--filter` | 在持续集成（CI）过程中始终使用 `--filter=...[origin/main]` 参数          |  
| 使用 `workspace:*` 引用导致发布失败           | 使用 `pnpm changeset publish` 替代 `workspace:*`，它会自动替换为实际版本            |  
| 无关文件更改时所有包都被重新构建       | 调整 `turbo.json` 中的 `inputs` 配置，排除文档和配置文件           |  
| 共享的 `tsconfig` 导致某些包的类型检查失败     | 正确使用 `extends` 规则，确保每个包独立配置 `rootDir` 和 `outDir`            |  
| 迁移过程中丢失 Git 历史记录         | 在合并前使用 `git filter-repo --to-subdirectory-filter`；切勿手动移动文件           |  
| 远程缓存功能在 CI 环境中失效         | 检查 `TURBO_TOKEN` 和 `TURBO_TEAM` 环境变量；使用 `turbo run build --summarize` 验证       |  
| CLAUDE.md 规则过于通用导致错误应用规则       | 为每个包的 CLAUDE.md 文件添加明确说明（例如：“处理 X 时，仅修改 apps/X 目录下的文件”）       |  

## 最佳实践  
1. **在根目录的 CLAUDE.md 中定义仓库结构**：记录每个包的用途及依赖关系。  
2. **为每个包创建单独的 CLAUDE.md 文件**：明确允许的操作、禁止的操作以及测试命令。  
3. **使用 `--filter` 参数限定命令范围**：避免对所有包执行不必要的操作。  
4. **远程缓存功能必不可少**：没有远程缓存，单仓库的 CI 流程会比多仓库慢很多。  
5. **优先使用变更集（Changesets）进行版本控制**：切勿手动修改单仓库中的 `package.json` 版本。  
6. **在根目录存放共享配置文件**，并在各包中自定义配置（如 `tsconfig.base.json`、`.eslintrc.base.js`、`jest.base.config.js`）。  
7. **在合并共享包更改前进行影响分析**：提前检查潜在问题，确保影响范围可控。  
8. **保持包和类型的纯 TypeScript 代码**：避免运行时代码和依赖关系，从而加快构建和类型检查速度。
