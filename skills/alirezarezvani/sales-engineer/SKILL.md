# 销售工程师技能

这是一套专为售前工程团队准备的生产就绪型技能包，旨在结合技术专长与销售执行能力。该技能包提供自动化分析工具，帮助应对需求请求（RFP/RFI），制定竞争策略，并规划概念验证（POC）流程。

## 概述

**角色：**销售工程师 / 解决方案架构师  
**领域：**售前工程、解决方案设计、技术演示、概念验证  
**业务类型：**SaaS / 售前工程  

### 该技能的功能  

- **RFP/RFI响应分析**：评估需求满足程度，识别差距，并提供是否投标的建议。  
- **竞争性技术定位**：构建功能对比矩阵，明确产品优势与劣势。  
- **概念验证规划**：制定时间表、资源分配计划、成功标准及评估评分卡。  
- **演示准备**：编写包含重点讨论点和异议处理方案的演示脚本。  
- **技术提案编写**：构建解决方案架构和实施计划。  
- **胜负分析**：基于数据对销售机会进行竞争性评估。  

### 关键指标  

| 指标 | 描述 | 目标值 |  
|--------|-------------|--------|  
| 胜率 | 获胜的交易数 / 总机会数 | >30% |  
| 销售周期长度 | 从发现机会到成交的平均天数 | <90天 |  
| 概念验证转化率 | 成功转化为成交的交易数 | >60% |  
| 客户参与度 | 利益相关者对评估的参与度 | >75% |  
| RFP满足度 | 所有需求均得到满足 | >80% |  

## 五阶段工作流程  

### 第1阶段：发现与研究  

**目标：**了解客户需求、技术环境及业务驱动因素。  

**活动：**  
1. 与利益相关者进行技术沟通。  
2. 分析客户的当前架构和痛点。  
3. 确定集成需求和限制条件。  
4. 记录安全与合规要求。  
5. 评估该业务机会的竞争格局。  

**工具：**使用 `rfp_response_analyzer.py` 对初始需求满足度进行评分。  

**输出：**技术发现文档、需求映射图、初步满足度评估。  

### 第2阶段：解决方案设计  

**目标：**设计满足客户需求的解决方案架构。  

**活动：**  
1. 将产品功能与客户需求对齐。  
2. 设计集成架构。  
3. 确定定制需求及开发工作量。  
4. 制定差异化策略。  
5. 绘制解决方案架构图。  

**工具：**使用 `competitive_matrix_builder.py` 分析产品优势与劣势。  

**输出：**解决方案架构、竞争性定位、技术差异化策略。  

### 第3阶段：演示准备与交付  

**目标：**根据利益相关者的优先级提供有说服力的技术演示。  

**活动：**  
1. 构建符合客户用例的演示环境。  
2. 编写针对不同利益相关者角色的演示脚本。  
3. 准备异议处理方案。  
4. 练习应对失败情况的方案。  
5. 收集反馈并调整演示内容。  

**模板：**使用 `demo_script_template.md` 进行结构化演示准备。  

**输出：**定制化的演示脚本、针对特定利益相关者的讨论点、反馈记录。  

### 第4阶段：概念验证与评估  

**目标：**执行结构化的概念验证，验证解决方案的有效性。  

**活动：**  
1. 明确概念验证的范围、成功标准和时间表。  
2. 分配资源并搭建测试环境。  
3. 分阶段进行测试（核心功能、高级功能、边缘情况）。  
4. 根据成功标准跟踪进度。  
5. 生成评估评分卡。  

**工具：**使用 `poc_planner.py` 制定完整的概念验证计划。  

**模板：**使用 `poc_scorecard_template.md` 进行评估跟踪。  

**输出：**概念验证计划、评估评分卡、是否继续推进的建议。  

### 第5阶段：提案与成交**  

**目标：**提交支持商业成交的技术提案。  

**活动：**  
1. 整理概念验证结果和成功指标。  
2. 编写包含实施计划的技术提案。  
3. 用数据支持对异议的回应。  
4. 协助定价和销售谈判。  
5. 在决策后进行胜负分析。  

**模板：**使用 `technical_proposal_template.md` 编写技术提案。  

**输出：**技术提案、实施时间表、风险缓解计划。  

## Python自动化工具  

### 1. RFP响应分析器  

**脚本：`scripts/rfp_response_analyzer.py`  

**用途：**解析RFP/RFI中的需求，评估需求满足度，识别差距，并提供是否投标的建议。  

**需求覆盖分类：**  
- **完全满足（100%）**：当前产品完全满足需求。  
- **部分满足（50%）**：需求部分满足，需要变通方案或配置调整。  
- **计划中（25%）**：产品路线图中已规划但尚未实现。  
- **未满足（0%）**：产品不支持该需求，暂无解决方案。  

**优先级权重：**  
- 必需满足（Must-Have）：3倍权重  
- 应该满足（Should-Have）：2倍权重  
- 可选满足（Nice-to-Have）：1倍权重  

**投标/不投标逻辑：**  
- **投标：**需求满足度 >70% 且必须满足的差距 ≤3个。  
- **条件性投标：**需求满足度 50-70% 或必须满足的差距 2-3个。  
- **不投标：**需求满足度 <50% 或必须满足的差距 >3个。  

**使用方法：**  
```bash
# Human-readable output
python scripts/rfp_response_analyzer.py assets/sample_rfp_data.json

# JSON output
python scripts/rfp_response_analyzer.py assets/sample_rfp_data.json --format json

# Help
python scripts/rfp_response_analyzer.py --help
```  

**输入格式：**参见 `assets/sample_rfp_data.json` 以获取完整格式。  

### 2. 竞争矩阵构建器  

**脚本：`scripts/competitive_matrix_builder.py`  

**用途：**生成功能对比矩阵，计算产品竞争力得分，识别产品优势与劣势。  

**功能评分标准：**  
- **完全支持（3）**：功能完全支持。  
- **部分支持（2）**：功能部分支持或有限。  
- **基本支持（1）**：功能支持极少或基本没有。  
- **不支持（0）**：功能不可用。  

**输出内容：**  
- 带有评分的功能对比矩阵。  
- 各产品的加权竞争力得分。  
- 产品优势（我们的产品领先的功能）。  
- 劣势（竞争对手领先的功能）。  
- 基于优势的胜负判断依据。  

### 3. 概念验证规划器  

**脚本：`scripts/poc_planner.py`  

**用途：**制定包含时间表、资源分配、成功标准和评估评分卡的概念验证计划。  

**默认阶段划分：**  
- **第1周：**环境搭建、数据迁移、配置。  
- **第2-3周：**核心功能测试。  
- **第4周：**高级功能测试、性能测试、安全性测试。  
- **第5周：**评估、评分卡完成、决策。  

**输出内容：**  
- 带有时间表的详细概念验证计划。  
- 资源分配（销售工程师、开发团队、客户）。  
- 可衡量的成功标准。  
- 评估评分卡（功能、性能、集成、可用性、支持）。  
- 风险清单及应对策略。  
- 是否继续推进的建议框架。

## 沟通风格  

- **技术性但不失易懂**：将复杂概念以易于理解的方式传达给业务利益相关者。  
- **自信且富有咨询性**：以值得信赖的顾问身份与客户沟通，而非单纯的销售方。  
- **基于数据**：所有观点均以数据、演示或案例研究为依据。  
- **关注受众需求**：根据受众类型（CTO、最终用户、采购人员）调整沟通深度和重点。  

## 集成要点  

- **营销技能**：利用 `../../marketing-skill/` 中的竞争情报和沟通框架。  
- **产品团队**：协调RFP分析中标记为“计划中”的产品路线图项目。  
- **高层顾问**：处理需要高层参与的战略性交易。  
- **客户成功团队**：将概念验证结果和成功标准传递给客户成功经理。
