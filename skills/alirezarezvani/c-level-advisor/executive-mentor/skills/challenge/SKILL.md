# /em:challenge — 事前评估（Pre-Mortem Analysis）

**命令:** `/em:challenge <计划名称>`

该工具能够在计划实际执行之前，系统地识别出其中的潜在弱点。目的不是为了否定这个计划，而是帮助它更好地应对现实中的挑战。

---

## 核心理念

大多数计划的失败都有可预测的原因：要么是假设错误，要么是对需求的估计过高，要么是对复杂性的低估，要么是没有对关键依赖因素进行充分的评估，要么是时间安排在理论上可行，但在现实中却行不通。

**事前评估的方法**是：**想象12个月后这个计划彻底失败了，然后从结果倒推原因。**

这并不是悲观主义，而是确保计划能够稳健运行的关键。

---

## 何时进行事前评估

- 在投入大量资源之前
- 在向董事会或投资者展示计划之前
- 当你只听到关于计划的正面反馈时
- 当计划依赖于多个外部因素时
- 当你感到时间紧迫、急于“以后再解决这些问题”时
- 当你对计划充满热情时（因为热情往往意味着需要更加谨慎地审视它）

---

## 评估框架

### 第一步：提取核心假设

在测试计划之前，首先需要明确计划所依赖的所有假设。

针对计划的每个部分，询问以下问题：
- 这个计划要成功，哪些条件必须成立？
- 我们对客户行为有什么假设？
- 我们对竞争对手的反应有什么假设？
- 我们对自己执行计划的能力有什么假设？
- 这个计划还依赖于哪些外部因素？

**常见的假设类别：**
- **市场假设**：市场规模、增长率、客户的支付意愿、购买周期
- **执行假设**：团队能力、项目进度、是否需要大量招聘
- **客户假设**：客户确实存在问题、他们意识到问题存在、并且愿意付费解决
- **竞争假设**：现有竞争对手不会采取行动、没有新竞争者出现、市场壁垒有效
- **财务假设**：资金消耗速度、收入时间点、客户获取成本（CAC）、客户生命周期价值（LTV）比率
- **依赖假设**：合作伙伴会按约定提供资源、API不会发生变化、法规不会改变

### 第二步：评估每个假设的可靠性

对每个提取出的假设，从两个维度进行评估：

**可靠性（你对这个假设的确定性）：**
- **高**：通过数据、客户访谈、市场研究验证过
- **中等**：方向正确但尚未验证
- **低**：看似合理但未经测试
- **未知**：我们根本不清楚

**假设错误的影响（如果这个假设不成立会怎样）：**
- **关键**：计划完全失败
- **高**：导致重大延误或成本超支
- **中等**：需要大量返工
- **低**：只需进行小幅度调整

### 第三步：识别风险点

将“可靠性低/未知”与“影响严重”相结合，找出风险最高的假设。

**风险点 = 可靠性低 + 影响严重**

这些风险不可忽视，因为它们直接关系到计划的成败。关键问题是：你是否清楚自己在做什么？

### 第四步：分析依赖关系

许多计划的失败并非因为某个单一假设错误，而是因为多个假设必须同时成立。

绘制依赖关系图：
- 假设B是否依赖于假设A的成立？
- 如果假设A出错，会引发多少后续问题？
- 关键路径是什么？哪些环节没有缓冲时间？

### 第五步：评估计划的可调整性

对于每个高风险点，思考如果某个假设在第三个月被证明是错误的，你应该如何应对：
- 你能调整计划方向吗？
- 你能缩减项目范围吗？
- 是否已经投入了资金？
- 是否已经做出了不可撤销的承诺？

计划的可调整性越低，你在投入之前就需要进行更严格的验证。

## 输出格式

**评估报告：[计划名称]**

```
CORE ASSUMPTIONS (extracted)
1. [Assumption] — Confidence: [H/M/L/?] — Impact if wrong: [Critical/High/Medium/Low]
2. ...

VULNERABILITY MAP
Critical risks (act before proceeding):
• [#N] [Assumption] — WHY it might be wrong — WHAT breaks if it is

High risks (validate before scaling):
• ...

DEPENDENCY CHAIN
[Assumption A] → depends on → [Assumption B] → which enables → [Assumption C]
Weakest link: [X] — if this breaks, [Y] and [Z] also fail

REVERSIBILITY ASSESSMENT
• Reversible bets: [list]
• Irreversible commitments: [list — treat with extreme care]

KILL SWITCHES
What would have to be true at [30/60/90 days] to continue vs. kill/pivot?
• Continue if: ...
• Kill/pivot if: ...

HARDENING ACTIONS
1. [Specific validation to do before proceeding]
2. [Alternative approach to consider]
3. [Contingency to build into the plan]
```

## 不同类型的计划对应的评估重点

### 产品路线图
- 我们开发的产品是否符合客户的实际需求，还是只是满足了他们的口头要求？
- 项目进度的估计是否基于团队的实际能力（而非理论预测）？
- 如果核心功能的开发时间比预期长3倍，会怎样？
- 当需求出现冲突时，谁来负责决策？

### 市场推广计划
- 实际的转化率（ICP）是多少，而不是预期值？
- 需要多少次接触才能完成销售？你的销售团队是否有足够的能力？
- 如果前10笔交易需要3个月而不是1个月才能完成，会怎样？
- “占领市场并扩大规模”只是一个目标，还是只是个幻想？

### 招聘计划
- 如果关键人才的招聘需要4个月而不是6周，会怎样？
- 计划是否依赖于某些可能会离职的关键员工？
- 计划是否考虑到了员工入职后的适应期（通常需要3-6个月才能达到满负荷工作状态）？
- 如果员工数量提前6个月就开始产生收入，会对资金消耗产生什么影响？

### 筹资计划
- 如果主要投资者退出，你的备选方案是什么？
- 你是否已经模拟过如果筹资时间延长到6个月的情况？
- 如果轮融资金额低于预期，你的资金储备能维持多久？
- 如果只筹集到目标金额的50%，你的运营计划会受什么影响？

---

## 需要深入探讨的问题

以下这些问题常常被忽略：
- “最坏的情况是什么？而不是最理想的情况？”
- “如果这个计划由一个我们不信任的团队执行，还会成功吗？”
- “有哪些因素让我们不敢把问题说出来？”
- “谁有动机让这个计划看起来比实际情况更好？”
- “这个计划的潜在对手会首先攻击哪些环节？”

---

## 评估结果的意义

`/em:challenge` 的输出结果并不是停止计划的理由，而是一份风险清单。现在你可以基于这份清单做出明智的决策：验证那些有风险的假设，对关键假设采取应对措施，或者坦然接受自己所做的风险判断。

未知的风险很危险，但已知的风险是可以管理的。
