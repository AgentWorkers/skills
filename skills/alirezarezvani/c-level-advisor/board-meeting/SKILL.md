# 董事会会议协议

这是一种结构化的多代理讨论机制，旨在防止群体思维、收集少数意见，并产生清晰、可执行的决策。

## 关键词
董事会会议、高管层审议、战略决策、高层管理团队（C-suite）、多代理、/cs:board、创始人审核、决策提取、独立观点

## 使用方法
`/cs:board [主题]` — 例如：`/cs:board 我们应该在第三季度扩展到西班牙吗？`

---

## 六阶段协议

### 第一阶段：收集背景信息
1. 加载 `memory/company-context.md`
2. 加载 `memory/board-meetings/decisions.md` **（仅限第二层数据——切勿使用原始记录）**
3. 重置会议状态——避免之前讨论的内容影响当前会议
4. 呈现议程并激活相关角色 → 等待创始人确认

**首席幕僚根据讨论主题选择相关角色**（并非每次会议都需要所有9个角色）：
| 主题 | 激活的角色 |
|-------|----------|
| 市场扩张 | CEO、CMO、CFO、CRO、COO |
| 产品方向 | CEO、CPO、CTO、CMO |
| 人力资源 | CEO、CHRO、CFO、COO |
| 定价 | CMO、CFO、CRO、CPO |
| 技术 | CTO、CPO、CFO、CISO |

---

### 第二阶段：独立贡献（各自独立思考）

**禁止相互影响。每个角色在看到其他人的意见之前先独立完成自己的思考。**

顺序：（如需要）进行研究 → CMO → CFO → CEO → CTO → COO → CHRO → CRO → CISO → CPO

**思考方法：**
- CEO：使用“思维树”方法（预测三种未来情景）
- CFO：使用“思维链”方法（展示计算过程）
- CMO：采用“递归思考”方法（草稿→批评→优化）
- CPO：基于“第一原则”进行分析
- COO：使用“逐步分析”方法（绘制流程图）
- CTO：采取“快速反应”策略（研究→分析→行动）
- CISO：采用“基于风险的决策方法”
- CHRO：运用同理心并结合数据分析

**贡献格式（最多5个关键点，需自行验证）：**
```
## [ROLE] — [DATE]

Key points (max 5):
• [Finding] — [VERIFIED/ASSUMED] — 🟢/🟡/🔴
• [Finding] — [VERIFIED/ASSUMED] — 🟢/🟡/🔴

Recommendation: [clear position]
Confidence: High / Medium / Low
Source: [where the data came from]
What would change my mind: [specific condition]
```

每个角色在贡献前需自行验证：确保信息来源正确、假设合理、信心度足够。禁止使用未经标记的陈述。

---

### 第三阶段：批判性分析
高管导师同时接收所有第二阶段的贡献。其角色是“批判性审查者”，而非信息整合者。

**检查清单：**
- 有哪些观点是所有参与者都轻易同意的？（这种共识可能隐藏问题）
- 有哪些共同假设尚未经过验证？
- 有哪些关键意见者没有参与讨论？（如客户声音或一线运营人员的意见）
- 有哪些风险被忽略了？
- 有没有角色超出了自己的专业领域进行讨论？

---

### 第四阶段：综合分析
首席幕僚按照 `agent-protocol/SKILL.md` 中定义的格式整理会议结果：
- 需要做出的决策（一句话）
- 各个参与者的观点（每个角色一条）
- 他们的共识点与分歧点
- 批判者的观点（直言不讳的真相）
- 建议的决策及行动事项（负责人及截止日期）
- 如果创始人不同意，你的建议是什么

---

### 第五阶段：等待创始人的决定 ⏸️

**此时停止讨论，等待创始人的意见。**

**规则：**
- 用户的修改意见优先于所有代理的建议。不得反驳或辩解（例如：“但是CFO这么说了……”）
- 如果30分钟内没有反馈，会议将自动标记为“待审核状态”
- 可以随时使用 `/cs:board resume` 重新开启会议

---

### 第六阶段：决策确认
获得创始人批准后：
- **第一层数据：** 编写完整的会议记录 → `memory/board-meetings/YYYY-MM-DD-raw.md`
- **第二层数据：** 将批准的决策添加到 `memory/board-meetings/decisions.md` 中
- 将被拒绝的提案标记为 `[DO_NOT_RESURFACE]`
- 向创始人确认决策结果，记录决策数量、跟踪行动事项，并添加相关标记

---

## 数据存储结构
**未来的会议只需加载第二层数据。** 切勿使用第一层数据，以避免产生错误的共识。

---

## 失败模式快速参考
| 失败原因 | 解决方法 |
|---------|-----|
| 群体思维（所有人意见一致） | 重新进行第二阶段的独立讨论；强制提出“最有力的反对意见” |
| 分析瘫痪（无法做出决策） | 限制每个角色的观点数量为5个；即使信心度低也要提出建议 |
- 无意义的讨论（重复讨论同一话题） | 将该讨论记录为异步行动事项，返回主议程 |
- 角色越权（例如CFO擅自做出产品决策） | 由批判者标记该行为；将其排除在综合结果之外 |
- 数据污染（不同阶段的记录相互干扰） | 严格限制只加载 `decisions.md` 文件 |
