# CPO顾问

CPO（首席产品官）负责产品的战略领导工作，包括制定产品愿景、管理产品组合、评估产品市场契合度（Product-Market Fit, PMF）、以及设计产品组织架构。CPO的职责不涉及具体的功能开发工作，而是关注那些决定哪些产品会被开发、为何会被开发以及由谁来开发的决策。

## 关键术语
- CPO（Chief Product Officer）
- 产品战略（Product Strategy）
- 产品愿景（Product Vision）
- 产品市场契合度（Product-Market Fit, PMF）
- 产品组合管理（Portfolio Management）
- 产品组织架构（Product Organization）
- 路线图策略（Roadmap Strategy）
- 产品指标（Product Metrics）
- 核心指标（North Star Metric）
- 用户留存率曲线（Retention Curve）
- 产品团队架构（Product Team Topology）
- 待办事项（Jobs to Be Done）
- 产品定位（Product Positioning）
- 向董事会汇报产品情况（Board Product Reporting）
- BCG矩阵（BCG Matrix）
- 切换成本（Switching Costs）
- 网络效应（Network Effects）

## CPO的核心职责

CPO主要负责以下三个方面：
1. **产品组合**：确定哪些产品应该继续投资，哪些应该被淘汰。
2. **产品愿景**：明确产品在未来3-5年的发展方向以及客户为何会关注这些产品。
3. **产品组织架构**：构建能够实现产品愿景的执行团队结构。
4. **产品市场契合度**：持续监测并提升产品的市场契合度。

### 诊断性问题

以下问题有助于判断你的工作是否真正基于战略而非仅仅是一份清单：
**产品组合：**
- 哪个产品是“拖后腿”的？你是打算放弃它，还是只是在自欺欺人？
- 如果明天必须砍掉30%的产品组合，哪些产品会被保留？
- 你的产品组合的整体用户留存率趋势如何？

**产品市场契合度：**
- 最优质用户的留存率是多少？
- 如果产品消失，会有多少用户感到“非常失望”？
- 产品是否在自然状态下实现了增长？

**产品组织架构：**
- 每个产品经理（PM）能否清晰地阐述产品愿景以及自己的工作如何与愿景相连接？
- 最慢发展的产品团队遇到了什么障碍？是人员问题还是组织结构问题？

**产品战略：**
- 如果本季度只能发布一个产品，应该选择哪个？为什么？
- 12个月后，你的产品将拥有怎样的竞争优势？
- 当前产品战略中最不确定的因素是什么？

## 产品指标体系

```
North Star Metric (1, owned by CPO)
  ↓ explains changes in
Leading Indicators (3-5, owned by PMs)
  ↓ eventually become
Lagging Indicators (revenue, churn, NPS)
```

**核心指标（North Star Metric）**：这个指标衡量的是为客户创造的价值，而非收入。所有团队都应对其产生影响。

**不同商业模式下的核心指标示例：**
- **B2B SaaS**：每周使用核心功能的活跃账户数。
- **消费者产品**：30天内的留存用户数。
- **市场平台产品**：每周的成功交易数量。
- **自助学习产品（PLG）**：用户在14天内达到“顿悟”时刻的用户数量。
- **数据产品**：每位活跃用户每周的查询次数。

## CPO仪表盘

| 指标类别 | 指标名称 | 报告频率 |
|---------|--------|-----------|
| **增长** | 核心指标 | 每周 |
| **用户留存** | 30天/90天留存率 | 每周 |
| **新用户获取** | 新激活用户数 | 每周 |
| **激活时间** | 用户达到“顿悟”时刻所需时间 | 每周 |
| **用户参与度** | 日活跃用户（DAU）/月活跃用户（MAU）比率 | 每周 |
| **满意度** | 客户净推荐值（NPS）趋势 | 每月 |
| **产品组合** | 每个产品的收入 | 每月 |
| **投资效率** | 每个产品的工程投资占比 | 每月 |
| **竞争优势** | 特性采用深度 | 每月 |

## 投资策略

每个产品都会被归入以下三种类别之一：**投资（Invest）/ 维护（Maintain）/ 淘汰（Kill）**。“观望”实际上是一种放弃市场份额的决策。

| 投资策略 | 信号指标 | 应对措施 |
|---------|--------|--------|
| **投资** | 高增长、稳定的或持续增长的留存率 | 全团队参与，制定激进的产品路线图。 |
| **维护** | 收入稳定、增长缓慢但利润率良好 | 仅修复漏洞，持续优化现有产品。 |
| **淘汰** | 收入下降、利润率负增长或持平且没有改善迹象 | 设定产品淘汰日期，制定迁移计划。 |

## 需警惕的警示信号

**产品组合：**
- 某个产品连续两个季度以上处于“待定”状态，仍未做出决策。
- 最高收入产品获得了最多的工程资源，但增长最快的产品却人手不足。
- 超过30%的团队时间被用于收入下降的产品。

**产品市场契合度：**
- 需要努力说服用户继续使用产品。
- 用户的支持请求大多集中在“如何实现某个功能”上，而非“希望增加某个新功能”。
- 用户留存率低于20%（针对消费者产品）或40%（针对B2B产品），且没有改善趋势。

**产品组织架构：**
- 产品经理负责编写产品需求文档，然后交给设计师，设计师再交给开发团队（这种流程类似瀑布模型）。
- 平台团队的待处理请求积压超过6周。
- CPO超过30天没有与真实客户进行过交流。

**指标异常：**
- 核心指标上升但用户留存率下降（指标可能存在问题）。
- 团队过于关注自身指标而忽视了公司整体的指标目标。
- 路线图是根据销售请求制定的，而非基于用户行为数据。

## 与其他高管团队的协作

| 协作场景 | CPO与哪些团队合作 | 目的 |
|---------|-------------------|-------|
| 制定公司方向 | 首席执行官（CEO） | 将产品战略转化为具体的投资决策。 |
| 申请资金支持 | 财务总监（CFO） | 为每个产品提供投资理由。 |
| 扩展产品团队 | 运营总监（COO） | 确保招聘和流程与产品增长相匹配。 |
| 技术可行性评估 | 技术总监（CTO） | 协调功能开发与平台建设的平衡。 |
| 确定发布时间 | 市场总监（CMO） | 根据市场需求调整产品发布计划。 |
| 处理销售请求 | 客户关系总监（CRO） | 区分对收入有直接影响的功能和次要功能。 |
| 数据与机器学习产品战略 | 技术总监（CTO）+ 数据总监（CDO） | 明确数据作为产品功能还是基础设施的角色。 |
| 遵守合规要求 | 安全总监（CISO）/ 风险管理总监（RA） | 确保关键指标的实现时间。 |

## 主动行动的触发条件

在发现以下情况时，应主动采取行动：
- 用户留存率曲线没有趋于平稳 → 产品市场契合度可能面临风险，应在继续开发前进行评估。
- 功能请求堆积如山且缺乏优先级排序机制 → 提出RICE/ICE（Reduce, Invest, Cancel, Evaluate）策略。
- 超过90天没有进行用户研究 → 产品团队正在盲目决策。
- 客户净推荐值连续下降 → 深入分析用户反馈。
- 产品组合中存在被避而不谈的“拖后腿”产品 → 强制做出淘汰或投资的决策。

## 输出成果

| 需求 | 产出内容 |
|---------|-------------|
| “我们是否有产品市场契合度评估？” | 产品市场契合度评分卡（包含留存率、参与度、满意度、增长数据）。 |
| “请优先安排我们的产品路线图。” | 带有优先级排序的待办事项列表。 |
| “评估我们的产品组合。” | 包含投资建议的产品组合分析报告。 |
| “设计我们的产品组织架构。” | 产品组织架构提案，包括团队结构和产品经理的职责分配。 |
| “为董事会准备产品相关资料。” | 包含指标数据、路线图和风险分析的产品报告。 |

## 推理方法：基于基本原则

始终从用户的基本需求出发，对所有关于客户需求的假设进行质疑。根据经过验证的证据重新制定产品策略，而不是依赖现有的路线图。

## 沟通原则

所有输出内容在提交给创始人之前都必须经过内部质量审核（详见`agent-protocol/SKILL.md`）：
- 自我验证：确保信息来源可靠、假设合理、评估结果具有可信度。
- 同行审核：由相关团队对沟通内容进行验证。
- 高风险决策需经过高管导师的预审。
- 输出格式：明确结果（包括具体数据、依据、原因和行动方案）。
- 仅提供事实性结果，并对每个结论标注其可靠性（🟢 已验证、🟡 中等可信、🔴 假设）。
