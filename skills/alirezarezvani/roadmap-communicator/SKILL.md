# 路线图沟通工具

为内部和外部利益相关者创建清晰的路线图沟通材料。

## 使用场景

适用于以下场景：
- 制作不同格式的路线图演示文稿
- 编写利益相关者更新报告（面向董事会、工程团队或客户）
- 生成发布说明（面向用户和内部团队）
- 从 Git 历史记录中提取变更日志
- 构建功能公告的内容结构

## 路线图格式

1. **当前/下一个/后续阶段**  
  - 适用于情况不明确或需要战略灵活性的场景。  
  - 以非精确的方式传达发展方向。

2. **时间线路线图**  
  - 适用于有固定发布日期的场景，有助于协调发布计划。  
  - 需要积极管理风险和依赖关系。

3. **主题式路线图**  
  - 适用于以结果为导向的规划和跨团队协作。  
  - 按问题领域或战略目标对项目进行分类。

相关模板请参见 `references/roadmap-templates.md`。

## 利益相关者更新模板

### 面向董事会/高管的更新报告  
  - 以结果和风险为核心  
  - 强调与战略目标的进展情况  
  - 突出需要做出的决策和权衡因素

### 面向工程团队的更新报告  
  - 明确项目范围、依赖关系及执行顺序  
  - 说明项目状态、存在的阻碍以及资源需求

### 面向客户的更新报告  
  - 介绍产品的价值及交付时间  
  - 区分现有功能与即将推出的新功能  
  - 设定明确的期望值

相关模板请参见 `references/communication-templates.md`。

## 发布说明指南

### 面向用户的发布说明  
  - 重点介绍产品对用户的价值，而非内部实现细节。  
  - 按工作流程或用户任务进行内容分类。  
  - 明确说明功能迁移或行为变化的内容。

### 内部发布的说明  
  - 包含技术细节、对运营的影响以及已知问题。  
  - 记录发布计划、回滚方案及监控要点。

## 变更日志生成工具  

使用方法：  
```bash
python3 scripts/changelog_generator.py --from v1.0.0 --to HEAD
```  

**功能特性：**  
- 读取 Git 日志范围  
- 解析常规的提交前缀  
- 按类型（如 `feat`、`fix`、`chore` 等）对变更进行分类  
- 以 Markdown 或纯文本格式输出结果

## 功能公告框架  

1. 问题背景  
2. 变更内容  
3. 变更的重要性  
4. 最受益的群体  
5. 如何开始使用新功能  
6. 行动建议及反馈渠道  

## 沟通质量检查清单  

- [ ] 沟通内容针对特定受众进行了明确区分。  
- [ ] 结果与权衡因素得到了清晰阐述。  
- [ ] 所使用的术语在所有沟通材料中保持一致。  
- [ ] 风险和依赖关系没有被隐瞒。  
- [ ] 下一步行动及负责人已明确指定。
