# 环境管理与密钥管理工具

**级别：** 高级  
**类别：** 工程技术  
**领域：** 安全性 / DevOps / 配置管理  

---

## 概述  
该工具提供全面的环境和密钥管理功能，支持 `.env` 文件在整个开发（dev）、测试（staging）和生产（prod）环境中的生命周期管理。包括自动生成 `.env.example` 文件、验证必需的配置变量、检测 Git 历史记录中的密钥泄露情况，以及实现密钥轮换机制。同时支持与 HashiCorp Vault、AWS SSM、1Password CLI 和 Doppler 等工具的集成。  

---

## 核心功能  
- **.env 文件生命周期管理**：支持创建、验证 `.env` 文件，并在各个环境间进行同步。  
- **.env.example 文件的自动生成**：会去除文件中的具体值，仅保留密钥和注释。  
- **配置变量验证**：在程序启动时，如果缺少必需的变量会立即报错并停止运行。  
- **密钥泄露检测**：通过正则表达式扫描 Git 历史记录和工作目录中的敏感信息。  
- **密钥轮换流程**：检测到泄露后，会确定受影响的范围，执行密钥轮换，并在轮换后进行验证。  
- **密钥管理工具集成**：支持与 HashiCorp Vault KV v2、AWS SSM、1Password 和 Doppler 的集成。  

---

## 使用场景  
- **新项目设置**：自动生成 `.env.example` 文件并执行配置变量验证。  
- **每次提交前**：扫描代码中是否意外包含了敏感信息。  
- **事件响应**：在密钥泄露后执行密钥轮换操作。  
- **新员工入职**：确保新员工能访问所有必要的配置变量。  
- **环境差异分析**：检查生产环境与测试环境之间的配置差异。

## 常见问题  
- **误将 `.env` 文件提交到代码库**：请在第一天就将 `.env` 文件添加到 `.gitignore` 文件中，并使用提交前钩子（commit-pre Hook）进行防护。  
- **将密钥信息存储在 CI/CD 日志中**：切勿在日志中直接输出敏感信息；应在 CI 配置中屏蔽这些变量。  
- **仅在一个地方轮换密钥**：密钥可能同时存在于多个系统（如 Heroku、Vercel、Docker、K8s、CI 等）中，因此需要同时更新所有相关系统。  
- **忘记在密钥轮换后失效旧密钥**：这可能导致所有用户被强制登出；务必及时通知用户。  
- **使用包含实际值的 `.env.example` 文件**：示例文件通常是公开的，应删除所有敏感信息。  
- **轮换后未进行监控**：轮换后需持续监控审计日志，以防止旧密钥被非法使用。  
- **使用弱密钥**：例如 `APP_SECRET=mysecret` 并不算是强密钥；建议使用 `openssl rand -base64 32` 生成随机密钥。

## 最佳实践  
1. **密钥管理工具是唯一的数据来源**：`.env` 文件仅用于本地开发环境，切勿用于生产环境。  
2. **定期轮换密钥**：特别是对于长期使用的密钥，至少每季度轮换一次。  
3. **最小权限原则**：为每个服务分配具有最小权限的 API 密钥。  
4. **审计访问记录**：记录所有从密钥管理工具中读取的密钥操作，并对异常访问行为进行报警。  
5. **禁止记录敏感信息**：在日志处理过程中屏蔽敏感密钥内容。  
6. **使用短生命周期的凭证**：优先使用 OIDC/实例角色而非长期有效的访问密钥。  
7. **按环境分离密钥**：切勿在开发环境和生产环境之间共享密钥。  
8. **记录密钥轮换流程**：应在事件发生前制定并执行轮换流程，而非在事件发生时才处理。
