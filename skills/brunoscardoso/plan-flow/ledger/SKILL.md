# 项目学习日志（Project Learning Log）

这是一个跨会话持续维护的、针对特定项目设计的学习日志。它记录了在开发过程中遇到的错误、解决方案、发现的规律以及关于该项目代码库的宝贵经验。

## 工作原理

该学习日志在后台默默运行：

1. **会话开始时**：读取 `flow/ledger.md` 文件并应用其中的学习内容。
2. **开发过程中**：实时记录新的学习成果。
3. **维护阶段**：定期整理和优化日志条目。

## 文件位置

```
flow/ledger.md
```

## 记录的内容

- 项目特有的特性和注意事项
- 成功或失败的开发方法
- 用户的偏好（超出官方文档规定的内容）
- 领域背景和业务逻辑
- 开发环境和工具链中遇到的意外问题
- 用户自己发现的错误及修正方法

## 不记录的内容

- 通用的编程知识
- 已经包含在规则文件中的开发规律
- 一次性的临时问题
- 应该记录在开发计划或文档中的内容

## 条目撰写要求

每条记录都应简洁明了且具有可操作性。记录的内容应体现“经验教训”，而非简单的日志记录。

**示例：**
- “`buildWidgets` 脚本需要 Node 18 及更高版本的 Node.js 环境——在 Node 16 上会无声失败。”
- “用户更喜欢使用明确的错误类型，而不是通用的 `Error` 类。”
- “针对流式端点的测试需要设置 3 秒的超时时间。”

## 分类

| 分类 | 用途 |
|---------|---------|
| 项目特性 | 项目中出现的意外行为或未记录的约束条件 |
| 成功的方法 | 经验证适用于该项目的开发方法 |
| 失败的方法 | 失败的尝试及其原因分析 |
| 用户偏好 | 用户对开发流程的具体要求 |
| 领域背景 | 项目的业务逻辑和术语 |

## 维护规则

- 每条记录的长度应控制在 150 行以内，确保内容具有较高的参考价值。
- 每 8 至 10 次会话后对日志进行一次整理。
- 当项目中的问题得到解决后，删除过时的条目。
- 将成熟的开发规律整理到官方的规则文件中。

## 集成方式

学习内容会自然地从所有开发活动（设置、问题发现、代码执行、代码审查等）中捕获。该学习日志会汇总这些跨各个开发阶段的元学习信息。
