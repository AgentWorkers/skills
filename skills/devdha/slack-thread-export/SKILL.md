# Slack主题导出

当需要从已登录的浏览器标签页中导出Slack数据（而非使用机器人令牌或管理员API）时，请使用此功能。

## 使用范围及不适用场景

该功能专为通过Chrome浏览器Relay标签页进行已登录Slack数据导出而优化。其主要特点包括：
- 能够导出指定频道内用户的主题回复；
- 可从Slack搜索结果中生成CSV/JSONL格式的文件；
- 可按频道/日期筛选结果，并可选择性地排除非工作相关内容；
- 即使部分请求失败，也能通过重试和延迟处理来保证数据完整性。

**不适用的场景包括：**
- 不能替代通用的Slack管理/导出工具；
- 不能作为基于机器人令牌或SCIM协议的导出工具；
- 不能保证导出的数据完全符合法律或合规性要求；
- 不能作为判断消息是否“与工作相关”的通用分类工具。

## 工作流程

1. 确保用户已通过Browser Relay附加了Slack标签页，并使用`profile="chrome"`进行配置。
2. 使用浏览器工具或`openclaw browser --browser-profile chrome`确认附加的标签页对应正确的Slack工作空间。
3. 从Slack标签页中读取`localStorage.localConfig_v2`以获取活跃团队的元数据和xoxc令牌。
4. 在页面上下文中调用Slack的内部`search.messages`接口（`fetch('/api/search.messages', ...)`，确保已登录的浏览器会话能够提供正确的cookies/会话信息。
5. 分批次导出数据：
   - 建议按频道逐一查询，而非一次性查询所有数据；
   - 使用`count=100`逐页获取数据，并逐步增加查询页数；
   - 遇到速率限制时，通过延迟或重试来处理；
   - 当查询结果数量为0或少于100条时停止导出。
6. 在写入文件前排除不需要的记录：
   - 删除即时消息（IM/MPIM）；
   - 仅保留包含`thread_ts=`的记录（如果用户需要主题回复）；
   - 应用用户指定的过滤条件（如频道白名单、关键词过滤、日期范围等）。
7. 保存两种格式的数据：
   - 原始的JSONL格式数据用于审计或调试；
   - 经过处理的CSV格式数据供用户使用。

## 工作原理

该功能并不直接抓取Slack页面的DOM内容，而是利用Slack Web内部的搜索流程：
- 附加到用户已登录的Slack浏览器标签页；
- 读取活跃工作空间的元数据；
- 仅在页面上下文中使用Web客户端的xoxc令牌；
- 从Slack页面发起`fetch('/api/search.messages', ...`请求，确保请求携带正确的会话信息；
- 遍历搜索结果并将其转换为可导出的格式。

这种方式比直接从服务器端发起请求更可靠，因为Slack的搜索机制要求同时提供令牌和已登录的浏览器会话信息。

## 运行参数及限制

除非有特殊原因，否则请使用以下默认参数：
- 每次`search.messages`请求最多获取100条数据。
- 正常请求之间延迟0.5秒。
- 遇到速率限制时，至少等待3-5秒后再尝试相同页面的查询。
- 每个频道最多查询60页。
- 建议每次查询一个频道，或仅查询白名单中的频道。

**实际使用建议：**
- **安全范围**：1个用户 + 1个频道 + 1-20页数据；
- **较为合理**：1个用户 + 3-10个频道 + 每页延迟0.5秒；
- **风险较高**：一次性查询整个工作空间的所有数据；
- **非常危险**：一次性查询多个频道和页面。

**注意事项：**
- 如果请求速度过快或查询范围过广，Slack会返回速率限制错误；
- 如果一次请求处理的数据量过大，浏览器可能会超时；
- 最安全的做法是分多次小批量请求。

## 错误处理机制**

- **invalid_auth**：表示标签页错误、用户未登录，或请求缺少浏览器会话信息；解决方法：确认标签页已登录，并从页面上下文重新发起请求。
- **ratelimited**：表示请求过多或查询范围过广；解决方法：延迟并重试相同页面，减少查询频道数量或添加日期过滤条件。
- **浏览器评估超时**：表示一次请求处理的数据量过大；解决方法：按频道和页面分批处理。
- **部分导出**：表示部分频道成功导出，部分失败；解决方法：保存原始数据，报告成功导出的频道，并仅重试失败频道。

## 推荐的查询模式**

- 用户过滤：`from:<@USER_ID>`
- 仅导出主题：`is:thread`
- 频道过滤：`in:#channel-name`
- 如需时间过滤：`after:YYYY-MM-DD`

**推荐的输出列**：
- `datetime_utc`、`channel_name`、`channel_id`、`thread_ts`、`username`、`text`、`permalink`、`source_query_channel`

**相关脚本**：
- `scripts/export_slack_threads.py`：通过Chrome Relay标签页按频道导出Slack主题的脚本。
- `scripts/retry_failed_channels.py`：使用之前的`failed-channels-out`文件重新执行导出操作。

**示例代码**：

```text
from:<@UXXXX> is:thread in:#tech-team after:2026-01-01
```

**优化导出范围的方法**：
- 在重试前缩小查询范围：
  - 使用更短的日期范围；
  - 先使用频道白名单；
  - 先导出原始数据，再对“与工作相关”的内容进行二次过滤；
  - 先抽样检查频道数据，再对符合条件的频道进行完整导出。

**重试流程**：
- 如果首次导出失败，保留原始的JSONL数据；
- 使用`--failed-channels-out`文件保存失败频道；
- 使用`scripts/retry_failed_channels.py`重新尝试导出；
- 如需，使用`--resume-from-jsonl`参数从之前的原始数据开始重试。

**支持的选项**：
- `--channel-file`：从文件中读取频道名称；
- `--after`/`--before`：指定日期范围；
- `--resume-from-jsonl`：从之前的导出结果继续导出；
- `--failed-channels-out`：保存失败频道以供重试；
- `--summary-json`：保存导出过程的摘要信息；
- `--preflight`：在完整导出前预览每个频道的部分数据；
- `--mode strict|raw|heuristic`：
  - `strict`：仅导出指定频道/日期范围内的数据，不使用启发式过滤；
  - `raw`：保留所有收集到的数据（适用于审计/审查）；
  - `heuristic`：在数据收集后应用与工作相关的过滤条件；
- `--keyword`/`--channel-hint`：针对特定工作空间调整启发式过滤。

**过滤建议**：
- 对于“与工作相关”的数据导出，优先使用以下顺序：
  - 频道白名单/频道名称；
  - 明确的日期范围；
  - 基于消息文本的关键词过滤；
- 对边缘频道进行人工审核。

**模式选择指南**：
- 当用户明确知道哪些频道重要时，使用`strict`模式；
- 当需要精确导出指定频道/日期范围内的数据时，使用`strict`模式；
- 当用户希望先获取数据集以供审核时，使用`raw`模式；
- 当需要人工审核或后续处理数据时，使用`raw`模式；
- 当数据包含多种类型的信息时，使用`heuristic`模式。

**提示**：
- 如果用户要求导出“所有与工作相关的主题”，请说明这是基于启发式的过滤结果，除非用户提供了具体的筛选条件。
