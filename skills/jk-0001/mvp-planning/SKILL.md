# MVP规划

## 概述
MVP（最小可行产品）并不是将所有你能想到的功能都剔除后的产品，而是能够验证你的核心业务假设并真正为用户创造价值的最小化版本。如果开发的内容过多，就会浪费时间在用户并不需要的功能上；如果开发的内容太少，那么推出的产品可能无法解决实际问题。本指南明确了这一界限，并为你提供了一个可重复的流程来找到正确的方向。

---

## 第一步：重新阐述你的核心假设
在开始规划之前，先写下你的MVP需要验证的单一假设。所有包含的功能都必须服务于这个假设，不符合假设的功能都应该被剔除。

**格式：**
```
IF we build [specific product that does X for Y customers],
THEN [specific outcome we expect — e.g., Z% will pay, W% will return weekly].
```

**示例：**
“如果我们为自由职业开发者开发一个自动化的客户进度报告工具，那么在第一个月内，至少有30%的测试用户会每周使用该工具，其中至少10%的用户会升级为付费用户。”

这个假设包含两部分：行为指标（使用情况）和收入指标（付费情况）。这两部分都必须能够被量化。如果无法量化，就无法从中获取有用的信息。

---

## 第二步：定义“可行”在你的具体情境下的含义
“可行”并不是一个普遍适用的标准，它取决于你的假设。你需要根据你的假设来确定实现该假设所需的最基本功能。

**请思考以下问题：**
1. 客户需要体验到什么，才能判断“这个产品有价值”或“这个产品没有价值”？
2. 哪一个核心交互能够传递产品的价值主张？
3. 有些功能可以手动完成、质量较差或暂时缺失，但仍然可以用来验证假设吗？（这些功能可以被剔除或暂时替代。）
4. 哪些功能是必不可少的？如果缺少它们，产品就会显得不完整或毫无用处？（这些功能是不可妥协的。）

**为每个功能标注标签：**
- 🔴 **必备功能**：没有这个功能，产品就没有意义；无法验证假设。
- 🟡 **可选功能**：虽然能提升用户体验，但即使没有也能验证假设。
- 🟢 **可剔除功能**：对于验证假设来说完全不必要；如果验证通过，可以后期再添加。

---

## 第三步：无情地剔除功能
查看你的功能清单，逐一应用上述标准进行筛选。要果断剔除不必要的功能。作为独立创业者，你的开发时间非常有限——每一个不必要的功能都会浪费掉本应用于核心功能的时间。

**四个剔除标准：**

### 第一标准：是否直接服务于核心假设的验证
- 如果某个功能直接有助于验证你的核心假设，保留它（暂时保留，待进一步评估）。
- 如果不能，直接剔除。没有例外。

### 第二标准：能否暂时替代
对于前N位用户，这个功能能否通过人工方式模拟或替代实现，而用户却毫不知情？
**可替代的例子：**
- 对于前10位用户，所谓的“仪表盘”实际上只是一个共享的Google表格。
- 所谓的“实时”更新实际上是在1小时后才发送的。
- 所谓的“人工智能推荐”实际上是你手动为测试用户筛选的内容。
- 所谓的“API集成”实际上只是每隔几小时自动运行一次的脚本。

如果可以通过人工方式让测试用户相信这些功能的存在，那就暂时使用人工替代方案。只有在确认用户确实需要这些功能后，再着手开发真实版本。

### 第三标准：是否可以延迟到后续版本添加
这个功能是否必须在产品发布时就存在？还是可以在发布后的2周内（v1.1版本）再添加？
如果这个功能可以延迟2周添加，且产品在没有它的情况下仍然可以使用和可测试，那就将其从MVP的范围内剔除，放入“第二周待办事项”中。

### 第四标准：是否属于“额外惊喜”
这些功能虽然很棒，但并非用户的核心需求。对于MVP来说，额外的惊喜功能反而会分散注意力。MVP的重点应该是功能性和清晰性，而不是惊喜性。惊喜功能属于后续版本（v2）的优化内容。

**经过这四个步骤后，你的功能列表应该会比最初大幅缩减。** 如果仍然觉得功能太多，继续剔除。最常见的MVP错误就是开发了过多的功能。

---

## 第四步：制定MVP范围文档
编写一份单页文档，明确产品的范围。这可以防止范围扩大，并为你提供一个明确的完成标准。

```
MVP SCOPE DOCUMENT
==================

HYPOTHESIS: [from Step 1]

CORE VALUE DELIVERED: [one sentence — what the user gets]

MUST-HAVE FEATURES (🔴):
  1. [Feature] — because [why it's essential to the hypothesis]
  2. [Feature] — because [why]
  3. [Feature] — because [why]

FAKED / MANUAL FEATURES (🟡 deferred to real implementation):
  1. [Feature] — faked as [how] — real build in [when]
  2. [Feature] — faked as [how] — real build in [when]

CUT FROM MVP (🟢 — revisit after launch):
  1. [Feature]
  2. [Feature]
  ...

LAUNCH CRITERIA (all must be true before you call this "launched"):
  - [ ] [Criterion 1]
  - [ ] [Criterion 2]
  - [ ] [Criterion 3]

WHAT SUCCESS LOOKS LIKE (measured in the first 30 days):
  - [Metric 1]: target = [number]
  - [Metric 2]: target = [number]
```

---

## 第五步：选择开发方式
对于MVP所需的每一项技术，决定以下方案：
- 自己开发；
- 购买现有的工具；
- 使用无代码/低代码解决方案。

**决策框架：**

| 如果这是... | 那么... |
|---|---|
| 你的核心差异化优势 | 自己开发。这是你的独特卖点；外包意味着放弃这一优势。 |
| 常见的基础基础设施（托管、支付、认证、邮件） | 购买成熟的工具（如Stripe、Auth0、SendGrid、Vercel等）。自行开发这些功能会浪费大量时间。 |
| 需要重复使用100多次的工作流程（但并非产品核心） | 使用无代码工具自动化（如Zapier、Make、n8n）。只有在自动化工具无法满足需求时才自行开发。 |
| 只需要使用一次或很少使用的功能 | 购买相关服务或雇佣自由职业者。不要为了一次性使用而开发专门的功能。 |

**独立创业者的原则：** 自定义开发的组件越少，MVP的发布速度就越快。除了那些真正属于你的独特功能外，尽可能使用现有的工具。

---

## 第六步：估算和安排开发计划
对于每个必备功能，进行以下估算：
- **开发时间**（以小时为单位，实事求是，并预留50%的缓冲时间）。
- **依赖关系**（功能B是否需要功能A先完成？）
- **复杂性风险**（是否有不确定的部分？先完成这部分以降低风险）。

**开发顺序规则：**
1. 先开发风险最高的代码部分。如果这部分无法实现或耗时远超预期，你需要提前了解情况。
2. 接着开发能够传递产品价值的核心功能。其他所有功能都围绕这个核心功能展开。
3. 最后开发辅助功能，如认证系统、用户引导材料、界面优化等。

**时间线：**
- 设定一个明确的发布日期（创造外部压力，比如告知他人并设置等待名单）。
- 从发布日期倒推到现在，评估开发时间是否合理。
- 如果时间不够，继续剔除功能（回到步骤3）或延长发布日期。但不要为了满足任意截止日期而牺牲核心功能的实现。

---

## 第七步：发布前的检查清单
在发布之前，确保满足以下所有条件：
- [ ] 核心功能能够完整运行（用户能够从注册到体验到产品价值）。
- [ ] 不存在会导致数据丢失的错误。
- [ ] 支付功能正常工作（即使是在发布后收取费用的情况下）。
- [ ] 你能够在5分钟内引导新用户使用产品（通过3位真实用户进行测试）。
- [ ] 你有收集用户反馈的渠道（应用内调查、电子邮件或Slack频道）。
- [ ] 你有监控产品运行状况的手段（正常运行时间、错误率、基本数据分析）。
- [ ] 你制定了发布后的48小时内的应对计划（通知对象、监控方式、反馈处理方式）。

---

## 第八步：发布后的学习循环
MVP只是一个开始，而不是终点。发布后应进入一个持续学习的循环。

**发布后第一周：**
- 与每一位早期用户进行深入交流（不要只是发问卷）。了解他们遇到的困惑、满意的地方以及未实现的需求。
- 如果可能的话，观察他们使用产品的过程（通过屏幕共享或会议录像）。他们在使用过程中遇到哪些困难？哪些环节让他们感到不满？

**第二至第四周：**
- 根据范围文档中的指标评估项目进展。
- 找出你开发的内容与用户实际需求之间的最大差距。
- 决定是继续优化当前的MVP，还是调整开发方向。

**决策规则：**
- 如果使用指标表现良好但收入不佳，可能是定价或转化率的问题，需要进一步优化。
- 如果使用指标不佳，说明核心功能没有达到预期效果，可能需要调整开发方向。
- 如果两者都表现良好，说明产品与市场需求匹配，可以扩大规模（增加用户数量、加大营销力度、添加更多功能）。

---

## 需避免的MVP常见错误：
- 仅仅因为功能开发起来有趣就添加它们，而不是因为它们能帮助验证假设。
- 先在朋友之间测试产品。朋友往往不会给出客观的反馈，应该先向陌生人发布产品。
- 在确认产品价值之前就追求完美的用户界面。一个能解决实际问题的丑陋产品总比一个美观但没人需要的产品更好。
- 将MVP视为最终产品。实际上，MVP只是一个学习工具，预计在后续版本中会有60-80%的部分需要重新设计。
- 不要推迟发布。最好的MVP就是已经发布的MVP；未发布的MVP无法带来任何收获。
