# 线性浏览器自动化

通过 OpenClaw 管理的浏览器中的 UI 操作来自动化线性工作流程。

## 执行模式

- 始终使用浏览器配置文件：`openclaw`
- 始终针对主机浏览器控制执行操作
- 在可能的情况下，重用已打开且已登录的 Linear 标签页
- 如果存在已认证的 Linear 标签页，则不要打开新的标签页
- 除非用户明确要求打开新标签页，否则将操作保留在一个标签页中

## 标准工作流程

1. 确保管理的浏览器正在运行（`profile="openclaw"`）。
2. 检查管理的浏览器中已打开的标签页，并优先选择已打开的 Linear 标签页。
3. 如果找到现有的 Linear 标签页：
   - 将焦点放在该标签页上并重用它
   - 截取快照并检测认证状态
4. 如果没有 Linear 标签页，则打开/导航到 `https://linear.app/`。
5. 如果处于公共登录页面或未认证状态：
   - 自动导航到 `/login`
   - 显示浏览器提示，要求用户登录以继续
   - 等待认证后的仪表板/工作区状态
6. 登录完成后且仪表板可见后，从相同的任务上下文和相同的标签页继续执行。
7. 解决快照中的引用问题。
8. 执行确定的操作（点击/输入/选择/按下）。
9. 在完成之前重新截图并验证预期状态。

## 必需输入策略（硬性规则）

对于所有 Linear 操作，在执行之前，务必在聊天中收集并确认所有必需的详细信息。

### 1) 创建问题

必填字段：
1. 团队（或暗示团队的项目上下文）
2. 标题

可选字段（如果缺失则提示）：描述、优先级、分配者、标签、截止日期。

在提交之前，始终显示一个总结框并询问：
- **`继续？（是/否）`**

只有在明确回答“是”时才继续。

### 2) 编辑问题和对问题进行评论

必填字段：
1. 团队名称
2. 问题编号/标识符（例如，`POW-123`）

硬性规则：
- 如果缺少团队名称或问题编号，则不要运行这些自动化操作。
- 如果用户只提供了标题或部分引用信息，请要求提供团队名称和问题编号。
- 在进行更改之前，先根据提供的团队名称和问题编号找到确切的问题。

### 3) 更改问题状态

必填字段：
1. 团队名称
2. 问题编号/标识符（例如，`POW-123`）
3. 新状态/状况（例如，`待办`、`进行中`、`已完成`）
4. 要添加到问题中的评论文本

有效的问题状态（仅接受这些状态）：
- `待办`
- `进行中`
- `已完成`
- `已取消`
- `重复`

硬性规则：
- 如果缺少任何一项，则不要运行更改状态的自动化操作。
- 如果未明确提供状态，请要求用户提供。
- 如果未明确提供评论文本，请要求用户提供。
- 如果状态值不在有效状态列表中，请拒绝并让用户选择一个有效的选项。
- 除非用户明确指示，否则不要自动选择状态。
- 先根据团队名称和问题编号找到确切的问题，然后应用请求的新状态并添加评论。

### 优先级验证规则

对于任何设置或更新问题优先级的自动化操作，只接受以下值：
- `紧急`
- `高`
- `中等`
- `低`

硬性规则：
- 拒绝任何其他优先级值，并要求用户选择一个有效的选项。

### 必需输入的对话契约

当用户请求创建新问题时：

1. 检查是否已经提供了所有必填字段。
2. 首先询问缺失的必填字段。
3. 询问可选字段（或接受默认值）。
4. 显示最终摘要：
   - 团队/项目
   - 标题
   - 描述
   - 优先级
   - 分配者
   - 标签
   - 截止日期
5. 询问 `继续？（是/否）`。
6. 仅在用户明确回答“是”后提交。

当用户请求编辑/评论问题时：

1. 首先检查团队名称和问题编号。
2. 询问缺失的字段。
3. 用一句话确认目标（例如，`目标问题：POW-123（Powerhouse）`）。
4. 只有在目标明确后才能执行操作。

当用户请求更改问题状态时：

1. 检查团队名称、问题编号、新的明确状态和评论文本。
2. 询问缺失的字段。
3. 用一句话确认目标、目的地和评论（例如，`目标问题：POW-123（Powerhouse）→ 进行中 | 评论：<text>`）。
4. 只有在目标明确、新状态和评论都确定后才能执行操作。

## 安全性 + 可靠性规则

- 在执行操作之前，优先使用来自最新快照的稳定选择器/引用。
- 在发生重大 UI 变更后重新截图。
- 如果出现未认证状态，自动导航到登录页面并触发浏览器内的提示，要求用户登录。
- 用户登录完成后且检测到仪表板后，自动恢复任务。
- 对于破坏性操作（删除/归档），需要明确的确认。
- 如果团队/项目上下文不明确，请在操作前询问。

## 支持的自动化操作

- 打开 Linear 并验证仪表板是否加载
- 搜索/过滤问题（团队、分配者、状态、标签）
- 创建新问题（遵循必需输入策略）
- 编辑问题字段（标题、描述、优先级、截止日期、标签）
- 更改问题状态（待办/进行中/已完成）
- 添加评论
- 批量分类（有限批次，需要确认）
- 视觉验证（操作后的快照检查）

## 建议的命令模式（代理端）

### 在管理的配置文件中打开 Linear

- 启动浏览器（`profile="openclaw"`）
- 打开 `https://linear.app/`

### 重用现有标签页

- 将当前 `targetId` 导航到目标 URL

### 交互周期

- 截取快照（优先使用 aria 参考）
- 执行操作（点击/输入/选择/按下）
- 截取快照并验证

## 失败处理

- 缺少元素/引用：
  - 重新获取快照
  - 使用更新的引用重新尝试一次
- 导航/登录中断：
  - 暂停并请求用户完成认证/多因素认证
- 目标不明确：
  - 请求澄清（团队/项目/问题）
- 缺少必需的问题字段：
  - 一直提示直到提供或用户取消
- 提交确认时没有明确的“是”回答：
  - 不要提交

## 完成报告格式

返回：
- 执行的操作
- 发生变化的项目（如果可见，则包括问题 ID/标题）
- 验证检查是否通过
- 是否需要任何手动跟进
